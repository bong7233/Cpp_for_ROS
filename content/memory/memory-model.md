# 2.1 메모리 모델: 스택, 힙, 정적 영역

::: lead
[1.10](#/linkage)에서 전역 변수가 프로그램 시작과 함께 "제로 초기화 상태"로 이미 존재한다는 것을 봤다 — 동적 초기화가 돌기도 전에 그 자리가 이미 있었다는 뜻이다. 그 자리가 정확히 어디인가. 지역 변수의 자리와는 왜 다른가. 이 절부터 Part I이 다루던 "소스 코드"를 떠나 **실행 중인 프로그램의 메모리**로 들어간다. int 하나를 선언해도 그 변수가 함수가 끝나자마자 증발할 수도, 프로그램이 끝날 때까지 살아남을 수도, `delete`할 때까지 버틸 수도 있다 — 같은 타입인데 수명이 완전히 다르다. 그 차이를 결정하는 것이 변수가 놓인 **영역**이다. 스택, 힙, 정적 영역, 이렇게 셋의 지도를 그리고, 각 영역에서 변수가 태어나고 죽는 과정을 위젯으로 직접 눈에 새긴다. 이 지도가 Part II 전체 — 포인터, RAII, 이동 시맨틱, 스마트 포인터 — 가 서 있는 땅이다.
:::

## 함수가 끝나면 사라지는 것, 사라지지 않는 것

센서값 하나를 읽어 그 주소를 돌려주는 함수를 짜 보자. 흔히 나오는 실수다.

```cpp title="sensor.cpp — 지역변수의 주소를 돌려준다"
#include <cstdio>

int* read_sensor() {
    int reading = 42;          // 센서값. read_sensor의 스택 프레임에 산다
    return &reading;           // 프레임이 사라지는 순간 죽는 주소를 돌려준다
}

int main() {
    int* p = read_sensor();
    std::printf("p  = %p\n", (void*)p);
    std::printf("*p = %d\n", *p);
    return 0;
}
```

```console
$ g++ -std=c++20 -Wall -Wextra sensor.cpp -o sensor
sensor.cpp: In function 'int* read_sensor()':
sensor.cpp:5:12: warning: address of local variable 'reading' returned [-Wreturn-local-addr]
    5 |     return &reading;
      |            ^~~~~~~~
sensor.cpp:4:9: note: declared here
    4 |     int reading = 42;
      |         ^~~~~~~
$ ./sensor
p  = (nil)
Segmentation fault (core dumped)
```

컴파일러가 경고까지 정확히 짚어 준 코드가 그대로 죽는다. 눈여겨볼 것은 **크래시의 모양**이다. "예전 값 42가 남아 있다가 다음 호출에 덮어써진다"는 흔한 예상과 다르게, g++ 13.3은 이 환경에서 `p`에 **아예 널 포인터**를 담아 돌려줬다(`p = (nil)`). `reading`의 주소를 계산해 돌려주는 코드가 함수를 벗어나는 순간 의미를 잃는다는 것을 컴파일러가 알고 있고, 그 반환값을 굳이 정확히 계산해 주지 않은 것이다 — [1.2](#/types)에서 만난 미정의 동작(UB)의 실물이다. "이 환경에서는 이렇게 나왔다"이지 "항상 이렇게 나온다"가 아니다 — 다른 컴파일러, 다른 최적화 레벨에서는 죽은 주소가 그대로 반환돼 옛 값이 우연히 남아 있는 것처럼 보일 수도 있다. 어느 쪽이든 결론은 같다. **`reading`이 사는 자리는 `read_sensor`가 끝나는 순간 회수된다.** 회수된 자리를 가리키는 포인터로 무엇을 하든 프로그램 전체의 의미가 사라진다.

그런데 똑같은 질문을 전역 변수에 던지면 답이 정반대다. `read_sensor` 100번을 호출해도 매번 죽는 지역변수와 달리, 전역 변수의 주소는 프로그램이 끝날 때까지 유효하다. 같은 "변수"인데 왜 하나는 함수 하나의 수명이고 하나는 프로그램 전체의 수명인가 — 답은 이 둘이 애초에 **다른 동네**에 산다는 것이다. 그 동네의 지도부터 그린다.

## 프로세스의 메모리 지도

실행 중인 프로그램은 자신의 메모리를 몇 개의 구역으로 나눠 쓴다. 함수의 명령어가 사는 **코드 영역**, 프로그램 시작과 함께 존재하는 전역·`static` 변수가 사는 **정적 영역**, `new`로 요청할 때마다 자라나는 **힙**, 함수 호출마다 프레임이 쌓이는 **스택**. 넷의 위치를 직접 찍어서 비교한다.

```cpp title="addrmap.cpp — 네 영역의 주소를 한 화면에"
#include <cstdio>

int global_var = 10;              // 정적 영역 (초기화 있음)
int global_uninit;                 // 정적 영역 (초기화 없음)

void some_function() {}            // 코드 영역

void print_addresses() {
    static int static_var = 20;    // 정적 영역 (함수-로컬 static)
    int local_var = 30;            // 스택
    int* heap_var = new int(40);   // 힙

    std::printf("코드          &some_function = %p\n", (void*)&some_function);
    std::printf("정적(전역)    &global_var    = %p\n", (void*)&global_var);
    std::printf("정적(static)  &static_var    = %p\n", (void*)&static_var);
    std::printf("힙            heap_var       = %p\n", (void*)heap_var);
    std::printf("스택          &local_var     = %p\n", (void*)&local_var);

    delete heap_var;
}

int main() { print_addresses(); }
```

```console
$ g++ -std=c++20 -Wall -Wextra addrmap.cpp -o addrmap
$ ./addrmap
코드          &some_function = 0x5643d29cd1a9
정적(전역)    &global_var    = 0x5643d29d0010
정적(static)  &static_var    = 0x5643d29d0014
힙            heap_var       = 0x5643e953d2b0
스택          &local_var     = 0x7fff2982820c
$ ./addrmap
코드          &some_function = 0x55d75d5ef1a9
정적(전역)    &static_var    = 0x55d75d5f2010
정적(static)  &static_var    = 0x55d75d5f2014
힙            heap_var       = 0x55d772a6c2b0
스택          &local_var     = 0x7ffd93886bec
```

(g++ 13.3 / Linux x86-64, 실행을 두 번 반복한 결과다.) 절대 주소는 실행마다 완전히 다르다 — 보안을 위해 리눅스가 매 실행마다 이 영역들의 시작 위치를 무작위로 흩뜨리기 때문이다(ASLR, Address Space Layout Randomization). 그런데 **순서는 두 실행 모두 똑같다.** 코드가 가장 낮은 주소, 그다음 정적 영역, 그다음 힙, 그리고 스택이 압도적으로 높은 주소(`0x7fff...`)에 홀로 떨어져 있다. 이 순서는 우연이 아니라 리눅스 프로세스 메모리 배치의 관례다.

```text nolines
높은 주소 (예: 0x7fff...)
+--------------------------------+
|   stack                        |  함수 호출마다 이 방향으로 자란다 (아래로)
|   (지역 변수, 매개변수)          |
+--------------------------------+
|              ...                |  (스택과 힙 사이의 빈 공간)
+--------------------------------+
|   heap                         |  new/malloc이 이 방향으로 자란다 (위로)
+--------------------------------+
|   static area (.data / .bss)   |  전역, static 변수
+--------------------------------+
|   code (.text)                 |  함수의 실행 명령어
+--------------------------------+
낮은 주소 (예: 0x5643...)
```

스택과 힙이 서로 마주 보고 자란다는 그림은 교육용으로 단순화한 관례다 — 실제로는 공유 라이브러리(libc, libstdc++)가 그 사이 어딘가에 별도로 매핑되고, 힙과 스택의 절대 간격도 ASLR과 스레드 개수에 따라 달라진다. 지금 기억할 것은 방향과 순서다. **코드와 정적 영역은 프로그램 시작 전에 이미 정해진 고정 구역이고, 힙과 스택만 실행 중에 자란다.** 그리고 그 "자라는 방식"이 정확히 다음 두 절의 주제다.

## 스택: 프레임이 쌓이고 사라진다

함수를 하나 호출할 때마다 그 함수의 지역 변수, 매개변수, 반환 주소를 담을 **프레임(frame)**이 스택 위에 하나 쌓인다. 함수가 끝나면 그 프레임 전체가 통째로 걷힌다 — 위에서 `reading`이 증발한 이유가 이것이다.

```cpp title="square.cpp — 프레임이 쌓이는 순간"
int square(int n) {
    int r = n * n;
    return r;
}

int main() {
    int a = 3;
    int b = square(a);   // 여기서 square의 프레임이 main 위에 쌓인다
    return 0;
}
```

`main`이 실행되며 프레임 하나가 생기고 `a`가 그 안에 자리 잡는다. `square(a)`를 호출하는 순간 `square`의 프레임이 그 위에 새로 쌓이고, 인자 `n`에는 `a`의 값이 **복사**된다([1.6](#/functions)에서 본 값 전달 그대로다). `square`가 `return r`에 도달하면 그 프레임은 즉시 걷히고, `r`과 `n`은 함께 사라진다 — 반환값 9만 `main`의 `b`로 넘어온다. 아래 위젯이 이 스텝 하나하나를 그대로 재생한다.

"함수 호출" 버튼이 눌린 상태에서 스텝을 한 칸씩 넘겨 봐라. `square`가 호출되는 순간 새 프레임이 `main` 위에 나타나는 것, `n`이 `a`와 같은 값 3을 갖지만 **다른 자리**라는 것, 그리고 `return` 이후 `square`의 프레임이 통째로 사라지며 `n`과 `r`도 함께 소멸하는 것을 눈으로 따라가라. 이 챕터를 텍스트로만 읽고 넘어가지 마라 — 프레임이 "쌓인다"와 "걷힌다"는 그림이 이 절 전체의 토대다.

::: deep 스택이 빠른 이유는 산수 한 줄이다
프레임을 만드는 비용이 얼마인지 어셈블리로 직접 확인한다. 위 `square.cpp`를 `-O0`로 컴파일해 `main`을 뜯어 보면 이렇다(g++ 13.3, x86-64 실측):

```console
401123: push   %rbp
401124: mov    %rsp,%rbp
401127: sub    $0x10,%rsp      <- a, b 두 개를 위한 자리 확보. 이게 전부다
40112b: movl   $0x3,-0x4(%rbp)
401132: mov    -0x4(%rbp),%eax
401135: mov    %eax,%edi
401137: call   401106 <square(int)>
40113c: mov    %eax,-0x8(%rbp)
40113f: mov    -0x8(%rbp),%eax
401142: leave                  <- rsp를 rbp로 되돌리는 것으로 회수 끝
401143: ret
```

`a`와 `b`를 위한 스택 공간을 만드는 명령은 `sub $0x10,%rsp` 딱 한 줄이다 — 스택 포인터(rsp)를 16만큼 밀어내리는 산술 연산 하나. 함수가 끝날 때 그 공간을 돌려주는 것도 `leave`(rsp를 rbp로 되돌리는 것) 한 줄이다. 힙 할당처럼 "빈 블록을 찾는" 탐색이 없다 — **포인터 하나를 움직이는 것**이 스택 할당의 전부라서 비용이 사실상 0에 가깝다. 아래 힙 절의 벤치마크가 이 차이를 숫자로 보여준다.
:::

빠르다는 것의 대가는 **한도**다. 스택은 프로세스마다(정확히는 스레드마다) 고정된 크기로 예약돼 있다. 이 환경에서 얼마나 되는지 셸에 직접 물어보자.

```console
$ ulimit -s
8192
```

8192KB, 즉 8MB다. 이 한도를 재귀로 실제로 뚫어 본다.

```cpp title="overflow.cpp — 바닥없는 재귀"
#include <cstdio>

long long recurse(long long depth) {
    char frame_padding[1024];       // 프레임을 두툼하게 만들어 바닥이 빨리 보이게 한다
    frame_padding[0] = static_cast<char>(depth);
    if (depth % 1000 == 0) {
        std::printf("depth = %lld\n", depth);
        std::fflush(stdout);
    }
    return recurse(depth + 1) + frame_padding[0];
}

int main() { recurse(0); }
```

```console
$ g++ -std=c++20 -Wall -Wextra overflow.cpp -o overflow
overflow.cpp: In function 'long long int recurse(long long int)':
overflow.cpp:3:11: warning: infinite recursion detected [-Winfinite-recursion]
    3 | long long recurse(long long depth) {
      |           ^~~~~~~
$ ./overflow
depth = 0
depth = 1000
...
depth = 7000
Segmentation fault (core dumped)
```

g++는 이 함수가 멈추는 조건이 없다는 것까지 정적으로 알아채고 `-Winfinite-recursion`을 낸다 — 실제로는 무한히 도는 게 아니라 스택이 바닥나며 강제 종료되는 것이지만, "이 재귀에는 탈출구가 없다"는 진단은 정확하다. 프레임 하나가 대략 1KB를 넘게 쓰므로 8MB ÷ 1KB, 즉 8000번 안팎에서 바닥이 드러났다 — 실측이 정확히 그 근처(7000번대)에서 죽는다. ASan(`-fsanitize=address`)을 붙이면 진단명까지 정확히 나온다.

```console
$ g++ -std=c++20 -Wall -Wextra -g -fsanitize=address overflow.cpp -o overflow_asan
$ ./overflow_asan
...
==20434==ERROR: AddressSanitizer: stack-overflow on address 0x7ffe5f757c98 ...
SUMMARY: AddressSanitizer: stack-overflow overflow.cpp:3 in recurse(long long)
```

`stack-overflow`라는 진단명 그대로다. 그런데 재귀 없이 **단 한 번의 호출**로도 이 한도를 뚫을 수 있다 — 큰 배열을 지역 변수로 잡는 실수다.

```cpp title="bigarray.cpp — 함수 하나가 스택 한도를 통째로 삼킨다"
#include <cstdio>

void process_scan() {
    double points[2'000'000];   // 16MB — 8MB 한도의 두 배
    points[0] = 1.0;
    points[1'999'999] = 2.0;
    std::printf("%f\n", points[0] + points[1'999'999]);
}

int main() {
    std::printf("process_scan 호출 전\n");
    std::fflush(stdout);
    process_scan();   // 여기서 즉시 죽는다 — 재귀조차 필요 없다
}
```

```console
$ g++ -std=c++20 -Wall -Wextra bigarray.cpp -o bigarray
$ ./bigarray
process_scan 호출 전
Segmentation fault (core dumped)
```

`process_scan`이 호출된 순간 바로 죽는다 — 함수 안의 코드가 한 줄도 실행되기 전에, 16MB짜리 배열을 위한 자리를 잡으려는 시도 자체가 8MB 한도를 뚫는다. 라이다 스캔 한 프레임, 포인트클라우드 버퍼처럼 크기가 큰 데이터를 "그냥 지역 배열로" 선언하고 싶은 유혹이 실무에서 실제로 이 사고를 낸다.

::: danger 큰 데이터는 절대 지역 배열로 잡지 마라
스택은 수천 바이트 단위의 지역 변수를 위한 공간이지, 수십·수백 KB 이상의 버퍼를 위한 공간이 아니다. 카메라 프레임, 포인트클라우드, 큰 룩업 테이블은 힙에 두거나(아래 절) 함수 밖에서 한 번만 할당해 재사용한다. "일단 지역 변수로 짜고 나중에 최적화하자"는 습관이 스택에서는 통하지 않는다 — 크기를 실수하면 최적화 이전에 크래시다.
:::

## 힙: new가 실제로 하는 일

`new`는 스택처럼 정해진 자리를 쓰는 게 아니라, 실행 중에 운영체제가 프로세스에 내준 **힙**이라는 넓은 구역에서 필요한 만큼을 매번 요청한다. 스택 절의 위젯과 짝을 이루는 예제로 확인한다.

```cpp title="heapex.cpp"
int main() {
    int* p = new int[4];   // 힙에 16바이트 할당
    p[0] = 42;
    int x = 7;              // 지역 변수는 여전히 스택에
    delete[] p;
    p = nullptr;
    return 0;
}
```

{ "scenario": "heap-alloc" }
:::

스텝을 넘겨 봐라. `new int[4]`가 실행되는 순간 **힙에** 새 블록이 생기고, 그 블록의 주소를 담는 포인터 `p` 자신은 **스택에** 산다 — 화살표가 스택에서 힙으로 건너가는 것이 그 증거다. `delete[] p` 직후에는 힙 블록이 점선으로 "해제됨" 표시가 되지만 `p`는 잠깐 옛 주소를 그대로 들고 있다(댕글링 상태) — 이 위험한 틈을 어떻게 다루는지는 [2.2 포인터](#/pointers)와 [2.11 댕글링, UB, 새니타이저](#/ub-sanitizers)의 몫이다. 지금 눈여겨볼 것은 하나다. **포인터 변수의 자리(스택)와 그 포인터가 가리키는 대상의 자리(힙)는 완전히 다른 동네다.**

`new`가 실제로 하는 일은 스택의 "포인터 한 번 밀기"보다 훨씬 무겁다 — 할당자(allocator)에게 "이만한 크기의 빈 블록이 있는가"를 물어야 한다. 할당자는 내부에 해제된 블록들의 목록(자유 리스트, free list)을 들고 있다가 크기가 맞는 조각을 찾아 내주고, 맞는 조각이 없으면 운영체제에 새 메모리를 요청한다. 게다가 스레드가 여러 개면 이 자유 리스트 자체가 공유 자원이라 접근을 잠가야 한다 — 락(또는 락 없는 자료구조)이 매 할당·해제마다 끼어든다. 스택은 이 중 어느 것도 하지 않는다. 얼마나 차이 나는지 직접 잰다.

::: perf 스택 대 힙, 같은 크기 128바이트를 1000만 번
128바이트짜리 객체 하나를 지역 변수로 만들었다 없애는 것과, `new`/`delete`로 힙에 만들었다 없애는 것을 각각 1000만 번 반복했다(g++ 13.3 / `-O2` / Linux x86-64 실측, 컴파일러가 결과를 지우지 못하게 값과 주소를 관찰 가능하게 묶어 뒀다):

```console
$ g++ -std=c++20 -Wall -Wextra -O2 bench.cpp -o bench
$ ./bench
stack  x 10000000  = 6183753 ns total, 0.62 ns/회
heap   x 10000000  = 115168947 ns total, 11.52 ns/회
비율(heap/stack) = 18.6배
```

여러 번 반복해도 힙이 스택보다 **18~34배** 느리게 나왔다(실행마다 할당자 내부 상태에 따라 변동한다 — 절대값보다 이 자릿수 차이를 기억해라). 128바이트라는 작은 크기에서도 이 정도다. 자유 리스트 탐색과 스레드 동기화 비용은 크기와 거의 무관하게 매번 붙는 고정비용이라, 작은 객체를 자주 할당·해제하는 코드일수록 이 배율의 타격이 크다.
:::

이 실측이 Part II 전체가 스마트 포인터·이동 시맨틱·`reserve()` 같은 "힙 할당 횟수를 줄이는" 관용구에 그렇게 공을 들이는 이유다. 힙이 나쁘다는 뜻이 아니다 — 크기가 실행 중에야 정해지거나 함수 경계를 넘어 살아야 하는 데이터는 힙 말고는 답이 없다. 다만 그 대가가 스택 대비 자릿수 하나 이상 크다는 것을 알고 써야 한다. `new`/`delete`를 직접 손으로 짝지어 관리하는 것의 위험(누수, 이중 해제)은 [2.4 new/delete와 동적 할당의 비용](#/dynamic-alloc)에서, 그 위험을 구조적으로 없애는 도구는 [2.9 unique_ptr](#/unique-ptr)에서 다룬다.

## 정적 영역: 프로그램과 함께 사는 데이터

위 `addrmap.cpp`가 이미 답을 보여줬다 — `global_var`(전역)와 `static_var`(함수-로컬 static)의 주소는 서로 이웃해 있었고, 둘 다 힙·스택과는 멀리 떨어진 낮은 주소대에 있었다. 이 구역이 **정적 영역**이다. 코드 영역처럼 프로그램이 시작되기 전에 크기와 위치가 정해지고, 프로그램이 끝날 때까지 그대로 남는다 — 함수 프레임처럼 걷히는 일도, 힙 블록처럼 명시적으로 해제되는 일도 없다.

전역 변수는 `main`이 실행되기도 전에 이 영역에 자리를 잡는다. 초기화 없는 전역(`global_uninit`)은 프로그램 로딩 시점에 자동으로 0으로 채워지고([1.3](#/variables)에서 본 "쓰지 않는 기능엔 비용도 없다"는 원칙과 대비해서, 이 제로 채움은 로더가 사실상 공짜로 해 준다), 초기화 있는 전역은 그 값 그대로 시작한다. 함수-로컬 `static`은 조금 다르다 — 자리는 프로그램 시작과 함께 정적 영역에 이미 있지만, **값을 채우는 시점**은 그 변수를 정의한 코드가 처음 실행될 때다([1.3](#/variables)에서 스코프와 수명이 별개의 축이라는 것을 이 변수로 확인한다). 번역 단위를 넘나드는 전역들 사이의 초기화 순서 문제는 [1.10](#/linkage)에서 이미 실측한 그대로다 — 지금 이 절에서 가져갈 것은 "정적 영역의 변수는 스택·힙과 달리 프로그램 실행 내내 같은 주소를 지킨다"는 사실 하나다.

## 어디에 둘 것인가

세 영역의 성질을 알면 선택은 규칙으로 정리된다.

- **기본은 스택이다.** 함수 안에서 쓰고 함수가 끝나면 필요 없는 데이터 — 대부분의 지역 변수, 루프 카운터, 작은 임시 객체 — 는 전부 스택에 둔다. 빠르고, 자동으로 회수되고, 실수할 여지가 적다.
- **크기가 실행 중에 정해지거나 수명이 함수 경계를 넘으면 힙이다.** 다만 이 책에서 `new`/`delete`를 손으로 직접 쓰는 코드는 [2.4](#/dynamic-alloc)의 위험을 이해하기 전까지만이다 — 실전에서는 [2.9 unique_ptr](#/unique-ptr)가 그 소유권을 대신 관리하게 한다.
- **로봇 제어 루프 안에서는 힙 할당 자체를 피한다.** 위에서 실측한 18~34배라는 배율도 문제지만, 진짜 문제는 힙 할당에 소요 시간의 **상한이 없다**는 것이다 — 자유 리스트가 마침 비어 있으면 운영체제에 새 페이지를 요청하는 시스템 콜까지 번질 수 있고, 이 지연은 예측이 안 된다. 1kHz로 도는 다리 제어 루프 안에서 매 주기 `new`를 부르면, 그 배율이 아니라 "이번 주기는 얼마나 걸릴지 아무도 모른다"는 것 자체가 실시간성을 깨뜨린다. 왜 그런지, 그리고 대신 무엇을 하는지는 [6.8 실시간 제약과 제어 루프](#/realtime)에서 실측으로 다룬다.

::: interview 스택과 힙의 차이
"스택과 힙의 차이를 설명하라"는 C++ 면접에서 가장 많이 나오는 질문이다. 답변 뼈대: ① **속도** — 스택 할당은 스택 포인터를 옮기는 산술 연산 한 줄이고(위에서 어셈블리로 확인했다), 힙 할당은 할당자가 자유 리스트를 뒤지고 필요하면 락까지 잡는다 — 실측으로 한 자릿수 이상 차이 난다. ② **수명** — 스택 변수는 자신이 속한 프레임이 걷히면 자동으로 사라지고, 힙 블록은 `delete`(또는 소유자 객체의 소멸)로 명시적으로 회수해야 한다. ③ **크기 한도** — 스택은 스레드마다 고정 크기로 예약되고(리눅스 기본 8MB 안팎, `ulimit -s`로 확인), 힙은 사실상 물리·가상 메모리가 허용하는 한 자란다. ④ **스레드별 스택** — 스레드를 하나 새로 만들면 그 스레드 전용 스택이 따로 생긴다([6.1](#/threads)의 몫이다), 반면 힙은 프로세스의 모든 스레드가 같은 구역을 공유한다 — 그래서 힙에 접근하는 할당자 내부 자료구조는 스레드 동기화가 필요하고, 이것이 ①에서 힙이 느린 이유의 절반을 차지한다. 이 넷을 순서대로, 실측 근거와 함께 말하면 상급 답변이 된다.
:::

## 요약

- 지역 변수는 자신이 속한 함수의 스택 프레임이 걷히면 함께 사라진다 — 그 주소를 돌려주면 g++ 13.3은 `-Wreturn-local-addr` 경고와 함께, 이 환경에서는 아예 널 포인터를 반환했다(실측). UB이므로 항상 이렇다는 보장은 없다.
- 프로세스 메모리는 코드 → 정적 영역 → 힙 → 스택 순으로 낮은 주소에서 높은 주소로 배치된다(실측, ASLR로 절대값은 실행마다 다르지만 순서는 같다). 코드·정적 영역은 시작 전에 고정되고, 힙·스택만 실행 중에 자란다.
- 스택 할당의 실체는 `sub $0x10,%rsp` 한 줄, 회수는 `leave` 한 줄이다(어셈블리 실측) — 그래서 사실상 공짜다. 대신 크기 한도가 있다(`ulimit -s`, 이 환경 8MB): 재귀로 뚫으면 `stack-overflow`(ASan 실측), 큰 배열 하나로도 함수 진입 즉시 크래시한다.
- `new`는 할당자가 자유 리스트를 탐색하고 필요하면 락을 잡는 무거운 연산이다 — 같은 128바이트를 1000만 번 반복한 실측에서 힙이 스택보다 18~34배 느렸다(자릿수 차이가 핵심이지 절대값이 아니다).
- 정적 영역(전역·`static`)의 변수는 프로그램 시작과 함께 자리를 잡고 끝날 때까지 같은 주소를 지킨다 — 스택·힙처럼 자동으로 걷히거나 명시적으로 해제되는 일이 없다.
- 기본 선택은 스택, 크기·수명이 함수를 넘으면 힙(단 손으로 `new`/`delete`를 짝짓기보다 [2.9 unique_ptr](#/unique-ptr)에 맡긴다), 실시간 제어 루프 안에서는 힙 할당 자체를 피한다 — [6.8](#/realtime).

::: quiz 연습문제
1~2번은 개념 문제, 3~5번은 네 컴퓨터에서 직접 확인하는 실습이다.

1. `read_sensor()`가 이 환경에서 널 포인터를 반환한 것과, "죽은 스택 주소가 그대로 반환돼 우연히 옛 값 42가 남아 있는" 경우를 비교하라. 두 결과 모두 "정상"이라고 부를 수 있는 이유를 UB의 정의로 설명하라.

2. 힙이 스택보다 느린 이유 두 가지를 이 절의 실측과 함께 대라. 그리고 그중 스레드가 여러 개일 때만 추가로 붙는 비용은 어느 것인가?

3. (실습) `addrmap.cpp`를 직접 쳐서 네 환경에서 코드·정적·힙·스택 네 주소를 찍어라. 성공 기준: 두 번 이상 실행해 절대 주소는 매번 다르지만 **순서**(코드 < 정적 < 힙 << 스택)는 항상 같다는 것을 확인했다.

4. (실습) 스택 오버플로 재현: 본문의 `overflow.cpp`를 그대로 치고 일반 빌드로 한 번, `-g -fsanitize=address`로 한 번 더 실행하라. 성공 기준: ① 일반 빌드가 `Segmentation fault`로 죽는 것 ② ASan 빌드가 진단명 `stack-overflow`를 정확히 찍는 것 ③ `ulimit -s`를 직접 확인해 대략 몇 번째 재귀에서 죽었는지 프레임 크기(1024바이트)로 설명할 수 있다.

5. (실습) `bigarray.cpp`를 직접 치고 배열 크기를 2,000,000에서 200,000으로 줄여 다시 컴파일하라(2MB, 8MB 한도 안). 성공 기준: 줄인 버전은 크래시 없이 정상 종료하는 것을 확인하고, "몇 MB부터 죽기 시작하는지" 네 `ulimit -s` 값을 근거로 직접 예측한 뒤 배열 크기를 조절해 검증했다.
:::

::: answer 해설
1. 둘 다 UB의 정의상 "허용된" 결과다. 표준은 스코프 밖 지역변수의 주소를 역참조했을 때 무엇이 나와야 하는지 **아무 요구도 하지 않는다** — 널 포인터가 나오는 것도, 죽은 스택 프레임의 잔여값이 우연히 읽히는 것도, 크래시하는 것도 전부 "표준을 어기지 않은" 결과다. 이 절의 실측(g++ 13.3, `-O0`)에서는 컴파일러가 UB를 이용해 반환값 계산 자체를 생략하고 이전에 다른 용도로 쓰던 레지스터 값(우연히 0)을 그대로 반환했다 — "우연히 42가 남아 있는" 시나리오보다 오히려 더 흔한 결과일 수 있다는 것이 이 실측의 요점이다.
2. 이유 둘: ① 할당자가 자유 리스트에서 맞는 크기의 빈 블록을 탐색해야 한다(스택은 포인터 산술 한 줄로 끝난다). ② 맞는 블록이 없으면 운영체제에 새 메모리를 요청하는 시스템 콜까지 번질 수 있다. 스레드가 여러 개일 때만 추가로 붙는 비용은 자유 리스트 접근을 위한 **동기화**(락 또는 락 없는 자료구조 오버헤드)다 — 단일 스레드 프로그램에도 ①·②는 있지만 동기화는 공유 자원일 때만 필요하다.
3. 실측 예시(이 환경): `0x5643d29cd1a9`(코드) < `0x5643d29d0010`(정적) < `0x5643e953d2b0`(힙) < `0x7fff2982820c`(스택). 두 번째 실행에서는 앞자리(`0x5643...` → `0x55d7...`)가 통째로 바뀌지만 네 값의 대소 관계는 그대로다.
4. 일반 빌드: `depth = 7000`대까지 찍고 `Segmentation fault`. ASan 빌드: `AddressSanitizer: stack-overflow`가 정확한 진단명으로 찍힌다. 프레임 하나가 최소 1024바이트(`frame_padding`) + 호출 관례 오버헤드를 쓰므로, 8MB(`ulimit -s` = 8192KB) ÷ 약 1KB ≈ 8000번 근처에서 바닥이 드러나는 것이 이 절의 실측과 일치한다.
5. 200,000개 `double` = 1,600,000바이트(약 1.5MB)로 8MB 한도 안이라 크래시 없이 정상 종료한다. `ulimit -s`가 8192(KB)라면 `double` 배열은 대략 1,000,000개(8MB) 근처부터 죽기 시작한다고 예측할 수 있고, 그 경계 위아래로 배열 크기를 바꿔 가며 직접 검증하는 것이 이 실습의 핵심이다.
:::

이 절의 코드는 전부 직접 쳐라. 특히 `sensor.cpp`와 `overflow.cpp`는 반드시 실행까지 해서 크래시를 눈으로 봐야 한다 — 크래시 메시지를 읽는 연습도 컴파일 에러를 읽는 연습만큼 중요하다. 기준 명령은 `g++ -std=c++20 -Wall -Wextra main.cpp -o main && ./main`, 새니타이저는 `g++ -std=c++20 -Wall -Wextra -g -fsanitize=address main.cpp -o main && ./main`, 벤치마크(`bench.cpp`)만 `-O2`를 붙인다.
