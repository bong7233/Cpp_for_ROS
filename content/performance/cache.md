# 8.2 캐시와 메모리 레이아웃

::: lead
`std::vector<int>` 백만 개를 열 바퀴 합산하면 2.4 ms, 같은 값을 담은 `std::list<int>`는 51.8 ms — 같은 $O(n)$ 순회가 이 환경에서 약 22배 갈린다. Big-O가 세는 "연산 횟수"는 같은데 시간이 다르다면, 비용은 연산이 아니라 다른 곳에 있다는 뜻이다. 이 절은 그 다른 곳 — CPU와 메모리 사이의 캐시 계층 — 을 이론이 아니라 도구로 다룬다. `valgrind --tool=cachegrind`로 캐시 미스를 한 번 단위로 세고, 포인터 체이싱의 지연을 나노초로 재고, 로봇 다리 상태 구조체의 배치(AoS vs SoA)가 미스 수를 정확히 4배 가르는 것을 확인한다. [8.4 데이터 지향 설계](#/data-oriented)가 이 원리 위에 설계 방법론을 세우니, 여기서는 원리와 측정을 확실히 잡는다.
:::

## 같은 복잡도, 22배의 시간

문제부터 재현한다. 두 컨테이너에 같은 정수 백만 개를 담고, 같은 순회를 열 번 돌린다.

```cpp title="vec_list.cpp — 컨테이너만 다른 동일한 합산"
#include <cstdio>
#include <vector>
#include <list>
#include <chrono>

constexpr int N = 1'000'000;
constexpr int PASSES = 10;

int main() {
#ifdef USELIST
    std::list<int> c;
    for (int i = 0; i < N; ++i) c.push_back(i);
#else
    std::vector<int> c(N);
    for (int i = 0; i < N; ++i) c[i] = i;
#endif
    long sum = 0;
    auto t0 = std::chrono::steady_clock::now();
    for (int p = 0; p < PASSES; ++p)
        for (int v : c) sum += v;
    auto t1 = std::chrono::steady_clock::now();
    printf("sum=%ld elapsed_ms=%.2f\n", sum,
           std::chrono::duration<double, std::milli>(t1 - t0).count());
}
```

```console
$ g++ -std=c++20 -Wall -Wextra -O2 -g vec_list.cpp -o vec_bench
$ g++ -std=c++20 -Wall -Wextra -O2 -g -DUSELIST vec_list.cpp -o list_bench
$ ./vec_bench
sum=4999995000000 elapsed_ms=2.36
$ ./list_bench
sum=4999995000000 elapsed_ms=51.82
```

(g++ 13.3.0 / `-O2` / Ubuntu 24.04 x86-64, 각 5회 실행 중앙값 — 이하 이 절의 모든 실측 동일 환경.) 방문 횟수 천만 번, 덧셈 천만 번, 결과까지 똑같다. [5.3](#/seq-containers)에서 이 격차를 "노드가 흩어져 있어서"라고 설명했는데, 이 절은 그 설명을 숫자로 증명하는 절이다 — 흩어져 있으면 **무엇이** 몇 번 더 일어나서 22배가 되는가.

## 메모리 벽 — 연산이 아니라 접근이 비용이다

원인은 CPU 바깥에 있다. 현대 CPU 코어는 사이클당 덧셈 여러 개를 처리하는데, 메인 메모리(DRAM)에서 값 하나를 가져오는 데는 수백 사이클이 걸린다 — 이 절 실측으로 이 환경에서 약 165 ns, 2.8 GHz 기준 약 460사이클이다(§포인터 체이싱에서 직접 잰다). 연산 능력은 수십 년간 메모리 지연보다 훨씬 빠르게 성장해 왔고, 그 벌어진 격차를 **메모리 벽(memory wall)**이라 부른다. 결과적으로 데이터 순회 코드의 시간은 대부분 "CPU가 계산하는 시간"이 아니라 "CPU가 데이터를 기다리는 시간"이다.

하드웨어의 대응이 **캐시(cache)** — 코어 옆에 둔 작고 빠른 SRAM 계층이다. 최근 쓴 데이터의 사본을 여기 두고, 다음 접근이 여기서 해결되면(히트) 몇 사이클, 없으면(미스) 다음 계층으로 내려간다. 이 구조가 이득이 되는 근거는 프로그램의 두 가지 통계적 성질이다.

- **시간 지역성(temporal locality)** — 방금 쓴 데이터는 곧 또 쓴다. 루프 변수, 반복 참조되는 상태가 그렇다.
- **공간 지역성(spatial locality)** — 방금 쓴 데이터의 **이웃**을 곧 쓴다. 배열 순회가 그렇다.

캐시는 이 두 성질에 베팅한 장치다. 그래서 지역성이 좋은 코드(연속 배열 순회)는 캐시가 거의 다 받아내고, 지역성이 없는 코드(흩어진 노드 추적)는 매번 벽 너머까지 간다 — 22배는 여기서 나온다.

::: hist "메모리 벽"이라는 이름
1994년 Wulf와 McKee의 논문 "Hitting the Memory Wall"이 굳힌 용어다. 당시의 경고는 "CPU가 아무리 빨라져도 결국 모든 프로그램이 메모리 대기 시간에 수렴한다"였고, 30년이 지난 지금 그 예측은 캐시 계층이라는 완충 장치 덕에 절반만 실현됐다 — 지역성이 좋은 코드에게는 벽이 없고, 나쁜 코드에게는 예고대로 벽이 있다. 어느 쪽이 될지는 하드웨어가 아니라 데이터 배치가 정한다는 것, 그게 이 절과 [8.4](#/data-oriented)의 전제다.
:::

## 이 기계의 캐시 계층 — getconf로 읽는다

계층의 실제 파라미터는 추측할 필요 없이 OS에 물으면 된다.

```console
$ getconf LEVEL1_DCACHE_LINESIZE
64
$ getconf -a | grep -i cache
LEVEL1_DCACHE_SIZE                 32768
LEVEL1_DCACHE_ASSOC                8
LEVEL1_DCACHE_LINESIZE             64
LEVEL2_CACHE_SIZE                  1048576
LEVEL2_CACHE_ASSOC                 16
LEVEL2_CACHE_LINESIZE              64
LEVEL3_CACHE_SIZE                  34603008
LEVEL3_CACHE_ASSOC                 11
LEVEL3_CACHE_LINESIZE              64
```

(일부 줄 생략. 이 환경: Intel Xeon 2.80 GHz, 4코어.) 정리하면 코어당 L1 데이터 캐시 32 KB, L2 1 MB, 전 코어 공유 L3 33 MB — 그리고 세 계층 모두 **라인 크기 64바이트**다. 용량이 한 단계에 30배씩 뛰는 대신 지연도 단계마다 뛴다. 이 절에서 워킹셋(working set — 한 루프가 반복해서 만지는 데이터 전체)의 크기를 계속 따지는 이유가 이 표다: 워킹셋이 32 KB에 들어가는 루프와 33 MB를 넘는 루프는 같은 코드라도 다른 기계에서 도는 것과 같다. [8.6](#/benchmarking)에서 이 계층이 벤치마크 결과표에 계단으로 찍히는 것을 본다.

::: tip lscpu로도 같은 정보를 본다
`lscpu | grep -i cache`는 같은 내용을 코어 수까지 곱해서 보여준다(예: "L1d: 128 KiB (4 instances)" — 코어당 32 KB × 4). 다른 기기의 수치를 확인할 때 `getconf -a | grep -i cache`와 함께 기본 장비다. 참고로 라인 크기 64바이트는 x86-64에서 사실상 표준이지만 아키텍처가 다르면(일부 ARM 서버는 128바이트) 다를 수 있다 — 그래서 하드코딩 대신 `std::hardware_destructive_interference_size`([2.12](#/object-layout))가 있다.
:::

## 캐시 라인 — 하나를 읽으면 열여섯 개가 온다

캐시와 메모리 사이의 전송 단위는 바이트가 아니라 **캐시 라인(cache line)** — 이 기계에서 64바이트다. `int` 하나(4바이트)를 읽어도 그 주소를 포함한 64바이트 정렬 블록 전체가 캐시로 올라온다. 즉 `int` 배열을 순서대로 읽으면 미스 한 번에 열여섯 개가 딸려 오고, 다음 열다섯 번의 접근은 공짜다. 이게 공간 지역성이 하드웨어에서 실현되는 방식이고, 반대로 말하면 — **64바이트씩 건너뛰며 읽으면 모든 접근이 미스가 된다.** 읽는 횟수를 똑같이 놓고 보폭만 바꿔서 확인한다.

```cpp title="stride.cpp — 읽기 백만 번은 동일, 보폭만 1 vs 16"
#include <cstdio>
#include <vector>

constexpr int N = 16'000'000;   // 64 MB — L3(33 MB)보다 크게 잡는다
constexpr int TOUCHES = 1'000'000;

int main(int argc, char**) {
    std::vector<int> v(N, 1);
    long sum = 0;
    // 인자 없음: 보폭 1 (한 라인의 int 16개를 전부 쓰고 다음 라인으로)
    // 인자 있음: 보폭 16 = 64바이트 (읽기마다 새 캐시 라인)
    const int stride = (argc >= 2) ? 16 : 1;
    for (int i = 0; i < TOUCHES; ++i) {
        sum += v[(long(i) * stride) % N];
    }
    printf("stride=%d sum=%ld\n", stride, sum);
}
```

cachegrind(다음 절에서 도구 자체를 설명한다)로 D1 — L1 데이터 캐시 — 읽기 미스만 뽑으면:

| 보폭 | 읽기 미스 (실측) | 산수 예상 |
| --- | --- | --- |
| 1 (연속) | 75,940 | 백만 읽기 ÷ 라인당 16개 = 62,500 + 초기화 잔여 |
| 16 (라인당 1개) | 1,013,425 | 읽기마다 미스 = 1,000,000 |

읽기 횟수도, 더한 값도 같은데 미스가 13배다 — 라인당 16개라는 산수가 실측과 그대로 맞는다. 시간도 따라온다: 읽기 횟수를 1억으로 늘려 네이티브로 재면 보폭 1이 99.4 ms, 보폭 16이 456.9 ms(각 3회 중앙값), 4.6배다. **캐시 라인은 "메모리를 얼마나 읽었는가"의 실제 단위다.** 배열에서 4바이트만 쓰고 60바이트를 버리는 접근 패턴은, 대역폭의 94%를 버리는 패턴이다 — 이 관점이 아래 AoS/SoA 실측의 전부다.

## cachegrind — 캐시 미스를 세는 결정론적 현미경

이 절의 주 도구는 valgrind의 **cachegrind**다(이 환경: valgrind 3.22.0). 프로그램을 가상 CPU 위에서 한 명령씩 실행하면서, 모든 메모리 접근을 L1/LL(last-level) 2단 캐시 모형에 통과시켜 히트/미스를 센다. 켜는 법:

```console
$ valgrind --tool=cachegrind --cache-sim=yes ./aos_bench
--3132-- warning: L3 cache found, using its data for the LL simulation.
--3132-- warning: specified LL cache: line_size 64  assoc 11  total_size 34,603,008
--3132-- warning: simulated LL cache: line_size 64  assoc 17  total_size 35,651,584
AoS sum=14999985000.000
==3132== D refs:        38,678,494  (32,509,885 rd   + 6,168,609 wr)
==3132== D1  misses:    16,015,754  (15,013,460 rd   + 1,002,294 wr)
==3132== LLd misses:       508,918  (     7,509 rd   +   501,409 wr)
==3132== D1  miss rate:       41.4% (      46.2%     +      16.2%  )
```

(출력 일부. `aos_bench`는 두 절 아래의 AoS 벤치마크다.) 세 가지를 읽어라. 첫째, valgrind 3.22에서 `--cache-sim=yes`는 **직접 켜야 한다** — 빼고 실행하면 명령 수(`I refs`)만 나온다(이 환경에서 실제로 확인했다. 3.20 이전 버전은 캐시 시뮬레이션이 기본값이었으므로 오래된 자료와 다를 수 있다). 둘째, 시뮬레이션할 캐시 크기는 **CPUID로 실행 중인 CPU에서 자동으로 읽어 온다** — 위 warning 줄이 그 증거다: 이 기계의 실제 L3(34,603,008바이트, 11-way)를 감지한 뒤, 시뮬레이터 제약에 맞게 35,651,584바이트 17-way로 보정해서 쓴다고 스스로 보고한다. 셋째, 자동 감지 대신 `--I1=<크기>,<연관도>,<라인>` `--D1=` `--LL=` 옵션으로 임의 캐시를 지정할 수 있다 — 타깃 보드(예: 라즈베리파이급 ARM)의 캐시 파라미터를 넣어 개발 PC에서 미리 시뮬레이션하는 용도로 쓴다.

cachegrind가 이 챕터에 특히 맞는 이유는 **결정론**이다. 실제 하드웨어 카운터 기반 프로파일러([8.1](#/profiling)의 perf)는 실행할 기계·순간마다 수치가 흔들리지만, cachegrind는 시뮬레이션이라 같은 프로그램이면 같은 숫자가 나온다 — 이 환경에서 같은 벤치마크를 두 번 돌려 D1 미스 15,013,460회가 마지막 자리까지 일치하는 것을 확인했다. "레이아웃을 바꿨더니 미스가 몇 회에서 몇 회로 줄었다"를 잡음 없이 비교할 수 있다는 뜻이다.

::: note cachegrind에 없는 것 두 가지 — 시간과 프리페처
cachegrind 아래서 프로그램은 수십 배 느리게 돈다. 그래서 **시간은 반드시 네이티브 실행으로 따로 재고, cachegrind에서는 횟수만 읽는다** — 이 절의 모든 표가 그 규칙을 따른다. 또 하나: 실제 CPU에는 연속 접근을 감지해 다음 라인을 미리 가져오는 하드웨어 프리페처(prefetcher)가 있는데 cachegrind는 이를 모의하지 않는다. 그래서 연속 순회의 미스 수는 실제 하드웨어에서 체감되는 것보다 과대평가된다 — 아래 AoS/SoA 실측에서 미스는 4배인데 시간은 1.6배인 이유 중 하나다.
:::

::: tip cg_annotate — 미스가 난 소스 줄을 찍는다
cachegrind는 `cachegrind.out.<pid>` 파일도 남긴다. `cg_annotate cachegrind.out.<pid>`를 돌리면 함수별·소스 줄별 미스 표가 나온다 — 이 절의 AoS 벤치마크에서 D1 읽기 미스 15,013,460회 중 15,000,001회(99.9%)가 합산 루프 한 줄에 찍히는 것까지 확인할 수 있다. 미스가 "프로그램 어딘가"가 아니라 "이 줄"에서 난다는 것을 못박는 도구다.
:::

## 실측 ①: 연속 배열 vs 링크드 리스트 — 미스를 센다

도구를 들고 첫 실측으로 돌아간다. `vec_bench`와 `list_bench`를 cachegrind에 통과시키면:

| 항목 | `vector` | `list` |
| --- | --- | --- |
| D1 읽기 미스 | 638,473 | 5,514,195 |
| D1 읽기 미스율 | 21.2% | 5.5% |
| 네이티브 시간 | 2.36 ms | 51.82 ms |

미스 수부터. `vector`는 `int` 천만 방문 ÷ 라인당 16개 = 62만 5천이라는 산수 그대로다(63만 8천 — 나머지는 초기화). `list`는 8.6배인 551만 회다: libstdc++의 리스트 노드는 앞뒤 포인터 16바이트 + `int` 4바이트 + 패딩으로 24바이트이고, 힙 할당자의 관리 공간까지 더하면 노드가 대략 32바이트 간격으로 놓인다 — 라인당 두 개, 천만 방문에 미스 약 5백만이라는 산수와 실측이 대략 맞는다. 페이로드 4바이트를 위해 라인 64바이트를 끌어오는 구조라, 공간 지역성 관점에서 리스트는 **최선의 경우에도** 배열보다 라인을 몇 배 더 태운다.

그런데 미스 8.6배로는 시간 22배가 다 설명되지 않는다. 남은 격차는 다음 절의 주제 — 리스트 순회는 다음 노드의 **주소가 현재 노드를 읽어야만 나오는** 의존 접근이라 미스 비용이 겹쳐지지 못하고 전부 직렬로 쌓인다는 것, 그리고 연속 배열 순회는 컴파일러가 SIMD로 묶고([8.5](#/simd)) 프리페처가 미리 채워 주는 반면 리스트에는 둘 다 불가능하다는 것이다.

::: warn 미스"율"은 비교 지표로 위험하다
위 표를 다시 봐라 — 22배 느린 `list`의 미스율(5.5%)이 `vector`(21.2%)보다 **낮다.** 미스율의 분모는 "전체 메모리 접근 수"인데, 리스트 순회는 노드마다 포인터도 읽어야 해서 분모 자체가 수십 배 크다(이 실측에서 `vector` 읽기 3백만 회 vs `list` 1억 회). 접근을 낭비할수록 미스율이 예뻐지는 왜곡이다. 레이아웃을 비교할 때는 **같은 작업량 기준의 미스 횟수**(그리고 네이티브 시간)를 봐라. 비율은 같은 프로그램 안에서 "이 루프가 캐시 친화적인가"를 볼 때만 쓴다.
:::

::: interview "배열과 링크드 리스트의 차이"의 상급 답변
이 질문에 삽입/삭제 $O(1)$ vs $O(n)$만 답하면 교과서 수준이다. 답변 뼈대: ① 현대 CPU에서 순회 비용은 연산이 아니라 캐시 미스가 지배한다 — 배열은 라인당 원소 여러 개를 공짜로 얻고, 리스트는 노드마다 라인 하나를 태운다(이 절 실측: 미스 8.6배). ② 리스트의 다음 주소는 읽기 결과에 의존해서 미스 지연이 직렬로 쌓이고, 프리페치·벡터화도 막힌다(실측: 시간 22배). ③ 그래서 "중간 삽입이 잦으면 list"라는 규칙도 실측 없이는 못 믿는다 — 삽입 지점까지 **가는** 비용이 순회라서, 실무에서는 vector가 이기는 경우가 대부분이다([5.3](#/seq-containers)). 여기까지 말하면 자료구조가 아니라 하드웨어를 이해한다는 신호가 된다.
:::

## 실측 ②: 포인터 체이싱 — 지연의 사다리를 나노초로 잰다

리스트 순회의 본질을 격리한 실험을 하나 만든다. **포인터 체이싱(pointer chasing)** — 다음에 읽을 주소가 지금 읽은 값에서 나오는 접근 패턴이다. 배열 `next`에 "다음 인덱스"를 담아 두면 링크드 리스트의 `->next` 추적과 같은 구조가 되고, `next`를 어떻게 채우는가로 접근 순서만 통제할 수 있다.

```cpp title="chase.cpp — 핵심 루프 (전체 코드는 next 배열 준비 포함)"
// 준비: next[i]를 (a) 순차: i+1, (b) 무작위 순열의 순환 고리로 채운다.
// 무작위 버전이 곧 "힙에 흩어진 링크드 리스트"의 모형이다.
long sum = 0;
int idx = 0;
for (int s = 0; s < STEPS; ++s) {
    idx = next[idx];   // 다음 주소가 이 읽기의 결과에 달려 있다
    sum += idx;
}
```

같은 코드에서 `next`의 내용과 배열 크기만 바꿔 세 가지를 쟀다(STEPS = 천만, 각 5회 중앙값):

| 구성 | 워킹셋 | 스텝당 시간 | 2.8 GHz 환산 |
| --- | --- | --- | --- |
| 순차 (`i+1`) | 4 MB | 3.6 ns | 약 10사이클 |
| 무작위, L3 안 | 8 MB (< 33 MB) | 40.9 ns | 약 115사이클 |
| 무작위, L3 밖 | 64 MB (> 33 MB) | 165.6 ns | 약 460사이클 |

같은 루프, 같은 스텝 수인데 **접근 패턴과 워킹셋 크기만으로 46배**다. 이 표가 메모리 벽의 실측이다: 순차 접근은 라인 재사용과 프리페치로 L1 속도에 붙고, L3 안의 무작위 접근은 L2/L3 왕복 지연을, L3 밖의 무작위 접근은 DRAM 왕복 지연(이 환경 실측 약 103~168 ns — 가상화 환경이라 TLB 미스 비용까지 섞인 값이고 실행 간 변동도 있었다)을 스텝마다 문다.

체이싱이 특히 잔인한 이유는 **의존성**이다. 독립적인 미스 여러 개는 CPU가 동시에 걸어 두고 겹쳐 기다릴 수 있어서(이걸 메모리 수준 병렬성이라 한다) 미스당 실효 비용이 크게 내려간다 — [8.4](#/data-oriented)의 hot/cold 실측에서 미스당 약 9 ns로 나온 게 그 경우다. 체이싱은 다음 주소를 모르니 겹칠 수가 없고, 왕복 지연 전액을 매 스텝 직렬로 낸다. `std::list`, `std::map`([5.2](#/assoc-containers)의 레드블랙 트리), 그리고 `shared_ptr` 그래프로 엮인 객체 구조가 전부 이 패턴이다.

::: perf 이 환경의 지연 사다리 (실측 요약)
순차 순회 스텝당 3.6 ns → L3 상주 무작위 40.9 ns(11배) → DRAM행 무작위 165.6 ns(46배). 절대값은 기기·가상화 여부에 따라 다르지만, "순차 대 무작위가 한 자릿수 배, 캐시 안 대 캐시 밖이 또 몇 배"라는 사다리 모양은 어디서나 유지된다. 네 기기의 사다리는 이 절 끝의 실습으로 직접 재라 — 그 숫자가 [6.8](#/realtime)의 지터 예산을 짤 때의 밑천이 된다.
:::

## 실측 ③: AoS vs SoA — 로봇 다리 상태 배열

이제 이 절의 본론인 **레이아웃 설계**다. 로봇 다리 상태를 담는 자연스러운 구조체와, 그 배열:

```cpp title="aos_soa.cpp — 같은 데이터, 두 가지 배치"
struct LegState {          // sizeof = 32, alignof = 8 (실측)
    double angle;
    double velocity;
    double torque;
    bool   active;         // 1바이트 + 꼬리 패딩 7바이트
};
std::vector<LegState> legs(N);            // ① AoS: Array of Structs

struct LegStates {                        // ② SoA: Struct of Arrays
    std::vector<double>  angle;
    std::vector<double>  velocity;
    std::vector<double>  torque;
    std::vector<uint8_t> active;
};
```

**AoS(Array of Structs)**는 "다리 하나의 모든 필드"가 붙어 있고, **SoA(Struct of Arrays)**는 "모든 다리의 같은 필드"가 붙어 있다. 어느 쪽이 좋은가는 순회가 무엇을 만지는가에 달렸다. 각도만 합산하는 루프 — 필터링, 통계, 한 필드만 쓰는 모든 패스의 모형 — 를 N = 백만, 30바퀴로 돌리면 두 배치가 캐시 라인을 이렇게 쓴다:

```text nolines
AoS: 64-byte line = LegState x 2
[angle|vel|torque|act+pad][angle|vel|torque|act+pad]
  8B use          24B waste  8B use         24B waste   -> 16/64 = 25%

SoA: 64-byte line = angle x 8
[angle|angle|angle|angle|angle|angle|angle|angle]
                                                        -> 64/64 = 100%
```

산수 예상: AoS는 라인당 원소 2개라 읽기 3천만 ÷ 2 = 미스 1,500만, SoA는 라인당 8개라 3천만 ÷ 8 = 375만. cachegrind 실측:

| 배치 | D1 읽기 미스 | D1 읽기 미스율 | 네이티브 시간 (중앙값 5회) |
| --- | --- | --- | --- |
| AoS | 15,013,460 | 46.2% | 58.9 ms |
| SoA | 3,763,595 | 12.3% | 37.7 ms |

미스는 예상과 자릿수까지 일치하는 **정확히 4.0배**다. `cg_annotate`로 확인하면 AoS 미스의 99.9%가 합산 루프 한 줄에 찍힌다. 필요 없는 `velocity`·`torque`·`active`가 캐시 라인에 끌려 들어와 라인의 75%를 버리는 비용이 그대로 미스 수가 된 것이다.

시간은 1.6배로, 미스 배수(4배)보다 작다 — 정직하게 짚고 간다. 이 워킹셋(AoS 32 MB)은 L3에 거의 들어가서 D1 미스 대부분이 L2/L3에서 해결됐고(LLd 읽기 미스는 7,509회뿐), 연속 순회라 실제 하드웨어의 프리페처가 미스 지연을 상당 부분 숨겼다 — cachegrind가 모의하지 않는 바로 그 효과다. 격차가 진짜 벌어지는 것은 워킹셋이 L3를 넘어 미스가 DRAM행이 될 때다: [8.4](#/data-oriented)의 hot/cold 실측이 같은 원리로 14배까지 가는 것을 보여준다. **미스 수는 레이아웃이 정하고, 미스의 가격은 워킹셋이 어느 계층에 얹히는가가 정한다.**

::: perf SoA가 이기는 조건, AoS가 이기는 조건
SoA의 이득은 "한 패스가 일부 필드만 만진다"에서 나온다. 반대로 다리 하나의 **모든** 필드를 한꺼번에 읽고 쓰는 접근(개별 다리의 IK 계산 등)이라면 AoS가 그 다리의 데이터를 라인 한두 개에 모아 주니 유리하고, SoA는 배열 네 곳으로 접근을 흩는다. 규칙: **순회가 필드 방향이면 SoA, 개체 방향이면 AoS.** 둘 다 있으면 가장 뜨거운 루프([8.1](#/profiling)이 찾아 주는)에 맞춘다. SoA의 추가 이득 — 같은 필드가 연속이면 자동 벡터화가 걸린다 — 은 [8.5](#/simd)에서 실측한다.
:::

## 정렬과 패딩 — 구조체가 라인에 눕는 모양

레이아웃 설계의 마지막 조각은 구조체 내부다. [2.12](#/object-layout)에서 본 정렬 규칙 — 각 멤버는 `alignof`의 배수 주소에 놓이고, 컴파일러가 그 사이를 패딩으로 메운다 — 을 캐시 관점에서 다시 본다. 멤버 **선언 순서만** 바꾼 두 구조체:

```cpp title="layout.cpp — 같은 멤버, 순서만 다르다"
// ❌ bool 뒤에 double이 올 때마다 7바이트 패딩이 낀다
struct LegStateBad {
    bool   active;      double angle;
    bool   contact;     double velocity;
    bool   calibrated;  double torque;
};

// ✅ 정렬 요구가 큰 멤버부터 — 패딩이 꼬리 5바이트로 모인다
struct LegStateGood {
    double angle; double velocity; double torque;
    bool   active; bool contact; bool calibrated;
};

static_assert(alignof(LegStateBad) == 8 && alignof(LegStateGood) == 8);
```

```console
$ g++ -std=c++20 -Wall -Wextra -O2 layout.cpp -o layout && ./layout
Bad : sizeof=48 alignof=8
Good: sizeof=32 alignof=8
```

멤버 합계는 27바이트로 같은데 `sizeof`가 48 vs 32다. 단독 객체라면 16바이트 낭비는 아무것도 아니지만, **배열이 되는 순간 캐시 라인 단위로 청구된다**: 64바이트 라인에 `Bad`는 1.3개, `Good`은 정확히 2개가 눕는다. 백만 개 배열이면 48 MB vs 32 MB — 워킹셋이 L3에 들어가느냐를 이 순서 하나가 가를 수 있다. 규칙은 하나뿐이다: **정렬 요구가 큰 멤버부터 선언한다.** 그 다음 단계 — 애초에 이 구조체에 이 필드가 같이 있어야 하는가 — 가 [8.4](#/data-oriented)의 hot/cold 분리다.

::: note 캐시 라인은 스레드 사이의 회계 단위이기도 하다
캐시 일관성 프로토콜은 라인 단위로 소유권을 관리한다. 그래서 서로 다른 스레드가 각자 갱신하는 두 변수가 **한 라인에** 있으면, 논리적으로 무관해도 라인이 코어 사이를 핑퐁하는 false sharing이 난다 — 정의와 `alignas(64)` 처방은 [2.12](#/object-layout)에서, 멀티스레드 카운터로의 실측은 [6.5](#/atomic)에서 이미 다뤘다. 이 절의 관점에서 한 줄로 요약하면: 단일 스레드에서는 라인에 **같이 쓰는 데이터를 모으는 게** 이득이고, 스레드가 갈리면 **각자 쓰는 데이터를 라인 단위로 떼는 게** 이득이다.
:::

## 로보틱스 연결 — 1 kHz 루프의 워킹셋과 지터

헥사포드 제어 루프에 그대로 대입한다. 다리 6개 × 관절 3개의 상태 — 각도·속도·토크·명령 — 는 SoA로 눕혀도 몇 KB가 안 된다. 이 워킹셋은 L1에 통째로 들어가고, 매 주기 같은 데이터를 만지니 시간 지역성도 완벽하다 — 즉 **제어 루프의 계산 자체는 캐시 문제가 없어야 정상이다.** 이게 중요한 이유는 지터([6.8](#/realtime)) 때문이다: 1 kHz 루프의 1 ms 예산에서 무서운 것은 평균이 아니라 최악 실행 시간인데, 이 절의 사다리가 보여주듯 캐시 미스는 접근 하나를 10사이클에서 460사이클로 만든다. 제어 스레드의 데이터가 L1/L2에 상주하도록 작고 연속으로 유지하는 것, 콜백 스레드와 공유하는 데이터를 라인 단위로 분리하는 것([6.9](#/thread-architecture))이 지터 예산의 캐시 항목이다. 그리고 `ros2_control`이 관절 상태를 객체 그래프가 아니라 `double` 배열 핸들로 노출하는 설계([10.9](#/ros2-control))는 정확히 이 절의 결론을 프레임워크가 먼저 실천한 것이다.

레이아웃이 승부를 가르는 규모는 인지 쪽에 있다. LiDAR 포인트클라우드는 스캔당 수만~수십만 포인트로 워킹셋이 MB 단위다 — PCL([11.6](#/pcl))의 포인트가 SIMD를 위해 16바이트 정렬로 패딩된 평범한 구조체의 연속 배열인 것, Eigen([9.1](#/eigen))이 행렬을 연속 메모리에 두는 것 모두 이 절의 산수 위에 서 있다. costmap 순회, 포인트 필터링처럼 "큰 배열을 매 주기 훑는" 코드를 쓰게 되면, 이 절의 cachegrind 워크플로 — 산수로 예상하고, 미스를 세고, 레이아웃을 바꿔 다시 세는 — 가 그대로 작업 순서가 된다.

## 요약

- 같은 $O(n)$ 순회가 `vector` 2.4 ms vs `list` 51.8 ms로 22배 갈렸다 — Big-O가 세지 않는 비용, 즉 메모리 접근이 지배해서다. CPU 연산과 DRAM 지연의 격차가 메모리 벽이고, 캐시 계층은 시간·공간 지역성에 베팅해 그 벽을 가리는 장치다.
- 캐시와 메모리의 전송 단위는 64바이트 캐시 라인이다(`getconf LEVEL1_DCACHE_LINESIZE`로 실측). 보폭 실험에서 같은 백만 번 읽기가 보폭 1이면 미스 7.6만, 보폭 16(64바이트)이면 101만 — 라인당 원소 수라는 산수 그대로였다.
- cachegrind는 CPUID로 감지한(또는 `--I1/--D1/--LL`로 지정한) 캐시를 시뮬레이션해 미스를 결정론적으로 센다 — valgrind 3.22에서는 `--cache-sim=yes`를 직접 켜야 하고, 시간 측정은 네이티브 실행으로 따로 한다.
- 포인터 체이싱 실측: 순차 3.6 ns → L3 상주 무작위 40.9 ns → DRAM행 무작위 165.6 ns/스텝. 의존 접근은 미스 지연이 겹쳐지지 못하고 직렬로 쌓인다 — 리스트·트리·포인터 그래프의 구조적 비용이다.
- 한 필드만 훑는 루프에서 AoS는 라인의 75%를 버려 D1 미스가 SoA의 정확히 4.0배였다(1,501만 vs 376만, 산수와 일치). 시간 차(1.6배)가 그보다 작은 것은 워킹셋이 L3 안이라 미스가 싸게 해결됐기 때문 — 미스 수는 레이아웃이, 미스 가격은 워킹셋 크기가 정한다.
- 구조체는 정렬 요구가 큰 멤버부터 선언한다 — 같은 멤버로 `sizeof` 48 vs 32를 실측했다. 배열이 되는 순간 패딩은 캐시 라인과 워킹셋으로 청구된다.

::: quiz 연습문제
1~3번은 개념·예측 문제, 4~5번은 네 컴퓨터에서 직접 실측하는 문제다.

1. 시간 지역성과 공간 지역성을 각각 정의하고, 이 절의 실측 중 각 지역성이 깨졌을 때의 비용을 보여주는 사례를 하나씩 골라 짝지어라.

2. 실측 ①에서 22배 느린 `list`의 D1 미스율(5.5%)이 `vector`(21.2%)보다 낮게 나왔다. 이 역전이 왜 생기는지 설명하고, 레이아웃 비교에는 어떤 지표를 써야 하는지 답하라.

3. (예측) `LegState`에 `double command;` 필드를 하나 추가해 `sizeof`가 40이 되면, 각도 합산 루프의 AoS D1 읽기 미스는 지금의 1,500만 회에서 어느 방향으로 얼마나 변할지 산수로 예측하라. 힌트: 40바이트 원소는 64바이트 라인 경계에 걸치기 시작한다 — 원소 8개가 라인 5개를 차지한다.

4. (실습, 코드 작성형) 이 절의 `stride.cpp`를 직접 타이핑해 `g++ -std=c++20 -Wall -Wextra -O2 -g stride.cpp -o stride`로 경고 없이 빌드하고, `valgrind --tool=cachegrind --cache-sim=yes ./stride`와 `./stride x`의 D1 읽기 미스를 비교하라. 네 기기의 `getconf LEVEL1_DCACHE_LINESIZE`가 64가 아니라면 보폭도 그에 맞춰 바꿔라. 미스 비율이 "라인 크기 ÷ sizeof(int)" 산수와 맞는 것까지 확인하는 게 성공 기준이다.

5. (실습) 이 절의 AoS/SoA 벤치마크를 타이핑해 네 기기에서 D1 미스와 네이티브 시간을 재라. 그다음 `--LL=1048576,16,64` 옵션으로 LL 캐시를 1 MB짜리 임베디드급으로 줄여 다시 돌려서, 같은 프로그램의 LLd 미스가 어떻게 변하는지 확인하라 — 타깃 보드의 캐시를 개발 PC에서 미리 시뮬레이션하는 연습이다.
:::

::: answer 해설
1. 시간 지역성: 방금 쓴 데이터를 곧 또 쓴다 — 제어 루프가 매 주기 같은 상태를 만지는 것이 활용 사례고, 워킹셋이 캐시보다 커서 한 바퀴 돌고 오면 쫓겨나 있는 상황(체이싱 실측의 64 MB 구성, 스텝당 165.6 ns)이 깨진 사례다. 공간 지역성: 방금 쓴 데이터의 이웃을 곧 쓴다 — 보폭 1 순회가 활용 사례고, 보폭 16 실측(같은 읽기 횟수에 미스 13배)이 깨진 사례다.

2. 미스율의 분모는 전체 메모리 접근 수다. 리스트 순회는 노드마다 다음 포인터까지 읽어 분모가 수십 배 크고(실측: 읽기 1억 회 vs 3백만 회), 그 결과 접근을 낭비할수록 비율은 오히려 예뻐진다. 레이아웃 비교에는 같은 작업량 기준의 미스 횟수와 네이티브 실행 시간을 쓴다 — 이 절의 모든 표가 횟수를 쓰는 이유다.

3. 늘어난다. 32바이트일 때는 라인당 정확히 2개라 3천만 읽기에 1,500만 미스였다. 40바이트면 원소 8개가 라인 5개를 차지하므로(8 × 40 = 320 = 5 × 64) 8개 읽을 때 새 라인을 5번 만난다 — 3천만 × 5/8 ≈ 1,875만 회로 약 25% 증가. 필드 하나가 순회 전체의 미스를 올리는 것, 그리고 원소 크기가 라인의 약수가 아니면 경계 걸침이 추가된다는 것이 요점이다. 예측했으면 실제로 재서 확인해라.

4. 성공 기준 그대로다. `getconf`가 64를 돌려주는 기계라면 보폭 16에서 읽기 미스가 TOUCHES(백만)에 수렴하고, 보폭 1에서는 백만 ÷ 16 = 6만 2천 근처가 나와야 한다. cachegrind는 결정론적이라 다시 돌려도 같은 숫자가 나온다는 것도 확인해 봐라 — perf 같은 하드웨어 카운터와의 결정적 차이다.

5. D1 미스는 두 구성에서 거의 그대로다(D1 파라미터를 안 바꿨으니까). 바뀌는 것은 LLd 미스다 — 이 환경 기준 AoS 워킹셋 32 MB는 34 MB LL에는 대부분 들어가지만 1 MB LL에는 전혀 못 들어가므로, D1에서 놓친 접근 대부분이 LLd 미스로 바뀐다. "미스 수는 레이아웃이, 미스 가격은 워킹셋이 어느 계층에 얹히는가가 정한다"는 본문 문장을 옵션 하나로 재현하는 실험이다.
:::

이 절의 벤치마크는 전부 짧다 — `vec_list.cpp`, `stride.cpp`, 체이싱 루프, AoS/SoA, `layout.cpp`를 직접 타이핑해서 네 기기의 지연 사다리와 미스 수를 얻어라. 절대값은 다르게 나와도 산수(라인당 원소 수)와 사다리 모양은 재현된다. 기준 명령: `g++ -std=c++20 -Wall -Wextra -O2 -g <파일>.cpp -o <출력> && ./<출력>`(시간 측정), `valgrind --tool=cachegrind --cache-sim=yes ./<출력>`(미스 계수, 없으면 `sudo apt install valgrind`), `getconf -a | grep -i cache`(네 기계의 계층). 시간과 미스를 같은 실행에서 재지 않는 것만 지켜라.
