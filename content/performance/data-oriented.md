# 8.4 데이터 지향 설계

::: lead
Part III는 "객체 하나가 자기 데이터와 동작을 캡슐화한다"는 설계를 가르쳤고, 그 설계는 지금도 옳다 — 객체가 소수이고 개별 로직이 복잡할 때는. 그런데 게임 엔진, 로봇 제어 루프, 포인트클라우드 처리처럼 **같은 모양의 데이터 수천~수십만 개를 매 주기 순회하는** 워크로드에서는 이 설계가 [8.2](#/cache)에서 본 캐시 계층과 정면으로 충돌한다. 이 절은 그 충돌을 세 번 실측한다 — 가상함수 디스패치가 약 6.6배, hot/cold 필드 혼합이 약 14배, 예측 불가능한 분기가 약 6배를 날리는 것을 이 환경에서 직접 확인하고, "코드가 아니라 데이터의 배치를 먼저 설계한다"는 데이터 지향 설계의 처방을 정리한다.
:::

## 객체 하나의 관점, 배열 전체의 관점

[3.1](#/classes)의 설계 단위는 객체 하나다 — 불변식을 지키는 하나의 `Sensor`, 하나의 `Joint`. 이 관점은 "이 객체가 뭘 할 수 있는가"를 묻는다. 데이터 지향 설계(data-oriented design)는 질문을 바꾼다: **"이 데이터 전체가 캐시를 어떤 모양으로 통과하는가."** CPU는 객체를 모른다. [8.2](#/cache)에서 봤듯 CPU가 아는 단위는 64바이트 캐시 라인이고, 메모리에서 뭘 하나 읽으면 그 주변 64바이트가 통째로 딸려 온다. 그렇다면 매 주기 순회하는 데이터는 **그 주기에 실제로 쓰는 값들이 연속으로 붙어 있도록** 배치하는 게 이득이고, 그 배치를 코드 구조보다 먼저 결정하자는 게 이 절의 전부다.

이 관점 전환이 필요한 워크로드는 명확하다 — 매 프레임 수만 개 엔티티를 갱신하는 게임 엔진, 1 kHz로 관절·센서 상태를 갱신하는 로봇 제어 루프, 수십만 포인트를 필터링하는 포인트클라우드 파이프라인. 공통점은 **대량 + 동종 + 매 주기 전체 순회**다. 이 세 조건이 안 갖춰진 곳에서는 이 절의 처방이 필요 없다 — 그 경계는 마지막 절에서 다시 긋는다.

::: hist 게임 업계에서 온 용어
"데이터 지향 설계"라는 이름은 2000년대 콘솔 게임 엔진 개발자들이 굳혔다. 콘솔은 CPU 대비 메모리가 극단적으로 느렸고, 매 프레임 16.6 ms 안에 수만 개 객체를 갱신해야 했다 — 캐시 미스가 곧 프레임 드랍이었다. 로봇 제어 루프는 조건이 똑같다: 주기 데드라인([6.8](#/realtime)) 안에 대량의 상태를 갱신한다. 용어만 게임 엔진 것일 뿐 원리는 도메인 중립이다.
:::

## 실측 ① — 기반 클래스 포인터의 바다 vs 타입별 연속 벡터

Part III 스타일의 자연스러운 설계부터. 센서마다 클래스를 만들고, `Sensor` 기반 클래스 포인터의 벡터에 담아 가상함수로 갱신한다. 비교 대상은 같은 일을 하는 데이터 지향 배치 — 타입별로 분리된 연속 벡터 두 개다.

```cpp title="sensors_bench.cpp — 같은 작업, 두 가지 배치"
#include <algorithm>
#include <chrono>
#include <cstdio>
#include <cstring>
#include <random>
#include <vector>

constexpr int N = 200'000;   // 센서 수 (두 타입 절반씩)
constexpr int FRAMES = 500;  // 제어 주기 수 — update 호출 총 1억 번

struct Sensor {
    virtual void update() = 0;
    virtual double value() const = 0;
    virtual ~Sensor() = default;
};

struct LidarSensor : Sensor {
    double range = 1.0, intensity = 0.5;
    void update() override {
        range = range * 0.9999 + 0.0001;
        intensity += range * 1e-9;
    }
    double value() const override { return range + intensity; }
};

struct ImuSensor : Sensor {
    double gx = 0.1, gy = 0.2, gz = 0.3;
    void update() override {
        gx = gx * 0.9999 + 0.0001;
        gy = gy * 0.9999 + 0.0001;
        gz = gz * 0.9999 + 0.0001;
    }
    double value() const override { return gx + gy + gz; }
};

static double run_virtual() {
    // ❌ 개별 힙 할당 + 셔플: 오래 돌아 할당·해제가 뒤섞인 프로세스의 힙을 흉내 낸다
    std::vector<Sensor*> sensors;
    sensors.reserve(N);
    for (int i = 0; i < N / 2; ++i) {
        sensors.push_back(new LidarSensor);
        sensors.push_back(new ImuSensor);
    }
    std::mt19937 rng(42);
    std::shuffle(sensors.begin(), sensors.end(), rng);

    auto t0 = std::chrono::steady_clock::now();
    for (int f = 0; f < FRAMES; ++f)
        for (Sensor* s : sensors) s->update();
    auto t1 = std::chrono::steady_clock::now();

    double sum = 0;
    for (Sensor* s : sensors) sum += s->value();
    for (Sensor* s : sensors) delete s;
    printf("checksum=%.6f\n", sum);
    return std::chrono::duration<double, std::milli>(t1 - t0).count();
}

static double run_split() {
    // ✅ 타입별 연속 벡터: 포인터도, 가상 호출도 없다
    std::vector<LidarSensor> lidars(N / 2);
    std::vector<ImuSensor> imus(N / 2);

    auto t0 = std::chrono::steady_clock::now();
    for (int f = 0; f < FRAMES; ++f) {
        for (auto& s : lidars) s.update();  // 타입이 정적으로 확정 — 비가상 호출
        for (auto& s : imus) s.update();
    }
    auto t1 = std::chrono::steady_clock::now();

    double sum = 0;
    for (auto& s : lidars) sum += s.value();
    for (auto& s : imus) sum += s.value();
    printf("checksum=%.6f\n", sum);
    return std::chrono::duration<double, std::milli>(t1 - t0).count();
}

int main(int argc, char** argv) {
    if (argc < 2) { printf("usage: %s virtual|split\n", argv[0]); return 1; }
    double ms = std::strcmp(argv[1], "split") == 0 ? run_split() : run_virtual();
    printf("%s elapsed_ms=%.1f\n", argv[1], ms);
}
```

두 경로 모두 `update()`를 정확히 1억 번 호출한다. 각 5회 실행해 중앙값을 취했다(g++ 13.3.0 / `-O2` / Ubuntu 24.04 x86-64, 이하 이 절의 모든 실측 동일).

```console
$ g++ -std=c++20 -Wall -Wextra -O2 -g sensors_bench.cpp -o sensors_bench
$ ./sensors_bench virtual
checksum=221705.558895
virtual elapsed_ms=1057.6
$ ./sensors_bench split
checksum=221705.558896
split elapsed_ms=160.7
```

**약 6.6배 차이다.** 같은 산술 연산을 같은 횟수 수행했는데도 그렇다(checksum 마지막 자리가 다른 건 부동소수점 합산 순서가 달라서다 — 계산량 차이가 아니다). 어디서 6.6배가 왔는지는 추측하지 않고 cachegrind로 해부한다.

## 6.6배를 해부한다 — 캐시 미스와 간접 분기

```console
$ valgrind --tool=cachegrind --cache-sim=yes --branch-sim=yes ./sensors_bench virtual
$ valgrind --tool=cachegrind --cache-sim=yes --branch-sim=yes ./sensors_bench split
```

(valgrind 3.22.0, 시뮬레이션 파라미터: D1 32 KB/64 B 라인/8-way, LL 약 35.7 MB. cachegrind 아래서는 실행 자체가 수십 배 느려지므로 시간은 위의 네이티브 실측만 쓰고, 여기서는 **횟수**만 읽는다.)

| 항목 | `virtual` | `split` |
| --- | --- | --- |
| D1 미스 | 139,271,009 (11.2%) | 44,030,612 (4.4%) |
| LLd 미스 | 158,948 | 96,435 |
| 간접 분기 | 101,405,222 | 205,228 |
| 간접 분기 예측 실패 | 50,283,466 (**49.6%**) | 1,134 |

세 가지 원인이 겹쳐 있다.

**첫째, 포인터 추적이 캐시를 흩뜨린다.** `virtual` 경로는 원소 하나를 갱신하려면 포인터 배열에서 주소를 읽고, 그 주소가 가리키는 힙 어딘가로 점프해야 한다. 객체들이 셔플된 순서로 흩어져 있으니 다음 객체가 어느 라인에 있을지 규칙이 없고, 호출 1억 번에 D1 미스가 1억 3천 9백만 번 — **호출당 약 1.4회** — 났다. `split`은 24~32바이트짜리 객체가 연속으로 붙어 있어서 미스가 "순회가 지나가는 라인 수"만큼만 난다(계산: 라이다 100,000개 × 24 B ÷ 64 B + IMU 100,000개 × 32 B ÷ 64 B = 프레임당 87,500라인, 500프레임 = 약 4,375만 — 실측 4,403만과 일치한다).

**둘째, 가상 호출은 간접 분기이고, 타입이 뒤섞이면 예측이 불가능하다.** [3.4](#/virtual-vtable)에서 본 대로 `s->update()`는 vtable에서 함수 주소를 읽어 뛰는 간접 호출이다. CPU 분기 예측기는 "이 호출 지점은 지난번에 어디로 뛰었나"로 다음 목적지를 찍는데, 셔플된 벡터에서는 라이다/IMU가 무작위로 번갈아 나와 목적지가 동전 던지기가 된다 — 실측 예측 실패율 49.6%, 1억 호출에 5천만 번 실패다. `split`은 목적지가 컴파일 타임에 박혀 있으니 간접 분기 자체가 사실상 없다.

**셋째, 간접 호출은 인라인을 차단한다.** `split` 루프에서는 컴파일러가 `update()`의 타입을 정적으로 알아 호출을 없애고 본문을 루프 안에 펼친다 — [8.3](#/codegen)에서 본 최적화 연쇄(인라인 후 재배치)가 전부 열린다. 함수 주소를 실행 시점에야 아는 가상 호출에는 이 연쇄가 원천적으로 막힌다.

```text nolines
vector<Sensor*>  (virtual)              vector<LidarSensor>  (split)
[p0][p1][p2][p3][p4]...                 [L0][L1][L2][L3][L4][L5]...
 |    |    \_________________           24 B each, contiguous ->
 v    v                      v          next object = next bytes
heap: ..[Imu]...[Lidar]..........[Imu]
      scattered, no pattern
```

::: note cachegrind는 프리페처를 모의하지 않는다
실제 CPU에는 "연속 주소를 순서대로 읽고 있네, 다음 라인을 미리 가져오자"를 하드웨어가 알아서 하는 프리페처가 있다. cachegrind 시뮬레이션에는 이게 없어서, `split`의 4,403만 미스는 실제 하드웨어에서는 상당 부분 미리 채워져 훨씬 싸게 먹힌다. 반대로 `virtual`의 흩어진 접근은 프리페처가 도울 방법이 없다. 즉 시뮬레이터 수치는 연속 배치의 이득을 **과소평가**하는 쪽으로 치우쳐 있다 — 그런데도 6.6배다.
:::

::: interview "가상함수는 왜 느린가"라는 질문의 함정
면접에서 이 질문에 "vtable을 한 번 더 거치니까 느리다"라고만 답하면 하수다. 간접 참조 한 번 자체는 몇 사이클짜리 문제고, vtable은 자주 쓰이면 캐시에 상주한다. 답변 뼈대: ① 진짜 비용은 **인라인 차단**이다 — 호출 너머로 최적화가 못 넘어간다. ② 호출 지점의 실제 타입이 뒤섞이면 **간접 분기 예측 실패**가 쌓인다(이 절 실측: 예측 실패율 49.6%). ③ 다형 객체는 포인터로만 담을 수 있어 **개별 힙 할당과 산포**를 강제한다 — 대량 순회에서는 이 배치 비용이 가장 크다. 그리고 "그래서 소수 객체의 아키텍처 경계에서는 가상함수가 여전히 옳다"까지 말하면 상급이다.
:::

::: deep 탈가상화(devirtualization)
컴파일러도 이 비용을 알아서, 실제 타입을 증명할 수 있으면 가상 호출을 직접 호출로 바꾼다 — 이걸 탈가상화라 한다. `run_split()`의 `s.update()`가 그 예다: `s`가 `LidarSensor` 값 그 자체이므로 vtable을 거칠 이유가 없다. `final` 키워드로 "더 파생될 일 없음"을 알려주면 포인터 경유 호출도 탈가상화될 여지가 생긴다. 단, `vector<Sensor*>`처럼 실행 시점까지 타입이 열려 있는 컨테이너에서는 컴파일러가 증명할 방법이 없다 — 탈가상화는 공짜 탈출구가 아니라 "타입을 정적으로 못박은 만큼만" 돌려받는 최적화다.
:::

## hot/cold 분리 — 매 주기 만지는 바이트만 모아라

가상함수를 안 써도 같은 함정이 구조체 **안에서** 재현된다. 관절 상태 구조체에 제어 루프가 매 주기 쓰는 필드(hot)와 초기화·진단 때만 쓰는 필드(cold)를 같이 담으면, hot 순회가 cold 바이트까지 캐시로 끌고 들어온다.

```cpp title="hotcold_bench.cpp — 핵심부 (main과 시간 측정은 실측 ①과 동일 패턴)"
// ❌ hot과 cold가 한 구조체에: 128바이트 중 매 주기 쓰는 건 앞 16바이트뿐
struct JointMixed {
    double position;        // hot — 매 주기 갱신
    double velocity;        // hot — 매 주기 갱신
    char   name[32];        // cold — 진단 로그에서만
    char   frame_id[32];    // cold — TF 설정에서만
    double effort_limit;    // cold — 초기화 검증에서만
    double gains[5];        // cold — 튜닝 때만
};
static_assert(sizeof(JointMixed) == 128);

// ✅ hot만 모은 구조체: 16바이트, 캐시 라인 하나에 4개
struct JointHot  { double position; double velocity; };
struct JointCold { char name[32]; char frame_id[32];
                   double effort_limit; double gains[5]; };

// 두 경로 모두 같은 갱신을 1억 번 수행한다 (N=1'000'000, FRAMES=100):
//   j.position += j.velocity * 0.001;
//   j.velocity *= 0.9999;
```

캐시 라인 관점에서 두 배치는 이렇게 다르다.

```text nolines
JointMixed: 1 element = 128 B = 2 cache lines
line #1: [position|velocity|      name[32]...      ]   hot 16 B + cold 48 B
line #2: [ frame_id[32] | effort_limit | gains[5]  ]   cold 64 B
-> 64 B loaded per element, 16 B used (25%)

JointHot: 16 B, 4 elements per cache line
line: [pos|vel|pos|vel|pos|vel|pos|vel]
-> 64 B loaded, 64 B used (100%)
```

실측(각 5회 중앙값):

| 배치 | 실행 시간 | D1 미스 | LLd 미스 |
| --- | --- | --- | --- |
| `JointMixed` | 1005.2 ms | 104,015,824 (32.6%) | **104,009,320** |
| hot/cold 분리 | 71.3 ms | 27,515,996 (8.6%) | 2,259,287 |

**약 14배.** 실측 ①보다 격차가 더 큰 이유가 표의 셋째 열에 있다. `JointMixed`는 원소마다 D1 미스가 나는데(1억 방문에 1억 4십만 — 원소당 정확히 1회), 배열 전체가 128 MB라 시뮬레이션된 35.7 MB LL 캐시에도 안 들어간다 — **D1 미스 전원이 LLd 미스, 즉 RAM행이다.** 분리 버전은 hot 배열이 16 MB라 LL 캐시 안에 머물고(LLd 미스 2백만), D1 미스도 4분의 1이다(라인당 4원소니까 — 실측 2,752만 ≈ 1억 ÷ 4). 같은 산술 연산 1억 번의 운명이 배치 하나로 "L2/L3에서 해결" vs "매번 RAM 왕복"으로 갈렸다.

::: perf RAM행 미스 한 번의 어림값
두 배치의 시간 차 933.9 ms를 LLd 미스 차이 약 1억 2백만 회로 나누면 미스당 약 9 ns다. 네이티브 시간과 시뮬레이터 횟수를 섞은 거친 산수지만, [8.2](#/cache)에서 본 메인 메모리 접근 지연의 자릿수와 맞는다 — "미스 한 번 = 수십에서 백수십 사이클"이라는 감각을 실측이 다시 확인해 준다. 절대값은 기기마다 다르다.
:::

::: warn 목표는 구조체를 작게 만드는 게 아니다
`sizeof`를 줄이는 게 목적이라면 패딩 정리로 충분하다. 여기서의 목표는 **"가장 자주 도는 순회가 만지는 바이트"를 연속으로 모으는 것**이다. cold 데이터는 어디 있어도 된다 — 다른 배열이든, 인덱스로 대응되는 별도 벡터든. 판단 기준은 언제나 "매 주기 루프가 캐시로 끌어들이는 라인 수"이고, 그걸 세는 도구가 cachegrind와 [8.1](#/profiling)의 perf다.
:::

::: tip 분리 요령: 같은 인덱스, 다른 배열
hot/cold를 쪼갤 때 cold를 `unique_ptr`로 매달면 다시 포인터 추적이 생긴다. 가장 단순한 방법이 가장 좋다 — `vector<JointHot> hot`과 `vector<JointCold> cold`를 같은 인덱스로 대응시킨다. i번 관절의 이름이 필요한 (드문) 순간에만 `cold[i].name`을 읽는다. 이 "같은 인덱스, 필드별 배열" 패턴을 끝까지 밀면 [8.2](#/cache)의 SoA가 된다 — hot/cold 분리는 SoA의 절반짜리 실용 버전이다.
:::

## Existence-based processing — 분기를 데이터 배치로 없앤다

세 번째 함정은 캐시가 아니라 분기 예측기다. "전체 중 일부만 활성"인 데이터를 순회할 때의 관성적 코드는 이렇다:

```cpp title="existence_bench.cpp — 핵심부 (N=1'000'000, FRAMES=200, active 비율 50%)"
struct Waypoint { float progress; float speed; bool active; };

// ❌ random: 전체를 돌며 매번 검사 — active가 무작위로 흩어져 있다
for (auto& w : all)
    if (w.active)
        w.progress += w.speed;

// ✅ compact: active인 것만 모아둔 벡터를 돌린다 — 분기 자체가 없다
for (auto& w : compact)
    w.progress += w.speed;
```

CPU 파이프라인은 `if`의 결과가 나오기 전에 한쪽 방향을 **찍고** 다음 명령들을 미리 실행한다 — 이게 분기 예측(branch prediction)이다. 틀리면 미리 한 일을 전부 버리고 다시 시작한다. `active`가 무작위 50%면 예측기는 동전 던지기를 하는 셈이다. 통제 실험을 위해 세 번째 변형을 하나 더 쟀다 — **같은 배열을 active인 것이 앞쪽에 오도록 정렬만 한 버전**(`sorted`)이다. 순회하는 메모리 양과 연산은 `random`과 완전히 같고, 분기가 예측 가능해지는 것만 다르다.

| 변형 | 실행 시간(중앙값, 5회) | 조건 분기 예측 실패 | D1 미스 |
| --- | --- | --- | --- |
| `random` | 947.4 ms | 100,218,832 (**24.3%**) | 38,078,740 |
| `sorted` | 158.1 ms | 526,559 (0.1%) | 38,640,417 |
| `compact` | 59.8 ms | 527,013 (0.5%) | 19,895,094 |

`random`과 `sorted`는 D1 미스가 사실상 같다 — **6배 차이는 순수하게 분기 예측 실패 1억 회의 값이다.** 시간 차 789.3 ms를 실패 횟수로 나누면 실패 1회당 약 7.9 ns, 2.8 GHz 기준 약 22사이클 — 파이프라인을 비우고 다시 채우는 비용의 실측치다. `compact`는 거기서 더 나아가 분기를 아예 없애고 순회하는 데이터도 절반으로 줄여(D1 미스 절반) 다시 2.6배를 번다. "active인지 매번 묻지 말고, active인 것들의 배열을 유지하라" — 이 패턴을 existence-based processing이라 부른다. 존재 여부를 플래그가 아니라 **어느 배열에 들어 있는가**로 표현하는 것이다.

::: note 컴파일러가 분기를 없애줄 때도 있다
단순한 조건 대입은 컴파일러가 분기 없는 조건부 이동(cmov)이나 SIMD 블렌드로 바꾸기도 한다([8.3](#/codegen)에서 본 것처럼 어셈블리를 열어보면 안다). 이 벤치마크는 `-O2`에서 분기가 그대로 남았고 — 위 표의 예측 실패 1억 회가 그 증거다 — 그래서 격차가 그대로 드러났다. "컴파일러가 알아서 해주겠지"는 이 지점에서도 측정으로만 확인된다.
:::

::: danger compact 배열 유지의 대가: 인덱스가 흔들린다
active 배열에서 원소를 뺄 때 순서 유지 삭제는 $O(n)$이라, 마지막 원소와 자리를 바꾸고 `pop_back()`하는 swap-and-pop을 쓰게 된다. 이 순간 **마지막 원소의 인덱스가 바뀐다** — 어딘가에 저장해 둔 "웨이포인트 7번"이 조용히 다른 원소를 가리키게 되는 버그는 크래시 없이 오답만 낸다. compact 배열 밖에서 원소를 참조해야 한다면 세대(generation) 카운터를 붙인 핸들이나 ID→인덱스 맵이 필요하다 — 이 관리 비용까지가 이 패턴의 가격표다.
:::

## std::vector가 데이터 지향 설계의 기본 컨테이너인 이유

지금까지의 세 실측이 전부 `std::vector` 위에서 이뤄진 건 우연이 아니다. [5.1](#/vector)에서 본 두 성질 — **원소가 메모리에 연속으로 놓인다는 표준 보장**과 `reserve()`로 재할당 시점을 통제할 수 있다는 것 — 이 데이터 지향 설계가 요구하는 전부다. 노드 기반 컨테이너(`list`, `map`)는 원소마다 개별 힙 할당이라 실측 ①의 `virtual` 배치와 같은 포인터 추적을 구조적으로 강제한다. 그래서 이 설계 스타일의 기본값은 언제나 "일단 `vector`, 필드가 갈리면 필드별 `vector` 여러 개(SoA — [8.2](#/cache))"다. 단 제어 루프 안에서의 `push_back`은 재할당을 부를 수 있다는 것, 그게 실시간 데드라인의 적이라는 것은 [6.8](#/realtime)에서 이미 정리했다 — 시작 시 `reserve`가 세트다.

## ECS — 이 원리를 끝까지 밀면

게임 엔진은 이 절의 패턴들을 하나의 아키텍처로 묶었다. **ECS(Entity Component System)**: 엔티티는 그냥 정수 ID이고, 데이터는 컴포넌트별 연속 배열(위치 배열, 속도 배열, 체력 배열…)에 담기며, 시스템은 "특정 컴포넌트 조합을 가진 엔티티 전부"를 배열 순회로 처리하는 함수다. 객체가 데이터를 소유하는 게 아니라, **배열들이 데이터를 소유하고 객체는 인덱스로만 존재한다.** hot/cold 분리는 컴포넌트 분리로, existence-based processing은 "그 컴포넌트 배열에 들어 있는 엔티티만 처리"로 자연스럽게 일반화된다. 이 책은 ECS를 여기까지만 소개한다 — 구현이 필요해지는 규모라면 이미 검증된 라이브러리를 검토할 시점이고, 로봇 애플리케이션 코드가 그 규모에 도달하는 일은 드물다. 중요한 건 어휘가 아니라 원리다: ROS 2 생태계에도 같은 결정이 이미 박혀 있다. `ros2_control`이 관절 상태를 객체 그래프가 아니라 `double` 값 핸들의 평평한 목록으로 노출하는 것([10.9](#/ros2-control))이 정확히 "제어 루프가 만질 hot 데이터를 얇게 분리한" 설계다.

## 로보틱스 연결 — 수십만 포인트, 그리고 18개의 관절

**포인트클라우드가 이 절의 규모다.** LiDAR 한 스캔이 수만~수십만 포인트이고, 필터·다운샘플·법선 추정 전부가 "전체 순회 + 균일 변환"이다 — 대량·동종·매 주기라는 세 조건에 정확히 부합한다. PCL([11.6](#/pcl))의 `pcl::PointCloud<PointT>`가 포인트를 가상함수 없는 평범한 구조체의 `std::vector`로 담는 것, 포인트 타입들이 SIMD 정렬을 위해 16바이트 경계에 패딩되는 것이 전부 이 절의 논리다. 포인트마다 `new`를 하고 기반 클래스 포인터로 다루는 포인트클라우드 라이브러리는 존재하지 않는다 — 실측 ①이 그 이유다.

**헥사포드 18관절은 정직하게 말하면 이 절의 규모가 아니다.** 18개 × 1 kHz = 초당 18,000회 갱신은 hot/cold를 안 갈라도 캐시에 다 들어간다. 그런데도 관절 상태를 `positions[18]`, `velocities[18]`, `commands[18]` 같은 필드별 배열로 두는 게 옳은 이유는 두 가지다. 첫째, 배치는 공짜다 — 나중에 상태 로깅·필터링 항목이 늘어도 구조를 안 바꾼다. 둘째, 필드별 배열은 Part IX에서 Eigen 벡터 연산으로, [8.5](#/simd)에서 SIMD로 넘어가는 자연스러운 입력 형태다. 18개 관절의 IK를 매 주기 푸는 루프는 작아도, 그 안에서 도는 수학은 연속 배열을 전제로 설계된 라이브러리 위에 선다.

## 언제 OOP로 돌아가는가

이 절을 "가상함수 금지"로 읽었다면 잘못 읽은 것이다. 판단 기준은 두 축이다:

| | 개체 수 적음 (수 개~수십 개) | 개체 수 많음 (수천 개 이상) |
| --- | --- | --- |
| **개별 로직이 복잡·이질적** | Part III OOP가 정답 | 타입별로 갈라서 각각 연속 배열 |
| **로직이 균일, 매 주기 전체 순회** | 아무거나 — 측정할 가치도 없다 | **이 절의 데이터 지향 설계** |

Nav2 플러그인이나 `hardware_interface` 구현체가 가상 인터페이스인 것은 왼쪽 위 칸이라서다 — 개체가 몇 개 안 되고, 호출이 주기당 몇 번뿐이고, 교체 가능성이라는 아키텍처 가치가 나노초보다 크다. 가상 호출이 문제가 되는 것은 오직 **안쪽 루프에서 수만 번** 일어날 때다. 그리고 그 판정은 언제나 측정이 내린다.

::: warn 측정 없이 구조를 바꾸지 마라
데이터 지향 재설계는 코드 구조를 크게 바꾸는 일이고, 가독성과 캡슐화라는 실제 비용을 치른다. [8.1](#/profiling)의 순서 — 프로파일로 병목을 찾고, 그 병목이 정말 메모리 접근·분기인지 확인하고, 바꾼 뒤 다시 측정 — 를 건너뛰고 "vector of pointers는 나쁘다더라"로 코드베이스를 뒤집는 것은 이 절이 비판한 관성적 OOP만큼이나 관성적이다. 이 절의 6.6배·14배는 "대량 순회 + 흩어진 배치"라는 조건에서 나온 수치이지, 모든 가상함수에 붙는 세금이 아니다.
:::

## 요약

- 데이터 지향 설계는 "객체가 뭘 하는가"보다 **"데이터 전체가 캐시를 어떤 모양으로 통과하는가"**를 먼저 설계한다 — 대량·동종·매 주기 전체 순회 워크로드에서만 의미가 있다.
- 기반 클래스 포인터 벡터(개별 힙 할당 + 가상 호출)와 타입별 연속 벡터의 같은 작업이 이 환경에서 약 6.6배 차이 났다 — 원인은 포인터 추적의 D1 미스(호출당 약 1.4회), 간접 분기 예측 실패(49.6%), 인라인 차단 세 겹이다.
- hot 필드 16바이트에 cold 112바이트를 섞은 구조체는 hot만 분리한 배치보다 약 14배 느렸다 — cold가 캐시 라인을 점유해 D1 미스가 4배가 되고, 워킹셋이 LL 캐시를 넘어 미스 전원이 RAM행이 된 결과다.
- "active인지 매번 if로 검사"는 active가 무작위일 때 분기 예측 실패(실측: 실패 1회당 약 22사이클)로 약 6배를 잃었다 — active인 것만 모은 배열을 순회하는 existence-based processing이 분기 자체를 없앤다. 단 swap-and-pop이 인덱스를 흔든다는 대가가 있다.
- 연속 저장이 보장되는 `std::vector`(+ 시작 시 `reserve`)가 이 설계의 기본 컨테이너이고, 끝까지 밀면 필드별 배열(SoA)과 ECS에 닿는다 — PCL 포인트클라우드와 `ros2_control`의 평평한 상태 핸들이 같은 계열의 실물이다.
- 소수 개체 + 복잡한 개별 로직에는 Part III의 OOP가 여전히 정답이다 — 구조 변경은 반드시 [8.1](#/profiling)의 측정이 앞선다.

::: answer 해설
1. ① 흩어진 힙으로의 포인터 추적 — D1 미스 139,271,009회(호출당 약 1.4회) vs 연속 배치 44,030,612회. ② 간접 분기 예측 실패 — 라이다/IMU가 무작위로 번갈아 나와 실패율 49.6%(50,283,466회) vs 연속 배치 1,134회. ③ 인라인 차단 — 이건 횟수로 안 찍히지만, `split`에서 간접 분기 자체가 20만 회 수준으로 사라진 것이 가상 호출이 직접 호출·인라인으로 대체됐다는 증거다.

2. `JointMixed` 배열은 128 B × 1,000,000 = 128 MB로 시뮬레이션된 LL 캐시(약 35.7 MB)보다 크다. 한 바퀴 돌고 오면 아까 쓴 라인은 이미 쫓겨나 있어서 D1에서 놓친 접근이 LL에서도 전부 놓친다 — 그래서 두 수치가 같고, 미스마다 RAM까지 간다. 분리 버전의 hot 배열은 16 MB로 LL 캐시에 들어가므로, D1에서 놓쳐도 대부분 LL에서 잡힌다 — LLd 미스가 D1 미스의 10분의 1 이하로 떨어진 이유다.

3. 실패율은 크게 내려가고 시간도 크게 줄어드는 방향이다. 무작위 50%는 예측기 입장에서 최악(정보가 0)이고, 95%는 "거의 항상 taken"이라 단순히 taken으로 찍기만 해도 실패가 5% 수준으로 떨어진다. 분기 예측 실패 비용은 분기의 **개수**가 아니라 **예측 불가능성**에 붙는다는 게 이 문제의 핵심이다. 예측이 맞는지 궁금하면 `bernoulli_distribution coin(0.95)`로 바꿔 `--branch-sim=yes`로 직접 재라.

4. 이 환경 실측(중앙값 5회): mixed 392.9 ms, split 66.8 ms — 약 5.9배로, 원래의 약 14배에서 크게 줄었다. mixed의 미스가 RAM행에서 LL 캐시행으로 바뀌어 미스당 비용이 내려간 것이다. 그래도 5.9배가 남는 이유: D1(32 KB)에는 여전히 안 들어가므로 "라인당 유용한 바이트 16/64"라는 D1 수준의 낭비는 그대로다. 격차의 절대값은 기기마다 다르지만 "줄어들되 사라지지 않는다"는 방향은 같아야 한다.

5. 이 환경 실측: 셔플을 빼면 `virtual`이 1057.6 ms에서 229.6 ms(중앙값 5회)로 떨어진다 — 방금 연달아 할당된 객체들은 힙에서 거의 순차 배치라 포인터 추적의 캐시 미스(세 원인 중 첫째)가 대부분 사라지기 때문이다. 그래도 `split`(160.7 ms)보다 약 1.4배 느린 것은 남은 두 원인 — 가상 호출의 간접 분기(할당 순서가 라이다/IMU 교대라 예측은 쉬워지지만 호출 자체는 남는다)와 인라인 차단 — 때문이다. 실전 함정도 여기 있다: 방금 만든 벤치마크는 힙이 깨끗해서 빠르게 나오지만, 몇 시간 돌며 할당·해제가 뒤섞인 실제 프로세스의 힙은 셔플 버전에 가깝다.
:::

이 절의 세 벤치마크는 전부 직접 타이핑해서 네 기기의 수치를 얻어라 — 절대값은 다르게 나와도 "연속 배치가 이긴다"는 방향과 자릿수는 재현된다. 기준 명령: `g++ -std=c++20 -Wall -Wextra -O2 -g <파일>.cpp -o <출력> && ./<출력> <모드>`, 캐시·분기 해부는 `valgrind --tool=cachegrind --cache-sim=yes --branch-sim=yes ./<출력> <모드>` (cachegrind가 없으면 `sudo apt install valgrind`). cachegrind 아래서는 수십 배 느려지니 시간은 반드시 네이티브 실행으로 따로 재라.

**다음 절**: [8.5 SIMD 기초](#/simd) — 데이터를 연속 배열로 눕혀 놓으면 CPU가 한 명령으로 여러 원소를 동시에 처리할 길이 열린다. 이 절의 배치가 왜 자동 벡터화의 전제조건인지, intrinsics는 언제 꺼내는지 다룬다.
