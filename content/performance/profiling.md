# 8.1 프로파일링: perf와 측정 방법론

::: lead
Part VII까지의 도구는 "이 코드가 올바른가"를 물었다. Part VIII의 첫 질문은 "이 코드가 어디서 느린가"다 — 그리고 이 질문에 감으로 답하면 거의 항상 틀린다. 이 절은 의도적으로 병목을 심은 프로그램 하나를 프로파일러 두 개(callgrind, gprof)로 실제로 해부해서, 직관이 지목하는 병목과 측정이 지목하는 병목이 어떻게 다른지를 숫자로 보여준다. 그 과정에서 프로파일러의 두 부류 — 샘플링과 계측 — 가 각각 무엇을 재고 무엇을 놓치는지, 그리고 측정값으로 "이걸 고치면 전체가 몇 % 빨라지는가"를 계산하는 Amdahl의 법칙까지 다룬다. 로봇 제어 루프처럼 마감시한이 있는 코드에서는 이 순서 — 측정, 그다음 최적화 — 가 선택이 아니라 규율이다.
:::

## 추측으로 최적화하면 무엇이 낭비되는가

헥사포드의 다리 제어 루프가 주기를 넘기기 시작했다고 하자. 1kHz 루프라면 예산은 1ms다([6.8 실시간 제약과 제어 루프](#/realtime)). 코드를 열어 보면 "느려 보이는" 곳이 눈에 들어온다 — 루프 안에서 `std::string`을 `+`로 이어 붙이는 로그 조립 함수. Part II와 [2.4](#/dynamic-alloc)에서 힙 할당의 비용을 배웠으니, 이 패턴이 나쁘다는 건 안다. 그래서 반나절을 들여 로그 조립을 최적화한다.

문제는 이것이다: **그 함수가 전체 시간의 5%만 차지하고 있었다면, 그 반나절의 이론적 최대 성과는 5% 단축이다.** 진짜 병목 — 예를 들어 매 주기 불필요하게 200번 반복되는 IK 수렴 루프 — 은 그대로 남는다. "나쁜 패턴을 고쳤다"는 만족감과 "루프가 주기 안에 들어온다"는 목표 사이에는 아무 관계가 없다. 최적화는 공짜가 아니다 — 가독성을 잃고, 버그 위험을 감수하고, 시간을 쓴다. 그 비용을 어디에 지불할지 정하는 유일하게 합리적인 방법이 **프로파일링(profiling)** — 프로그램의 실행 시간이 실제로 어느 코드에서 소비되는지 측정하는 것 — 이다.

이 절의 실측이 정확히 이 시나리오를 재현한다. "누가 봐도 나쁜" 문자열 연결 함수와 계산 집약적인 IK 루프를 한 프로그램에 넣고, 측정이 직관을 어떻게 뒤집는지 본다.

## 프로파일러는 두 부류다 — 샘플링과 계측

모든 프로파일러는 같은 질문("시간이 어디로 갔나")에 답하지만, 답을 얻는 방식이 근본적으로 둘로 갈린다.

**샘플링(sampling) 프로파일러**는 프로그램을 그대로 실행시켜 놓고 주기적으로 끼어들어 "지금 어느 명령을 실행 중인가"(프로그램 카운터)를 기록한다. 통계의 논리다 — 1초에 수천 번 무작위로 들여다봤을 때 어떤 함수가 40%의 확률로 목격됐다면, 그 함수가 시간의 40%를 쓰고 있다고 추정한다. 리눅스의 표준 도구 `perf`가 이 방식이고, CPU의 **PMU(Performance Monitoring Unit)** — 사이클·명령어·캐시 미스를 하드웨어가 직접 세는 카운터 — 의 인터럽트를 트리거로 쓴다. 프로그램 자체는 건드리지 않으므로 재컴파일이 필요 없고, 오버헤드가 작아서(끼어드는 순간에만 비용이 든다) 실행 시간이 거의 왜곡되지 않는다. 대가는 통계적 불확실성이다 — 짧게 실행되는 프로그램에서는 표본이 부족하고, 호출 횟수 같은 정확한 카운트는 원리적으로 얻을 수 없다.

**계측(instrumentation) 프로파일러**는 프로그램에 측정 코드를 심는다. [7.5](#/sanitizers)의 새니타이저가 모든 메모리 접근 앞에 검사 코드를 끼워 넣었던 것과 같은 발상이다. valgrind의 `callgrind` 도구가 극단적인 예다 — 프로그램을 가상 CPU 위에서 재실행하면서 **실행된 명령어 하나하나를 센다.** 결과는 통계가 아니라 전수조사다: 함수별 명령어 수, 호출 횟수, 호출 관계가 전부 정확하다. 대가는 속도다 — 이 절의 실측에서 callgrind는 프로그램을 **약 109배** 느리게 만들었다. `gprof`는 두 방식의 혼합이다: 컴파일 시 `-pg` 플래그가 모든 함수 입구에 호출 기록 코드(`mcount`)를 심고(계측 — 호출 횟수는 정확하다), 시간은 100Hz 타이머로 프로그램 카운터를 샘플링해서 잰다.

어느 쪽이 좋은 도구인가는 틀린 질문이다. **무엇을 재야 하는가에 따라 도구가 정해진다** — 실행 시간을 왜곡하면 안 되는 상황(실기 로봇)에서는 샘플링만 가능하고, 정확한 호출 횟수·명령어 수가 필요한 오프라인 분석에서는 계측이 강하다. 이 절 끝까지 가면 두 부류의 사각지대를 각각 실측으로 확인하게 된다.

::: interview "샘플링 프로파일러와 계측 프로파일러의 차이를 설명해 보라"
성능 직군 면접의 단골이다. 답변 뼈대: ① 샘플링은 주기적 인터럽트로 프로그램 카운터를 수집하는 통계적 방식이라 오버헤드가 낮고 실행 왜곡이 작지만 호출 횟수 같은 정확한 카운트를 못 준다(예: perf). ② 계측은 코드에 측정 로직을 삽입해 전수조사하므로 명령어 수·호출 횟수가 정확하지만 수십~백 배 느려질 수 있다(예: callgrind — 이 절 실측 약 109배). ③ 그래서 프로덕션/실시간 환경 프로파일링은 샘플링, 정밀한 오프라인 분석은 계측으로 역할이 갈린다는 것까지 답하면 상급이다. "오버헤드가 큰 도구가 왜 여전히 쓰이는가"라는 후속 질문에는 "측정치가 결정론적이라 실행 간 비교·회귀 검출에 강하다"가 답이다.
:::

::: hist 1982년의 gprof, 2009년의 perf
gprof는 1982년 버클리에서 나온 도구로, "플랫 프로파일에 콜 그래프를 더한다"는 아이디어 자체가 이 도구의 논문 제목이었다 — 40년 넘게 살아남은 인터페이스다. valgrind 계열(callgrind 포함)은 2000년대 초의 동적 바이너리 계측 프레임워크이고, 실행 출력의 저작권 배너에서 보이듯 callgrind는 Josef Weidendorfer가 만들었다. perf는 2009년 리눅스 커널 2.6.31에 합류한 커널 서브시스템 + 사용자 도구로, 하드웨어 PMU를 커널이 직접 관리한다 — 커널과 한 몸이라는 사실이 곧 뒤에서 볼 설치 제약의 원인이기도 하다.
:::

## 이 절의 도구 선택 — perf가 막힌 환경에서

정직하게 밝힌다. 이 절의 실측 환경(컨테이너)에서는 perf를 쓸 수 없었다. perf의 사용자 도구는 커널 버전과 짝을 맞춘 `linux-tools` 패키지로 배포되는데, 컨테이너는 호스트의 커널을 빌려 쓰므로 배포판 저장소에 그 커널 버전용 패키지가 없으면 설치 자체가 막힌다 — 이 환경(커널 6.18)이 정확히 그 경우였다. 게다가 이 환경의 `/proc/sys/kernel/perf_event_paranoid`는 `2`(실측)로, 일반 권한 프로세스의 하드웨어 이벤트 접근이 제한돼 있다. perf가 하드웨어 카운터와 커널에 깊이 결합한 대가다.

그런데 이 제약이 이 절의 주제와 정확히 맞물린다 — **계측 프로파일러는 이런 제약이 없다.** callgrind는 프로그램을 가상 CPU에서 재실행할 뿐이라 커널 버전도, PMU 접근 권한도 필요 없다. 어디서나 돌고, 어디서 돌리든 같은 명령어 카운트를 준다. 그래서 이 절의 실측은 callgrind와 gprof로 하고, perf는 명령과 원리를 정리해 둔다 — 네 우분투 데스크톱과 로봇 보드에서는 perf가 정상적으로 깔리고, 그때 이 절의 방법론이 그대로 적용된다.

| 도구 | 방식 | 오버헤드 | 재빌드 | 환경 제약 |
| --- | --- | --- | --- | --- |
| perf | 샘플링 (PMU 인터럽트) | 수 % 수준 | 불필요 | 커널 짝 패키지, `perf_event_paranoid` |
| gprof | 혼합 (`-pg` 계측 + 100Hz 샘플링) | 작음 (이 절 실측에서 잡음에 묻힘) | `-pg` 필요 | 공유 라이브러리 시간을 못 본다 (실측으로 확인) |
| callgrind | 완전 계측 (가상 CPU 재실행) | **약 109배** (실측) | 불필요 (`-g` 권장) | 사실상 없음 |

::: note 컨테이너와 perf — 로보틱스 개발에서 실제로 마주치는 제약
Hexpider의 개발 환경은 ROS2 Humble Docker다. 컨테이너 안에서 perf를 쓰려면 호스트 커널용 perf 바이너리가 컨테이너 안에 있어야 하고, 실행 시 `--privileged` 또는 `perf_event_paranoid` 완화가 필요할 수 있다. "도커 안에서 perf가 안 된다"는 상황은 예외가 아니라 흔한 기본값이다 — 그럴 때 callgrind가 환경 무관한 대안이라는 것, 그리고 최종 수치는 어차피 실기(로봇 보드)에서 perf로 다시 재야 한다는 것(뒤의 로봇 절 참고)을 기억해 두면 된다.
:::

## 실험 대상 — 병목이 "뻔해 보이는" 프로그램

두 함수를 한 프로그램에 넣는다. `solve_leg_ik`는 다리 하나의 역기구학을 뉴턴 방식으로 수렴시키는 계산 루프를 흉내 낸 함수다(진짜 IK는 [9.5 역기구학](#/inverse-kinematics)에서 다룬다 — 여기서는 "삼각함수가 많은 수치 반복"이라는 비용 구조만 빌린다). `build_status_log`는 매 프레임 로그 문자열을 `+` 연결로 다시 조립하는, 교과서적인 나쁜 패턴이다.

```cpp title="ikbench.cpp — 어느 쪽이 병목인지 읽기 전에 예측해 보라"
#include <cmath>
#include <cstdio>
#include <string>
#include <vector>
#include <chrono>

// 가짜 6족 로봇 다리 하나의 역기구학(IK) 반복 계산 — 실제 계산 집약적 작업을 흉내낸다.
// 뉴턴-랩슨 스타일로 목표 각도에 여러 번 수렴시키는 루프라고 가정한다.
double solve_leg_ik(double target_x, double target_y, double target_z) {
    double theta1 = 0.0, theta2 = 0.0, theta3 = 0.0;
    for (int iter = 0; iter < 200; ++iter) {
        double fx = std::cos(theta1) * std::sin(theta2) + std::sin(theta3);
        double fy = std::sin(theta1) * std::cos(theta2) - std::cos(theta3);
        double fz = std::sin(theta2) * std::sin(theta3) + std::cos(theta1);
        double ex = target_x - fx;
        double ey = target_y - fy;
        double ez = target_z - fz;
        theta1 += 0.001 * ex;
        theta2 += 0.001 * ey;
        theta3 += 0.001 * ez;
        // 야코비안 역행렬 근사를 대신하는 추가 삼각함수 보정(의도적으로 계산량을 늘린다)
        theta1 = std::atan2(std::sin(theta1), std::cos(theta1));
        theta2 = std::atan2(std::sin(theta2), std::cos(theta2));
        theta3 = std::atan2(std::sin(theta3), std::cos(theta3));
    }
    return theta1 + theta2 + theta3;
}

// 로그 문자열을 매 프레임 다시 조립하는 흔한 실수 — 반복적인 문자열 연결.
std::string build_status_log(int frame, double angle_sum) {
    std::string log = "";
    log = log + "frame=" + std::to_string(frame);
    log = log + " leg=hex_leg_0";
    log = log + " angle_sum=" + std::to_string(angle_sum);
    log = log + " status=";
    for (int i = 0; i < 20; ++i) {
        log = log + "ok,";
    }
    return log;
}

int main() {
    constexpr int FRAMES = 20000;
    std::vector<double> results;
    results.reserve(FRAMES);
    std::vector<std::string> logs;
    logs.reserve(FRAMES);

    auto t0 = std::chrono::steady_clock::now();
    for (int f = 0; f < FRAMES; ++f) {
        double angle_sum = solve_leg_ik(0.3, 0.15, -0.2);
        results.push_back(angle_sum);
        logs.push_back(build_status_log(f, angle_sum));
    }
    auto t1 = std::chrono::steady_clock::now();
    double ms = std::chrono::duration<double, std::milli>(t1 - t0).count();

    double checksum = 0.0;
    for (double r : results) checksum += r;
    printf("frames=%d checksum=%.6f elapsed_ms=%.1f last_log_len=%zu\n",
           FRAMES, checksum, ms, logs.back().size());
    return 0;
}
```

```console
$ g++ -std=c++20 -Wall -Wextra -O2 -g ikbench.cpp -o ikbench
$ ./ikbench
frames=20000 checksum=941.630154 elapsed_ms=474.8 last_log_len=112
```

(g++ 13.3.0 / `-O2 -g` / Ubuntu 24.04 x86-64 실측. 10회 실행 중앙값 478ms — 절대값은 기기마다 다르지만 이 절의 비율 분석은 어디서나 같은 결론이 나온다.) `-O2`와 `-g`를 같이 켰다는 데 주의하라 — [8.3 컴파일러 최적화와 코드 생성](#/codegen)에서 보겠지만 성능 측정은 반드시 최적화 빌드로 하고, `-g`는 프로파일러가 결과를 소스 줄에 대응시키는 데 필요할 뿐 코드 생성에는 영향이 없다.

`build_status_log`는 프레임당 수십 번의 임시 `std::string` 생성과 힙 할당을 일으킨다. 직관은 이쪽을 가리킨다. 측정해 본다.

## callgrind 실측 — 직관과 다른 병목

```console
$ valgrind --tool=callgrind ./ikbench
==1660== Callgrind, a call-graph generating cache profiler
frames=20000 checksum=941.630154 elapsed_ms=51920.8 last_log_len=112
==1660== Events    : Ir
==1660== I   refs:      4,882,019,035
```

(valgrind 3.22.0 실측, 배너 일부 생략.) 프로그램 자신이 잰 실행 시간이 478ms에서 51,921ms가 됐다 — **약 109배.** 이게 완전 계측의 비용이다. 대신 결과는 전수조사다: 이 실행이 소비한 명령어는 정확히 4,882,019,035개(**Ir** — Instruction read, 실행된 명령어 수)이고, 이제 그 한 개 한 개가 어느 함수 소속인지 물을 수 있다. 결과 파일 `callgrind.out.<pid>`를 `callgrind_annotate`로 읽는다.

```text nolines
   26,339,872 ( 0.54%)  .../malloc.c:_int_free [libc.so.6]
   21,600,172 ( 0.44%)  .../malloc.c:malloc [libc.so.6]
   21,329,960 ( 0.44%)  .../basic_string.h:build_status_log(int, double)
```

(상위 10행, 지면상 경로 축약. 이 표의 수치는 함수 **자신이** 실행한 명령어만 센 self 비용이다.) 직관이 지목한 문자열 쪽 — `build_status_log`와 `malloc`/`free` — 은 전부 합쳐도 2%가 안 된다. 압도적 1~3위는 전부 libm의 삼각함수 구현(`__sincos_fma`, `__ieee754_atan2_fma`)으로, 셋이 합쳐 **73%**다. 여기에 그 함수들을 부르는 `solve_leg_ik` 자신(7.3%)과 atan2 래퍼까지 더하면 IK 계산 계열이 사실상 전부를 먹는다.

함수별 표는 self 비용이라 "누가 시켰는가"가 안 보인다. `callgrind_annotate <출력파일> ikbench.cpp`로 소스에 주석을 달면 **inclusive 비용**(그 줄에서 출발한 호출이 아래에서 쓴 것까지 전부)이 나온다 — `main`의 두 호출 줄이 핵심이다.

```text nolines
   100,000 ( 0.00%)          double angle_sum = solve_leg_ik(0.3, 0.15, -0.2);
4,702,520,000 (96.32%)  => ikbench.cpp:solve_leg_ik(...) (20,000x)
   100,000 ( 0.00%)          logs.push_back(build_status_log(f, angle_sum));
174,162,503 ( 3.57%)  => ikbench.cpp:build_status_log(...) (20,000x)
```

판결이 났다. **IK 루프가 96.3%, 문자열 로그가 3.6%다.** 호출 횟수도 정확히 20,000번씩 — 추정이 아니라 계수다. `build_status_log`를 반나절 들여 완벽하게 고쳐도 이 프로그램은 3.6% 이상 빨라질 수 없다. 반대로 IK 쪽은 수렴 반복을 200회에서 100회로 줄이거나, `sin`/`cos` 쌍을 한 번에 계산하거나([9.5](#/inverse-kinematics)에서 실제 IK로 다시 만난다), 근사 테이블을 쓰는 어떤 시도든 전체에 거의 그대로 반영된다. 최적화의 우선순위가 측정 한 번으로 뒤집혔다.

::: perf callgrind 오버헤드 실측 — 109배
같은 바이너리, 같은 입력: 비계측 10회 중앙값 478ms → callgrind 아래에서 51,921ms (약 109배, 이 환경 실측). 흔히 인용되는 20~50배보다 큰데, 이 프로그램이 libm 호출 밀도가 극단적으로 높기 때문이다 — 계측 오버헤드는 코드 성격에 따라 달라진다는 것 자체가 교훈이다. 이 배수 때문에 callgrind는 "오프라인에서 알고리즘의 명령어 예산을 해부하는 도구"이지, 실행 시간이 의미를 갖는 상황(실시간 루프, 타임아웃 있는 통신)에 붙이는 도구가 아니다.
:::

::: tip 같은 계열의 도구 — cachegrind와 KCachegrind
valgrind에는 형제 도구 `cachegrind`가 있다 — 명령어 수 대신 캐시 히트/미스를 세는 계측기로, [8.2 캐시와 메모리 레이아웃](#/cache)에서 주력으로 쓴다. 그리고 `callgrind.out.*` 파일은 KCachegrind(리눅스 GUI)로 열면 콜 그래프를 시각적으로 탐색할 수 있다 — 텍스트 출력이 답답해지는 큰 프로그램에서 유용하다.
:::

## gprof 교차 검증 — 같은 결론, 다른 사각지대

한 도구의 결과는 한 도구의 관점이다. 같은 프로그램을 gprof로 다시 잰다. `-pg`로 재컴파일하면 실행 시 `gmon.out`이 생기고, `gprof <바이너리>`가 그걸 해석한다.

```console
$ g++ -std=c++20 -Wall -Wextra -O2 -g -pg ikbench.cpp -o ikbench_pg
$ ./ikbench_pg
frames=20000 checksum=941.630154 elapsed_ms=469.4 last_log_len=112
$ gprof ikbench_pg gmon.out
```

```text nolines
Each sample counts as 0.01 seconds.
  %   cumulative   self              self     total
 time   seconds   seconds    calls  us/call  us/call  name
100.00      0.04     0.04   100000     0.40     0.40  solve_leg_ik(double, double, double)
  0.00      0.04     0.00    20000     0.00     1.60  build_status_log(int, double)
```

(GNU gprof 2.42 실측, 플랫 프로파일 상위 2행.) 실행에 469ms가 걸렸는데 gprof가 계상한 시간은 **0.04초뿐이다.** 100Hz 샘플링이니 이 실행 전체에서 표본이 47개쯤 생겼을 텐데 그중 4개만 잡혔다 — 나머지는 어디로 갔나. **gprof의 시간 샘플링은 `-pg`로 계측된 코드, 즉 실행 파일 자신의 코드만 계상한다.** callgrind가 73%를 차지한다고 알려준 libm(공유 라이브러리)의 시간이 gprof에게는 통째로 보이지 않는 것이다. 이것이 gprof의 첫 번째 사각지대이고, 표본을 늘리려 작업량을 10배(`FRAMES = 200000`)로 올려도 구조는 같다:

```text nolines
  %   cumulative   self              self     total
 time   seconds   seconds    calls  us/call  us/call  name
 56.00      0.28     0.28  1000000     0.28     0.28  solve_leg_ik(double, double, double)
 36.00      0.46     0.18                             _init
  8.00      0.50     0.04   200000     0.20     1.32  build_status_log(int, double)
```

(같은 환경 실측. 이 실행의 벽시계 시간은 4.59초, gprof가 계상한 시간은 0.50초 — 시간의 약 89%가 여전히 안 보인다.) 그래도 **보이는 범위 안에서의 순위는 callgrind와 일치한다**: `solve_leg_ik`가 `build_status_log`의 7배(0.28s 대 0.04s)다. 교차 검증의 목적은 이것이다 — 측정 방식이 전혀 다른 두 도구가 같은 함수를 병목으로 지목하면, 그 결론은 도구의 인공물이 아니다. 한편 2위의 `_init`(36%)는 진짜 초기화 코드가 아니라, libm으로 건너가는 PLT 점프대 주소들이 인근 심벌인 `_init`로 귀속된 것이다(`objdump -h` 실측: `.plt` 구간 0x1020~0x1240이 `_init`(0x1000) 바로 뒤다) — 안 보이는 libm 시간의 그림자가 이런 식으로 새어 나온다.

::: warn gprof의 호출 횟수가 틀렸다 — -O2 콜 그래프를 그대로 믿지 마라
위 플랫 프로파일에서 `solve_leg_ik`의 호출 횟수를 보라: 1,000,000회. 실제 호출은 200,000회다(callgrind가 계수로 확인해 준다 — FRAMES=20000 실행에서 정확히 20,000회였다). 콜 그래프를 열어 보면 존재하지 않는 호출 관계까지 나온다 — `build_status_log`가 `solve_leg_ik`를 800,000회 호출했다고 보고하는데, 소스에서 보듯 `build_status_log`는 `solve_leg_ik`를 부르지 않는다. 같은 바이너리를 `-O0 -pg`로 다시 빌드하면 호출 수가 정확히 200,000회로 나온다(실측) — 왜곡은 최적화가 켜졌을 때 생긴다. 원인은 아래 deep 상자에서 실측으로 추적한다. 실무 규칙은 이것이다: **gprof의 시간 비율은 참고하되, 최적화 빌드의 콜 그래프 호출 횟수는 callgrind(정확한 계수)로 확인하라.**
:::

::: deep 800,000회의 정체 — 컴파일러 클론과 심벌 테이블
`nm ikbench_pg`로 심벌을 보면 `-O2`가 만든 지역(local) 클론이 보인다 — `std::operator+(const string&, const char*) [clone .isra.0]`이 주소 0x1e40, 바로 앞 0x1bf0이 `solve_leg_ik`다(실측). 이 클론은 `build_status_log`가 프레임당 4번, 총 800,000회 호출하는 함수다(callgrind 계수와 정확히 일치). 그런데 gprof 출력 어디에도 `.isra.0` 행이 없다 — gprof가 이 컴파일러 생성 지역 심벌을 함수 목록에 넣지 못했고, 그 주소 구간이 바로 앞 함수 `solve_leg_ik`의 범위로 흡수됐다. 클론 입구의 `mcount`가 기록한 호출 800,000회가 고스란히 `solve_leg_ik`의 것으로 계수된 이유다. 계측이라고 무조건 정확한 게 아니라, **계측 결과를 심벌에 되붙이는 단계에서 최적화된 바이너리의 현실(클론, 인라인, 콜드 분할 — [8.3](#/codegen))과 충돌할 수 있다**는 사례다.
:::

## 명령어 수는 시간이 아니다 — 삭제 실험과 측정 규율

callgrind의 3.6%는 **명령어 수**의 비율이다. 시간의 비율도 같을까? 이번엔 도구 없이 가장 원초적인 방법으로 검증한다 — `build_status_log` 호출과 `logs` 벡터를 아예 들어낸 변형을 만들어 원본과 비교한다. 단, 비교 방법이 중요하다. 두 바이너리를 **한 세션에서 번갈아(A-B-A-B) 10회씩** 실행해 중앙값을 취했다.

| 빌드 | 10회 교차 실행 중앙값 | 원본 대비 |
| --- | --- | --- |
| 원본 (로그 포함) | 477.6 ms | 1.000 |
| 로그 제거 | 453.7 ms | 0.950 |

(같은 환경 실측.) 로그 조립의 실제 시간 비용은 약 5.0% — callgrind의 명령어 비율 3.6%와 크게 다르지 않지만 같지도 않다. 문자열 경로의 명령어(할당자 내부의 포인터 추적, 메모리 복사)는 IK 루프의 FMA 연산보다 명령어당 시간이 더 들기 때문이다 — 명령어마다 비용이 다르고, 그 차이를 만드는 최대 요인인 캐시가 [8.2 캐시와 메모리 레이아웃](#/cache)의 주제다. 규칙으로 정리하면: **Ir는 "코드가 시킨 일의 양"을 재는 정확하고 재현 가능한 대리 지표이고, 시간과의 환율은 코드 성격에 따라 달라진다.** 이 프로그램에서는 두 지표가 같은 순위, 비슷한 자릿수를 줬으므로 결론(IK가 압도적 병목)은 흔들리지 않는다.

::: warn 다른 세션의 측정값을 비교하지 마라
이 절을 작성하며 같은 원본 바이너리의 10회 중앙값이 측정 세션에 따라 594ms와 478ms로 나왔다(실측 — 24% 차이). 바이너리도 입력도 같다. 달라진 것은 머신의 상태(다른 부하, 클럭 상태)뿐이다. 서로 다른 시점에 잰 두 수치를 비교해 "빨라졌다/느려졌다"를 판정하면 이 24%가 통째로 결론을 뒤집을 수 있다. 그래서 A/B 비교는 반드시 **같은 세션에서 교차 실행 + 중앙값**으로 한다. 실행 간 변동을 통계적으로 다루는 정식 도구(Google Benchmark)와 나머지 함정들은 [8.6 마이크로벤치마크의 함정](#/benchmarking)에서 다룬다.
:::

## Amdahl의 법칙 — 이 최적화의 상한은 얼마인가

측정으로 비율을 얻었으면, 최적화를 시작하기 전에 그 최적화의 **이론적 상한**을 계산할 수 있다. 전체 실행에서 어떤 부분이 차지하는 비율이 $p$이고 그 부분을 $s$배 빠르게 만들면, 전체 가속비는:

$$S = \frac{1}{(1-p) + \dfrac{p}{s}}$$

**Amdahl의 법칙**이다. 원래 병렬화의 한계를 논한 공식이지만, "부분 최적화가 전체에 주는 효과"라는 구조는 동일하다. 극한 $s \to \infty$ — 그 부분을 **완전히 공짜로** 만들어도 — 에서 상한은 $\frac{1}{1-p}$다. 이 절의 실측값을 대입한다.

| 최적화 대상 | 측정된 비율 $p$ | $s=2$ | $s=4$ | $s=\infty$ (상한) |
| --- | --- | --- | --- | --- |
| `build_status_log` (시간 기준) | 0.050 | 1.026배 | 1.038배 | **1.053배** |
| `solve_leg_ik` (시간 기준) | 0.950 | 1.905배 | 3.478배 | **20배** |

문자열 로그를 우주 최고 수준으로 최적화해도 전체는 5.3% 이상 못 빨라진다. IK 루프는 2배만 개선해도 전체가 1.9배 빨라진다. "어디를 고칠 것인가"가 산수로 정리됐다.

::: perf Amdahl 예측 대 실측 — 1.053배 대 1.053배
위 표의 첫 행 상한은 검증까지 됐다. $p = 0.050$을 대입한 상한 $\frac{1}{1-0.05} = 1.053$배 — 그리고 앞 절의 삭제 실험(로그를 완전히 제거 = $s=\infty$를 물리적으로 실현)의 실측 가속비가 $\frac{477.6}{453.7} = 1.053$배다. 공식이 맞아떨어지는 게 당연해 보이지만, 이 당연함이 실무에서 힘이 된다 — **삭제 실험 없이도, 프로파일 비율만으로 "이 최적화에 시간을 쓸 가치가 있는가"를 착수 전에 판정할 수 있다.**
:::

## 로봇 제어 루프에서는 왜 더 무거운 문제인가

일반 프로그램에서 병목을 놔두면 "느리다". 실시간 제어 루프에서 병목을 놔두면 **오작동한다.** [6.8](#/realtime)에서 정리했듯 SCHED_FIFO로 도는 1kHz 제어 루프는 매 주기 1ms 안에 센서 읽기 → 계산 → 명령 쓰기를 끝내야 하고, 마감시한을 놓친 주기(deadline miss)는 서보에 명령이 늦게 도착한다는 뜻이다 — 여섯 다리가 각자 조금씩 늦은 명령으로 움직이면 그건 성능 문제가 아니라 자세 붕괴다. 그래서 로봇 코드에서 프로파일링은 "더 빠르게"가 아니라 **"예산 안에 들어가는가"**를 위한 도구이고, 우선순위 판정(이 절의 Amdahl 계산)이 곧 "어느 최적화가 주기를 구하는가"의 판정이 된다.

두 가지가 일반 프로그램과 다르다. 첫째, **실기에서는 계측 프로파일러를 붙일 수 없다.** 둘째, 판정 기준이 평균이 아니라 최악 지연이다 — 프로파일러는 "시간이 평균적으로 어디에 쓰이는가"를 주므로 병목 함수를 찾는 데 쓰고, 마감시한 충족의 증명은 제어 루프 안에 직접 심는 주기별 계측(max/백분위)으로 따로 한다([6.8](#/realtime), [8.6](#/benchmarking)). 그리고 개발 PC(x86-64)에서 잰 비율은 로봇 보드(ARM)에서 달라질 수 있다 — libm 구현도, 캐시 구조도 다르다. 병목 후보는 개발기에서 callgrind로 좁히고, 최종 수치는 실기에서 perf로 확인하는 이중 구조가 실전 워크플로다.

::: danger 계측 프로파일러를 실기 제어 루프에 붙이지 마라
이 절 실측의 109배를 1kHz 루프에 대입하면 주기당 계산이 1ms에서 109ms가 된다 — 제어 루프는 즉시, 완전히 무너진다. 로봇이 서 있는 채로 callgrind를 붙이는 것은 측정이 아니라 사고 유발이다. 실기 프로파일링은 오버헤드가 수 % 수준인 샘플링(perf)으로만 하고, 그마저도 안전 조치(로봇을 스탠드에 올리기, 비상 정지 준비) 위에서 한다. 계측 프로파일러의 자리는 개발 머신의 오프라인 분석이다.
:::

perf가 있는 환경(네 우분투 데스크톱, 로봇 보드)에서의 기본기는 세 명령이다. 이 절의 측정 환경에서는 실행하지 못했으므로 출력 예시 없이 명령만 남긴다 — 네 머신에서 직접 돌려 보라.

```bash
sudo apt install linux-tools-common linux-tools-$(uname -r)
perf stat ./ikbench            # 사이클·명령어·IPC 등 PMU 카운터 요약
perf record -g ./ikbench       # 콜 스택 포함 샘플링 (perf.data 생성)
perf report                    # 함수별 시간 비율을 대화형으로 탐색
```

`perf stat`이 보여주는 "instructions"가 callgrind의 Ir에 대응하는 값이고(단, 하드웨어가 직접 센 것), "cycles"와의 비율(IPC)이 이 절에서 본 "명령어 수 대 시간"의 간극을 하드웨어 수준에서 설명해 준다. `perf record`는 재빌드 없이, 이미 돌고 있는 프로세스에도(`-p <pid>`) 붙는다 — 샘플링 방식이라 가능한 일이다.

## 요약

- 프로파일링은 최적화의 우선순위를 정하는 측정이다 — 감으로 고른 "나쁜 패턴"이 실제 병목이라는 보장은 없고, 이 절의 실측에서 직관이 지목한 문자열 연결은 시간의 5%, 명령어의 3.6%에 불과했다(진짜 병목은 IK 루프의 삼각함수 — libm 셋이 명령어의 73%).
- 프로파일러는 샘플링(perf — PMU 인터럽트, 낮은 오버헤드, 통계적)과 계측(callgrind — 가상 CPU 전수조사, 정확한 계수, 실측 약 109배 감속)으로 갈리고, gprof는 `-pg` 계측(호출 수) + 100Hz 샘플링(시간)의 혼합이다.
- 도구마다 사각지대가 있다 — gprof는 공유 라이브러리 시간을 못 본다(실측: 벽시계 4.59초 중 0.50초만 계상). 최적화 빌드에서는 컴파일러 클론 심벌 때문에 gprof 호출 계수가 왜곡된다(실측: 200,000회가 1,000,000회로). 서로 다른 방식의 두 도구가 같은 병목을 지목할 때 결론이 단단해진다.
- callgrind의 Ir(명령어 수)는 정확하고 재현 가능하지만 시간이 아니다 — 명령어당 비용은 코드 성격(캐시 행동)에 따라 다르고, 그 간극은 [8.2](#/cache)의 주제다.
- A/B 성능 비교는 같은 세션에서 교차 실행 + 중앙값으로 한다 — 같은 바이너리의 중앙값이 세션에 따라 24% 흔들리는 것을 실측했다.
- Amdahl의 법칙 $S = 1/((1-p) + p/s)$로 최적화의 상한을 착수 전에 계산한다 — 실측 대입: 5% 부분의 상한은 1.053배(삭제 실험으로 검증 일치), 95% 부분은 2배 개선만으로 전체 1.9배.
- 실시간 제어 루프에서 병목은 deadline miss, 곧 오작동이다 — 실기에는 샘플링 프로파일러만 붙이고, 마감시한 증명은 주기별 계측([6.8](#/realtime))으로 따로 한다.

4. (계산) 프로파일 결과 어떤 함수가 전체 시간의 60%를 차지한다. 이 함수를 3배 빠르게 만들면 전체 가속비는 얼마인가? 이 함수를 아무리 최적화해도 넘을 수 없는 전체 가속비의 상한은? Amdahl의 법칙으로 계산하라.

5. (실습, 코드 작성형) 이 절의 `ikbench.cpp`를 직접 타이핑해 `-O2 -g`로 빌드하고, `valgrind --tool=callgrind ./ikbench` → `callgrind_annotate callgrind.out.<pid>`로 함수별 Ir 비율을 뽑아라. 그다음 `solve_leg_ik`의 수렴 루프를 200회에서 100회로 줄인 버전을 만들어 같은 측정을 반복하고, (a) Ir 총량이 얼마나 줄었는지 (b) 교차 실행 중앙값으로 잰 실제 시간 단축이 Amdahl 예측(p ≈ 0.95, s ≈ 2)과 자릿수에서 일치하는지 확인하라. 성공 기준은 두 측정의 비율이 실제로 네 화면에 찍히는 것이다.

::: answer 해설
1. 샘플링은 주기적 인터럽트(perf는 PMU 카운터 오버플로)로 프로그램 카운터를 수집하는 통계 방식이라 오버헤드가 수 % 수준이고 재빌드 없이 실행 중인 프로세스에도 붙는다. 계측은 코드에 측정 로직을 삽입(callgrind는 가상 CPU 재실행)해 명령어·호출을 전수조사하므로 정확하지만 이 절 실측 기준 약 109배 느려진다. 실기 로봇의 제어 프로세스에는 샘플링만 가능하다 — 계측의 감속은 제어 주기를 무너뜨려 측정 자체가 오작동을 만든다. 알고리즘 후보 비교는 계측이 강하다 — Ir는 머신 상태 잡음과 무관하게 재현되는 결정론적 지표라 두 실행의 비교가 깨끗하다.
2. Ir는 실행된 명령어의 개수이고 시간은 명령어 수 × 명령어당 평균 소요의 합이다. 문자열 경로는 할당자 내부의 포인터 추적과 메모리 복사가 많아 명령어당 시간이 IK 루프의 연속적인 FMA 연산보다 길다 — 그래서 명령어 비율(3.6%)보다 시간 비율(5.0%)이 크게 나온다. 두 지표는 서로 다른 통화일 뿐 같은 순위를 줬으므로 모순이 아니라 상보적이다. 명령어당 비용 차이를 만드는 최대 요인(캐시)은 8.2에서 측정한다.
3. 사라진 시간은 libm(공유 라이브러리)의 `__sincos_fma`/`__ieee754_atan2_fma` 안에 있었다 — callgrind 실측으로 명령어의 약 73%다. gprof의 시간 히스토그램은 `-pg`로 계측된 실행 파일 자신의 코드 영역만 계상하므로 libm에서 보낸 시간이 표본에 안 잡힌다(그 흔적이 PLT 스텁 주소가 `_init`로 귀속된 36%로 새어 나왔다). callgrind는 프로그램이 실행하는 모든 명령어를 — 어느 라이브러리 소속이든 — 가상 CPU 위에서 직접 세기 때문에 libm 내부까지 보인다.
4. $p = 0.6$, $s = 3$: $S = 1/((1-0.6) + 0.6/3) = 1/(0.4 + 0.2) = 1.67$배. 상한($s \to \infty$): $1/(1-0.6) = 2.5$배 — 이 함수를 완전히 없애도 전체는 2.5배 이상 못 빨라진다.
5. 수렴 루프를 반으로 줄이면 IK 계열 Ir(원본에서 약 96%)가 대략 절반이 되므로 Ir 총량은 절반 가까이(약 48% 감소) 줄어야 한다. 시간은 Amdahl 예측 $S = 1/(0.05 + 0.95/2) \approx 1.9$배 근처가 나와야 정상이다 — 정확히 1.9가 아니어도 자릿수(수십 % 단축)가 맞으면 성공이다. 비교는 반드시 같은 세션에서 두 바이너리를 교차 실행한 중앙값으로 하라 — 이 절에서 실측했듯 세션이 다르면 24%짜리 유령 차이가 낀다.
:::

이 절의 `ikbench.cpp`는 직접 타이핑해서 전 과정을 재현해 보라 — 병목이 "어느 쪽일 것 같은지" 먼저 적어 두고 측정 결과와 대조하는 것까지가 실습이다. 기준 명령: `g++ -std=c++20 -Wall -Wextra -O2 -g ikbench.cpp -o ikbench && valgrind --tool=callgrind ./ikbench && callgrind_annotate callgrind.out.<pid>` (valgrind가 없으면 `sudo apt install valgrind`), gprof는 `g++ ... -O2 -g -pg`로 빌드 후 실행하고 `gprof ikbench gmon.out`. perf가 되는 머신이라면 본문의 세 명령(`perf stat`/`record`/`report`)으로 같은 프로그램을 다시 재서, gprof가 못 보던 libm의 시간이 perf에는 잡히는 것까지 확인하라.
