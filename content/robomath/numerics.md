## 부동소수점 간격의 실감: ULP, 흡수, 결합법칙 붕괴

double은 63비트(부호 제외)로 실수 전체를 근사한다. 그래서 표현 가능한 값들은 실수직선 위에 촘촘하지만 **균일하지 않게** 흩어져 있다 — 간격이 값의 크기에 비례해서 벌어진다. `std::nextafter`로 "바로 다음 표현 가능한 값"까지의 거리를 재면 이 비례관계가 숫자로 보인다.

```cpp title="ulp.cpp — 1.0 근처와 1e15 근처의 간격"
#include <cmath>
#include <cstdio>
#include <initializer_list>
#include <limits>

int main() {
    for (double x : {1.0, 1000.0, 1.0e9, 1.75e9, 1.0e15}) {
        double gap = std::nextafter(x, 2 * x) - x;
        std::printf("x = %-10.6g  next(x) - x = %.6g\n", x, gap);
    }
    std::printf("epsilon<double> = %.6g\n", std::numeric_limits<double>::epsilon());
    return 0;
}
```

```console
$ g++ -std=c++20 -Wall -Wextra -O2 ulp.cpp && ./a.out
x = 1           next(x) - x = 2.22045e-16
x = 1000        next(x) - x = 1.13687e-13
x = 1e+09       next(x) - x = 1.19209e-07
x = 1.75e+09    next(x) - x = 2.38419e-07
x = 1e+15       next(x) - x = 0.125
epsilon<double> = 2.22045e-16
```

(g++ 13.3.0 / -O2 / x86-64 실측.) 이 최소 간격을 **ULP**(unit in the last place, 최종 자리 단위)라 부른다. `1e15` 근처에서는 ULP가 **0.125** — 이 스케일에서 double은 소수 이하는커녕 0.1 단위조차 정확히 표현하지 못한다. [9.4](#/forward-kinematics)의 발끝 좌표(수백 mm)나 [9.6](#/state-estimation)의 공분산(대략 1 근처)은 ULP가 $10^{-13}$ 이하라 실용적으로 무해하지만, 큰 누적 카운터(나노초 타임스탬프, 장시간 오도메트리)를 double로 들면 이 간격이 현실이 된다.

간격이 값에 비례한다는 사실은 두 가지 구체적 함정으로 이어진다. 첫째, **흡수(absorption)** — 큰 수에 그 수의 ULP보다 작은 수를 더하면 더한 것이 사라진다.

```console
1e16 + 1.0 == 1e16 ?  true
1e16 + 2.0 == 1e16 ?  false
```

`1e16`의 ULP는 2다. 1.0을 더해 봐야 반올림으로 증발하고, 2.0을 더해야 겨우 한 계단 움직인다. [1.2](#/types)에서 본 float 1kHz 시계가 한 시간에 70초를 잃은 것도 정확히 이 흡수다 — 경과 시간이 커질수록 0.001초짜리 증분이 표현 간격보다 작아진다. 둘째, 흡수는 **결합법칙을 깨뜨린다.** 수학에서는 $(a+b)+c = a+(b+c)$이지만 부동소수점은 아니다.

```console
a = 1e16, b = -1e16, c = 1.0
(a + b) + c = 1
a + (b + c) = 0
```

$(a+b)$를 먼저 계산하면 $10^{16}$끼리 상쇄돼 정확히 0이 남고, 거기에 1을 더하면 1이 남는다. 반대로 $b+c$를 먼저 하면 $-10^{16} + 1$이 $-10^{16}$의 ULP(2)보다 작아 흡수되어 그대로 $-10^{16}$이고, 거기에 $10^{16}$을 더하면 0이다. **같은 세 수, 다른 순서, 다른 답.** [8.3](#/codegen)과 [8.5](#/simd)에서 "컴파일러가 부동소수점 리덕션의 순서를 함부로 못 바꾼다"고 한 이유, `-ffast-math`가 결과 비트를 바꾸는 이유가 전부 이 한 줄의 반례다 — 재배열이 곧 다른 답을 낼 수 있다는 것을 컴파일러는 알고, 당신도 이제 안다.

::: note 흡수는 우연이 아니라 설계다
IEEE 754는 "반올림 오차가 생긴다"가 아니라 "매 연산의 결과는 무한 정밀도 참값을 가장 가까운 표현 가능값으로 반올림한 것과 같아야 한다"를 못박은 표준이다. 흡수는 그 표준을 정확히 지킨 결과다 — 버그가 아니라 유한 정밀도의 필연이다. [9.1](#/eigen)에서 Eigen의 표현식 템플릿이 임시 없이 정확히 소스 코드 순서대로 연산을 융합했던 것도 이 원칙 위에 있다 — 순서가 바뀌면 값도 바뀔 수 있으므로, "제로 오버헤드"는 되지만 "순서 자유"는 아니다.
:::

## 재앙적 상쇄: 가까운 두 수를 빼면 유효자리가 사라진다

흡수가 "덧셈이 정보를 삼키는" 함정이라면, **재앙적 상쇄(catastrophic cancellation)**는 "뺄셈이 정보를 드러내는 대신 잡음을 드러내는" 함정이다. 크기가 비슷한 두 수를 빼면 앞자리들이 서로 지워지고, 원래 각 수에 실려 있던 반올림 오차만 유효숫자 자리로 튀어 올라온다. 교과서 사례 — 이차방정식 근의 공식 — 로 실측하자. $x^2 - 10^8 x + 1 = 0$의 작은 근은 참값 근처 $10^{-8}$이다.

```cpp title="cancel.cpp — 근의 공식: 나쁜 형태 vs 안정 형태"
double b = -1.0e8, c = 1.0;
double disc = std::sqrt(b * b - 4 * c);
double x_naive  = (-b - disc) / 2.0;        // ❌ b와 disc가 거의 같은 크기 -> 뺄셈에서 상쇄
double q        = -0.5 * (b - disc);        // 부호가 다른 쪽끼리 더해 상쇄를 피한다
double x_stable = c / q;                    // ✅ 비에트(Vieta) 공식: 두 근의 곱 = c/a
```

```console
small root naive  = 7.4505805969238281e-09
small root stable = 1.0e-08
small root true   = 1.0000803740695119e-08   (long double 기준)
rel err naive     = 0.255
rel err stable    = 8.04e-05
```

(g++ 13.3.0 / -O2 / x86-64 실측, 참값은 `long double`로 계산.) `b`와 `disc`는 둘 다 $10^8$ 근처인데 그 차는 $10^{-8}$ 근처 — 8자리 유효숫자가 뺄셈 한 번에 통째로 상쇄돼 사라지고, 남은 것은 `disc`의 `sqrt` 반올림 잡음뿐이다. 결과는 참값 대비 **25.5% 상대오차** — 소수점 자릿수 낭비가 아니라 답이 틀렸다고 불러야 할 수준이다. 부호가 반대인 쪽끼리 더해(비에트 공식) 상쇄를 원천 차단한 안정 형태는 상대오차 $8 \times 10^{-5}$로, 3,000배 이상 정확하다.

같은 병리가 [9.6](#/state-estimation)의 분산 계산에도 있다. 통계 교과서의 "간편식" $\mathrm{Var}(X) = E[X^2] - E[X]^2$는 두 큰 수의 차다 — 데이터가 큰 오프셋 위에 작은 산포로 실려 있으면(예: 위치 좌표 `1e9 + 노이즈`) 재앙적 상쇄가 그대로 터진다.

```console
variance of {1e9+0.0, 1e9+0.1, ..., 1e9+0.9}  (참 분산 = 0.0825)
  true (오프셋 제거 후 두 번 훑기) = 0.082499995231629542
  naive  E[X^2] - E[X]^2         = 0
  Welford (온라인, 한 번 훑기)     = 0.082500035762791413
```

(실측.) naive 공식은 분산을 **0**으로 보고한다 — $E[X^2]$와 $E[X]^2$가 둘 다 $10^{18}$ 근처라 그 차이(진짜 분산, $10^{-1}$대)가 완전히 반올림 잡음 아래 묻혔다. Welford 알고리즘(온라인 평균·분산을 한 번의 훑기로 갱신하는 관용구, [9.6](#/state-estimation)의 $P$ 갱신과 같은 "누적하며 조금씩 고친다" 정신이다)은 오프셋을 사전에 뺄 필요 없이 정확한 값에 붙는다. **오프셋이 있는 데이터(큰 좌표 위의 작은 진동)의 분산·표준편차를 구하는 코드에서 naive 공식을 보면 그 자리에서 의심하라.**

::: interview "0.1 + 0.2 != 0.3인 이유"
C++ 면접에서 부동소수점을 아는지 확인하는 첫 질문이다. 답변 뼈대: ① double은 이진 분수의 유한 합이고, 0.1과 0.2는 십진법의 1/3처럼 이진법으로 무한소수라 애초에 반올림된 근사값으로 저장된다([1.2](#/types) 실측: `0.1`은 `0.10000000000000001`). ② 두 근사값을 더하면 그 반올림 오차가 합쳐진 결과가 나오는데, `0.3`도 독립적으로 반올림된 또 다른 근사값이라 둘이 정확히 같을 이유가 없다. ③ 실제로 `0.1+0.2`와 `0.3`의 차는 ULP 거리로 딱 **1**이다 — 오차라기보다 반올림 방향이 살짝 다른 정상적인 결과다. ④ 그래서 결론은 "버그가 아니라 설계"이고, 올바른 비교는 `==`가 아니라 허용 오차 비교라는 것. 여기까지 말하고 아래 절의 절대오차/상대오차/ULP 구분까지 붙이면 상급 답변이다.
:::

## 합산 순서의 함정과 Kahan summation

흡수는 "쌓이는" 합산에서 특히 잔인하다. 균등분포 float 1억 개를 그냥 더하면 무슨 일이 일어나는지 실측했다.

```cpp title="kahan.cpp — naive vs 정렬 후 vs Kahan (핵심부)"
float sum_naive(const std::vector<float>& v) {
    float s = 0.0f;
    for (float x : v) s += x;
    return s;
}

float sum_kahan(const std::vector<float>& v) {
    float s = 0.0f, comp = 0.0f;              // comp: 직전 덧셈에서 잘려나간 하위 비트
    for (float x : v) {
        float y = x - comp;                   // 이번 항에 보상을 얹어 더할 준비
        float t = s + y;
        comp = (t - s) - y;                   // (t-s)가 실제로 반영된 양 -- 그 차이가 새 보상
        s = t;
    }
    return s;
}
```

```console
$ g++ -std=c++20 -Wall -Wextra -O2 kahan.cpp && ./a.out
N = 100000000, uniform [0,1) floats
double reference = 49997554.572665133
naive  float     = 16777216         rel err = -0.664   165.7 ms
sorted float     = 16777216         rel err = -0.664
kahan  float     = 49997556         rel err = +2.85e-08   526.4 ms  (3.2x naive)
```

(g++ 13.3.0 / -O2 / x86-64, 3회 실행 중앙값.) naive 합산은 **참값의 33.6%밖에 안 되는 값**에서 멈춘다 — `16777216`은 정확히 $2^{24}$, float 가수부가 표현할 수 있는 마지막 정수다. 누적값이 그 문턱을 넘는 순간 ULP가 2를 넘어서고, [0,1) 범위의 항은 전부 ULP의 절반 아래라 반올림으로 증발한다. 흥미로운 건 **오름차순 정렬도 구하지 못한다**는 것 — "작은 수부터 더해 큰 수에 흡수되지 않게 하라"는 상식적 처방이 여기서는 안 통한다. 항들의 크기가 전부 [0,1)로 고만고만해서, 누적값이 커지면 정렬 순서와 무관하게 같은 흡수 문턱에 부딪히기 때문이다. **정렬은 항들의 크기가 서로 크게 다를 때만 듣는 처방이고, 이 실험처럼 크기가 비슷한 항이 압도적으로 많은 리덕션에는 안 듣는다.**

Kahan summation은 다른 전략이다 — 매 덧셈에서 잘려나가는 하위 비트를 `comp`에 별도로 추적해 다음 항에 되돌려준다. 결과는 상대오차 $2.85 \times 10^{-8}$로 float epsilon 근처, 사실상 정확한 값이다.

::: perf Kahan의 가격표 (g++ 13.3.0 / -O2 / x86-64, 1억 원소, 3회 중앙값)
naive **165.7 ms**, Kahan **526.4 ms** — **약 3.2배**. 각 항마다 뺄셈 두 번·덧셈 두 번이 추가되니(코드 4줄이 원래 1줄을 대체) 자릿수로도 타당한 배수다. 절대값은 기기마다 다르지만 "값이 틀리는 것을 막는 데 3배 정도의 값을 치른다"는 구조는 재현된다. 1억 개를 매 제어 주기 더할 일은 로봇 코드에 없다 — 이 대가가 실제로 문제 되는 곳은 장시간 실행되는 로그 집계·오프라인 캘리브레이션의 대량 리덕션 정도다. 실시간 루프의 몇 개짜리 합(자코비안 원소, PID 적분)에는 naive로 충분하고, Kahan은 배보다 배꼽이 크다.
:::

::: hist 1965년, 오차의 행방을 쫓다
William Kahan은 IEEE 754 부동소수점 표준(1985년)을 설계한 핵심 인물이자, 이 보상 합산 기법을 1965년에 발표했다 — 표준이 생기기 20년 전이다. 아이디어의 핵심은 "매 연산에서 버려지는 오차도 계산 가능한 값"이라는 통찰이다: `t - s`가 하드웨어 부동소수점 뺄셈으로 정확히 계산되므로(같은 부동소수점 연산 안에서), 그 결과가 곧 "이번에 실제로 반영된 양"이 되고, `y`와의 차가 "잘려나간 양"으로 정확히 복원된다. 이 기법이 `-ffast-math`([8.3](#/codegen), [8.5](#/simd))와 상극인 이유도 여기서 나온다 — `comp = (t - s) - y`는 수학적으로는 0이어야 할 것 같은 식이라, 결합·재배열을 허용하는 컴파일러가 "최적화"랍시고 이 식을 통째로 0으로 접어 버리면 보상 항 전체가 증발한다. Kahan summation을 쓰는 파일은 `-ffast-math`에서 반드시 제외해야 한다.
:::

## 로봇 수학의 함정 회수 I — 각도 정규화와 acos 클램프

각도를 $[-\pi, \pi]$로 접는 코드는 로봇 어디에나 있다 — 헤딩 오차, IK 출력, IMU 적분 전부 이 wrap이 필요하다. 가장 흔히 시도되는 `fmod` 방식과, [9.5](#/inverse-kinematics)의 IK가 실제로 쓰는 `atan2` 방식의 경계 동작을 실측으로 대조했다.

```cpp title="angle.cpp — fmod 방식 vs atan2 방식"
double wrap_fmod(double a)  { return std::fmod(a + pi, 2 * pi) - pi; }
double wrap_atan2(double a) { return std::atan2(std::sin(a), std::cos(a)); }
```

```console
input               fmod-way                atan2-way
3.241592654         -3.0415926535897935     -3.0415926535897935
-3.241592654        -3.2415926535897932     3.0415926535897935      <- fmod 실패!
```

(g++ 13.3.0 / -O2 실측.) 양수 입력(π보다 살짝 큰 각)은 두 방식이 일치한다. 그런데 **음수** 입력에서 `fmod` 방식이 무너진다 — 결과 `-3.2415...`는 $[-\pi, \pi]$ 범위 **밖**이다. 원인은 C++의 `fmod`가 나머지의 부호를 피제수(왼쪽 인자)의 부호로 맞추는 규칙에 있다: `a + pi`가 음수이고 크기가 $2\pi$보다 작으면 `fmod`는 그 값을 **그대로** 돌려주고, 거기서 `pi`를 또 빼면 범위 밖으로 밀려난다. `atan2` 방식은 `sin`/`cos`를 한 번 거치므로 이런 부호 함정이 애초에 없다 — 입력이 얼마나 크든, 부호가 무엇이든 결과는 항상 $[-\pi, \pi]$ 안이다.

::: warn "정규화했으니 범위 안"이라고 넘겨짚지 마라
`fmod` 기반 wrap 함수는 테스트를 양수 각으로만 돌리면 통과하고, 음수 헤딩이 들어오는 순간(로봇이 반시계 방향으로 돈 뒤) 범위 밖 값을 흘린다 — 그 값이 다음 계산의 `sin`/`cos`에 들어가면 계산 자체는 여전히 옳지만(주기함수라 $2\pi$ 밖이어도 값은 맞다), **비교·보간·로그**에서 각도가 튄 것처럼 보이는 버그가 된다. 각도 wrap 함수는 반드시 음수 입력으로도 단위 테스트를 짜라 — `EXPECT_GE`/`EXPECT_LE`로 결과가 $[-\pi, \pi]$ 안에 있는지 직접 검사하는 것이 `fmod`류 함정의 표준 방어다.
:::

[9.5](#/inverse-kinematics)에서 실측한 `acos` 오차 증폭 표를 다시 가져온다 — $c \to \pm 1$ 근처에서 입력의 작은 오차가 출력에서 크게 불어나는 구조다.

```console
c2 = 0            오차 증폭      1배
c2 = 0.99         오차 증폭      7배
c2 = 0.999999     오차 증폭    707배
c2 = 0.999999999  오차 증폭  21828배
```

증폭의 정체는 미분이다 — $\frac{d}{dc}\arccos(c) = -1/\sqrt{1-c^2}$가 $c \to \pm 1$에서 발산하므로, **연산 자체의 민감도(조건수, condition number)**가 경계 근처에서 무한대로 치닫는다. IK가 계산한 $\cos\theta_2 = 1 + 6.66 \times 10^{-16}$(정규화된 벡터의 자기 내적조차 이 환경에서 100만 회 중 18.6%가 1을 넘겼다 — 실측) 같은 반올림 잔여물을 그대로 `acos`에 넣으면:

```console
worst dot      = 1.0000000000000007  (1 + 6.66e-16)
acos(worst)    = nan
acos(clamped)  = 0
sqrt(-1e-9)    = -nan
```

`acos`가 정의역을 $10^{-16}$만큼 넘은 값에도 가차없이 NaN을 준다 — [9.5](#/inverse-kinematics)에서 "클램프가 필수"였던 이유가 여기서 숫자로 확인된다. `sqrt(-1e-9)`도 같은 이야기다 — 이론적으로 0이어야 할 계산이 반올림으로 아주 작은 음수가 되면 `sqrt`는 즉시 NaN이다.

::: danger NaN은 조용히 퍼진다 — isfinite로 막아라
NaN의 성질 자체가 위험하다: `nan == nan`은 **false**고, `nan < 1.0`도 `nan > 1.0`도 둘 다 **false**다 — 비교 기반의 이상값 필터(`if (x > threshold) reject();`)는 NaN을 통과시킨다. 그리고 NaN이 낀 연산은 전부 NaN이다(`0.0 * nan == nan`). 지수이동평균 필터에 NaN이 **단 한 번** 섞이면 어떻게 되는지 실측했다: 정상 샘플 50개 뒤에 `acos(1 + 2.3e-16)` 하나(NaN)가 끼고, 다시 정상 샘플이 이어져도 —

```console
EMA after one NaN at k=50, 50 clean samples later: nan
std::isfinite(ema) = false
```

— 필터는 영원히 NaN이다. `ema = 0.9*ema + 0.1*z`의 재귀 구조상 한 번 오염되면 그 뒤의 정상 데이터로 절대 씻기지 않는다. [9.6](#/state-estimation)의 칼만 필터 상태·공분산도 같은 재귀 구조라 동일하게 취약하다. 방어는 하나 — **경계를 넘나드는 모든 계산(acos, sqrt, 나눗셈) 직후에 `std::isfinite()`로 검사**하고, 실패하면 그 자리에서 이전 값을 유지하거나 예외적 경로로 보낸다. NaN이 필터 안으로 들어간 다음에 잡으려 하면 이미 늦다.
:::

## 로봇 수학의 함정 회수 II — 회전 드리프트, 공분산 대칭성, 동차 변환 역

[9.2](#/rotations)의 float 드리프트 실측을 다시 세워 두자. 작은 회전을 100만 번 곱하면:

```console
float:  det(Rf) = 0.980109, ||Rf^T Rf - I|| = 0.0236869,  norm(qf) = 1.00313
double: det(R)  = 1 + 1.58e-11,                            norm(q) = 1 + 4.29e-11
```

float는 100만 곱에 행렬식이 1에서 **2% 이탈**하고 쿼터니언 norm도 1.003으로 흐른다. double은 같은 반복에 $10^{-11}$ 수준으로 훨씬 여유롭지만 **방향은 같다** — 반복 누적은 정밀도와 무관하게 SO(3) 밖으로 밀어낸다. [9.2](#/rotations)가 실측한 복구 비용 — 쿼터니언 `normalize()` **16 ns**, 행렬 SVD 직교화 **621 ns**(약 39배) — 이 절의 결론에 그대로 들어온다: 나노초 단위 비용은 1kHz 제어 루프 예산([6.8](#/realtime))에서 반올림 오차이므로, **오도메트리·자세 필터는 매 주기 재정규화해도 아깝지 않다.** "얼마나 자주"에 대한 답은 "튈 때까지 기다리지 말고 매번"이다.

[9.6](#/state-estimation)의 공분산 대칭성 붕괴도 같은 반올림 뿌리를 갖는다. 갱신식 $(I-KH)P$는 수학적으로만 대칭을 보존하는데, float 2×2 필터를 100만 스텝 돌리며 잰 $|P - P^T|$의 최대 원소는:

```console
naive (I-KH)P        : 9.313e-10   (rel to P(0,0)=6.972e-05)
naive + symmetrize   : 0.000e+00
Joseph form          : 2.794e-09
```

강제 대칭화(`P = 0.5*(P+P.transpose()).eval()`) 한 줄로 비대칭이 정확히 0이 된다. 더 근본적인 조셉 형식(Joseph form)이 왜 필요한지는 극단적인 사례가 보여준다 — 사전 분산 $P=100$에 초정밀 센서 $R=10^{-6}$이 들어온 float 갱신:

```console
exact (double)  P+ = 1.000000e-06
naive  (float)  P+ = 0.000000e+00   (K=1.000000000, 1-K=0.000e+00)
Joseph (float)  P+ = 1.000000e-06
```

naive 공식은 $1-K$에서 **재앙적 상쇄**가 일어나 $K$가 float 반올림으로 정확히 1.0이 되고, $P^+ = 0$ — 필터가 "위치를 무한히 확신한다"고 선언한 뒤 모든 후속 측정에 귀를 닫는다. 조셉 형식은 뺄셈이 아니라 두 양수 항의 **덧셈**이라(위 절의 비에트 공식과 같은 원리 — 상쇄가 일어날 뺄셈 구조를 아예 피한다) 같은 float로 정확한 답을 낸다. **재앙적 상쇄가 실제 로봇 필터를 죽이는 구체적 사례가 이것이다.**

[9.3](#/transforms)의 SE(3) 역변환도 이 절의 어휘로 다시 읽힌다. 닫힌 형태 $[R^T, -R^T t]$는 `Isometry3d::inverse()` **18 ns**, 일반 4×4 역행렬은 **25~30 ns**(1.4~1.7배) — double 정확도는 둘 다 $2\times 10^{-15}$대로 동급이지만, **float**로 내려가면 닫힌 형태 최악 오차 **29 µm**, 일반 역행렬 **115 µm**(4배)로 벌어진다. 이유는 이 절에서 배운 그대로다 — 일반 역행렬(가우스 소거·LU)은 여러 단계의 곱셈·뺄셈을 거치며 반올림이 누적되는 반면, 닫힌 형태는 회전 블록이 원본의 **전치 그 자체**라 새로운 반올림이 낄 자리가 구조적으로 없다. **구조를 아는 공식이 일반 알고리즘보다 정확한 것은 우연이 아니라, 반올림이 낄 연산 횟수 자체가 다르기 때문이다.**

::: warn 조셉 형식은 기본값이 아니라 처방이다
[9.6](#/state-estimation)이 이미 정리했듯 조셉 형식은 naive+대칭화보다 몇 배 비싸다. 기본은 "naive + 매 스텝 강제 대칭화"로 충분하고, 필터가 이유 없이 발산하거나(공분산이 음의 고윳값을 갖게 되거나) 센서 신뢰도 차이가 극단적인 구성(위 $R=10^{-6}$급)에서만 조셉 형식으로 바꾼다. 모든 곳에 최고 비용의 방어를 까는 것은 [8.1](#/profiling)의 "측정 없이 최적화하지 않는다"의 반대쪽 오류 — "측정 없이 방어하지 않는다"도 같은 원칙이다.
:::

## 비교 연산의 규율: ==, 절대오차, 상대오차, ULP

이 절 전체가 가리키는 결론은 하나다 — **부동소수점에 `==`를 쓰지 마라**([1.2](#/types)에서 이미 예고했다). 그런데 "허용 오차를 두고 비교하라"는 조언은 그 오차를 **어느 척도로 잴 것인가**를 답하지 않으면 미완성이다. 세 가지 척도가 서로 다른 곳에서 무너진다.

```cpp title="compare.cpp — ULP 거리와 절대/상대 오차가 갈리는 지점"
int64_t ulp_distance(double a, double b) {
    int64_t ia, ib;
    std::memcpy(&ia, &a, sizeof ia);
    std::memcpy(&ib, &b, sizeof ib);
    if (ia < 0) ia = INT64_MIN - ia;         // 부호 있는 크기(sign-magnitude)로 변환
    if (ib < 0) ib = INT64_MIN - ib;
    return llabs(ia - ib);
}
```

```console
0.1+0.2 vs 0.3       : ULP distance = 1      (a-b = 5.55e-17)
near 1.0 pair        : abs diff = -1e-10,   ULP distance = 450360
near 0 pair          : abs diff = -2e-12 (< 1e-9!),  rel diff = -0.667
near 1e15 pair       : abs diff = 0.125 (already 1 ULP!), ULP distance = 1
```

(실측.) 셋째 줄이 **절대오차의 함정**이다 — $10^{-12}$대의 두 수는 절대차가 $2\times10^{-12}$로 아주 작아 보이지만 **상대차는 66.7%**, 완전히 다른 값이다. 절대 tolerance $10^{-9}$로 비교하면 이 둘을 "같다"고 잘못 판정한다. 넷째 줄이 **절대오차의 반대쪽 함정**이다 — $10^{15}$ 근처에서는 표현 가능한 **가장 가까운 이웃끼리도** 절대차가 0.125나 되므로, 절대 tolerance $10^{-9}$는 항상 실패하는 비교가 된다(ULP 거리로는 정확히 1, 완벽하게 같은 값의 이웃). **원칙: 스케일이 일정하고 그 스케일에서 의미 있는 물리 단위 오차가 있으면 절대오차(mm, rad), 스케일이 넓게 변하면 상대오차나 ULP 거리.**

[7.6 GoogleTest](#/googletest)의 두 매크로가 이 두 척도에 각각 대응한다. `EXPECT_NEAR(a, b, tol)`은 **절대오차** 비교 — [9.4](#/forward-kinematics)가 다리 검증에 쓴 `EXPECT_NEAR(..., 1e-9)`처럼 물리 단위(mm)로 tolerance를 말할 수 있을 때 쓴다. `EXPECT_DOUBLE_EQ(a, b)`는 절대값이 아니라 **ULP 거리**(gtest 문서가 명시하는 기본 한계는 4 ULP)로 비교한다 — 스케일과 무관하게 "부동소수점 연산이 낼 수 있는 정상적인 반올림 차이인가"를 판정하는 것이라, 크기를 모르는 값이나 크기가 큰 값에 안전하다. 이 절의 `1e15` 실측이 왜 `EXPECT_DOUBLE_EQ`가 그 스케일에서도 안 깨지는지를 보여준다 — ULP 거리 1은 4 이하이므로 통과다.

::: tip 어느 매크로를 쓸지는 스케일이 정한다
좌표·각도처럼 스케일이 일정하고 "이 정도 오차는 물리적으로 무의미하다"는 기준이 있으면 `EXPECT_NEAR`([9.4](#/forward-kinematics)의 253.0mm 검증처럼). 순수 연산 결과(캐스팅 왕복, 대수적 항등식)라 스케일이 다양하거나 모르면 `EXPECT_DOUBLE_EQ`. 절대 tolerance를 아무 근거 없이 `1e-9`로 박아 넣는 테스트는 값이 `1e15` 근처로 커지면 영원히 실패하거나(위 실측처럼 ULP 1도 tolerance보다 크다), 값이 아주 작으면 있으나 마나 한 검사가 된다 — tolerance 숫자에는 항상 "이 스케일에서"라는 전제가 숨어 있다는 것을 기억하라.
:::

## float vs double: 언제 무엇을 쓰는가

이 절 전체가 반복해서 보여준 것 — float는 double보다 흡수·상쇄가 훨씬 빨리 시작된다(ULP가 double보다 약 $2^{29}$배 성기다, `epsilon`의 비 $10^{-7}$ 대 $10^{-16}$) — 이 [9.1](#/eigen)·[9.2](#/rotations)의 타입 선택 기준으로 이어진다. **좌표·각도·시간·공분산은 double.** [9.2](#/rotations)의 float 드리프트(2% 이탈)와 이 절의 float Kahan/상쇄 실험이 전부 같은 결론을 가리킨다. float가 정당한 자리는 [8.5](#/simd)·[9.1](#/eigen)이 이미 표시해 둔 곳 — 포인트클라우드·이미지·신경망 가중치처럼 **정밀도보다 대역폭과 SIMD 레인 수가 지배하는 대량 데이터**뿐이다. Eigen에서 `Vector3f`/`Matrix3f`를 쓰고 싶은 유혹이 들면 [9.2](#/rotations)의 실측 표를 다시 열어라 — "메모리를 반으로 줄인다"의 대가가 "100만 번째 곱에서 방향이 2% 어긋난다"라면 로봇 자세에는 성립하지 않는 거래다.

임베디드 보드는 이 계산을 뒤집을 수 있다 — FPU가 단정밀도(float)만 하드웨어로 빠르고 배정밀도(double)는 소프트웨어 에뮬레이션으로 떨어지는 저가형 MCU(Cortex-M4F급)가 실재한다. 그런 보드에서 double 연산 하나가 float 대비 수십 배 느려질 수 있다 — 이 경우 정밀도와 속도의 저울은 데이터시트를 실측해서 다시 달아야 한다. **원칙은 그대로다: 기본은 double이고, float로 내려가는 것은 측정된 이유(대역폭, 또는 실측된 하드웨어 FPU 제약)가 있을 때만, 그리고 이 절에서 배운 흡수·상쇄의 대가를 알고 하는 결정이다.**

::: deep -ffast-math는 이 절 전체를 무효로 만드는 스위치다
[8.3](#/codegen)·[8.5](#/simd)·[9.1](#/eigen)이 각각 다른 각도에서 경고했던 것을 이 절이 근거로 완성한다. `-ffast-math`는 결합법칙이 성립한다고 가정하고 덧셈 순서를 재배열하는데, 이 절 첫머리의 $(a+b)+c \ne a+(b+c)$ 실측이 바로 그 가정이 거짓인 이유다. Kahan summation의 보상 항(`comp = (t-s)-y`)은 `-ffast-math`가 대수적으로 0이라 착각해 소거해 버릴 수 있는 식이고, NaN 처리 완화(`-ffinite-math-only`)는 이 절의 `isfinite` 방어망 자체를 무력화할 수 있다. [9.6](#/state-estimation)의 칼만 필터, 이 절의 조셉 형식처럼 반올림 순서 자체가 정확성의 일부인 코드에는 전역으로 켜지 말고, 켜더라도 파일 단위로 결과를 이 절의 척도(절대/상대/ULP)로 검증한 뒤에만 켠다.
:::

## Part IX를 닫으며 — 유한 정밀도 위의 수학

[9.1 Eigen](#/eigen)이 도구를 쥐여 줬고, [9.2](#/rotations)·[9.3](#/transforms)이 회전과 좌표 변환을 표현하는 법을, [9.4](#/forward-kinematics)·[9.5](#/inverse-kinematics)가 관절과 발끝을 잇는 법을, [9.6](#/state-estimation)이 거짓말하는 센서에서 진실을 추정하는 법을, [9.7 PID](#/pid)가 그 추정을 원하는 상태로 밀어붙이는 법을 다뤘다. 이 절이 마지막에 하는 일은 새 수학을 더하는 것이 아니라 **바닥을 보여주는 것**이다 — 지금까지의 모든 행렬 곱, 모든 자코비안, 모든 필터 갱신은 실수 위에서가 아니라 63비트 유한 정밀도 위에서 돈다. 그 바닥이 새는 지점(흡수, 상쇄, 조건수 폭발, NaN 전파)을 알면 Part IX의 모든 절에서 본 그 이상한 숫자들이 더 이상 이상하지 않다 — 예측 가능하고, 방어 가능하고, 어떤 경우엔 테스트로 박제할 수 있는 정상적인 동작이다. Part X부터는 이 수학이 전부 ROS 2 노드 안으로 들어간다 — `rclcpp`의 콜백 하나가 이 절의 필터를, 이 절의 IK를, 이 절의 좌표 변환을 실행하게 된다. 유한 정밀도라는 바닥은 노드 경계를 넘어도 사라지지 않는다.

## 요약

- Part IX의 모든 "이상한 숫자"(6.12323e-17, float 드리프트, acos 증폭, 공분산 비대칭)는 하나의 원인 — double이 유한 정밀도로 실수를 근사한다는 사실 — 의 서로 다른 증상이다.
- ULP(표현 가능한 최소 간격)는 값의 크기에 비례해 벌어진다(`1.0` 근처 $2.2\times10^{-16}$, `1e15` 근처 0.125 실측) — 그래서 큰 값에 작은 값을 더하면 **흡수**가 일어나고(`1e16+1.0==1e16`), 결합법칙이 깨진다(`(a+b)+c != a+(b+c)` 실측) — `-ffast-math`가 결과를 바꾸는 근본 이유다.
- **재앙적 상쇄**는 가까운 두 수의 뺄셈이 유효자리를 날리는 것이다. 근의 공식 나쁜 형태는 상대오차 25.5%, 안정 형태(비에트 공식)는 $8\times10^{-5}$로 3,000배 차이(실측). naive 분산 공식은 오프셋 있는 데이터에서 참값 대신 0을 준다 — Welford가 처방이다.
- 1억 개 float 합산 실측: naive/정렬 후 둘 다 $2^{24}$에서 흡수돼 멈추고(상대오차 66.4%), Kahan summation만 참값에 붙는다(상대오차 $2.85\times10^{-8}$) — 대가는 약 3.2배(165.7ms → 526.4ms 실측).
- 로봇 함정 회수: `fmod` 기반 각도 wrap은 음수 입력에서 범위 밖 결과를 준다(`atan2` 방식은 안전) — [9.5](#/inverse-kinematics)의 acos 오차 증폭(최대 21,828배)과 NaN 전파(한 번 섞이면 EMA·필터가 영구히 오염, `isfinite` 방어 필수)로 이어진다.
- [9.2](#/rotations)의 float 드리프트(100만 곱에 det 2% 이탈, normalize 16ns vs SVD 621ns), [9.6](#/state-estimation)의 공분산 대칭성 붕괴(강제 대칭화로 0, 조셉 형식이 $P=100,R=10^{-6}$ 캔슬레이션의 처방)와 [9.3](#/transforms)의 SE(3) 역변환 정확도(닫힌 형태가 float에서 4배 정밀)가 전부 이 절의 흡수·상쇄 어휘로 다시 읽힌다.
- 부동소수점 비교에 `==` 금지. 절대오차는 스케일이 일정할 때만(`EXPECT_NEAR`, [9.4](#/forward-kinematics)의 mm tolerance), 상대오차·ULP 거리(`EXPECT_DOUBLE_EQ`, 기본 4 ULP)는 스케일이 다양하거나 클 때 — 절대 tolerance 하나로 모든 스케일을 덮으려는 테스트는 반드시 어느 스케일에서 깨진다.
- float는 대역폭·SIMD가 지배하는 대량 데이터에만, 좌표·각도·시간·공분산은 double — 임베디드 FPU 제약이 있을 때만 이 기본을 실측 근거로 뒤집는다.

::: quiz 연습문제
1~3번은 개념·판단 문제, 4번은 코드 작성형, 5번은 코드 리뷰형이다.

1. `1e16 + 1.0 == 1e16`은 true인데 `1e16 + 2.0 == 1e16`은 false다. `1e16`의 ULP가 2라는 사실로 두 결과를 모두 설명하라.

2. 재앙적 상쇄와 흡수는 둘 다 "정보가 사라지는" 현상이지만 발생 조건이 다르다. 각각 어떤 연산(덧셈/뺄셈)과 어떤 피연산자 관계(크기 차이)에서 일어나는지 한 문장씩으로 구분하라. 이 절의 어느 실측이 상쇄이고 어느 것이 흡수인지 하나씩 짝지어라.

3. `EXPECT_NEAR(x, y, 1e-9)`로 짠 테스트가 있다. 이 테스트가 값의 스케일이 커지면(가령 `x, y`가 $10^{12}$ 근처) 왜 항상 실패하게 되는지, 이 절의 ULP 실측(`1e15` 근처 간격 0.125)을 근거로 설명하라. 이런 상황에 더 적합한 매크로는 무엇인가?

4. (실습, 코드 작성형) 본문의 `sum_kahan`을 직접 타이핑해 완결 프로그램으로 만들고, 균등분포 float 1,000만 개(1억 대신 시간을 줄인 크기)에 대해 naive·Kahan 합을 double 기준값과 비교해 상대오차를 출력하라. 그다음 `-ffast-math`를 추가해 같은 바이너리를 다시 빌드하고 Kahan의 상대오차가 어떻게 변하는지 관찰하라. 컴파일: `g++ -std=c++20 -Wall -Wextra -O2 kahan.cpp && ./a.out` 그리고 `g++ -std=c++20 -Wall -Wextra -O2 -ffast-math kahan.cpp && ./a.out`. 성공 기준: 두 실행의 Kahan 상대오차가 눈에 띄게 달라지는 것(대개 `-ffast-math`에서 나빠진다)을 직접 확인했다.

5. (코드 리뷰) 동료가 IMU 자이로 바이어스 추정 필터에 다음 코드를 추가했다: `if (residual > 0.0 && residual < 1e-6) { /* 잔차가 거의 0, 수렴했다고 판단 */ }`. `residual`이 NaN이 될 수 있는 경로가 있다는 것을 알게 됐다. 이 조건문의 문제를 NaN의 비교 성질로 지적하고, 안전한 형태로 고쳐라.
:::

::: answer 해설
1. `1e16`의 ULP는 2다. `1e16 + 1.0`을 정확히 계산하면 `1e16`과 다음 표현 가능값 사이(간격 2) 중간보다 아래에 있어 가장 가까운 표현 가능값인 `1e16`으로 반올림된다 — 그래서 `== 1e16`이 true. `1e16 + 2.0`은 정확히 다음 표현 가능값(`1e16` + ULP)과 일치하므로 반올림 없이 그 값이 그대로 저장되고, 원래 `1e16`과 다르므로 `==`가 false다. 흡수의 경계가 정확히 "더하는 값이 ULP의 절반보다 작은가"로 갈린다는 것을 보여주는 예다.

2. 흡수는 **크기가 크게 다른 두 수를 더할 때** 작은 쪽이 큰 쪽의 ULP보다 작아 반올림으로 사라지는 것이다(`1e16+1.0` 실측). 재앙적 상쇄는 **크기가 비슷한 두 수를 뺄 때** 앞자리가 서로 지워지고 각 수에 실려 있던 반올림 잡음만 유효숫자로 남는 것이다(근의 공식 나쁜 형태, 상대오차 25.5% 실측). 짝짓기: `1e16+1.0==1e16`과 float Kahan 실험의 naive 합산 정체($2^{24}$ 문턱)는 흡수, 근의 공식과 naive 분산 공식($E[X^2]-E[X]^2$)은 상쇄, `P=100,R=1e-6` 캔슬레이션(naive $1-K=0$)도 상쇄다.

3. `1e15` 근처에서 표현 가능한 값들끼리의 최소 간격(ULP)이 이미 0.125로 실측됐다 — `1e12` 근처도 같은 자릿수 구조로 ULP가 $10^{-9}$대보다 크다(값이 커질수록 ULP는 값에 비례해 커진다). tolerance `1e-9`가 그 스케일에서의 최소 표현 간격보다 작으므로, 완전히 같은 값의 바로 다음 이웃끼리 비교해도 tolerance를 통과하지 못하는 상황이 생긴다 — 테스트는 항상 실패한다. 이런 큰 스케일이거나 스케일을 모르는 값에는 절대오차가 아니라 ULP 기반의 `EXPECT_DOUBLE_EQ`(또는 상대오차 비교)가 맞다.

4. `sum_kahan`은 본문 그대로 네 줄이면 된다(`y = x - comp; t = s + y; comp = (t-s) - y; s = t;`). 기본 빌드에서는 이 환경 기준 naive 상대오차가 크게(수십 % 대) 나오고 Kahan은 float epsilon 근처($10^{-7}$~$10^{-8}$대)로 훨씬 정확해야 한다. `-ffast-math`를 추가하면 컴파일러가 `comp = (t-s) - y`를 대수적 항등식으로 보고 소거하거나 재배열할 수 있어 Kahan의 오차가 naive에 가깝게 나빠지는 것이 관찰돼야 한다(정확한 정도는 컴파일러·버전에 따라 다르다 — 그래서 "직접 확인"이 이 문제의 요점이다). 이것이 이 절의 deep 상자가 경고한 "Kahan summation과 `-ffast-math`는 같은 파일에 있으면 안 된다"의 실증이다.

5. `residual`이 NaN이면 `residual > 0.0`도 `residual < 1e-6`도 **둘 다 false**이므로 조건문 전체가 false로 평가돼 "수렴 안 함" 취급된다 — 겉보기엔 안전해 보이지만, NaN이 발생했다는 사실 자체가 숨겨진 채로 필터가 그냥 계속 돈다(이 절의 EMA 실험처럼 NaN이 상태에 이미 섞였다면 이후 모든 잔차도 NaN이 되어 영원히 이 분기를 벗어난다). 안전한 형태: 조건 이전에 `if (!std::isfinite(residual)) { /* 로그, 이전 상태 유지 또는 필터 리셋 */ return; }`을 먼저 두어 NaN을 침묵 대신 명시적으로 처리하고, 그 뒤에야 수렴 판정 `residual < 1e-6`(NaN이 걸러졌으므로 `residual > 0.0`은 잔차가 절대값이 아니라면 별도 검토가 필요하다)을 수행한다.
:::

이 절의 코드는 전부 직접 쳐라. `ulp.cpp`와 `cancel.cpp`는 짧으니 그대로 타이핑해 결과를 눈으로 확인하고, `kahan.cpp`는 원소 수를 1,000만 정도로 줄여서 먼저 감을 잡은 뒤 1억으로 올려 시간을 재라. `angle.cpp`는 음수 입력 몇 개를 직접 추가해 `fmod` 방식이 범위를 벗어나는 지점을 네 눈으로 봐라. 기준 명령은 `g++ -std=c++20 -Wall -Wextra -O2 -I/usr/include/eigen3 main.cpp && ./a.out`(Eigen이 필요 없는 파일은 `-I` 없이도 된다). ASan/UBSan([2.11](#/ub-sanitizers), [7.5](#/sanitizers))으로 `acosnan.cpp`류 코드를 돌려도 NaN 자체는 UB가 아니라 검출되지 않는다는 것도 함께 확인해 두면 좋다 — NaN은 정의된 IEEE 754 값이지, 미정의 동작이 아니다.
