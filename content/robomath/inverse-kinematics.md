# 9.5 역기구학

::: lead
[9.4 순기구학](#/forward-kinematics)은 함수였다 — 관절 각도를 넣으면 발끝 위치가 하나 나온다. 이 절은 그 함수를 거꾸로 뒤집는다: "발끝을 여기 두려면 관절을 몇 도로 꺾어야 하는가." 방향만 바뀌었는데 문제의 성격이 완전히 달라진다 — 답이 없을 수도, 두 개일 수도, 무한히 많을 수도 있다. 헥사포드 보행의 매 스텝이 이 문제의 호출이고, [8.1](#/profiling)에서 프로파일링 실험용으로 흉내만 냈던 그 IK가 여기서 진짜가 된다. 2링크 해석해를 코사인 법칙에서 끝까지 유도해 C++로 구현하고, FK로 왕복 검산하고, 3자유도 헥사포드 다리로 확장한 뒤, 해석해가 없는 일반 체인용 수치법(자코비안·뉴턴·DLS)까지 실측으로 비교한다.
:::

## 방향을 뒤집으면 문제의 성격이 바뀐다

순기구학은 언제나 잘 정의된 함수다. $(\theta_1, \theta_2)$를 넣으면 발끝 $(x, y)$가 정확히 하나 나온다 — [9.4](#/forward-kinematics)에서 유도한 그 식이다.

$$x = l_1\cos\theta_1 + l_2\cos(\theta_1+\theta_2), \qquad y = l_1\sin\theta_1 + l_2\sin(\theta_1+\theta_2)$$

역기구학(inverse kinematics, IK)은 이 연립방정식을 $(x, y)$가 주어졌을 때 $(\theta_1, \theta_2)$에 대해 **푸는** 것이다. 함수 평가가 방정식 풀이로 바뀌는 순간, 세 가지 병리가 동시에 생긴다.

- **해가 없을 수 있다.** 다리 길이 합($l_1+l_2$)보다 먼 곳, 또는 길이 차($|l_1-l_2|$)보다 가까운 곳에는 발끝이 물리적으로 닿지 않는다. 닿는 영역 — 반지름 $|l_1-l_2|$와 $l_1+l_2$ 사이의 환형 — 을 **작업공간(workspace)**이라 부른다.
- **해가 여러 개일 수 있다.** 같은 발끝 위치를 무릎을 이쪽으로 꺾어도, 저쪽으로 꺾어도 만들 수 있다 — elbow-up/elbow-down 두 해다. 이 절에서 실측으로 확인한다.
- **해가 무한히 많을 수 있다.** 관절 수가 목표의 자유도보다 많으면(사람 팔은 7자유도로 6자유도 목표를 잡는다) 해가 연속체를 이룬다 — **여유 자유도(redundancy)**다. 헥사포드 다리 하나는 3관절로 3차원 점을 잡으므로 여기 해당하지 않지만, 다리 18관절로 몸통 6자유도 자세를 잡는 전신 문제는 해당한다.

이게 왜 로봇 코드의 일상 문제인가. 보행 제어의 데이터 흐름이 정확히 이 방향이기 때문이다. 걸음새(gait) 계획이 내놓는 것은 "스윙을 마친 발이 **이 좌표에** 착지해야 한다"는 목표 위치이지 관절 각도가 아니다. 모터에 보내야 하는 것은 관절 각도다. 그 사이를 잇는 변환이 IK이고, 다리 6개가 매 제어 주기마다 이 변환을 요구한다. [8.1](#/profiling)에서 프로파일 대상으로 썼던 가짜 `solve_leg_ik`가 흉내 낸 게 바로 이 호출이다 — 이 절이 끝나면 그 자리에 진짜가 들어간다.

## 코사인 법칙에서 θ2가 나온다

2링크부터 푼다. 목표점까지의 거리 제곱을 $d^2 = x^2 + y^2$라 하자. FK 식의 $x^2 + y^2$를 전개하면 ($\cos$ 덧셈정리로 $c_1 c_{12} + s_1 s_{12} = \cos\theta_2$):

$$d^2 = l_1^2 + l_2^2 + 2 l_1 l_2 \cos\theta_2$$

$\theta_1$이 **소거됐다.** 정리하면 이 절의 중심 공식이 나온다.

$$\cos\theta_2 = \frac{x^2 + y^2 - l_1^2 - l_2^2}{2\, l_1 l_2}$$

기하로 읽으면 코사인 법칙 그 자체다. 고관절–무릎–발끝이 변 $l_1$, $l_2$, $d$의 삼각형을 이루고, 세 변이 정해지면 무릎의 내각은 하나로 정해진다. 여기서 중요한 통찰 하나 — **무릎 각도는 목표까지의 거리에만 의존하고, 방향과는 무관하다.** 목표가 어느 방향에 있든 거리만 같으면 무릎은 같은 각도로 꺾인다. 방향을 담당하는 것은 고관절이다.

$$\theta_2 = \pm\arccos\left(\frac{d^2 - l_1^2 - l_2^2}{2 l_1 l_2}\right)$$

$\arccos$는 $[0, \pi]$의 값 하나를 주지만 $\cos(-\theta_2) = \cos(\theta_2)$이므로 **부호가 다른 두 해**가 존재한다 — 무릎이 한쪽으로 꺾인 해와 반대쪽으로 꺾인 해. 그리고 우변이 $[-1, 1]$을 벗어나면 $\arccos$의 정의역 밖 — **해가 없다.** 세 병리 중 둘("두 개" 와 "없음")이 이 한 줄에 다 들어 있다.

## θ1은 atan2 두 번이다

$\theta_2$를 정했으면 $\theta_1$은 각도 두 개의 차다.

$$\theta_1 = \operatorname{atan2}(y, x) - \operatorname{atan2}(l_2 \sin\theta_2,\; l_1 + l_2\cos\theta_2)$$

첫 항은 고관절에서 목표를 바라보는 방향각이다. 둘째 항은 보정각 — 발끝은 femur 방향이 아니라 무릎에서 한 번 더 꺾인 곳에 있으므로, femur 방향과 고관절→발끝 직선 사이의 각도만큼 빼 준다(femur 기준 좌표계에서 발끝이 $(l_1 + l_2\cos\theta_2,\; l_2\sin\theta_2)$에 있다는 것에 atan2를 적용한 것이다). $\theta_2$의 부호를 뒤집으면 보정각의 부호도 뒤집혀 $\theta_1$이 달라진다 — 두 해는 $\theta_2$만 다른 게 아니라 $(\theta_1, \theta_2)$ 쌍 전체가 다르다. 기하적으로 두 해는 **고관절→목표 직선을 거울로 삼아 무릎이 서로 반대편에 있는** 대칭 자세다.

아래 위젯이 이 수식 그대로다 — 캡션의 $\cos\theta_2$ 공식이 방금 유도한 식이고, 목표를 **드래그**하면 매 픽셀 이동마다 이 두 줄이 다시 계산된다. 직접 만져 보라.

:::

위젯에서 확인할 것 세 가지. ① 목표 십자를 끌면 각도가 즉시 풀린다 — 캡션에 $\theta_2$, $\theta_1$이 수식과 함께 갱신된다. ② **무릎 방향 전환** 버튼이 $\pm\arccos$의 부호 선택이다 — 발끝은 제자리인데 자세 전체가 거울처럼 뒤집힌다. ③ 목표를 점선 원(도달 한계 $r = l_1+l_2 = 208$mm) 밖으로 끌면 십자가 붉게 바뀌고 "도달 불가"가 뜬다 — $\cos\theta_2$가 1을 넘은 것이다. 이때 다리가 이전 자세를 유지하는 것도 눈여겨봐라. 곧 다룰 "도달 불가 처리 전략" 중 하나를 위젯이 이미 선택하고 있다.

## C++ 구현과 왕복 검산

수식을 코드로 옮긴다. 링크 길이는 위젯과 같은 femur 80mm / tibia 128mm다.

```cpp title="ik2link.cpp — 2링크 해석해와 FK 왕복 검산" {17,19-22}
#include <Eigen/Dense>
#include <cmath>
#include <iostream>
#include <numbers>
#include <optional>

constexpr double l1 = 80.0, l2 = 128.0;   // femur, tibia [mm]

struct Joints { double t1, t2; };

Eigen::Vector2d fk(double t1, double t2) {
    return { l1 * std::cos(t1) + l2 * std::cos(t1 + t2),
             l1 * std::sin(t1) + l2 * std::sin(t1 + t2) };
}

std::optional<Joints> ik(double x, double y, bool elbow_up) {
    const double c2 = (x * x + y * y - l1 * l1 - l2 * l2) / (2.0 * l1 * l2);
    if (c2 < -1.0 || c2 > 1.0) return std::nullopt;          // 작업공간 밖
    double t2 = std::acos(c2);                                // [0, pi]
    if (elbow_up) t2 = -t2;                                   // 두 번째 해
    const double t1 = std::atan2(y, x)
                    - std::atan2(l2 * std::sin(t2), l1 + l2 * std::cos(t2));
    return Joints{t1, t2};
}

int main() {
    constexpr double pi = std::numbers::pi;
    // 왕복 검산: theta -> FK -> IK -> theta 가 돌아오는가
    const double t1_in = 35 * pi / 180, t2_in = 55 * pi / 180;
    const Eigen::Vector2d p = fk(t1_in, t2_in);
    const auto s = ik(p.x(), p.y(), false);
    std::cout << "FK(35, 55) = (" << p.x() << ", " << p.y() << ")\n"
              << "IK -> (" << s->t1 * 180 / pi << ", " << s->t2 * 180 / pi << ")\n";
    return 0;
}
```

반환 타입이 `std::optional<Joints>`인 것이 설계다 — [5.9](#/vocabulary-types)에서 배운 "부재를 값으로 표현"이 IK에서는 선택이 아니라 필수다. 해가 없는 입력이 정상 운용 중에 실제로 들어오기 때문이다. 실행하면:

```console
$ g++ -std=c++20 -Wall -Wextra -O2 -I/usr/include/eigen3 ik2link.cpp && ./a.out
FK(35, 55) = (65.5322, 173.886)
IK -> (35, 55)
```

넣은 각도가 그대로 돌아왔다. 이 **왕복 검산 — IK(FK($\theta$)) = $\theta$** 이 IK 코드의 기본 테스트 패턴이다. 정답을 아는 문제를 스스로 만들 수 있기 때문이다: 아무 각도나 찍고, FK로 발끝을 구하고, 그 발끝을 IK에 넣어 원래 각도가 나오는지 본다. 이 환경 실측으로 각도 오차는 $\theta_1$이 0, $\theta_2$가 $2.2 \times 10^{-16}$ rad — double의 반올림 한 계단이다. 더 넓게, 작업공간 안 격자점 전체에 두 해 모두로 돌린 **2,568회 검산에서 최악의 발끝 오차가 $1.0 \times 10^{-13}$ mm**였다(실측) — 원자 반지름보다 작다. 해석해는 근사가 아니라 정확한 답이다.

두 해도 수치로 확인했다. 같은 발끝 $(65.5, 173.9)$에 대해:

```console
elbow-down: t1 = 35,      t2 =  55
elbow-up:   t1 = 103.700, t2 = -55
FK(down) = (65.5322, 173.886)
FK(up)   = (65.5322, 173.886)     <- 두 자세의 발끝 차이: 2.8e-14 mm
```

$\theta_2$의 부호만 뒤집힌 게 아니라 $\theta_1$이 35°에서 103.7°로 함께 바뀌었다 — 위에서 말한 "쌍 전체가 다른" 두 해다. 위젯의 무릎 방향 전환 버튼을 누를 때 일어나는 일이 정확히 이 두 줄 사이의 점프다.

::: interview "IK 해가 여러 개면 어떻게 고르나"
로봇 팔·다리를 다루는 포지션의 단골 질문이다. 모범 답변 뼈대: ① **연속성** — 직전 제어 주기의 해와 가까운 쪽을 고른다. 매 주기 무릎 방향이 널뛰면 모터가 감당할 수 없는 순간 속도가 나오므로, 궤적을 따라가는 동안 같은 가지(branch)에 머무는 게 기본이다. ② **관절 한계** — 수학적 해가 하드웨어 가동 범위 밖이면 버린다(아래 3자유도 실측에서 femur가 −152°로 꺾이는 "유효한" 해가 실제로 나온다). ③ **장애물·자기 충돌** — 남은 후보 중 몸통이나 옆 다리와 부딪히지 않는 자세. ④ 여유 자유도가 있으면 그 위에서 2차 목표(특이점 회피, 관절 중앙 유지)를 최적화한다. "일단 elbow-down 고정"이 답이 되는 경우도 많다고 덧붙이면 실무 감각까지 전달된다 — 헥사포드는 곤충처럼 무릎 방향을 한쪽으로 고정하고 시작한다.
:::

## 도달 불가: |cosθ2| > 1을 어떻게 처리하는가

`if (c2 < -1.0 || c2 > 1.0)` 한 줄이 작업공간 검사의 전부다 — 수학이 판정을 공짜로 준다. 실측으로 경계를 확인하면:

```console
target (208, 0):     cos(t2) = 1            -> ok       (딱 도달 한계)
target (208.001, 0): cos(t2) = 1.0000203    -> unreachable
target (209, 0):     cos(t2) = 1.0203613    -> unreachable
target (250, 0):     cos(t2) = 1.9392578    -> unreachable
```

경계 1mm 밖에서 $\cos\theta_2$가 1을 $2 \times 10^{-5}$만큼 넘는다. 판정 자체는 간단한데, **판정 이후에 무엇을 하는가**가 설계 결정이다. 실전 선택지는 세 개다.

| 전략 | 하는 일 | 대가 |
| --- | --- | --- |
| **에러 반환** (`nullopt`) | 호출자에게 실패를 알리고 처리를 위임 | 호출자가 매번 분기해야 한다. 걸음새 계획이 실수로 1mm 밖을 요구하면 다리가 아예 안 움직인다 |
| **클램프** (`std::clamp(c2, -1.0, 1.0)`) | 가장 가까운 도달 가능 자세(다리를 완전히 뻗은 자세)로 대체 | 요청과 다른 곳에 발이 놓이는 것을 **조용히** 허용한다. 오차가 로그 없이 삼켜진다 |
| **이전 해 유지** | 새 목표를 무시하고 직전 자세를 지킨다 | 발이 멈춘다. 목표가 계속 밖이면 계속 멈춰 있다 — 위 위젯의 선택 |

정답은 하나가 아니고 **계층마다 다르다.** 궤적 생성기처럼 목표를 만드는 쪽에 가까운 계층은 에러 반환이 맞다 — 도달 불가는 계획의 버그이고, 일찍 시끄럽게 터져야 고친다([0.3](#/first-build)의 에러 철학 그대로다). 반대로 모터에 직결된 최종 계층은 클램프가 흔하다 — 실시간 루프 한복판에서 "해 없음"으로 멈추는 것보다 최근접 자세로 버티는 게 하드웨어에 안전하기 때문이다. 절충안은 **클램프하되 보고하는 것**: 값은 클램프해서 돌려주고, 클램프가 일어났다는 사실을 반환값이나 카운터로 남긴다. 조용한 클램프가 가장 위험하다 — 발끝이 요청보다 안쪽에 닿는 오차가 지형 접촉 판정을 서서히 오염시키는데, 어디서도 비명이 없다.

::: warn 경계 판정에 epsilon 여유를 두지 마라 — 대신 무엇이 문제인지 알고 클램프하라
"부동소수점이니까 `c2 > 1.0 + 1e-9`처럼 여유를 주자"는 유혹이 있다. 방향이 틀렸다. $c2 = 1 + 10^{-10}$은 "아슬아슬하게 도달 가능"이 아니라 **딱 경계에 있는 목표가 반올림 오차로 밀려난 것**이므로, 올바른 대응은 판정 완화가 아니라 $[-1, 1]$로의 클램프다 — 그러면 $\theta_2 = 0$(완전히 뻗음)이라는 옳은 답이 나온다. 진짜로 1을 유의미하게 넘는 입력(위 실측처럼 경계 1mm 밖부터 $10^{-5}$ 단위로 커진다)과 반올림 잔여물($10^{-16}$ 근처)은 크기가 10차수 이상 달라 혼동될 일이 없다.
:::

여기에 수치 안정성 문제가 하나 겹쳐 있다. $\theta_2$를 $\arccos$로 뽑는 것 자체가 경계 근처에서 나쁜 선택이다 — $\arccos$의 미분은 $-1/\sqrt{1-c^2}$라 $c \to \pm 1$에서 발산한다. 입력 오차가 출력에서 얼마나 증폭되는지 실측했다:

```console
c2 = 0            오차 증폭      1배
c2 = 0.99         오차 증폭      7배
c2 = 0.999999     오차 증폭    707배
c2 = 0.999999999  오차 증폭  21828배
```

다리를 거의 뻗은 자세에서는 $c_2$의 $10^{-10}$짜리 오차가 각도에서 $10^{-6}$ rad로 불어난다. 대안은 사인을 따로 구해 `atan2(±sqrt(1 - c2*c2), c2)`로 뽑는 것 — atan2는 두 인자의 **비율**로 각도를 정하므로 한쪽이 0에 붙어도 조건이 무너지지 않는다. $\theta_1$ 공식에 처음부터 acos가 아니라 atan2를 쓴 것도 같은 이유다(사분면 문제까지 겸해서). 부동소수점이 언제 어떻게 이런 짓을 하는가는 [9.8 수치 안정성](#/numerics)에서 정면으로 다룬다 — 이 증폭 표는 그 절의 예고편이다.

## 3자유도 헥사포드 다리: yaw를 분리하면 2링크로 떨어진다

이제 진짜 다리로 간다. [9.4](#/forward-kinematics)에서 FK를 세운 그 사슬 — coxa(yaw 회전) → femur(pitch, 80mm) → tibia(pitch, 128mm) — 이고, coxa 링크 길이는 45mm로 둔다. 3관절·3차원 목표라 방정식 3개에 미지수 3개다. 일반적으로 이런 비선형 연립은 손으로 안 풀리는데, 이 사슬은 **구조 덕분에** 풀린다.

핵심 관찰: coxa만 yaw(z축)이고 femur·tibia는 둘 다 pitch(y축)다. 즉 **femur와 tibia는 언제나 coxa가 정한 수직 평면 안에서만 움직인다.** 발끝이 목표 $(x, y, z)$에 닿으려면 그 수직 평면이 목표를 포함해야 하므로, coxa 각도는 다른 관절과 무관하게 즉시 확정된다:

$$\theta_0 = \operatorname{atan2}(y, x)$$

$\theta_0$이 빠지고 나면 남는 것은 그 수직 평면 안의 2차원 문제다. 평면 좌표는 수평 거리 $r = \sqrt{x^2+y^2} - l_0$ (coxa 링크만큼 빼 준다)와 수직 거리이고, 이건 방금 다 푼 2링크 그 자체다. 3자유도 문제가 "1차원 문제 + 2링크 문제"로 **분해**된 것이다 — 6자유도 산업용 팔의 해석해가 손목 3축의 교점을 이용해 "위치 3 + 자세 3"으로 쪼개지는 것과 같은 패턴이고, 해석해가 존재하는 로봇은 대부분 이런 분해 구조를 갖도록 **의도적으로 설계**된다.

```cpp title="leg3dof.cpp — 3자유도 다리 해석해 (FK는 9.4의 Isometry3d 사슬)" {28-30}
#include <Eigen/Dense>
#include <Eigen/Geometry>
#include <cmath>
#include <iostream>
#include <numbers>
#include <optional>

constexpr double l0 = 45.0, l1 = 80.0, l2 = 128.0;   // coxa, femur, tibia [mm]

struct LegJoints { double t0, t1, t2; };

// FK: 9.4에서 만든 Isometry3d 사슬 — 왕복 검산의 심판
Eigen::Vector3d fk_leg(const LegJoints& q) {
    using namespace Eigen;
    Isometry3d T = Isometry3d::Identity();
    T *= AngleAxisd(q.t0, Vector3d::UnitZ());      // coxa yaw
    T *= Translation3d(l0, 0, 0);
    T *= AngleAxisd(q.t1, Vector3d::UnitY());      // femur pitch (+면 발이 내려간다)
    T *= Translation3d(l1, 0, 0);
    T *= AngleAxisd(q.t2, Vector3d::UnitY());      // tibia pitch
    T *= Translation3d(l2, 0, 0);
    return T.translation();
}

std::optional<LegJoints> ik_leg(const Eigen::Vector3d& p, bool knee_up) {
    const double t0 = std::atan2(p.y(), p.x());            // ① yaw 분리
    const double r  = std::hypot(p.x(), p.y()) - l0;       // ② 평면으로 사영
    const double zp = -p.z();                              //    Ry(+)가 발을 -z로 내리므로
    const double c2 = (r * r + zp * zp - l1 * l1 - l2 * l2) / (2.0 * l1 * l2);
    if (c2 < -1.0 || c2 > 1.0) return std::nullopt;
    double t2 = std::acos(c2);                             // ③ 여기부터는 2링크 그대로
    if (knee_up) t2 = -t2;
    const double t1 = std::atan2(zp, r)
                    - std::atan2(l2 * std::sin(t2), l1 + l2 * std::cos(t2));
    return LegJoints{t0, t1, t2};
}
```

FK를 IK와 **다른 방법**(각도 산술이 아니라 [9.3](#/transforms)의 변환 행렬 사슬)으로 구현한 것이 검산의 요점이다 — 같은 실수를 양쪽에 심을 수 없다. 보행 스텝에서 나올 법한 착지 목표들로 돌리면:

```console

( 150.0   40.0  -90.0)   ->   14.931  -23.965   97.098     7.1e-15 mm
( 120.0  -60.0 -110.0)   ->  -26.565  -12.648   97.671     1.6e-14 mm
( 180.0    0.0  -40.0)   ->    0.000  -47.594   98.307     7.1e-15 mm
(  90.0   90.0 -100.0)   ->   45.000  -20.330  107.077     3.2e-14 mm
```

첫 네 줄이 전형적인 지지·착지 자세다 — femur가 수평보다 살짝 들리고($\theta_1 < 0$) tibia가 90°대로 꺾여 내려가는, 곤충 다리의 그 실루엣이다. 3D 격자 2,446점 왕복 검산의 최악 오차도 $10^{-14}$ mm — 2링크와 같은 급이다. IK가 각도를 내면 FK 사슬이 그 각도로 발끝을 재조립해 목표와 대조하는 이 구조를 그대로 GoogleTest 픽스처([7.6](#/googletest))로 옮기면 다리 기구학 테스트가 된다.

마지막 줄 직전을 봐라. 스윙 정점 목표 $(100, 20, 30)$ — 발이 고관절보다 위 — 에 대해 IK는 $\theta_1 = -152°$라는 해를 내놨다. 수학적으로 완벽하다(오차 0.0). 그런데 femur를 뒤로 152° 젖히는 이 자세는 실물 서보의 가동 범위 밖이다. **해가 존재한다는 것과 그 해를 쓸 수 있다는 것은 다른 문제다** — IK 다음 단계에 반드시 관절 한계 검사가 붙는 이유이고, 위 interview 상자의 선택 기준 ②가 실측에 나타난 순간이다.

::: danger 수치 IK의 해를 검사 없이 모터에 보내지 마라
아래 수치법 절에서 실측하겠지만, 반복법은 초기 추정에 따라 $-520°$ 같은 **감긴(wound-up) 각도**나 관절 한계 밖 자세로 태연히 수렴한다. 발끝 오차만 보면 완벽한 해다. 그 각도를 그대로 서보에 쓰면 다리가 몸통을 후려치거나 케이블을 감아 뜯는다 — 시뮬레이션에서는 안 보이고 실물에서만 보이는 종류의 사고다. IK 출력과 모터 명령 사이에는 예외 없이 ① 각도 정규화(unwrap) ② 관절 한계 클램프 ③ 직전 명령 대비 변화율 제한, 세 겹의 검사를 둔다.
:::

## 자코비안: 관절 속도를 발끝 속도로 보내는 지도

여기까지의 해석해는 이 다리가 "yaw + 평면 2링크"로 분해되는 구조라서 가능했다. 분해가 안 되는 체인 — 축이 뒤섞인 6자유도 팔, 사용자가 URDF로 조립한 임의 구조 — 은 다르게 접근해야 한다. 그 열쇠가 **자코비안(Jacobian)**이다.

FK를 $p = f(\theta)$라 쓰면, 자코비안 $J$는 이 함수의 미분 — 관절이 **조금** 움직일 때 발끝이 **어느 방향으로 얼마나** 움직이는지를 담은 행렬이다:

$$\dot{p} = J(\theta)\, \dot{\theta}, \qquad J_{ij} = \frac{\partial p_i}{\partial \theta_j}$$

관절 속도 공간에서 발끝 속도 공간으로 가는 **선형 지도**다. FK 자체는 비선형이지만, 한 자세 근처에서의 미소 변화는 선형이다 — 그 국소 선형화가 $J$다. 2링크면 FK 두 식을 $\theta_1, \theta_2$로 편미분해서 손으로 쓸 수 있다:

```cpp title="jacobian.cpp (조각) — 2링크 자코비안"
Eigen::Matrix2d jacobian(const Eigen::Vector2d& q) {
    const double s1 = std::sin(q[0]), c1 = std::cos(q[0]);
    const double s12 = std::sin(q[0] + q[1]), c12 = std::cos(q[0] + q[1]);
    Eigen::Matrix2d J;
    J << -l1 * s1 - l2 * s12, -l2 * s12,      // dx/dt1, dx/dt2
          l1 * c1 + l2 * c12,  l2 * c12;      // dy/dt1, dy/dt2
    return J;
}
```

이 행렬 하나에 이 절 후반부가 전부 걸려 있다. 두 가지를 읽어내라.

**첫째, $J$를 뒤집으면 IK의 방향이 나온다.** "발끝을 $e$만큼 옮기고 싶다"에 $\dot\theta = J^{-1} e$가 "그러려면 관절을 이만큼"으로 답한다 — 다음 절의 반복법이 이 한 줄의 반복이다.

**둘째, $J$가 뒤집히지 않는 자세가 존재한다.** 2링크 자코비안의 행렬식을 손으로 정리하면 $\det J = l_1 l_2 \sin\theta_2$ — 실측으로도 확인했다($\theta_2 = 90°$에서 10240 = $80 \times 128$, 공식과 일치. $5°$에서 892, $0.1°$에서 17.9로 붕괴). $\theta_2 = 0$, 즉 **다리를 완전히 뻗은 자세에서 $\det J = 0$이다.** 이런 자세를 **특이점(singularity)**이라 부른다. 실측한 그 자세의 자코비안을 보면 물리적 의미가 보인다:

```console
J at fully stretched pose (t1=0, t2=0):
 -0  -0
208 128        <- 첫 행(x 방향)이 전부 0
det = 0
```

두 관절 어느 쪽을 어떻게 움직여도 발끝의 x 성분(뻗은 방향)은 **1차 근사로 전혀 움직이지 않는다.** 완전히 뻗은 다리는 더 뻗을 수도, 그 방향으로 당길 수도 없다 — 잃어버린 방향이 생긴 것이다. 작업공간 경계에 발이 닿는 순간이 정확히 이 자세이므로, **도달 한계와 특이점은 같은 사건의 두 이름**이다.

::: note 특이점은 위치의 문제가 아니라 자세의 문제다
$\det J = l_1 l_2 \sin\theta_2$에 $\theta_1$이 없다는 것에 주목하라 — 특이점 여부는 발끝이 어디 있느냐가 아니라 관절이 어떻게 꺾여 있느냐로 정해진다. 2링크는 작업공간 안쪽 경계($\theta_2 = \pi$, 완전히 접힘)에서도 특이하다. 6자유도 팔은 작업공간 한가운데에서도 손목 축이 정렬되는 손목 특이점(wrist singularity)이 생긴다 — MoveIt2로 팔 궤적을 짤 때([11.4](#/moveit2)) 계획이 실패하거나 관절이 폭주하는 지점의 정체가 대부분 이것이다.
:::

## 뉴턴 반복: 해석해 없이 IK를 푸는 법

자코비안이 있으면 IK를 **반복적으로** 풀 수 있다. 아이디어는 뉴턴법 그대로다 — 현재 추정 자세에서의 발끝 오차를 자코비안의 역으로 관절 보정량으로 바꾸고, 수렴할 때까지 반복한다:

```cpp title="newton.cpp (조각) — 자코비안 기반 반복 IK"
Eigen::Vector2d ik_newton(Eigen::Vector2d q, const Eigen::Vector2d& target) {
    for (int i = 0; i < 200; ++i) {
        const Eigen::Vector2d e = target - fk(q);     // 발끝 오차
        if (e.norm() < 1e-9) break;                   // 수렴
        q += jacobian(q).inverse() * e;               // 오차를 관절 보정으로
    }
    return q;
}
```

해석해가 있는 2링크에 굳이 이걸 돌리는 이유는 **답안지가 있는 문제로 채점**하기 위해서다. 실측:

```console
target ( 120,   60): newton 7 iters (잔차 7.5e-10), 해석해와 각도 일치 (-41.508049, 103.508744)
target ( -40,  150): newton 6 iters, 해석해와 일치 (49.563682, 86.315760)
target (  30, -100): newton 8 iters, 발끝 오차 2.9e-10 — 그런데
   newton q   = (-520.157, 845.470) deg
   analytic q = (-160.157, 125.470) deg
```

정상 케이스에서 6~8회 반복으로 double 정밀도까지 수렴하고, 각도가 해석해와 소수점 여섯 자리까지 일치한다. 그런데 셋째 줄 — 발끝은 정확히 맞는데 각도가 $-520°$와 $845°$다. $-520 = -160 - 360$, $845 = 125 + 720$: **같은 자세의 한 바퀴, 두 바퀴 감긴 표현**으로 수렴한 것이다. 반복법은 초기 추정에서 연속적으로 흘러간 곳에 도착할 뿐, 각도를 $[-\pi, \pi]$로 예쁘게 접어 주지 않는다 — 위 danger 상자의 unwrap 검사가 필요한 실증이다.

특이점 근처에서는 어떻게 되나. 목표를 도달 한계 208mm로 밀어붙이며 쟀다:

```console
r_target    det(J) at 해     newton     DLS(λ=0.5)     DLS(λ=5)
180.000     9041             7회         7회             10회
206.000     2882             9회         9회             17회
207.900      652            11회        13회             80회
207.999       65            14회        65회            200회 실패 (잔차 1.4e-3)
208.000        0            21회       200회 실패        200회 실패
그리고 도달 불가 목표 (230, 0):
newton: 200회 후 잔차 273mm, q = (4507.99, -8764.18) deg   <- 폭주
```

$\det J$가 0으로 무너질수록 뉴턴의 반복 횟수가 7 → 21회로 늘어난다 — $J^{-1}$이 오차를 점점 거대한 관절 보정으로 뻥튀기해 목표 주변을 몸부림치는 것이다. 그리고 마지막 줄, **도달 불가 목표를 주면 뉴턴은 실패를 선언하는 대신 4,500°까지 감기며 폭주한다.** 해석해는 `nullopt` 한 줄로 끝났던 판정이, 반복법에서는 "최대 반복 도달 + 잔차 큼"이라는 간접 증거로만 나타난다 — 수치법을 쓰면 도달 불가 처리도 판정 로직을 직접 설계해야 한다.

**DLS(damped least squares, 댐핑 최소제곱)**가 이 폭주의 표준 처방이다. $J^{-1}$ 대신

$$\Delta\theta = J^\top (J J^\top + \lambda^2 I)^{-1} e$$

를 쓴다 — $\lambda^2 I$가 행렬을 항상 가역으로 만들어 특이점에서도 보정량이 유한하게 남는다. 위 표의 DLS 열이 그 대가를 보여준다: $\lambda$가 클수록 특이점 근처에서 안정적인 대신 **수렴이 느려지고, 목표에 정확히 닿는 것을 포기한다**($\lambda = 5$는 207.999mm 목표에서 잔차 1.4e-3mm로 멈춘다). 댐핑은 "정확도를 조금 팔아 폭주를 막는" 거래이고, $\lambda$ 튜닝이 그 가격 협상이다. 도달 불가 목표 (230, 0)에서도 뉴턴은 4,507°까지 감겼지만 DLS($\lambda=5$)는 2,222° 수준에서 진동했다 — 덜 폭주할 뿐 우아하게 실패하는 것은 아니다.

::: deep DLS의 족보 — Levenberg–Marquardt
$J^\top(JJ^\top + \lambda^2 I)^{-1}$은 비선형 최소제곱의 Levenberg–Marquardt 방법을 IK에 적용한 것이다. 최적화 관점으로 보면 발끝 오차의 제곱 $\|target - f(\theta)\|^2$을 내려가는 것인데, $\lambda \to 0$이면 뉴턴(빠르지만 특이점에서 폭주), $\lambda \to \infty$면 경사하강(느리지만 안정)이 되는 연속 스펙트럼의 중간이다. MoveIt2에서 KDL 기본 솔버와 나란히 제공되는 LMA 솔버가 이름 그대로 이 계열이다. [9.6](#/state-estimation)의 EKF도 "비선형을 자코비안으로 국소 선형화해서 다룬다"는 같은 뼈대 위에 있다 — Part IX 후반부는 이 한 가지 아이디어의 변주다.
:::

## 해석해 vs 수치해: 1kHz 예산의 산수

두 방법이 다 손에 있으니 선택 기준을 숫자로 만들자. 같은 조건(-O2, 함수 `noipa` 격리, 작업공간 안 무작위 목표 1만 개, 3회 실행)에서 호출당 비용을 쟀다.

::: perf 해석해는 나노초, 반복법은 마이크로초 (g++ 13.3 / -O2 / x86-64 실측)
| 방법 | 호출당 | 해석해 대비 |
| --- | --- | --- |
| 해석해 2링크 | 92 ns | 1.0배 |
| 해석해 3자유도 다리 | 136 ns | 1.5배 |
| 뉴턴 (고정 초기 추정, 평균 ~7회 반복) | 3,327 ns | **36배** |
| DLS λ=5 (고정 초기 추정) | 3,456 ns | 37배 |

1kHz 제어 루프의 주기 예산은 1,000,000ns다. 다리 6개 IK가 해석해면 **6 × 136 = 818ns, 주기의 0.08%** — 반올림 오차 수준이다. 뉴턴이면 6 × 3.3µs = 20µs, 주기의 2% — 아직 수용 가능하지만 25배 크고, 특이점 근처에서 반복 횟수가 3배로 뛰는 변동성([8.6](#/benchmarking)에서 경계한 꼬리 지연)까지 안는다. 절대값은 기기마다 다르지만 "닫힌 수식 몇 줄 vs 행렬 역산을 낀 반복"이라는 자릿수 구조는 어디서나 같다.
:::

반복법에도 자기 무기가 있다 — **warm start**다. 보행 궤적은 연속이라 이번 주기의 해가 다음 주기의 훌륭한 초기 추정이 된다. 1000스텝짜리 스윙 궤적으로 실측하면 직전 해에서 시작한 뉴턴은 평균 **2.1회** 반복(최대 4회, 호출당 ~240ns)으로, 고정 초기 추정의 5.0회(~520ns)보다 배 이상 싸진다. 단 이 이득은 연속성이 전제다 — 같은 warm start를 무작위 순서 목표에 쓰면 직전의 감긴 해가 다음 풀이의 나쁜 출발점이 되어 오히려 18µs까지 악화되는 것도 실측됐다. warm start는 궤적 위에서만 공짜 점심이다.

결론은 이 절 전체의 요지다. **해석해가 존재하면 해석해가 정답이다** — 36배 빠르고, 반복 횟수 변동이 없고(실시간 루프가 사랑하는 성질, [6.8](#/realtime)), 도달 불가 판정이 공짜고, 두 해를 명시적으로 쥐고 고를 수 있다. 헥사포드 다리는 해석해가 존재하도록 설계된 구조이므로 고민의 여지가 없다. 수치법의 자리는 해석해가 **없는** 곳이다 — 임의 URDF 체인을 받아야 하는 범용 소프트웨어가 그렇다. MoveIt2의 기본 IK 플러그인(KDL)이 자코비안 기반 수치 솔버인 이유가 정확히 이것이다: 어떤 로봇이 올지 모르니 일반해법을 기본으로 깔고, 특정 로봇에 최적화하고 싶은 사용자가 해석해 솔버(IKFast처럼 코드 생성으로 만든)를 플러그인으로 꽂는 구조다([11.1](#/pluginlib)의 플러그인 아키텍처, [11.4](#/moveit2)에서 실물을 만진다).

::: tip 헥사포드 실전 배치: 해석해 + 고정 무릎 방향 + 한계 검사
Hexpider 보행 루프에 들어갈 최종 형태는 이 절의 재료를 그대로 쌓은 것이다: `ik_leg`(해석해, 136ns) → 무릎 방향은 곤충 자세로 고정(`knee_up` 한쪽만 사용, 연속성 고민 소거) → 관절 한계 검사 → 실패 시 직전 해 유지 + 실패 카운터 증가. 걸음새 계획이 내놓는 스윙 착지점이 이 함수의 입력이고, 출력이 [10.9](#/ros2-control)의 hardware_interface로 흘러간다. [8.1](#/profiling)에서 병목 후보였던 그 함수의 실물이 주기 예산의 0.014%(다리당)로 끝나는 것 — 이게 해석해를 가진 로봇의 사치다.
:::

## 요약

- IK는 FK의 역방향 — 함수 평가가 아니라 방정식 풀이라서 해가 없거나($|\cos\theta_2| > 1$, 작업공간 밖), 둘이거나(elbow-up/down), 무한할 수 있다(여유 자유도). 보행의 매 스텝(착지점 → 관절 각도)이 IK 호출이다.
- 2링크 해석해: $\cos\theta_2 = (x^2+y^2-l_1^2-l_2^2)/(2l_1l_2)$ — 무릎은 거리만으로 정해진다. $\theta_1 = \operatorname{atan2}(y,x) - \operatorname{atan2}(l_2 s_2, l_1+l_2 c_2)$. 두 해는 $\pm\arccos$의 부호 선택이고 위젯의 무릎 방향 전환이 그것이다. 왕복 검산 IK(FK($\theta$)) = $\theta$가 기본 테스트 — 격자 2,568회 최악 오차 $10^{-13}$mm 실측.
- 도달 불가 처리 3전략: 에러 반환(계획 계층 — 일찍 시끄럽게), 클램프(모터 직전 계층 — 단 보고와 함께), 이전 해 유지(위젯의 선택). 경계 근처의 $\arccos$는 오차를 최대 2만 배 증폭한다(실측) — atan2 형태가 안전하고, [9.8](#/numerics)에서 계속된다.
- 3자유도 다리는 coxa가 yaw 하나라 $\theta_0 = \operatorname{atan2}(y, x)$로 분리되고 나머지가 수직 평면 2링크로 환원된다 — 해석해가 존재하는 로봇은 이런 분해가 되도록 설계된 것이다. 3D 격자 2,446회 왕복 검산 최악 $10^{-14}$mm. 단 수학적 해가 관절 한계 안이라는 보장은 없다($\theta_1 = -152°$ 실측).
- 자코비안은 관절 속도 → 발끝 속도의 선형 지도(국소 선형화)다. $\det J = l_1 l_2 \sin\theta_2 = 0$인 자세(다리를 완전히 뻗음)가 특이점 — 잃어버린 이동 방향이 생기고, 도달 한계와 같은 사건이다.
- 뉴턴 반복($\Delta\theta = J^{-1}e$)은 정상 목표에 6~8회로 해석해와 같은 답에 수렴하지만, 특이점 접근 시 21회로 늘고 도달 불가 목표엔 4,500°까지 폭주한다(실측). DLS는 $\lambda^2 I$ 댐핑으로 폭주를 막는 대신 수렴 속도와 최종 정확도를 판다.
- 선택 기준: 해석해 136ns vs 뉴턴 3.3µs — 36배(실측). 1kHz × 6다리에서 해석해는 주기의 0.08%다. 해석해가 있으면 해석해, 임의 체인을 받아야 하는 범용 소프트웨어(MoveIt2/KDL)만 수치법이 기본이다.

::: quiz 연습문제
1~3번은 개념 문제, 4번은 코드 작성형(성공 기준: 네 컴퓨터에서 컴파일 통과 + 검산 수치), 5번은 설계 판단이다.

1. 2링크 IK에서 무릎 각도 $\theta_2$가 목표의 방향과 무관하게 거리만으로 정해지는 이유를 코사인 법칙의 관점에서 한 문장으로 설명하라. 그렇다면 목표의 방향 정보는 어느 관절이 전담하는가?

2. 위젯에서 목표를 도달 한계 원 바로 안쪽에 두고 무릎 방향 전환을 눌러 보면 두 해의 차이가 거의 없다. 왜인가? $\theta_2$와 특이점의 관계로 설명하라.

3. 뉴턴 반복이 발끝 오차 $10^{-10}$으로 수렴했는데 관절 각도가 해석해와 360°의 배수만큼 다를 수 있음을 실측으로 봤다. 이 해를 "틀렸다"고 할 수 있는가? 이 해가 실물 로봇에서 문제가 되는 지점과, IK와 모터 사이에 둬야 하는 검사를 답하라.

4. (실습, 코드 작성형) 본문의 `ik_leg`/`fk_leg`를 완결된 프로그램으로 작성하고, 세 겹 for 루프로 작업공간 안 3D 격자점 1,000개 이상에 대해 왕복 검산을 돌려 최악 오차를 출력하라. 그다음 `knee_up = true` 가지도 같은 격자로 검산하라. 컴파일: `g++ -std=c++20 -Wall -Wextra -O2 -I/usr/include/eigen3 leg_ik.cpp && ./a.out`. 성공 기준: ① 두 가지 모두 최악 오차가 $10^{-9}$mm 아래 ② 도달 불가 목표 하나를 넣어 `nullopt` 반환을 확인했다.

5. (설계 판단) 걸음새 계획이 가끔 작업공간을 1~2mm 벗어나는 착지점을 요구한다는 것이 로그로 확인됐다. 동료가 "IK에서 $\cos\theta_2$를 조용히 클램프하면 끝"이라고 제안한다. 이 제안의 문제를 지적하고, 계층(궤적 생성 / 다리 IK / 모터 직전)별로 어떤 처리를 둘지 당신의 설계를 제시하라.
:::

::: answer 해설
1. 고관절–무릎–발끝 삼각형의 세 변($l_1$, $l_2$, 목표 거리 $d$)이 정해지면 무릎 내각은 코사인 법칙으로 유일하게 결정된다 — 삼각형의 모양은 변 길이만의 함수이고, 그 삼각형이 고관절 둘레로 어느 방향으로 돌아 앉는지는 별개다. 방향은 $\theta_1$의 첫 항 $\operatorname{atan2}(y, x)$가 전담한다.

2. 도달 한계에서 $\cos\theta_2 \to 1$, 즉 $\theta_2 \to 0$이다. 두 해는 $\pm\theta_2$이므로 $\theta_2$가 0에 가까우면 두 해 자체가 서로 접근하고, 한계 위에서는 하나로 합쳐진다(중근). 이 합쳐지는 지점이 정확히 $\det J = l_1 l_2 \sin\theta_2 = 0$인 특이점이다 — 두 해의 융합, 작업공간 경계, 특이점이 같은 사건의 세 얼굴이라는 것이 이 절의 구조다.

3. 발끝 기준으로는 틀리지 않았다 — $\theta + 2\pi k$는 같은 자세이고 FK 오차도 $10^{-10}$이다. 문제는 실물에서 터진다: ① 서보 명령이 각도 자체를 받으므로 $-520°$는 가동 범위 밖 명령이 되고 ② 직전 주기 해가 $-160°$였다면 한 주기 만에 360° 점프하는 속도 명령이 된다. 검사는 본문 danger 상자의 세 겹 — $[-\pi, \pi]$ unwrap, 관절 한계 클램프, 변화율 제한 — 이고, unwrap이 이 문제의 직접 처방이다.

4. 본문 `leg3dof.cpp`가 골격이다. 격자는 `x: 40~260, y/z: ±200` 정도를 20mm 간격으로 돌면 2,000개 이상이 작업공간 안에 들어온다(이 환경 실측: knee-down 2,446개 해, 최악 $5.9 \times 10^{-14}$mm). `knee_up = true`도 같은 자릿수가 나와야 한다 — 두 가지 모두 FK라는 독립 구현이 심판이므로, IK 부호 실수는 즉시 mm 단위 오차로 폭로된다. 도달 불가는 `(300, 100, -150)`처럼 $r - l_0 > 208$인 점이면 된다.

5. 조용한 클램프의 문제: 1~2mm의 발끝 오차가 **보고 없이** 계통에 들어간다 — 지형 접촉 판정·오도메트리가 실제와 다른 발 위치를 참으로 믿게 되고, 무엇보다 "계획이 작업공간을 벗어난다"는 상류의 버그가 영원히 은폐된다. 계층별 설계: 궤적 생성 단계에서는 작업공간 마진 검사를 두고 벗어나면 시끄럽게 실패시켜 계획 버그를 고치게 한다(근본 수리). 다리 IK는 `optional` 반환으로 판정만 정직하게 전달한다. 모터 직전 계층만 클램프를 허용하되 클램프 발생 카운터를 증가시켜 텔레메트리로 노출한다 — "버티되 기록한다"가 원칙이다.
:::

이 절의 코드는 전부 직접 쳐라. `ik2link.cpp`를 먼저 완성해 왕복 검산이 $10^{-13}$대로 나오는 것을 확인하고, 4번 문제의 `leg_ik.cpp`로 확장하라 — 기준 명령은 `g++ -std=c++20 -Wall -Wextra -O2 -I/usr/include/eigen3 leg_ik.cpp && ./a.out`. 여력이 있으면 `newton.cpp`도 만들어 목표를 (207, 0) → (207.9, 0) → (208, 0)으로 밀며 반복 횟수가 부풀어 오르는 것을 네 눈으로 봐라 — 특이점은 정의를 외우는 게 아니라 몸부림을 목격하는 쪽이 오래 남는다.

**다음 절**: [9.6 상태 추정: 칼만 필터에서 EKF까지](#/state-estimation) — IK가 "어디에 둘 것인가"였다면 다음은 "지금 어디에 있는가"다. 센서는 거짓말을 하고, 그 거짓말들을 섞어 진실에 다가가는 수학이 칼만 필터다 — 비선형을 자코비안으로 선형화하는 EKF에서 이 절의 $J$를 다시 만난다.
