# 9.4 순기구학

::: lead
서보에 "고관절 35°, 무릎 55°"라고 명령했다. 그래서 발끝은 지금 어디에 있는가? 이 질문에 답하는 것이 순기구학(forward kinematics, FK)이다 — 관절 각도를 넣으면 말단의 위치·자세가 나오는 함수. 보행은 이 함수 없이는 시작도 못 한다: 발이 지면 어디를 딛는지 모르면 몸통을 얼마나 밀었는지도 모른다. 좋은 소식은 FK가 로보틱스 수학에서 가장 "쉬운 방향"이라는 것이다 — 답이 항상, 정확히 하나 존재한다. 이 절은 2링크 다리를 삼각함수로 손 유도한 뒤, 같은 계산을 [9.3](#/transforms)의 동차 변환 곱으로 다시 해서 **FK = 변환 곱의 사슬**이라는 일반 원리에 도달하고, 헥사포드 3자유도 다리와 URDF 스타일 파라미터까지 확장한다. [9.5 역기구학](#/inverse-kinematics)이 왜 어려운지는 이 절의 "쉬움"을 먼저 체감해야 보인다.
:::

## 각도는 아는데, 발은 어디에 있는가

로봇 다리의 상태를 코드가 아는 방식은 관절 공간(joint space)이다. 서보 18개에 명령한 각도, 엔코더가 돌려준 각도 — 전부 관절 각도의 배열이다. 그런데 로봇이 해야 하는 일은 작업 공간(task space)에서 정의된다. "발끝을 지면의 이 점에 딛어라", "몸통을 10mm 앞으로 밀어라". 두 공간을 잇는 다리가 기구학이고, 방향이 둘이다.

- **순기구학 (FK)**: 관절 각도 → 말단 위치. $f(\theta_1, \theta_2, \ldots) = (x, y, z, \ldots)$
- **역기구학 (IK)**: 말단 위치 → 관절 각도. $f^{-1}$

둘의 난이도는 극단적으로 비대칭이다. FK는 **함수**다 — 각도 조합 하나에 발끝 위치가 정확히 하나 대응하고, 예외가 없다. 로봇의 각 링크는 강체이고 각 관절은 정해진 축으로만 도니까, 각도를 전부 지정하면 모든 링크의 위치가 기하학적으로 확정된다. 계산도 삼각함수와 곱셈뿐이라 실패 모드가 없다. 반면 IK는 함수의 역방향 문제라 해가 없을 수도(도달 불가), 여러 개일 수도(무릎 위/아래), 무한히 많을 수도(여유 자유도) 있다 — 그 지뢰밭은 [9.5](#/inverse-kinematics)의 몫이다. 이 절에서는 "항상 되는 방향"을 완전히 손에 넣는다.

::: note FK가 항상 풀리는 이유는 강체 가정이다
"각도만 알면 위치가 확정된다"는 링크가 휘지 않는다는 가정 위에 서 있다. 링크가 탄성 변형하는 로봇(연속체 로봇, 소프트 로봇)에서는 FK조차 단순한 함수가 아니게 된다. 취미급 헥사포드에서도 이 가정은 미세하게 깨진다 — 서보 브래킷이 부하로 휘면 FK가 계산한 발끝과 실제 발끝이 어긋난다. 그래서 실기체에서는 FK 모델의 예측과 실측(자로 잰 발끝 위치)을 비교하는 캘리브레이션이 따라붙는다. 모델은 모델이다.
:::

## 2링크 다리 — 삼각함수로 손 유도

헥사포드 다리의 측면도를 보자. 몸통에 붙은 고관절에서 femur 링크(길이 $l_1 = 80$mm)가 나가고, 무릎에서 tibia 링크($l_2 = 128$mm)가 이어져 발끝으로 간다. 관절 각도는 두 개: 고관절 각 $\theta_1$(수평 기준), 무릎 각 $\theta_2$(**femur의 연장선 기준** — 이전 링크에 상대적인 각도라는 것이 핵심이다. 서보가 실제로 그렇게 붙어 있다).

무릎의 위치는 바로 나온다. 반지름 $l_1$인 원 위의 점이다.

$$k_x = l_1\cos\theta_1, \qquad k_y = l_1\sin\theta_1$$

발끝은 무릎에서 tibia만큼 더 간 점인데, tibia가 세계 기준으로 향한 각도는 $\theta_1 + \theta_2$다 — $\theta_2$가 femur에 상대적인 각도이므로 femur의 각도에 누적된다. 그래서:

$$x = l_1\cos\theta_1 + l_2\cos(\theta_1 + \theta_2), \qquad y = l_1\sin\theta_1 + l_2\sin(\theta_1 + \theta_2)$$

이게 2링크 평면 FK의 전부다. 관례로 $c_1 = \cos\theta_1$, $c_{12} = \cos(\theta_1+\theta_2)$라고 줄여 $x = l_1 c_1 + l_2 c_{12}$로 쓴다 — 기구학 교과서·논문에서 계속 만나는 표기다. 구체 수치로 확인하자. $\theta_1 = 35°$, $\theta_2 = 55°$면 $\theta_1 + \theta_2 = 90°$라 tibia가 정확히 수직으로 꺾인다 — $c_{12} = 0$이므로 x에는 femur만, y에는 tibia 전체가 들어가는 깔끔한 검산 케이스다.

```cpp title="fk2d.cpp — 손 유도 식과 변환 사슬의 대조" {14-15,19-24}
#include <Eigen/Dense>
#include <Eigen/Geometry>
#include <cmath>
#include <iostream>
#include <numbers>

int main() {
    using namespace Eigen;
    constexpr double pi = std::numbers::pi;
    const double l1 = 80.0, l2 = 128.0;                    // femur, tibia [mm]
    const double t1 = 35.0 * pi / 180.0;                   // 고관절
    const double t2 = 55.0 * pi / 180.0;                   // 무릎

    double x = l1 * std::cos(t1) + l2 * std::cos(t1 + t2);
    double y = l1 * std::sin(t1) + l2 * std::sin(t1 + t2);
    std::cout << "trig : x = " << x << ", y = " << y << "\n";

    // 같은 다리를 동차 변환의 사슬로 — 다음 섹션의 주제
    Isometry3d T_base_j1(AngleAxisd(t1, Vector3d::UnitZ()));
    Isometry3d T_j1_j2 = Translation3d(l1, 0, 0)
                       * AngleAxisd(t2, Vector3d::UnitZ());
    Isometry3d T_j2_foot(Translation3d(l2, 0, 0));

    Isometry3d T_base_foot = T_base_j1 * T_j1_j2 * T_j2_foot;
    Vector3d foot = T_base_foot * Vector3d::Zero();        // 발끝 프레임의 원점
    std::cout << "chain: " << foot.transpose() << "\n";
    std::cout << "diff : " << (foot.head<2>() - Vector2d(x, y)).norm() << "\n";
    return 0;
}
```

```console
$ g++ -std=c++20 -Wall -Wextra -O2 -I/usr/include/eigen3 fk2d.cpp && ./a.out
trig : x = 65.5322, y = 173.886
chain: 65.5322 173.886       0
diff : 0
```

손 계산 검증: $x = 80\cos 35° + 128\cos 90° = 65.53 + 0$, $y = 80\sin 35° + 128\sin 90° = 45.89 + 128 = 173.89$. 맞다.

## 위젯으로 몸에 새긴다

아래 위젯이 방금 그 다리다 — femur 80mm, tibia 128mm, 초기 각도가 정확히 $\theta_1 = 35°$, $\theta_2 = 55°$로 설정돼 있다. 캡션의 발끝 좌표가 위 실행 결과 (65.5, 173.9)와 일치하는 것부터 확인하라(위젯 화면은 로봇 다리가 아래로 늘어지는 방향, 즉 y 아래쪽이 양수다 — 수식과 부호 관례가 같다). 그다음 슬라이더로 두 각도를 돌려 보라. **어떤 각도 조합을 넣어도 발끝은 즉시, 유일하게 결정된다** — 이것이 FK가 "쉬운 방향"이라는 말의 체감이다. θ1만 돌리면 다리 전체가 고관절 중심의 원호를 그리고, θ2만 돌리면 무릎 중심의 원호를 그린다. 점선으로 표시된 두 원이 이 다리가 도달할 수 있는 영역(작업공간, workspace)의 경계다: 바깥 한계 $l_1 + l_2 = 208$mm(다리를 곧게 뻗은 반지름), 안쪽 한계 $|l_1 - l_2| = 48$mm(무릎을 완전히 접은 반지름).

각도를 이리저리 돌리면서 궤적이 두 원 사이의 고리 모양(환형) 영역을 절대 벗어나지 못하는 것을 확인하라. IK 모드 버튼도 눌러 볼 수 있다 — 목표점을 끌면 각도가 "풀려서" 나오는 반대 방향인데, 도달 불가 영역과 무릎 방향 두 해가 바로 IK의 난점이다. 지금은 눈도장만 찍고 [9.5](#/inverse-kinematics)에서 제대로 다룬다.

## FK는 변환 곱의 사슬이다

`fk2d.cpp`의 후반부가 이 절의 진짜 주제다. 같은 답 (65.53, 173.89)를 삼각함수 식 없이, [9.3](#/transforms)의 동차 변환 곱 세 개로 얻었다.

$$T_{base \to foot}(\theta_1, \theta_2) = T_{base \to j1}(\theta_1) \cdot T_{j1 \to j2}(\theta_2) \cdot T_{j2 \to foot}$$

각 인수를 뜯어 보면 전부 "관절 하나" 아니면 "링크 하나"다.

- $T_{base \to j1}(\theta_1)$ — 고관절의 회전. 순수 회전, $\theta_1$의 함수.
- $T_{j1 \to j2}(\theta_2)$ — femur 링크를 따라 $l_1$만큼 이동한 뒤 무릎 회전. 코드의 `Translation3d(l1,0,0) * AngleAxisd(t2, ...)`.
- $T_{j2 \to foot}$ — tibia 링크를 따라 $l_2$만큼 이동. 각도와 무관한 **상수 변환**이다.

발끝 위치는 발끝 프레임의 원점 $(0,0,0)$을 base 프레임으로 옮긴 것, 즉 `T_base_foot * Vector3d::Zero()` — 결국 $T$의 이동 성분이라 `T_base_foot.translation()`과 같은 값이다. 삼각함수 유도에서 "각도가 누적된다"고 손으로 챙겼던 $\theta_1 + \theta_2$가, 변환 곱에서는 회전 행렬끼리의 곱으로 **자동으로** 처리된다는 것에 주목하라. 덧셈 공식($\cos(\theta_1+\theta_2) = c_1 c_2 - s_1 s_2$)을 전개해 보면 행렬 곱이 하고 있는 일이 정확히 그것이다.

이 재작성이 주는 것은 우아함이 아니라 **확장성**이다. 삼각함수 유도는 관절이 하나 늘 때마다 식을 처음부터 다시 세워야 하고, 3D로 가서 회전축이 섞이면 손 유도는 사실상 불가능해진다. 변환 사슬은 관절이 늘면 곱셈 인수가 하나 늘 뿐이다. **어떤 로봇이든 — 6축 팔이든, 18관절 헥사포드든, 휴머노이드든 — FK는 "관절 변환과 링크 변환을 뿌리부터 말단까지 곱한다" 한 문장이다.** 이 관절-링크의 연결 구조를 기구학 체인(kinematic chain)이라 부른다.

::: interview "FK와 IK의 차이를 설명하라 — 어느 쪽이 왜 어려운가"
로보틱스 직군 면접의 첫 관문 단골이다. 모범 답변의 뼈대: ① FK는 관절 각도 → 말단 자세, IK는 그 역방향. ② FK는 각 관절의 변환을 체인 순서로 곱하기만 하면 되는 **잘 정의된 함수**다 — 해가 항상 유일하게 존재하고, 닫힌 형태로 계산되며, 실패 모드가 없다. ③ IK는 비선형 연립방정식의 역문제라 해가 없거나(작업공간 밖), 유한히 여러 개거나(elbow-up/down 같은 자세 분기), 무한개(자유도가 남는 로봇)일 수 있고, 특이점 근처에서는 수치적으로도 불안정하다. ④ 그래서 실무는 FK는 그냥 곱하고, IK는 구조가 단순하면 해석해(닫힌 식), 아니면 자코비안 기반 수치 반복으로 푼다 — 여기까지 말하면 충분하다. "IK 해가 여러 개일 때 무엇으로 고르나"(관절 한계, 현재 자세와의 연속성)까지 덧붙이면 실전 경험이 있다는 신호가 된다.
:::

## 3D로 — 헥사포드 3자유도 다리

실제 헥사포드 다리는 평면에 살지 않는다. 관절이 세 개다: **coxa**(고관절 yaw — 다리를 수평으로 휘두른다, z축 회전), **femur**(pitch — 다리를 들었다 내린다, y축 회전), **tibia**(pitch — 무릎, y축 회전). 링크 길이는 coxa 45mm, femur 80mm, tibia 128mm로 잡는다. 회전축이 z, y, y로 섞여 있어 삼각함수 손 유도는 벌써 고통스럽지만, 변환 사슬은 인수가 둘 늘었을 뿐이다.

```cpp title="leg3d.cpp — 헥사포드 3자유도 다리 FK"
#include <Eigen/Dense>
#include <Eigen/Geometry>
#include <iostream>
#include <numbers>

using namespace Eigen;
constexpr double pi = std::numbers::pi;
constexpr double deg = pi / 180.0;

// 링크는 전부 자기 프레임의 +x 방향으로 뻗는다. 단위 mm.
constexpr double Lc = 45.0, Lf = 80.0, Lt = 128.0;

Isometry3d leg_fk(double t1, double t2, double t3) {
    return Isometry3d(AngleAxisd(t1, Vector3d::UnitZ()))   // coxa yaw
         * Translation3d(Lc, 0, 0)
         * AngleAxisd(t2, Vector3d::UnitY())               // femur pitch
         * Translation3d(Lf, 0, 0)
         * AngleAxisd(t3, Vector3d::UnitY())               // tibia pitch
         * Translation3d(Lt, 0, 0);
}

int main() {
    struct { double t1, t2, t3; } sets[] = {
        {0, 0, 0},
        {0, 30 * deg, 60 * deg},
        {30 * deg, 30 * deg, 60 * deg},
        {0, 0, 90 * deg},
    };
    for (auto [t1, t2, t3] : sets) {
        Vector3d foot = leg_fk(t1, t2, t3) * Vector3d::Zero();
        std::cout << "(" << t1/deg << ", " << t2/deg << ", " << t3/deg
                  << ") -> " << foot.transpose() << "\n";
    }
    Vector3d zero = leg_fk(0, 0, 0) * Vector3d::Zero();
    std::cout << "zero pose |foot| = " << zero.norm()
              << " (Lc+Lf+Lt = " << Lc + Lf + Lt << ")\n";
    return 0;
}
```

```console
$ g++ -std=c++20 -Wall -Wextra -O2 -I/usr/include/eigen3 leg3d.cpp && ./a.out
(0, 0, 0) -> 253   0   0
(0, 30, 60) -> 114.282       0    -168
(30, 30, 60) -> 98.9711  57.141    -168
(0, 0, 90) ->  125    0 -128
zero pose |foot| = 253 (Lc+Lf+Lt = 253)
```

좌표계는 REP-103([9.2](#/rotations)): x 전방(다리가 뻗는 방향), z 상방. pitch 양수는 링크를 아래로 꺾는다($R_y(\theta)$가 x축을 $-z$ 쪽으로 돌린다). 출력을 한 줄씩 검산하자.

- **(0, 0, 0)**: 다리를 곧게 뻗은 영점 자세. 발끝 $(253, 0, 0)$ — 링크 길이 합 $45+80+128 = 253$ 그대로다. 첫 번째 검산 통과.
- **(0, 30, 60)**: coxa가 0이라 평면 문제로 퇴화한다. $x = 45 + 80\cos 30° + 128\cos 90° = 114.28$, $z = -(80\sin 30° + 128\sin 90°) = -168$. 2링크 손 유도가 x-z 평면에서 그대로 재현된다 — femur·tibia의 pitch가 서로 누적되는 것도 동일하다.
- **(30, 30, 60)**: 위 자세에서 coxa만 30° 돌렸다. $z = -168$은 **그대로**고, 수평 성분 $(114.28, 0)$이 yaw 30°로 회전해 $(114.28\cos 30°, 114.28\sin 30°) = (98.97, 57.14)$가 됐다. coxa 회전은 발끝 높이를 건드리지 못한다 — z축 회전이니까. 이런 구조적 성질이 코드 검증의 무기가 된다(아래 검산 섹션).
- **(0, 0, 90)**: femur는 수평, 무릎만 직각. $x = 45 + 80 = 125$, $z = -128$. tibia가 수직으로 늘어진 자세다.


::: danger 각도 단위를 섞으면 FK는 조용히 틀린 자세를 준다
`AngleAxisd`는 라디안을 받는다. 30°를 의도하고 `leg_fk(0, 30, 60)`이라고 도(degree) 값을 그대로 넣으면 — 컴파일러는 아무 말 없다. double은 double이니까. 이 환경 실측: 올바른 `leg_fk(0, 30*deg, 60*deg)`는 발끝 $(114.28, 0, -168)$인데, 단위를 빼먹은 버전은 $(-0.013, 0, -35.4)$ — 다리가 몸통 밑으로 완전히 접힌 엉뚱한 자세다. 30 rad는 약 1719°, 즉 네 바퀴 반이다. 이런 값이 서보 명령 경로에 들어가면 하드웨어가 부서진다. 방어책은 경계에서 단위를 못박는 것이다: 외부 입력(설정 파일, 메시지)은 어느 단위인지 명시하고, **내부 계산은 전부 라디안으로 통일**하고, 변환은 입구에서 한 번만 한다. 타입으로 막고 싶다면 `std::chrono`가 시간에 한 일을 각도에 하는 강타입 래퍼(`Radian` 타입)를 만들 수도 있다 — double 하나를 감싼 래퍼는 [8.3](#/codegen)에서 본 인라이닝으로 기계어에서 사라지므로, 이 방어의 런타임 비용은 없다.
:::

## URDF 스타일로 조립하는 FK

`leg_fk()`는 다리 하나에 하드코딩된 사슬이다. 실전 로봇 소프트웨어는 로봇 구조를 코드가 아니라 **데이터**로 기술한다 — ROS 2에서는 URDF가 그 데이터다. URDF의 revolute joint 하나는 기하학적으로 딱 세 가지 정보다.

```xml title="URDF joint의 뼈대 (femur 관절 예시)"
<joint name="femur_joint" type="revolute">
  <origin xyz="0.045 0 0" rpy="0 0 0"/>   <!-- 부모 링크에서 조인트까지의 고정 변환 -->
  <axis xyz="0 1 0"/>                      <!-- 회전축: y (pitch) -->
  <parent link="coxa_link"/> <child link="femur_link"/>
</joint>
```

이것이 뜻하는 변환은 정확히 다음 구조다:

$$T_{parent \to child}(q) = T_{origin} \cdot R(\hat{a}, q)$$

$T_{origin}$은 `<origin>`의 xyz·rpy로 만드는 **상수** 변환(우리 사슬의 `Translation3d(Lc,0,0)` 자리), $R(\hat{a}, q)$는 `<axis>` 둘레의 관절 각도 $q$ 회전(우리 사슬의 `AngleAxisd` 자리)이다. 즉 URDF 파일은 변환 사슬의 인수 목록을 XML로 적어 놓은 것이고, FK 라이브러리는 그것을 읽어 곱한다. 직접 조립해 보면 이 대응이 확실해진다.

```cpp title="urdf_fk.cpp — joint origin+axis 방식의 미니 FK (조각: main은 아래 설명대로)"
struct RevoluteJoint {
    Vector3d xyz;    // <origin xyz> : 부모 링크 프레임에서 본 조인트 원점
    Vector3d rpy;    // <origin rpy> : 조인트 원점의 고정 회전 (9.2의 URDF 관례)
    Vector3d axis;   // <axis>       : 회전축 단위벡터
};

Isometry3d joint_transform(const RevoluteJoint& j, double q) {
    Isometry3d T_origin = Translation3d(j.xyz)
        * AngleAxisd(j.rpy.z(), Vector3d::UnitZ())   // rpy -> 행렬 (9.2 관용구)
        * AngleAxisd(j.rpy.y(), Vector3d::UnitY())
        * AngleAxisd(j.rpy.x(), Vector3d::UnitX());
    return T_origin * AngleAxisd(q, j.axis);
}

// FK = 조인트 변환을 뿌리부터 말단까지 곱하는 루프 하나. 어떤 체인이든 같다.
Isometry3d chain_fk(const std::vector<RevoluteJoint>& joints,
                    const std::vector<double>& q,
                    const Isometry3d& T_last_ee) {
    Isometry3d T = Isometry3d::Identity();
    for (std::size_t i = 0; i < joints.size(); ++i)
        T = T * joint_transform(joints[i], q[i]);
    return T * T_last_ee;
}
```

`main`에서 `leg3d.cpp`와 같은 다리를 데이터로 기술하고 — coxa `{xyz(0,0,0), axis z}`, femur `{xyz(45,0,0), axis y}`, tibia `{xyz(80,0,0), axis y}`, 발끝 오프셋 `Translation3d(128,0,0)` — 각도 $(30°, 30°, 60°)$를 넣으면:

```console
$ g++ -std=c++20 -Wall -Wextra -O2 -I/usr/include/eigen3 urdf_fk.cpp && ./a.out
urdf-style foot: 98.9711  57.141    -168
diff vs direct chain: 0
```

하드코딩 사슬과 행렬 단위로 완전히 같다(차 노름 0). **로봇 구조가 데이터가 되는 순간, FK 코드는 로봇과 무관해진다** — `chain_fk()`는 다리 개수도 링크 길이도 모른다. KDL, MoveIt2, robot_state_publisher가 URDF만 주면 임의의 로봇 FK를 즉시 제공하는 이유가 이 구조다. [11.4](#/moveit2)에서 MoveIt2의 `RobotState::getGlobalLinkTransform()`을 부를 때, 그 밑에서 도는 것이 방금 짠 이 루프의 산업용 버전이다.

::: note tf2가 하는 일도 절반은 FK다
[10.7](#/tf2)에서 만날 robot_state_publisher는 `/joint_states` 토픽의 관절 각도를 받아 URDF로 위 계산을 하고, 결과 $T_{parent \to child}$들을 TF 트리로 방송한다. 즉 RViz에서 로봇 모델이 관절대로 움직이는 것은 누군가가 매 갱신마다 이 절의 FK를 돌리고 있다는 뜻이다. FK는 "가끔 부르는 유틸리티"가 아니라 로봇 미들웨어의 심장 박동에 들어 있는 연산이다.
:::

## DH 파라미터 — 이름은 알아 둔다

기구학 교과서를 펴면 체인 기술 방법으로 십중팔구 **Denavit-Hartenberg(DH) 파라미터**가 먼저 나온다. 관절마다 네 숫자 $(a, \alpha, d, \theta)$로 변환을 기술하는 1955년의 관례로, 프레임 배치 규칙을 강하게 제약해서 관절당 파라미터를 4개로 압축한 것이 요점이다(일반 변환은 6개 자유도).

::: hist DH가 절약한 것과 대가로 치른 것
1955년, 컴퓨터 메모리와 계산이 모두 귀하던 시절에 자유도 6개를 4개로 줄이는 것은 실질적 이득이었고, 이후 반세기 동안 논문·교과서의 표준 표기가 됐다. 대가는 가독성이다 — DH 프레임은 "z축은 반드시 관절축, x축은 반드시 이웃 z축들의 공통 수선"이라는 규칙 때문에 링크의 물리적 형상과 동떨어진 곳에 배치되기 일쑤고, 관례의 변종(distal/proximal)까지 있어 논문 간 표가 호환되지 않는 사고가 흔하다. 현대 로봇 소프트웨어(URDF, SDF, MJCF)가 DH를 버리고 joint origin+axis 방식을 택한 이유가 그것이다: 파라미터 몇 개를 아끼는 것보다 CAD에서 읽은 좌표를 그대로 적을 수 있는 것이 압도적으로 가치 있다. 오늘의 실무 기준: **DH는 읽을 줄 알면 되고(교과서·논문·일부 산업용 로봇 매뉴얼이 여전히 쓴다), 쓸 때는 URDF 방식으로 쓴다.** 이 책도 이후 전부 origin+axis 방식이다.
:::

## FK 구현을 검증하는 세 가지 습관

FK 코드의 무서운 점은 **틀려도 그럴듯한 숫자가 나온다**는 것이다. 부호 하나, 축 하나가 틀린 FK는 크래시하지 않고, 어딘가 있을 법한 좌표를 내놓는다. 그 좌표가 서보 명령이 되어 다리가 이상하게 꺾이고 나서야 발견된다. 실기체에 올리기 전에 기계적으로 돌릴 수 있는 검산 세 개를 습관으로 만들어라 — 전부 이 절에서 이미 실측했다.

1. **영점 자세 검산.** 모든 관절 0°에서 말단 위치는 링크 기하로 암산이 된다. 우리 다리: $(253, 0, 0)$, 링크 길이 합. 이게 틀리면 상수 변환($T_{origin}$들)이 틀린 것이다 — 각도가 전부 0이라 회전은 용의선상에서 빠진다.
2. **단일 관절 스윕.** 관절 하나만 움직이면 말단은 그 관절축 중심의 **원호**를 그려야 한다. 이 환경 실측: coxa만 $-45°$~$45°$ 스윕하면 발끝의 수평 반지름은 114.282mm에서 스프레드 $5.7 \times 10^{-14}$(부동소수점 먼지), z는 정확히 $-168$ 고정. 반지름이 출렁이면 그 관절의 축 방향이, z가 움직이면 축 위치가 틀린 것이다 — **어느 관절을 스윕했느냐가 버그의 주소를 알려 준다.**
3. **작업공간 경계 검산.** 2링크 부분(femur-tibia)의 도달 거리는 $[|l_1 - l_2|, l_1 + l_2] = [48, 208]$mm 안에 있어야 한다. 실측으로 무릎을 $\pm 150°$(서보 가동 범위) 스윕하면 femur 관절 기준 거리는 $[71.05, 208]$ — 최대는 정확히 208에 닿고, 최소가 48이 아니라 71.05인 것도 맞다: 48은 무릎을 180°까지 완전히 접어야 나오는 값인데 서보가 거기까지 못 가기 때문이다. **기하학적 작업공간과 관절 한계가 만드는 실효 작업공간은 다르다** — 보행 궤적 생성기가 참조해야 하는 것은 후자다.

::: tip 검산을 GoogleTest로 박제하라
위 세 검산은 한 번 눈으로 확인하고 끝낼 것이 아니라 [7.6](#/googletest)의 테스트로 박아 두는 것이 맞다. `EXPECT_NEAR(leg_fk(0,0,0).translation().norm(), 253.0, 1e-9)` 한 줄이면 이후 누가 링크 길이 상수를 건드리든, 사슬 순서를 리팩터링하든 영점 검산이 CI에서 자동으로 돈다. 기하 코드는 리팩터링 사고가 조용히 나는 대표 지역이라 이 안전망의 가성비가 특히 높다. 부동소수점 비교는 `EXPECT_DOUBLE_EQ`가 아니라 허용 오차를 명시하는 `EXPECT_NEAR`를 쓴다 — 왜 그래야 하는지는 [9.8](#/numerics)에서 다룬다.
:::

## FK가 실전에서 뛰는 자리

이 절의 계산이 헥사포드에서 실제로 도는 자리를 짚고 끝내자.

**매 제어 주기, 다리 여섯 개.** 보행 컨트롤러는 주기마다 여섯 다리의 발끝 위치를 알아야 한다 — 스윙 중인 다리의 궤적 추종 오차, 지지 중인 다리의 하중 분배 전부 발끝 좌표 기준이다. 6다리 × 3관절 = 18관절의 FK가 매 주기 도는 것인데, 비용을 실측하면 걱정할 거리가 아니라는 것을 알 수 있다.

::: perf 3자유도 다리 FK 비용 실측 (g++ 13.3.0 / -O2 / Eigen 3.4.0 / x86-64, 1천만 회 반복 3회 중앙값)
- `leg_fk()` (Isometry3d 사슬 곱 + translation 추출): **138 ns/호출**
- 다리 6개 = **0.83 µs/주기** — 100Hz 제어 주기(10ms 예산)의 0.008%

절대값은 기기마다 다르지만 결론은 어디서나 같다: FK는 공짜에 가깝다. 로봇 코드에서 계산 예산을 잡아먹는 것은 FK가 아니라 IK의 수치 반복, 충돌 검사, 포인트클라우드다. "FK를 캐싱해야 하나" 같은 고민은 하지 마라 — 측정 없이 최적화하지 않는다는 [8.1](#/profiling)의 규율이 여기서는 "그냥 매번 계산하라"로 귀결된다.
:::

**leg odometry — FK를 거꾸로 읽는 응용.** 지지 중인 다리는 발끝이 지면에 붙어 있다(미끄러지지 않는 한 세계 좌표로 고정점이다). 그 상태에서 관절 각도가 변하면? FK가 준 발끝 위치는 몸통 기준 좌표이므로, "발끝(고정) 기준으로 몸통이 반대로 움직였다"고 읽을 수 있다 — 지지 다리들의 FK 변화량에서 몸통의 이동을 역산하는 것이 leg odometry다. GPS도 카메라도 없이 관절 엔코더만으로 "얼마나 걸어왔나"를 추정하는 이 기법은 FK의 정확도가 곧 오도메트리의 정확도가 되는 구조이고, 그래서 위의 검산 습관이 학습 예제가 아니라 항법 품질 문제가 된다. 추정치가 시간에 따라 흐르는 문제는 [9.6](#/state-estimation)의 센서 융합으로 잡는다.

**상류 소프트웨어는 FK를 자동으로 얻는다.** [10.7](#/tf2)의 robot_state_publisher, [11.4](#/moveit2)의 MoveIt2는 URDF에서 FK를 스스로 조립한다 — 당신이 `chain_fk()`를 다시 짤 일은 없다. 그런데도 이 절을 손으로 통과해야 하는 이유는, 그 도구들이 틀린 답을 줄 때(십중팔구 URDF의 origin·axis를 잘못 적은 것) 어디를 봐야 하는지 아는 사람과 모르는 사람의 디버깅 속도가 자릿수로 다르기 때문이다.

## 요약

- 순기구학(FK)은 관절 각도 → 말단 위치·자세의 함수다. 해가 **항상 유일하게 존재**한다 — 역방향인 IK([9.5](#/inverse-kinematics))는 해가 0개·여러 개·무한개일 수 있는 것과 대비된다.
- 2링크 평면 FK: $x = l_1 c_1 + l_2 c_{12}$, $y = l_1 s_1 + l_2 s_{12}$. 무릎 각이 femur에 **상대적**이라 세계 기준 각도는 $\theta_1 + \theta_2$로 누적된다. femur 80/tibia 128, $(35°, 55°)$에서 발끝 $(65.53, 173.89)$ 실측 — 위젯과 동일.
- 일반 원리는 **FK = 변환 곱의 사슬**: $T_{base \to foot} = T_{base \to j1}(\theta_1) \cdot T_{j1 \to j2}(\theta_2) \cdot T_{j2 \to foot}$. 관절이 늘면 인수가 늘 뿐이고, 각도 누적은 행렬 곱이 자동으로 처리한다. 3D 헥사포드 다리(coxa 45/femur 80/tibia 128)도 인수 여섯 개짜리 같은 사슬이다.
- URDF joint의 `<origin xyz rpy>` + `<axis>`는 정확히 $T_{parent \to child}(q) = T_{origin} \cdot R(\hat{a}, q)$다. 로봇 구조를 데이터로 내리면 FK 코드는 로봇과 무관한 루프 하나가 된다 — KDL·MoveIt2·robot_state_publisher의 구조.
- DH 파라미터는 관절당 4개 숫자로 체인을 기술하는 고전 관례다. 읽을 줄만 알면 된다 — 현대 도구(URDF/SDF)는 origin+axis 방식을 쓴다.
- FK 검증 3종 세트: **영점 자세**(링크 합 253mm 검산), **단일 관절 스윕**(원호 불변량 — 반지름 스프레드 $10^{-14}$ 실측), **작업공간 경계**($[|l_1-l_2|, l_1+l_2]$, 단 관절 한계로 실효 범위는 더 좁다 — $\pm150°$ 한계에서 최소 71mm 실측).
- FK 비용은 다리당 138ns 실측, 6다리 0.83µs — 제어 주기 예산에서 반올림 오차다. 헥사포드에서는 매 주기 18관절 FK가 돌고, 지지 다리 FK로 몸통 이동을 역산하는 leg odometry의 기반이 된다.

::: quiz 연습문제
1~3번은 개념·예측 문제, 4번은 네 컴퓨터에서 컴파일이 통과하고 수치가 재현되어야 성공인 실습, 5번은 위젯과 코드를 함께 쓰는 문제다.

1. FK는 해가 항상 정확히 하나인데 IK는 그렇지 않다. 두 사실을 "함수"라는 단어를 써서 각각 한 문장으로 설명하라. 그리고 2링크 다리에서 IK 해가 정확히 두 개가 되는 기하학적 이유를 위젯에서 본 것으로 답하라.

2. femur 80/tibia 128 다리에서 $\theta_1 = 90°$, $\theta_2 = 0°$일 때 발끝 좌표를 **계산하지 말고 기하로** 먼저 말하고, FK 식으로 검산하라. 이 자세는 작업공간의 어떤 특별한 위치에 있는가?

3. `chain_fk()`의 루프는 `T = T * joint_transform(...)`을 뿌리부터 말단 순서로 곱한다. 곱하는 순서를 말단부터 뿌리로 뒤집으면(결과적으로 $T_{j2} T_{j1} T_{j0}$) 일반적으로 왜 틀린 답이 나오는가? [9.2](#/rotations)에서 실측한 성질 하나로 답하라.

4. (실습) `leg3d.cpp`를 완결 프로그램으로 직접 타이핑해 컴파일하고, 세 가지 검산을 전부 코드로 수행하라: ① 영점 자세 norm이 253.0인가, ② coxa만 $-45°$~$45°$ 스윕할 때 수평 반지름과 z의 스프레드가 $10^{-10}$ 이하인가, ③ 무릎만 $\pm150°$ 스윕할 때 femur 관절 기준 거리의 최대가 208.0에 닿는가. 컴파일: `g++ -std=c++20 -Wall -Wextra -O2 -I/usr/include/eigen3 leg3d.cpp && ./a.out`. 성공 기준: 세 검산이 전부 통과하고, 본문 표의 발끝 좌표 네 세트가 재현된다.

5. (실습) 4번의 다리에서 tibia 길이를 128 → 120으로 바꾸면 영점 자세 norm과 작업공간 경계가 어떻게 변하는지 **먼저 예측해 적고**, 코드로 확인하라. 그다음 위젯의 파라미터가 80/128이라는 사실을 이용해 답하라: 위젯에서 $\theta_1 = 35°$, $\theta_2 = 55°$로 맞췄을 때와 네가 수정한 코드의 2링크 부분(coxa 제외, femur-tibia 평면)이 다른 값을 주는 이유는 무엇이고, 그 차이는 정확히 얼마인가?
:::

::: answer 해설
1. FK는 각도 조합 하나에 말단 위치를 하나 대응시키는 **함수**라 정의역 전체에서 답이 유일하다. IK는 그 함수의 **역방향**을 묻는 것인데, 함수는 일반적으로 단사가 아니므로 역이 함수라는 보장이 없다 — 상(작업공간) 밖의 점은 해가 0개, 상 안의 점은 원상이 여러 개일 수 있다. 2링크에서 두 해가 되는 이유: 고관절과 목표점을 잇는 선분을 기준으로 무릎이 위로 꺾인 자세와 아래로 꺾인 자세가 거울상으로 존재하고, 둘 다 같은 발끝을 준다(위젯의 "무릎 방향 전환" 버튼이 그 두 해 사이를 오간다).
2. 기하로: $\theta_2 = 0$이면 다리가 일직선이고, 그 직선이 $\theta_1 = 90°$로 수직이므로 발끝은 고관절 바로 "아래"(위젯 좌표로 y 양의 방향) 거리 208mm — $(0, 208)$이다. 검산: $x = 80\cos 90° + 128\cos 90° = 0$, $y = 80 + 128 = 208$. 이 자세는 도달 한계 원($r = l_1 + l_2 = 208$) **위의 점**이다 — 작업공간의 바깥 경계는 전부 "다리를 곧게 뻗은" 자세들이다.
3. 행렬 곱은 비가환이다 — [9.2](#/rotations)에서 $R_z R_x \ne R_x R_z$를 실측했다($(1,0,0)$이 순서에 따라 y축 방향 또는 z축 방향이 됐다). 변환 사슬의 곱 순서는 "부모 프레임에서 자식 프레임으로"라는 기하학적 관계의 순서 그 자체라서, 뒤집으면 각 관절의 회전이 엉뚱한 프레임에서 적용된다. (특수한 각도 조합에서 우연히 같아질 수는 있지만 일반적으로는 다른 변환이다.)
4. 이 환경 실측 기준 ① 253.0, ② 반지름 스프레드 $5.7 \times 10^{-14}$·z 스프레드 0, ③ 최대 208.0이 나온다. ②의 스프레드가 정확히 0이 아니라 $10^{-14}$대인 것은 부동소수점 반올림 때문이고, 그래서 성공 기준을 "== 0"이 아니라 "$10^{-10}$ 이하"로 준 것이다 — 기하 코드 테스트에 허용 오차가 필수인 이유([9.8](#/numerics))를 미리 체감하는 지점이다.
5. 영점 norm은 $45+80+120 = 245$, femur-tibia 작업공간은 $[|80-120|, 80+120] = [40, 200]$이 된다(바깥 경계는 8mm 줄고 안쪽 한계는 48 → 40으로 오히려 넓어진다 — 두 링크 길이가 가까울수록 몸쪽 깊이 접힌다). 위젯(80/128)과 수정 코드(80/120)의 $(35°, 55°)$ 발끝 차이: $\theta_1 + \theta_2 = 90°$라 tibia는 전부 y 성분이므로 x는 65.53으로 같고 y만 $173.89 - 8 = 165.89$ — 차이는 정확히 tibia 차이 8mm, 방향은 tibia가 가리키는 수직 방향이다. 링크 길이 오차가 발끝 오차로 **어느 방향으로** 전파되는지가 자세에 따라 달라진다는 것 — 실기체 캘리브레이션에서 중요한 감각이다.
:::

이 절의 코드는 전부 직접 쳐라. `fk2d.cpp`는 trig와 chain의 diff가 0인 것을 확인하고, `leg3d.cpp`는 4번 문제의 검산 세 개를 꼭 코드로 박아라. 시간이 남으면 `urdf_fk.cpp`의 `RevoluteJoint`에 관절 한계(min/max)를 추가하고 `joint_transform()`이 범위 밖 각도에서 클램프하도록 확장해 보라 — [9.5](#/inverse-kinematics)에서 IK 해를 고를 때 바로 쓴다. 기준 명령은 `g++ -std=c++20 -Wall -Wextra -O2 -I/usr/include/eigen3 main.cpp && ./a.out` (Eigen 설치는 [9.1](#/eigen) 참고).

**다음 절**: [9.5 역기구학](#/inverse-kinematics) — 이제 방향을 뒤집는다. "발끝을 여기 딛고 싶다"에서 관절 각도를 풀어내는 문제 — 해가 없기도, 두 개이기도 한 그 지뢰밭을 해석해와 수치해 양쪽으로 뚫는다. 헥사포드 다리 IK가 이 책 Part IX의 정점이다.
