# 9.6 상태 추정: 칼만 필터에서 EKF까지

::: lead
[9.4](#/forward-kinematics)의 leg odometry는 관절 각도로 몸통의 이동을 역산했다 — 그리고 마지막에 "추정치가 시간에 따라 흐른다"는 문제를 이 절로 미뤄 뒀다. 이 절이 그 빚을 갚는다. 센서는 노이즈가 있고 모델은 불완전하므로 "로봇이 지금 어디에 있는가"는 측정할 수 있는 값이 아니라 **추정해야 하는 값**이다. 예측과 측정이라는 두 정보원을 노이즈 크기에 반비례해 섞는 것이 칼만 필터의 전부다 — 1차원 다섯 줄 구현부터 시작해 Eigen 행렬 형태로 확장하고, 비선형 오도메트리를 위한 EKF까지 전부 고정 시드 시뮬레이션으로 실측한다. 유도는 하지 않는다. 각 수식이 "무슨 일을 하는가"를 숫자로 확인하는 것이 목표다.
:::

## 측정도 모델도 믿을 수 없다

로봇의 위치를 아는 방법은 두 가지뿐이고, 둘 다 결함이 있다.

**방법 1 — 측정한다.** 센서를 읽는다. 문제: 센서는 전부 노이즈가 있다. IMU 가속도를 두 번 적분한 위치는 바이어스 때문에 수십 초 만에 미터 단위로 흐르고, 관절 엔코더는 분해능만큼 양자화돼 있으며, GPS류 절대 측정은 수 cm~수 m가 튄다. 측정값을 그대로 위치로 쓰면 로봇의 "현재 위치"가 매 주기 튀어 다닌다 — 그 값을 [9.7](#/pid)의 제어기에 넣으면 튐이 그대로 모터 명령이 된다.

**방법 2 — 예측한다.** 물리 모델로 계산한다. "0.2 m/s로 걸으라고 명령했고 0.02초 지났으니 4 mm 전진했다." 문제: 모델은 불완전하다. 바퀴는 미끄러지고, 헥사포드의 발끝은 지면에서 밀리고, 모터는 명령을 정확히 따르지 않는다. 예측만 누적하는 것이 [9.4](#/forward-kinematics)에서 본 dead reckoning(추측 항법)이고, 오차가 **누적**된다 — 매 스텝의 작은 오차가 상쇄되지 않고 쌓여서, 아래 EKF 실측에서 보듯 60초 만에 3 m를 벗어난다.

핵심 관찰은 이것이다. **두 정보원의 결함은 성격이 반대다.** 측정은 튀지만 흐르지 않고(오차가 누적되지 않는다), 예측은 매끄럽지만 흐른다(누적된다). 서로의 약점을 서로의 강점이 지운다 — 남는 문제는 "얼마씩 섞을 것인가"뿐이고, 그 배합 비율을 매 스텝 자동으로 계산해 주는 기계가 칼만 필터다.

::: note 불확실성의 언어는 분산이다
"측정을 얼마나 믿을까"를 숫자로 만들려면 노이즈의 크기를 정량화해야 한다. 칼만 필터는 모든 불확실성을 **분산**(여러 값이면 공분산 행렬)으로 표현한다 — 센서 노이즈의 분산 $R$, 모델 오차의 분산 $Q$, 현재 추정치의 분산 $P$. "노이즈가 정규분포"라는 가정이 이론적 최적성의 조건이지만, 실무 감각은 더 단순하다: 분산은 신뢰도의 역수다. 분산이 작은 쪽을 더 믿는다 — 이 절의 모든 수식이 이 한 문장의 변주다.
:::

## 가중 평균과 칼만 이득

추정치 $\hat{x}$(분산 $P$)가 있고 새 측정 $z$(분산 $R$)가 들어왔다. 어떻게 섞나? 분산에 반비례하는 가중 평균이 답이다. 이를 "추정치에서 출발해 잔차를 일부 수용한다"는 꼴로 다시 쓰면:

$$\hat{x}_{new} = \hat{x} + K(z - \hat{x}), \qquad K = \frac{P}{P + R}$$

$z - \hat{x}$는 **잔차(innovation)** — "측정이 내 예상과 어긋난 정도"다. $K$가 **칼만 이득**이고, 정체는 신뢰도 배분율이다:

- $P \gg R$ (내 추정이 측정보다 훨씬 불확실): $K \to 1$, 측정을 거의 그대로 받아들인다.
- $P \ll R$ (내 추정이 측정보다 확실): $K \to 0$, 측정을 거의 무시한다.

그리고 섞고 나면 아는 것이 늘었으므로 분산이 준다: $P_{new} = (1-K)P$. 여기에 시간 축을 붙이면 필터가 완성된다. 매 주기 두 단계를 반복한다.

```text nolines
             ┌────────────────────┐
  x, P ────▶ │      predict       │   x += v*dt      (모델로 전진)
             │                    │   P += Q         (모델을 썼으니 불확실성 증가)
             └─────────┬──────────┘
                       ▼
             ┌────────────────────┐
  z ───────▶ │      update        │   K = P/(P+R)    (신뢰도 배분율 계산)
             │                    │   x += K*(z-x)   (잔차의 K배만 수용)
             └─────────┬──────────┘   P *= (1-K)     (측정을 얻었으니 감소)
                       ▼
                 x', P' ──▶ (다음 주기의 입력)
```

predict는 $P$를 키우고 update는 줄인다 — 필터가 돌면 $P$는 두 힘이 평형을 이루는 값으로, $K$는 그에 대응하는 고정 배분율로 수렴한다. 지금부터 이 전부를 코드로 확인한다.

## 1차원 칼만 필터: 전체 구현과 실측

50 Hz로 직선 보행하는 로봇 하나를 시뮬레이션한다. 명령 속도 0.2 m/s, 실제 움직임에는 미끄러짐(표준편차 5 mm/스텝), 위치 센서에는 노이즈(표준편차 5 cm). 난수는 `<random>`의 `mt19937`에 시드 42를 박아 **완전히 재현 가능**하다.

```cpp title="kf1d.cpp — 필터 본체는 다섯 줄이다" {28-33}
#include <cmath>
#include <cstdio>
#include <random>

int main() {
    std::mt19937 rng(42);                                // 고정 시드 -- 재현 가능
    std::normal_distribution<double> slip(0.0, 0.005);   // 실제 보행의 미끄러짐 [m/스텝]
    std::normal_distribution<double> sensor(0.0, 0.05);  // 위치 센서 노이즈 [m]

    const double dt = 0.02, v_cmd = 0.20;                // 50 Hz, 명령 속도 0.2 m/s
    const int    N  = 500;                               // 10초
    const double q  = 0.005 * 0.005;                     // 프로세스 노이즈 분산
    const double r  = 0.05 * 0.05;                       // 측정 노이즈 분산

    double x_true = 0.0;
    double x = 0.0, P = 1.0;                             // 추정치와 그 분산 (P0 크게 = 모른다)

    double se_meas = 0.0, se_kf = 0.0;
    for (int k = 1; k <= N; ++k) {
        x_true += v_cmd * dt + slip(rng);                // 실제 세계 (필터는 모른다)
        double z = x_true + sensor(rng);                 // 측정

        x += v_cmd * dt;                                 // [predict] 모델로 전진
        P += q;                                          //           불확실성 증가

        double K = P / (P + r);                          // [update] 칼만 이득
        x += K * (z - x);                                //          잔차의 K배만 수용
        P *= (1.0 - K);                                  //          불확실성 감소

        se_meas += (z - x_true) * (z - x_true);
        se_kf   += (x - x_true) * (x - x_true);
    }
    std::printf("RMSE raw measurement = %.4f m\n", std::sqrt(se_meas / N));
    std::printf("RMSE kalman estimate = %.4f m\n", std::sqrt(se_kf / N));
}
```

읽을 것이 세 가지다. **첫째, 필터가 이겼다.** 같은 센서, 같은 노이즈인데 원시 측정의 RMSE 5.25 cm가 필터 출력에서 1.64 cm로 — 3.2배 — 줄었다. 하드웨어를 바꾼 게 아니라 "명령 속도로 움직이고 있을 것"이라는 모델 지식을 측정에 섞었을 뿐이다. **둘째, $K$의 궤적이 신뢰도 배분 그 자체다.** 시작 시 $P_0 = 1$(아무것도 모른다)이라 첫 측정은 $K = 0.9975$로 통째로 수용되고, 추정이 쌓일수록 $K$가 떨어져 50스텝부터는 0.095에 고정된다 — 정착 후에는 매 측정의 9.5%만 받아들인다는 뜻이다. **셋째, $P$는 자기 오차의 예고다.** 정착한 $\sqrt{P} = 0.0154$ m는 실측 RMSE 0.0164 m와 거의 같다 — 필터는 자기가 얼마나 틀리고 있는지도 추정한다. 이 성질이 아래 공분산 건강 관리가 중요한 이유가 된다: $P$가 거짓말을 하면 $K$가 엉뚱한 배분을 한다.

:::

## 행렬로 확장: 다섯 행렬이 각자 맡는 일

실제 로봇 상태는 스칼라가 아니다 — 위치·속도·자세를 묶은 **상태 벡터** $x$다. 위 다섯 줄을 행렬로 일반화하면 칼만 필터의 표준형이 된다. 새로 등장하는 것은 없다 — 나눗셈이 역행렬이 됐을 뿐, 각 줄의 역할은 1차원과 자리까지 같다.

$$\text{predict:}\quad x = Fx, \qquad P = FPF^T + Q$$
$$\text{update:}\quad K = PH^T(HPH^T + R)^{-1}, \qquad x \mathrel{+}= K(z - Hx), \qquad P = (I - KH)P$$

각 행렬이 담는 것과 — 더 중요하게 — **그 값을 키우면 필터가 어떻게 변하는지**를 한 표로 정리한다. 아래 두 절에서 전부 실측으로 확인한다.

| 기호 | 이름 | 담는 것 | 10배 키우면 생기는 일 |
| --- | --- | --- | --- |
| $F$ | 상태 전이 | 물리 모델: 한 스텝 뒤 상태 예측 ("위치 += 속도×dt") | 크기 조절 대상이 아니라 **설계** 대상. 틀리면 잔차가 한쪽 부호로 쏠린다 |
| $H$ | 관측 모델 | 상태 → 측정값의 사영. "센서가 상태의 어느 성분을 보는가" | 역시 설계 대상. 단위 변환이 틀리면 필터 전체가 조용히 틀린다 |
| $Q$ | 프로세스 노이즈 | 모델을 얼마나 **불신**하는가 (미끄러짐, 외란, 모델링 안 한 가속) | 예측을 덜 믿음 → 측정 추종이 빨라지고 출력이 거칠어진다 |
| $R$ | 측정 노이즈 | 센서를 얼마나 불신하는가 (데이터시트의 노이즈 밀도가 출발점) | 측정을 덜 믿음 → 출력이 매끄러워지고 변화에 둔해진다 |
| $P$ | 추정 공분산 | 지금 추정치를 얼마나 불신하는가. 필터가 스스로 갱신한다 | 초기값을 키우면 초반에 측정을 통째로 수용 (위 실측의 $K=0.9975$) |

$F$와 $H$는 세계에 대한 사실이고, $Q$와 $R$은 신뢰에 대한 선언이며, $P$는 필터의 자기 평가다. 튜닝이라 부르는 행위는 $Q$와 $R$ 두 개만 만진다.

## 재지 않은 것이 추정된다: 위치만 보고 속도를

행렬 형태가 주는 첫 번째 보상은 성능이 아니라 **관측하지 않은 상태의 추정**이다. 상태를 (위치, 속도)로 잡고 센서는 위치만 재게 하자 — 속도계는 없다. Eigen으로 쓰면 predict/update가 수식 그대로 옮겨진다. [9.1](#/eigen)의 규칙대로 차원이 고정돼 있으므로 전부 고정 크기 타입이다 — 필터 한 사이클에 힙 할당 0회.

```cpp title="kf2d.cpp — 상태 (p, v), 관측은 p만 (핵심부)"
Eigen::Matrix2d F; F << 1, dt,                    // p += v*dt
                        0, 1;                     // v는 유지 (등속 모델)
Eigen::RowVector2d H; H << 1, 0;                  // 센서는 위치만 본다
Eigen::Vector2d G; G << 0.5 * dt * dt, dt;        // 가속 교란 -> (p, v) 반영 벡터
Eigen::Matrix2d Q = G * G.transpose() * (sigma_a * sigma_a);

x = F * x;                                        // predict
P = F * P * F.transpose() + Q;

double y = z - H * x;                             // update: 잔차
double S = (H * P * H.transpose())(0) + R;        // 잔차의 분산
Eigen::Vector2d K = P * H.transpose() / S;
x += K * y;
P = (Eigen::Matrix2d::Identity() - K * H) * P;
```

진짜 속도는 0.2 m/s에서 시작해 작은 가속 교란(σ=0.05 m/s²)으로 흔들리는 랜덤 워크다. 필터는 속도를 0으로 초기화하고, 위치 측정(σ=5 cm)만 500스텝 받는다.

```console
velocity RMSE (last 5s) = 0.0069 m/s   (never measured directly)
```

(시드 42 실측.) 속도계 없이 속도가 나왔다 — 정착 후 RMSE 0.0069 m/s, 진짜 속도의 3.5% 수준이다. 마법이 아니라 $F$의 결합이 한 일이다: 모델이 "위치 변화 = 속도×dt"라고 선언했으므로, 위치 측정의 **시간에 따른 경향**이 속도 추정으로 흘러 들어간다. t=0.1 s에는 측정 다섯 개의 노이즈에 속아 부호까지 틀리지만(−0.28), 0.5초 만에 참값에 붙고 $\sqrt{P_{vv}}$도 0.62 → 0.0084로 함께 떨어진다 — 필터가 "이제 속도를 안다"고 스스로 보고하는 것이다. 이 구조가 로봇 상태 추정의 상수 패턴이다: 자세는 직접 못 재도 IMU와 leg odometry의 결에서 나오고, 자이로 바이어스는 어떤 센서에도 안 잡히지만 상태 벡터에 넣어 두면 추정된다.

::: interview "칼만 필터를 직관적으로 설명해 보라"
로보틱스·자율주행 면접의 개막 질문이다. 수식 암송이 아니라 이해를 본다. 답변 뼈대: ① 로봇의 상태는 측정이 아니라 추정의 대상이다 — 센서는 튀고(측정 노이즈), 모델은 흐른다(누적 오차). ② 칼만 필터는 두 정보원을 **분산에 반비례하는 가중 평균**으로 섞는다 — predict가 모델로 전진하며 불확실성을 키우고, update가 측정으로 보정하며 줄인다. ③ 칼만 이득 $K$가 그 배분율이다: $P/(P+R)$, "내 추정과 센서 중 누가 덜 불확실한가". 필터 초반엔 $K \approx 1$로 측정을 통째로 받고, 정착하면 일정 비율로 수렴한다. ④ 위치만 관측해도 상태 전이 모델의 결합을 통해 속도처럼 재지 않은 상태까지 추정된다 — 여기까지 말하면 충분하다. "1차원 구현이 다섯 줄"이라며 predict 두 줄, update 세 줄을 직접 써 보이면 확실히 통과다.
:::

## Q/R 튜닝: 비율이 필터의 성격이다

$Q$와 $R$을 잘못 잡으면 어떻게 되나 — 같은 시드, 같은 측정 수열로 배율만 바꿔 4케이스를 쟀다. 시나리오에 함정을 하나 심었다: t=10 s에 보행 속도가 0.2 → 0.5 m/s로 바뀐다(등속 모델이 모르는 기동이다). 표의 마지막 열은 속도 변경 직후 3초간의 최대 위치 오차 — "필터가 변화를 따라잡는 속도"다.

| 케이스 | 위치 RMSE | 속도 RMSE | 기동 후 최대 오차 |
| --- | --- | --- | --- |
| 원시 측정 (참고) | 0.0525 m | — | — |
| 기준 (Q, R) | 0.0382 m | 0.0755 m/s | 0.128 m |
| **Q ×10** | 0.0182 m | 0.0611 m/s | 0.080 m |
| **Q ÷10** | 0.0918 m | 0.0970 m/s | 0.239 m |
| **R ×10** | 0.0918 m | 0.0944 m/s | 0.239 m |
| **R ÷10** | 0.0183 m | 0.0843 m/s | 0.080 m |

(시드 42 실측.) 표가 두 가지를 증언한다.

**첫째, Q×10과 R÷10이 사실상 같은 필터다** (0.0182 vs 0.0183 m — 기동 후 최대 오차는 소수 셋째 자리까지 동일). Q÷10과 R×10도 마찬가지다(0.0918 = 0.0918). 칼만 이득이 $P$와 $R$의 **비율**로 정해지므로, 정착한 필터의 성격을 결정하는 것은 Q와 R의 절대값이 아니라 $Q/R$ 비율이다 — "모델과 센서 중 누구를 상대적으로 더 믿는가" 하나가 실질 튜닝 노브다.

**둘째, "정직한" Q가 항상 이기지 않는다.** 기준 케이스의 Q는 시뮬레이션의 실제 교란과 일치하는 정직한 값인데, Q×10에 완패했다(0.0382 vs 0.0182 m). 이유는 기동이다 — 등속 모델은 t=10 s의 속도 변경을 표현할 수 없고, 그 모델 결함을 흡수하는 유일한 통로가 Q다. **Q는 "확률적 노이즈의 크기"가 아니라 "모델이 모르는 모든 것의 예산"으로 읽어야 한다.** 미끄러짐, 기동, 모델링 안 한 동역학 — 전부 Q행이다.

::: tip 튜닝의 실전 시작점
R은 측정할 수 있다: 로봇을 정지시켜 놓고 센서를 수천 샘플 기록해 분산을 구하면 그게 R이다(데이터시트 노이즈 밀도는 출발점, 실측이 정답). Q는 측정할 수 없으므로 — 위에서 봤듯 모델 결함의 예산이라 — R을 고정하고 Q를 로그 스케일로(×10씩) 움직이며 위 표 같은 트레이드오프 곡선을 그리는 것이 정석이다. 판단 기준도 표의 두 열 그대로다: 정상 구간의 RMSE(매끄러움)와 기동 추종(반응성) 중 어느 쪽이 아픈 애플리케이션인가. 제어 루프에 들어가는 추정치는 대개 반응성 쪽이 아프다 — 지연된 추정은 [9.7](#/pid)에서 볼 위상 지연이 되어 제어를 불안정하게 만든다.
:::

::: warn 잔차를 감시하지 않는 필터는 자기 오류를 모른다
필터가 잘 도는지 확인하는 창은 잔차 $z - Hx$다. 건강한 필터의 잔차는 평균 0 주변에서 분산 $S = HPH^T + R$ 안에 들어온다. 잔차가 한쪽 부호로 쏠리면 F나 H가 틀린 것이고(모델 바이어스), 잔차가 $\sqrt{S}$의 3배를 넘는 일이 잦으면 Q/R 선언이 현실과 안 맞는 것이다. robot_localization을 포함한 실전 구현들이 이 검사를 마할라노비스 거리 게이트로 자동화한다 — 튀는 측정 하나가 상태를 끌고 가는 사고를 막는 문지방이다.
:::

## 비선형의 벽: 자코비안으로 국소 선형화 — EKF

여기까지의 필터는 $x = Fx$ — 상태가 **행렬 곱 한 번**으로 전진한다는 가정 위에 서 있다. 2D 로봇의 오도메트리는 이 가정을 즉시 깬다. 상태 $(x, y, \theta)$에 속도 $v$·요레이트 $\omega$ 입력이 들어오는 유니사이클 모델은:

$$x \mathrel{+}= v\,dt\cos\theta, \qquad y \mathrel{+}= v\,dt\sin\theta, \qquad \theta \mathrel{+}= \omega\,dt$$

$\cos\theta$가 문제다 — 위치 변화가 상태 $\theta$의 **비선형 함수**라서, 이 관계를 표현하는 상수 행렬 $F$는 존재하지 않는다. [9.2](#/rotations)에서 회전이 선형대수의 특별 취급을 받았던 것과 같은 뿌리다: 자세가 끼는 순간 세계는 비선형이 된다.

EKF(확장 칼만 필터)의 답은 실용주의다. **상태는 비선형 모델 $f$로 그대로 전진시키고, 공분산 $P$만은 현재 추정치 주변에서 $f$를 1차 근사한 자코비안 $F = \partial f/\partial x$로 전파한다.** 자코비안은 [9.5](#/inverse-kinematics)의 수치 IK가 쓰는 그 행렬 — 비선형 함수의 국소 기울기 행렬이다. 구조는 KF와 완전히 같고, $F$와 $H$가 "상수 행렬"에서 "매 스텝 현재 상태에서 다시 계산하는 자코비안"으로 바뀌는 것이 전부다.

```cpp title="ekf.cpp — 유니사이클 EKF의 predict (핵심부)"
double c = std::cos(x[2]), s = std::sin(x[2]);
x[0] += vm * dt * c;                              // 상태는 비선형 모델 f 그대로
x[1] += vm * dt * s;
x[2] += wm * dt;

Eigen::Matrix3d F = Eigen::Matrix3d::Identity();  // 공분산은 f의 자코비안으로
F(0, 2) = -vm * dt * s;                           // d(x)/d(theta)
F(1, 2) =  vm * dt * c;                           // d(y)/d(theta)
P = F * P * F.transpose() + Q;                    // update는 KF와 한 글자도 안 다르다
```

시뮬레이션: 완만한 원호를 60초 걷는다. 오도메트리(50 Hz)는 속도 σ=0.02 m/s, 요레이트 σ=0.15 rad/s로 노이즈가 심하고(자이로 없이 다리 미끄러짐이 요레이트를 흔드는 상황), 1 Hz로 σ=10 cm짜리 GPS류 절대 위치가 들어온다. 네 명의 경쟁자를 같은 시드로 돌렸다 — 예측만(dead reckoning), 측정만(GPS 원시값), EKF, 그리고 "자코비안을 초기 자세 $\theta = 0$에서 한 번만 계산하고 고정한" 대조군.

```console
$ ./ekf
60 s, odometry @50Hz + GPS @1Hz (sigma 0.10 m), seed 42
GPS raw fix RMSE          = 0.1488 m (60 fixes)
dead reckoning pos RMSE   = 1.6839 m, final err = 3.2736 m
EKF pos RMSE              = 0.0695 m, final err = 0.0444 m
frozen-lin KF pos RMSE    = 4.6694 m, final err = 10.2217 m
EKF heading RMSE          = 0.0612 rad (heading never measured)
```

(시드 42 실측.) 네 줄이 이 절 전체의 요약이다. **예측만 쓰면 흐른다** — dead reckoning은 60초 만에 3.27 m 밖이고, 계속 걸으면 계속 벌어진다. **측정만 쓰면 튄다** — GPS 원시값은 매 픽스 15 cm 언저리에서 영원히 튄다. **섞으면 둘 다 이긴다** — EKF는 RMSE 6.9 cm로 GPS 원시값보다 2.1배, dead reckoning보다 24배 정확하고, 오차가 누적되지도 않는다(최종 오차 4.4 cm). 그리고 위치만 관측했는데 **heading이 0.06 rad 정확도로 추정됐다** — 앞 절의 "재지 않은 상태" 패턴이 비선형에서도 그대로 성립한다.

마지막 줄이 EKF의 존재 이유다. 선형화를 고정한 대조군은 RMSE 4.7 m — dead reckoning보다도 나쁘다. 로봇이 돌아서 $\theta$가 커질수록 "$\theta = 0$ 근처"라는 가정이 거짓이 되고, 매초 들어오는 GPS 보정으로도 못 구한다. **EKF의 핵심은 자코비안을 쓴다는 것이 아니라, 매 스텝 현재 추정치에서 다시 선형화한다는 것이다.** 선형 근사는 국소적으로만 유효하고, 그래서 근사점이 상태를 따라다녀야 한다.

::: deep EKF의 한계와 그 너머
EKF의 1차 근사는 비선형성이 한 스텝 안에서 완만할 때만 유효하다 — 초기 추정이 크게 틀렸거나(전역 위치 초기화), 한 스텝에 자세가 수십 도 돌거나, 분포가 다봉이 되면(복도가 여러 개인 건물에서의 위치 추정) 발산한다. 그 너머가 UKF(시그마 포인트로 분포 자체를 통과시킨다)와 파티클 필터(분포를 샘플 무리로 표현 — 다봉 가능)다. 자세를 쿼터니언으로 들면 "4성분·norm 1 제약"이 공분산과 충돌하는 문제가 추가되는데, 실전 해법은 오차 상태(error-state) EKF — 상태는 쿼터니언으로, 공분산은 3차원 오차각으로 드는 구성이다. 그리고 [9.2](#/rotations)의 이중 덮개 경고가 여기서 실전이 된다: 쿼터니언 잔차를 계산하기 전에 $q_1 \cdot q_2 < 0$이면 부호를 뒤집어야 한다 — 안 그러면 5° 차이가 355° 차이로 계산돼 필터가 튄다.
:::

## 공분산 행렬의 건강 관리

$P$는 정의상 대칭이고 양의 정부호여야 한다(분산이 음수인 방향이 존재할 수 없다). 그런데 갱신식 $(I - KH)P$는 수학적으로만 대칭을 보존한다 — 부동소수점 반올림은 봐주지 않는다. float 2상태 필터를 100만 스텝 돌리며 $|P - P^T|$의 최대 원소를 추적했다:

```console
$ ./phealth
float 2x2, 1e6 steps, max |P - P^T| element:
  naive (I-KH)P        : 9.313e-10   (rel to P(0,0)=6.972e-05)
  naive + symmetrize   : 0.000e+00
  Joseph form          : 2.794e-09
```

(실측.) 비대칭이 $P$ 자체의 만분의 일 수준에서 생겼다 — 2×2에서는 무해하지만, 상태가 15차원이고([9.8](#/numerics)에서 볼) 조건수가 나쁘면 이 부스러기가 자라서 $P$의 고윳값을 음수로 밀 수 있고, 음의 분산은 칼만 이득의 부호를 뒤집어 필터를 발산시킨다. 방어는 한 줄이다 — 갱신 후 강제 대칭화:

```cpp
P = 0.5 * (P + P.transpose()).eval();   // .eval()은 9.1의 앨리어싱 규칙
```

(실측 표의 둘째 줄 — 비대칭이 정확히 0이 된다. `.eval()`이 없으면 [9.1](#/eigen)에서 재현한 그 앨리어싱 함정이다.) 더 근본적인 처방이 **조셉 형식(Joseph form)** $P = (I-KH)P(I-KH)^T + KRK^T$이다 — "대칭행렬을 양쪽에서 같은 행렬로 감싼 것의 합" 꼴이라 반올림이 있어도 양의 정부호가 구조적으로 보존된다. 연산이 몇 배 비싸므로 기본은 "naive + 대칭화", 필터가 이유 없이 발산하는 증상이 보이면 조셉 형식이 순서다.

::: danger 잘 맞는 센서가 필터를 죽인다 — 캔슬레이션
조셉 형식이 이론적 결벽이 아니라는 것을 float 한 줄로 재현할 수 있다. 사전 분산 $P = 100$(위치를 거의 모름)에 초정밀 측정 $R = 10^{-6}$이 들어오면:

```console
float, P=100, R=1e-6 (one update):
  exact (double)  P+ = 1.000000e-06
  naive  (float)  P+ = 0.000000e+00   (K=1.000000000, 1-K=0.000e+00)
  Joseph (float)  P+ = 1.000000e-06
```

(실측.) float에서 $K = P/(P+R)$이 정확히 1.0으로 반올림돼 $1-K = 0$, naive 갱신은 $P^+ = 0$을 만든다 — 필터가 "위치를 무한히 확신한다"고 선언한 것이고, 이후 칼만 이득이 0에 붙어 **모든 측정에 귀를 닫는다.** 조셉 형식은 같은 float로 정답 $10^{-6}$을 낸다 — $K^2R$ 항이 빼기가 아니라 더하기라 캔슬레이션이 없다. "크기가 크게 다른 두 수의 뺄셈이 정보를 지운다"는 이 병리의 일반형은 [9.8](#/numerics)에서 정면으로 다룬다.
:::

## robot_localization: 이 절이 ROS 2에서 도는 곳

이 절에서 만든 구조가 그대로 패키지로 존재한다. ROS 2의 `robot_localization`이 제공하는 `ekf_node`는 15차원 상태(위치 3 + 자세 3 + 선속도 3 + 각속도 3 + 선가속 3)의 EKF다. yaml 설정에서 오도메트리 토픽·IMU 토픽을 나열하고 각 센서가 상태의 어느 성분을 관측하는지 불리언 행렬로 선언하면 — 그게 정확히 $H$를 채우는 일이다 — 융합된 오도메트리와 [10.7](#/tf2)의 `odom → base_link` 변환을 발행한다. 설정 파일의 `process_noise_covariance`가 이 절의 $Q$이고, 각 센서 메시지에 실려 오는 covariance 필드가 $R$이다 — 위 튜닝 표의 감각이 yaml 파라미터의 감각으로 직결된다.

Hexpider의 추정 문제가 정확히 이 틀이다: [9.4](#/forward-kinematics)의 leg odometry는 흐르는 예측원이고(발끝 미끄러짐 = 프로세스 노이즈), IMU는 튀는 측정원이다 — 둘을 이 절의 필터로 융합한 것이 몸통 상태 추정이 되고, 전체 그림에서의 위치는 [12.5](#/hexpider-map)에 매핑돼 있다. 비용 걱정은 없다. 이 절의 3상태 EKF 한 사이클(predict + update + 대칭화)을 실측하면 **285 ns**(g++ 13.3.0 / -O2, 100만 회 평균 3회 중앙값)다 — 전부 고정 크기 Eigen이라 힙 할당 0회, [6.8](#/realtime)의 제어 루프 규칙 안에서 1 kHz로 돌려도 주기 예산의 0.03%다.

::: note 직접 짤 일과 안 짤 일
평지 주행 로봇의 오도메트리+IMU 융합이라면 `ekf_node` 설정으로 끝난다 — 검증된 구현을 두고 다시 짜는 것은 [9.1](#/eigen)에서 역행렬을 손으로 짜지 않던 것과 같은 이유로 피한다. 직접 구현이 정당한 것은 상태나 측정 모델이 표준 틀 밖일 때다 — 다리 접촉 상태가 스위칭되는 leg odometry의 공분산을 접촉 수에 따라 바꾸는 것 같은 도메인 로직이 그 예다. 어느 쪽이든 이 절의 어휘($Q$, $R$, 잔차, 자코비안) 없이는 설정 파일 한 줄도 읽을 수 없다.
:::

## 요약

- 로봇의 상태는 측정값이 아니라 추정값이다. 측정은 튀지만 안 흐르고(IMU·GPS 노이즈), 예측은 매끄럽지만 흐른다(미끄러짐 누적 — dead reckoning 60초에 3.27 m 실측). 칼만 필터는 둘을 분산에 반비례하는 가중 평균으로 섞는다.
- 1차원 필터는 다섯 줄이다: predict(`x += v*dt; P += q`)와 update(`K = P/(P+r); x += K*(z-x); P *= 1-K`). 실측에서 원시 측정 RMSE 5.25 cm → 필터 1.64 cm, 3.2배 개선. 칼만 이득 $K$는 신뢰도 배분율이고, $P_0$가 크면 첫 측정을 통째로 수용($K=0.9975$)한 뒤 정착값(0.095)으로 수렴한다.
- 행렬 형태의 다섯 행렬: $F$(모델)·$H$(관측 사영)는 설계 대상, $Q$(모델 불신)·$R$(센서 불신)는 튜닝 대상, $P$(자기 평가)는 필터가 스스로 갱신. 위치만 관측해도 $F$의 결합으로 속도가 추정된다 — 속도 RMSE 0.0069 m/s 실측, 속도계 없이.
- 튜닝의 실체는 $Q/R$ 비율 하나다 — Q×10과 R÷10이 소수 셋째 자리까지 같은 필터였다(실측). Q는 확률 노이즈가 아니라 "모델이 모르는 모든 것의 예산"이라, 기동이 있는 현실에서는 정직한 Q보다 부풀린 Q가 이겼다(0.0382 → 0.0182 m).
- EKF는 상태를 비선형 모델로 전진시키고 공분산만 자코비안으로 전파하는 KF다 — 구조는 같고 $F$/$H$가 매 스텝 다시 계산될 뿐. 융합 결과(6.9 cm)는 GPS만(14.9 cm)보다도, 예측만(1.68 m)보다도 정확했고, 선형화를 고정하면 4.7 m로 무너졌다 — 핵심은 재선형화다.
- $P$는 대칭·양의 정부호를 유지해야 한다. `P = 0.5*(P + P.transpose()).eval()` 한 줄이 기본 위생이고, float에서 $P=100, R=10^{-6}$ 한 번의 갱신으로 naive 식이 $P^+=0$(측정에 귀를 닫는 필터)을 만드는 것을 실측했다 — 조셉 형식이 처방이다([9.8](#/numerics)로 이어진다).
- ROS 2 `robot_localization`의 `ekf_node`가 이 구조의 15차원 구현이다. yaml의 센서 설정이 $H$, `process_noise_covariance`가 $Q$, 메시지의 covariance가 $R$ — Hexpider의 leg odometry + IMU 융합이 이 필터의 실전형이다. 3상태 EKF 한 사이클 285 ns 실측, 힙 할당 0회.

::: quiz 연습문제
1~3번은 개념·판단, 4번은 코드 작성형, 5번은 코드 리뷰형이다.

1. 1차원 실측에서 칼만 이득은 $K=0.9975$로 시작해 0.095로 정착했다. 두 값 각각이 "필터가 무엇을 믿고 있는가"로 무엇을 말하는지 설명하고, 정착 후에도 $K$가 0으로 떨어지지 않는 이유를 predict 단계에서 찾아라.

2. 동료가 "필터 출력이 너무 튄다"며 R을 10배 키웠더니 매끄러워졌다고 만족한다. 이 튜닝의 숨은 청구서를 이 절의 실측 표에서 찾아 지적하라. 같은 매끄러움을 얻는 다른 방법이 있는가?

3. 위치만 관측하는 2상태 필터에서 속도가 추정되는 이유를 $F$ 행렬의 구조로 설명하라. 반대로, 상태에 요레이트 자이로의 바이어스를 추가한 필터에서 로봇이 **정지해 있는 동안**에는 위치 관측만으로 바이어스 추정이 개선되지 않는 이유는 무엇이라고 예상되는가 — 관측과 상태를 잇는 경로의 관점에서 답하라.

4. (실습, 코드 작성형) 본문 `kf1d.cpp`를 보지 말고 1차원 칼만 필터를 처음부터 작성하라 — `mt19937 rng(42)`, 미끄러짐 σ=0.005, 센서 σ=0.05, dt=0.02, 500스텝, 본문과 같은 q/r. 원시 측정 RMSE와 필터 RMSE를 둘 다 출력한다. 성공 기준: `g++ -std=c++20 -Wall -Wextra -O2 kf.cpp && ./a.out`이 경고 없이 컴파일되고, libstdc++ 기준 측정 RMSE ≈ 0.0525, 필터 RMSE ≈ 0.0164가 재현되며, 시드를 바꿔도 필터 RMSE < 측정 RMSE가 유지될 것. 그다음 P의 초기값을 1.0에서 0.0001로 바꿔 첫 50스텝의 오차가 어떻게 변하는지 관찰하라.

5. (코드 리뷰) 유니사이클 EKF에서 "자코비안 계산이 비싸다"며 시작 시 한 번 계산한 F를 캐시해 재사용하는 코드를 발견했다. 이 최적화의 문제를 본문 실측 수치로 지적하고, 진짜 비용 절감이 필요하다면 무엇을 먼저 재야 하는지 말하라.
:::

::: answer 해설
1. $K=0.9975$: $P_0=1$이 $R=0.0025$보다 압도적으로 커서 "내 추정보다 측정이 훨씬 믿을 만하다" — 첫 측정을 사실상 그대로 수용한다. $K=0.095$: 정착 후에는 추정 분산이 측정 분산보다 작아져 매 측정의 9.5%만 섞는다. $K$가 0으로 안 떨어지는 이유는 predict의 `P += q` — 매 스텝 모델 불확실성이 주입되므로 $P$는 update가 줄이는 양과 평형을 이루는 값 아래로 내려가지 못하고, 따라서 $K$도 하한을 갖는다. $q=0$이면 $K \to 0$으로 필터가 점점 귀를 닫는다.

2. 실측 표에서 R×10의 청구서: 위치 RMSE가 0.0382 → 0.0918 m로 나빠지고(원시 측정 0.0525보다도 나쁘다 — 필터를 달고 더 틀리는 상태), 기동 후 최대 오차가 0.128 → 0.239 m로 늘었다. 매끄러움은 지연으로 산 것이다 — 제어에 들어가면 위상 지연이 된다. "같은 매끄러움"이 정말 필요하면 R이 아니라 필터 뒤에 별도 저역 필터를 두거나, 상태에 가속을 추가해 모델 자체를 좋게 만드는 것이 정도다. 그리고 R은 애초에 튜닝 노브가 아니라 정지 로봇으로 실측하는 값이다.

3. $F$의 (0,1) 원소 $dt$가 "다음 위치 = 현재 위치 + 속도×dt"를 선언하므로, 위치 잔차가 $K$를 통해 속도 성분에도 배분된다 — 위치 측정의 시간 경향이 속도로 흘러드는 경로가 $F$에 있다. 자이로 바이어스: 바이어스는 각속도 적분을 거쳐 자세로, 자세는 이동 방향을 거쳐 위치로 이어지는데, 정지 중에는 "이동"이라는 다리가 끊긴다 — 바이어스가 관측(위치)에 도달하는 경로가 움직임 없이는 사라지므로 추정이 정체된다. 관측 불가 상태의 추정은 공짜가 아니라 상태→관측 경로가 살아 있을 때만 작동한다(관측 가능성).

4. 구조는 본문과 같다: predict 두 줄, update 세 줄, RMSE 누적 두 줄. libstdc++에서 시드 42면 0.0525/0.0164가 자리수까지 재현된다(다른 표준 라이브러리면 값은 다르되 비율은 유사). $P_0 = 0.0001$로 바꾸면 첫 $K$가 0.997이 아니라 0.04 수준으로 시작해 — 필터가 "이미 위치를 안다"고 거짓 선언한 상태라 — 초기 오차(추정 0 vs 실제 이동 시작)를 수십 스텝에 걸쳐 천천히만 흡수한다. 초기 P는 겸손하게(크게) 잡는 것이 안전한 이유다.

5. 본문의 frozen-lin 대조군이 정확히 그 코드다 — RMSE 4.67 m로, 보정을 아예 안 하는 dead reckoning(1.68 m)보다도 나빴다. 선형 근사는 현재 추정치 근처에서만 유효하므로 $\theta$가 초기값에서 멀어질수록 공분산 전파가 거짓이 되고, 잘못된 $P$는 잘못된 $K$를 만든다. 비용 논거도 성립하지 않는다: 자코비안 갱신을 포함한 EKF 한 사이클 전체가 285 ns 실측이다 — sin/cos 두 번과 원소 두 개 대입을 아끼는 것은 측정 없이 추측으로 최적화하지 말라는 [8.1](#/profiling)의 규칙 위반이기도 하다.
:::

이 절의 시뮬레이션은 전부 네 IDE에서 재현해라. 명령은 하나다: `g++ -std=c++20 -Wall -Wextra -O2 -I/usr/include/eigen3 main.cpp -o main && ./main` (1차원 필터는 Eigen이 없어도 된다). `kf1d.cpp`를 직접 치고 시드를 42, 7, 2026으로 바꿔 "수치는 달라져도 필터가 항상 이긴다"를 확인하고, 튜닝 실험에서는 Q×10/R÷10 두 케이스의 출력이 정말 같은 자리까지 일치하는지 눈으로 봐라. EKF에서는 GPS 주기를 1 Hz에서 0.2 Hz(`k % 250 == 0`)로 늦춰 보라 — 보정 사이 구간에서 오차가 자라다가 픽스마다 꺾이는 톱니 모양이 dead reckoning과 융합의 관계를 그대로 보여준다.
