# 5.3 deque, list, array

::: lead
[5.1 vector](#/vector)의 `push_back`은 뒤쪽에서 상각 상수 시간이다. 그런데 앞쪽에 넣어야 한다면 얘기가 다르다 — `vector`의 메모리는 통째로 하나의 연속 블록이라, 맨 앞에 원소 하나를 끼워 넣으려면 기존 원소 전부를 한 칸씩 뒤로 밀어야 한다. IMU 샘플을 매 주기 앞에 쌓고 오래된 샘플을 뒤에서 버리는 슬라이딩 윈도우를 `vector`로 짜면, 원소 수가 늘수록 그 밀어내기 비용이 선형으로, 반복 횟수까지 곱하면 제곱으로 불어난다. 이 절은 그 문제를 실제로 재서 보여준 뒤, `vector`가 못 하는 일을 대신하는 세 컨테이너 — 양끝이 다 빠른 `deque`, 중간 삽입/삭제가 빠른 `list`, 크기를 컴파일 타임에 못박는 `array` — 가 내부적으로 무엇을 다르게 하길래 그런 성질을 갖는지를 다룬다.
:::

## 1. vector가 못 하는 것: 앞쪽 삽입의 실제 비용

말로 하지 말고 재보자. `vector`의 `insert(begin(), x)`와 `deque`의 `push_front(x)`를 각각 3만 번, 6만 번 반복했다.

```cpp title="front_insert.cpp — vector 맨 앞 삽입 vs deque push_front"
#include <chrono>
#include <deque>
#include <vector>

int main() {
    for (int N : {30000, 60000}) {
        std::vector<int> v;
        auto t0 = std::chrono::steady_clock::now();
        for (int i = 0; i < N; ++i) v.insert(v.begin(), i);
        auto t1 = std::chrono::steady_clock::now();
        // ms 출력
    }
    for (int N : {30000, 60000}) {
        std::deque<int> d;
        auto t0 = std::chrono::steady_clock::now();
        for (int i = 0; i < N; ++i) d.push_front(i);
        auto t1 = std::chrono::steady_clock::now();
        // ms 출력
    }
}
```

```console
$ g++ -std=c++20 -Wall -Wextra -O2 front_insert.cpp -o front_insert
$ ./front_insert
N=30000 vector.insert(begin()): 28.4299 ms
N=60000 vector.insert(begin()): 127.648 ms
N=30000 deque.push_front(): 0.11156 ms
N=60000 deque.push_front(): 0.161167 ms
```

(g++ 13.3.0 / `-O2` / Linux x86-64 실측. 반복 실행에서 vector 쪽은 28~35ms, 116~130ms 사이로 흔들렸다.) `vector`는 `N`이 두 배(3만→6만)가 되자 시간이 4.5배(28ms→128ms)로 뛰었다 — $O(n^2)$의 흔적이다. `deque`는 같은 배율에서 0.11ms→0.16ms, 거의 그대로다. **`vector`의 맨 앞 삽입은 원소 하나당 나머지 전체를 밀어야 해서 $O(n)$이고, 그걸 $n$번 반복하면 $O(n^2)$이 된다. `deque`의 `push_front`는 매번 상수 시간이라 총 비용이 $O(n)$이다.** 이 차이가 이 절 전체의 출발점이다 — 왜 `deque`는 되고 `vector`는 안 되는가.

## 2. deque 내부 구조: 연속 블록이 아니라 청크의 배열

`deque`가 앞뒤 모두에서 빠른 이유는 **메모리를 하나의 연속 블록으로 관리하지 않기 때문**이다. 대신 고정 크기 청크(chunk) 여러 개를 만들고, 그 청크들의 주소를 담은 작은 포인터 배열("맵")로 전체를 엮는다.

```text nolines
deque 내부 구조 (libstdc++ 구현 기준)

map (포인터 배열, 필요하면 재할당됨)
  +------+------+------+------+
  | ptr0 | ptr1 | ptr2 | ptr3 | ...
  +--|---+--|---+--|---+--|---+
     v      v      v      v
   chunk0  chunk1  chunk2  chunk3      <- 각 청크는 고정 크기, 내부는 연속
   [.....] [.....] [.....] [.....]

  front 쪽에 넣을 공간이 없으면 -> map 앞에 새 chunk 포인터 하나 추가
  back  쪽에 넣을 공간이 없으면 -> map 뒤에 새 chunk 포인터 하나 추가
  (기존 chunk 안의 원소는 단 하나도 옮기지 않는다)
```

이걸 눈으로만 믿지 말고 실제 주소를 찍어서 청크 경계를 찾아본다. `int` 2000개를 `push_back`한 뒤, 인접 인덱스의 주소 차이가 `sizeof(int)`(4바이트)가 아닌 지점을 전부 찾으면 그게 청크 경계다.

```cpp title="deque_chunk.cpp — 인접 원소 주소를 비교해 청크 경계를 찾는다"
std::deque<int> d;
for (int i = 0; i < 2000; ++i) d.push_back(i);

for (int i = 1; i < 2000; ++i) {
    auto p0 = reinterpret_cast<std::uintptr_t>(&d[i - 1]);
    auto p1 = reinterpret_cast<std::uintptr_t>(&d[i]);
    if (p1 - p0 != sizeof(int)) {
        // 청크 경계 -- 직전 청크 길이 출력
    }
}
```

```console
$ g++ -std=c++20 -Wall -Wextra -O2 deque_chunk.cpp -o deque_chunk
$ ./deque_chunk
청크 경계: 인덱스 127 -> 128 (직전 청크 길이 128개 원소, 간격 20 바이트)
청크 경계: 인덱스 255 -> 256 (직전 청크 길이 128개 원소, 간격 20 바이트)
청크 경계: 인덱스 383 -> 384 (직전 청크 길이 128개 원소, 간격 20 바이트)
...
총 청크 경계 수: 15 / 원소 2000개
```

(g++ 13.3.0 / libstdc++ 실측 — 이 청크 크기는 표준이 정하지 않은 **구현별 세부사항**이다. 다른 표준 라이브러리는 다른 값을 쓸 수 있다.) 정확히 128개 원소마다 청크가 끊긴다 — `int`가 4바이트이니 청크 하나가 512바이트라는 뜻이다. libstdc++는 원소 타입 크기가 512바이트보다 작으면 "청크 하나 = 512바이트를 채우는 개수"로 정하는 규칙을 쓴다. 이 숫자 자체을 외울 필요는 없다 — 기억할 건 **원소가 몇 개의 고정 크기 블록으로 나뉘어 저장되고, 그 블록들을 가리키는 포인터 배열이 따로 있다는 구조**뿐이다.

이 구조가 앞쪽 삽입을 상수 시간으로 만드는 이유는 이제 명확하다. `push_front`가 맨 앞 청크에 여유가 있으면 그 청크 안에 원소 하나를 채워 넣고 끝이다 — 다른 원소는 건드리지 않는다. 여유가 없으면 새 청크를 하나 할당해서 맵의 앞에 포인터를 추가하면 그만이다. **기존 원소를 단 하나도 옮기지 않는다는 것**이 `vector`와의 근본적인 차이다. `vector`는 맨 앞에 자리를 만들려면 그 뒤 모든 원소를 물리적으로 옮겨야 하지만, `deque`는 애초에 "전부가 하나의 연속 블록"이라는 전제 자체가 없어서 옮길 이유가 없다.

::: perf deque의 operator[]가 vector보다 미묘하게 느린 이유
`deque::operator[](i)`는 여전히 $O(1)$이지만, 상수항이 `vector`보다 크다 — 인덱스 `i`가 어느 청크에 속하는지 나누기·나머지 연산으로 계산한 뒤, 맵에서 그 청크의 포인터를 한 번 더 읽어야 원소에 닿는다. `vector`는 `data() + i`로 끝나는 계산을 `deque`는 두 단계로 한다. 순수 순차 순회(`sort`, `for`)에서 이 차이는 미미하지만, 무작위 접근이 극단적으로 많은 핫패스에서는 체감될 수 있다 — 자세한 반복자 비용은 [5.4 반복자와 무효화 규칙](#/iterators)에서 이어간다.
:::

::: tip deque는 참조 안정성이 vector보다 유리하다
`vector`는 재할당이 일어나면 기존 원소를 가리키던 포인터·레퍼런스·반복자가 전부 무효화된다([5.1 vector](#/vector)). `deque`는 앞뒤에 원소를 추가해도 **기존 원소를 가리키던 레퍼런스와 포인터는 무효화되지 않는다** — 기존 청크의 원소는 애초에 옮겨지지 않기 때문이다(단, 반복자는 여전히 무효화될 수 있다). 원소를 가리키는 포인터를 오래 들고 있어야 하는 코드에서 이 차이가 실전에 영향을 준다.
:::

## 3. list: 이중 연결 리스트, 그리고 그 대가

`deque`도 못 하는 게 있다 — **컨테이너 중간**에 원소를 넣거나 뺄 때, 그 위치를 이미 반복자로 들고 있어도 앞뒤 원소를 옮겨야 한다(청크 내부는 연속이므로). 중간 삽입/삭제 자체를 $O(1)$로 만들려면 원소끼리를 포인터로 직접 엮어야 한다 — 이게 `std::list`다.

```text nolines
list 내부 구조 (이중 연결 리스트, 원소는 힙 어딘가에 흩어져 있다)

  head                                                    tail
   |                                                        |
   v                                                        v
 [prev|10|next] <-> [prev|20|next] <-> [prev|30|next] <-> [prev|40|next]
    0x7f01a0          0x7f03c8          0x7f00e4          0x7f0510

  각 노드는 별도로 new된 힙 블록 -- 주소가 순서와 무관하게 흩어질 수 있다
  중간에 끼워 넣기: 앞뒤 노드의 next/prev 포인터 4개만 바꾸면 끝 (원소 이동 없음)
```

노드 하나는 데이터 앞뒤로 포인터 두 개(`prev`, `next`)를 더 들고 있다 — `int` 하나(4바이트, 패딩 포함 8바이트)에 포인터 두 개(16바이트)를 더하면 원소 하나가 최소 24바이트다. `vector<int>`의 원소 하나(4바이트)와 비교하면 순수 오버헤드만 6배다. 이 오버헤드는 메모리 낭비로 끝나지 않는다 — **순회할 때마다 다음 노드의 주소를 몰라서, 매번 포인터를 따라가 봐야 한다.** 이게 순차 순회 성능에 실제로 얼마나 영향을 주는지 재보자.

```cpp title="traverse.cpp — 1000만 개 원소를 vector와 list로 각각 10회 순회"
#include <chrono>
#include <list>
#include <numeric>
#include <vector>

int main() {
    constexpr std::size_t N = 10'000'000;
    std::vector<int> v(N);
    std::iota(v.begin(), v.end(), 0);
    std::list<int> l(v.begin(), v.end());

    // vector 순회 10회, list 순회 10회, 각각 합계 시간 측정
    long long s = 0;
    for (int x : v) s += x;   // 이 루프를 10회 반복해서 잰다
    for (int x : l) s += x;   // 이 루프도 10회 반복해서 잰다
}
```

```console
$ g++ -std=c++20 -Wall -Wextra -O2 traverse.cpp -o traverse
$ ./traverse
N=10000000 원소, 10회 순회 합계
vector 순회 총합: 54.3207 ms (1회 평균 5.43207 ms)
list   순회 총합: 467.515 ms (1회 평균 46.7515 ms)
list/vector 비율: 8.60657x
```

(g++ 13.3.0 / `-O2` / Linux x86-64 실측. 반복 실행에서 비율은 7.9~9.7배 사이로 흔들렸다.) `l`은 `v`의 원소를 순서 그대로 복사해 만들었으니 논리적 순서는 완전히 같다. 그런데도 순회 1회 평균이 `vector`는 5.4ms, `list`는 46.8ms — **8배 이상 느리다.** 둘 다 "원소 N개를 한 번씩 읽어서 더한다"는 같은 $O(n)$ 알고리즘인데도 실제 걸리는 시간이 이렇게 벌어지는 이유는 알고리즘 복잡도가 아니라 **메모리 접근 패턴**에 있다.

::: perf 캐시 미스가 정확히 무엇을 느리게 만드는가
CPU는 메모리를 캐시 라인(보통 64바이트) 단위로 끌어온다. `vector`를 순회하면 다음에 읽을 주소가 바로 직전 주소 + 4바이트라, CPU의 하드웨어 프리페처가 다음 캐시 라인을 미리 끌어와 둔다 — 대부분의 읽기가 이미 캐시에 있는 데이터를 그냥 읽는 것이다. `list`는 다음 원소의 주소가 현재 노드 안의 `next` 포인터를 읽어야 알 수 있고, 그 포인터가 가리키는 주소는 힙 어디에 있을지 예측이 안 된다 — 매 노드마다 메인 메모리까지 갔다 와야 하는 캐시 미스가 발생할 확률이 높다. 캐시 히트가 몇 사이클이면 끝날 일을, 미스 한 번은 수백 사이클을 태운다 — 이 절의 8배 차이는 정확히 이 "포인터 체이싱"의 대가다.
:::

::: danger 실시간 제어 루프에서 list를 쓰면 안 되는 이유는 캐시 미스만이 아니다
[6.8 실시간 제약과 제어 루프](#/realtime)의 1kHz 제어 루프처럼 매 주기 시간 예산이 정해진 코드에서 `list`는 이중으로 위험하다. 첫째, 방금 실측한 캐시 미스가 순회할 때마다 반복돼 **주기마다 걸리는 시간이 들쑥날쑥해진다**(지터). 둘째, `list`는 원소 하나를 넣을 때마다 노드 하나를 `new`로 힙에 할당한다 — 할당기 락과 실행 시간이 예측 불가능한 `malloc` 호출이 제어 루프 안에 그대로 들어간다는 뜻이다. 반면 `deque`는 청크 단위로만 가끔 할당하고, `array`는 아예 할당이 없다. **제어 주기 안에서는 list보다 deque, 가능하면 array를 먼저 고려해라.**
:::

## 4. array\<T, N\>: 스택에 고정된, 크기를 잃지 않는 배열

`vector`, `deque`, `list` 셋 다 힙에 데이터를 둔다. `std::array<T, N>`은 다르다 — 크기 `N`이 [4.6 constexpr와 컴파일 타임 계산](#/constexpr)에서 본 **비타입 템플릿 매개변수**로 타입 자체에 새겨지고, 데이터는 일반 C 배열과 똑같이 **선언된 자리(대개 스택)에 그대로 놓인다.** 실제로 스택에 있는지 주소로 확인해본다.

```cpp title="array_stack.cpp — 스택 지역변수와의 주소 거리로 배치를 확인한다"
int stack_marker = 0;
std::array<int, 8> a{};
std::vector<int> v(8);

auto d_a = /* |&a - &stack_marker| */;
auto d_v = /* |v.data() - &stack_marker| */;
```

```console
$ g++ -std=c++20 -Wall -Wextra array_stack.cpp -o array_stack
$ ./array_stack
|&a - &stack_marker|      = 60 바이트
|v.data() - &stack_marker| = 46490638024308 바이트
```

(g++ 13.3.0 실측 — 정확한 오프셋은 스택 프레임 레이아웃에 따라 실행마다 다를 수 있다.) `stack_marker`와 `a`는 같은 스택 프레임 안에 있으니 주소 차이가 60바이트에 불과하다. `v`의 실제 데이터(`v.data()`)는 힙에 있어서 주소 차이가 46조 바이트대 — 자릿수 자체가 완전히 다른 메모리 영역이라는 뜻이다. `array`가 힙 할당·해제 비용이 전혀 없는 이유가 여기 있다.

`array`는 그냥 크기만 고정된 게 아니라 **컨테이너로서의 인터페이스**를 갖췄다는 점에서 C 배열과 또 다르다 — `.size()`, `.begin()`/`.end()` 반복자, `std::get<I>()`를 전부 지원한다. 이게 왜 중요한지는 [1.7 배열과 문자열](#/arrays-strings)에서 본 **배열의 붕괴(decay)** 문제와 나란히 놓아야 보인다.

```cpp title="array_decay.cpp — 함수 경계를 넘을 때 크기 정보가 살아남는가"
void print_c_array_size(int arr[]) {           // 배열 매개변수 -- 사실상 포인터
    // sizeof(arr) -- 여기선 포인터 크기만 나온다
}

template <typename T, std::size_t N>
void print_array_size(const std::array<T, N>& arr) {
    // arr.size() -- N이 타입에 있으니 여기서도 정확하다
}

int c_arr[10] = {};
print_c_array_size(c_arr);

std::array<int, 10> s_arr{};
print_array_size(s_arr);
```

```console
$ g++ -std=c++20 -Wall -Wextra array_decay.cpp -o array_decay

return size of 'int*' [-Wsizeof-array-argument]
$ ./array_decay

s_arr.size() main 안에서: 10

```

(g++ 13.3.0 실측 — `-Wsizeof-array-argument` 경고 자체가 컴파일러가 직접 짚어 주는 붕괴의 증거다.) `main` 안에서 `c_arr`의 `sizeof`는 40바이트(`int` 10개)였지만, 함수 매개변수로 넘어가는 순간 8바이트(포인터 하나)로 쪼그라든다 — [1.7](#/arrays-strings)에서 본 배열 붕괴가 함수 경계에서 그대로 재현된 것이다. `std::array`는 함수에 **레퍼런스나 값**으로 넘기는 한(포인터로 넘기지 않는 한) 이 문제 자체가 생기지 않는다 — `N`이 템플릿 인자로 타입의 일부이니, 붕괴할 "포인터로의 변환"이 애초에 일어나지 않는다.

`std::get<I>()`는 `operator[]`보다 한 걸음 더 간다 — 인덱스가 컴파일 타임 상수이기 때문에, 범위를 벗어난 인덱스를 그 자리에서 컴파일 에러로 잡는다.

```cpp title="array_get_oob.cpp — std::get의 범위 검사는 컴파일 타임에 끝난다"
std::array<int, 5> arr{1, 2, 3, 4, 5};
return std::get<10>(arr);   // 인덱스 10, 크기는 5
```

```console
$ g++ -std=c++20 -Wall -Wextra array_get_oob.cpp -o array_get_oob
/usr/include/c++/13/array:386:26: error: static assertion failed:
array index is within bounds
  386 |       static_assert(_Int < _Nm, "array index is within bounds");
      |                     ~~~~~^~~~~
```

(g++ 13.3.0 실측.) `operator[]`로 같은 실수를 했다면 미정의 동작으로 조용히 넘어갔을 것이다. `std::get<10>`은 `_Int < _Nm`이라는 `static_assert`가 컴파일 타임에 실패해서, 프로그램이 만들어지기도 전에 잡힌다 — 인덱스가 실행 중에 바뀌지 않는 자리에서는 `operator[]`보다 `std::get`을 쓸 이유가 여기 있다.

::: warn std::array\<T, N\> a; 는 원소를 초기화하지 않는다
`std::array`는 내부적으로 C 배열을 감싼 집성체(aggregate)일 뿐이다. `std::array<int, 5> a;`처럼 중괄호 없이 선언하면 C 배열과 똑같이 **값이 무엇인지 정해지지 않은 채로** 남는다 — `vector<int>(5)`가 자동으로 0으로 채우는 것과 다르다. 반드시 `std::array<int, 5> a{};`처럼 빈 중괄호를 붙여 값 초기화해라.
:::

::: note 중괄호 하나로도 충분한 이유
`std::array<int, 3> a = {1, 2, 3};`처럼 중괄호를 하나만 써도 컴파일된다. `std::array`가 실제로 감싸는 건 C 배열 멤버 하나뿐인 집성체라 원칙적으로는 `{{1, 2, 3}}`(바깥은 `array` 자체, 안쪽은 그 C 배열 멤버)이 정확한 형태지만, 표준은 이런 경우 중괄호 생략(brace elision)을 허용한다. 실무에서는 거의 항상 중괄호 하나만 쓴다.
:::

## 5. 컨테이너 선택 가이드

| 상황 | 선택 | 이유 |
|---|---|---|
| 기본값, 특별한 이유가 없으면 | `vector` | 연속 메모리, 최고의 순회 성능, 캐시 친화적 ([5.1](#/vector)) |
| 양쪽 끝에서 자주 넣고 뺀다 (슬라이딩 윈도우, 큐) | `deque` | 앞뒤 모두 상각 $O(1)$, 청크 구조 덕에 재할당 시 기존 원소 이동 없음 |
| 중간 위치(반복자로 이미 확보한 자리)에서 삽입/삭제가 잦고, 그 반복자가 계속 유효해야 한다 | `list` | 삽입/삭제가 위치 무관 $O(1)$, `splice`로 노드째 이동 가능. 순회 성능은 포기 |
| 크기가 컴파일 타임에 고정이고 힙 할당 자체를 피하고 싶다 | `array` | 스택 배치, 할당 없음, `vector`와 동일한 반복자/알고리즘 인터페이스 |

실전에서는 `vector`가 압도적 기본값이고, 나머지 셋은 각자 딱 하나의 명확한 이유가 있을 때만 골라 쓴다 — "일단 list가 유연해 보여서" 같은 이유는 이 절의 실측이 보여주듯 근거가 안 된다.

## 6. 로보틱스 도메인: 세 컨테이너가 실제로 쓰이는 자리

- **`deque`**: IMU나 조인트 인코더의 최근 N개 샘플을 들고 있는 슬라이딩 윈도우 필터(이동평균, 노이즈 완화)가 정확히 이 절 1장의 문제다 — 새 샘플은 `push_back`, 오래된 샘플은 `pop_front`로 버린다. 둘 다 상수 시간이라 윈도우 크기가 커져도 매 주기 비용이 그대로다.
- **`list`**: 3장의 캐시 미스·할당 지연 문제 때문에 [6.8 실시간 제약과 제어 루프](#/realtime)의 제어 루프 안에서는 사실상 배제 대상이다. 이벤트 큐처럼 순회 빈도가 낮고 중간 삭제가 잦은 비실시간 경로에서만 후보로 남는다.
- **`array`**: [9.4 순기구학](#/forward-kinematics)·[9.5 역기구학](#/inverse-kinematics)에서 다루는 다리 하나의 관절 각도(3자유도)나 헥사포드 전체 관절(6다리 × 3자유도 = 18개)처럼, **로봇 구조가 바뀌지 않는 한 개수가 절대 안 바뀌는 값**은 `std::array<double, 18>`이 정답이다. 힙 할당이 없어 제어 루프 안에서 안전하고, 크기가 타입에 박혀 있어 실수로 다른 크기의 배열과 섞어 쓰면 컴파일이 막힌다.

::: interview vector, deque, list 중 언제 무엇을 쓰는가
컨테이너 선택 질문의 표준 답변 뼈대. ① **`vector`가 기본값**이다 — 연속 메모리라 캐시 친화적이고, 반복자 무효화 규칙도 가장 단순하다([5.1](#/vector)). ② **`deque`**는 양쪽 끝 삽입이 잦을 때만 고른다 — 내부가 고정 크기 청크의 배열이라 앞뒤 모두 기존 원소를 옮기지 않고 상수 시간으로 확장된다(청크 경계 실측: libstdc++에서 `int` 128개당 하나). ③ **`list`**는 "반복자를 오래 들고 있으면서 그 위치에서 자주 삽입/삭제해야 한다"는 조건이 실제로 성립할 때만 고른다 — 이중 연결이라 위치 무관 $O(1)$ 삽입/삭제가 되지만, 노드가 힙에 흩어져 있어 순회는 포인터 체이싱 때문에 `vector`보다 실측 8배 이상 느리다. 마지막에 "실무에서는 90% 이상 `vector`로 끝난다"까지 붙이면 좋은 답이다.
:::

## 요약

- `vector`의 맨 앞 삽입은 $O(n)$이라 반복하면 $O(n^2)$이다(실측: `insert(begin())` 3만→6만회에서 4.5배로 증가). `deque::push_front`는 상각 $O(1)$이라 같은 배율에서 시간이 거의 그대로다.
- `deque`는 연속 메모리 하나가 아니라 **고정 크기 청크들의 배열("맵")**이다. 앞뒤 청크에 여유가 없을 때만 새 청크를 할당하고, 기존 원소는 절대 옮기지 않는다(libstdc++ 실측: `int` 기준 청크당 128개, 512바이트 — 구현별 세부사항).
- `deque`는 재할당이 있어도 기존 원소의 포인터/레퍼런스를 무효화하지 않는다(`vector`와 다르다). 대신 `operator[]`는 청크 인덱스 계산이 한 단계 더 있어 `vector`보다 상수항이 크다.
- `list`는 이중 연결 리스트다 — 임의 위치 삽입/삭제가 반복자만 있으면 $O(1)$이지만, 노드가 힙에 흩어져 있어 순회할 때마다 포인터 체이싱으로 캐시 미스가 잦다(실측: 1000만 원소 순회에서 `vector` 대비 8배 이상 느림).
- 실시간 제어 루프에서는 `list`의 순회 지연뿐 아니라 삽입마다 발생하는 예측 불가능한 `new` 호출 자체가 문제다 — 주기 지터의 원인이 된다.
- `std::array<T, N>`은 `N`이 비타입 템플릿 매개변수로 타입에 새겨진, 스택에 놓이는 고정 크기 배열이다(실측: 스택 지역변수와의 주소 거리 60바이트, `vector` 힙 버퍼와는 자릿수가 다른 거리). 함수 경계를 넘어도 C 배열과 달리 크기가 붕괴하지 않는다(실측: `sizeof`가 함수 안에서 포인터 크기로 줄어드는 것과 대조).
- `std::get<I>(arr)`는 컴파일 타임에 범위를 검사한다(실측: 범위를 벗어난 인덱스가 `static_assert` 컴파일 에러로 잡힘). 단, `std::array<T,N> a;`처럼 중괄호 없이 선언하면 원소는 초기화되지 않는다 — 반드시 `{}`를 붙여라.

::: quiz 연습문제
1~2번은 개념 문제, 3~5번은 네 컴퓨터에서 직접 코드를 짜고 컴파일해서 확인하는 실습이다.

1. `vector::insert(begin(), x)`가 $O(n)$인 이유와 `deque::push_front(x)`가 $O(1)$인 이유를 각각 컨테이너의 메모리 구조와 연결해서 한 문장씩 설명하라.
2. `list`가 임의 위치 삽입/삭제는 $O(1)$인데도 순회는 왜 `vector`보다 느린지, 이 절의 캐시 미스 설명을 근거로 답하라.
3. (실습, 코드 작성형) `front_insert.cpp`를 그대로 타이핑하되 `N`을 30000, 60000, 120000 세 단계로 늘려서 돌려라. 성공 기준: `vector`쪽 시간이 대략 $N$이 두 배가 될 때마다 4배 가까이 늘어나는 것을 네 눈으로 확인하고, `deque`쪽은 거의 그대로임을 확인한다.
4. (실습) `deque_chunk.cpp`를 그대로 타이핑해서 돌려라. 성공 기준: 네 환경의 libstdc++가 `int` 하나당 청크에 몇 개씩 담는지(이 절에서는 128개였다) 직접 출력으로 확인한다 — 다른 환경이면 다른 숫자가 나올 수 있다는 것 자체가 "구현별 세부사항"의 의미다.
5. (실습) `std::array<int, 5> a;`(중괄호 없음)를 선언하고 바로 `std::cout`으로 다섯 원소를 출력하는 코드를 짜고, `g++ -std=c++20 -Wall -Wextra -fsanitize=address -g`로 컴파일해 실행하라. 성공 기준: 값이 0이 아닌 쓰레기값이 나올 수 있음을 확인하고(정확한 값은 실행마다 다를 수 있다), `a{}`로 바꾸면 항상 0이 나오는 것과 대조한다.
:::

::: answer 해설
1. `vector`는 모든 원소가 하나의 연속 블록에 있어서, 맨 앞에 자리를 만들려면 그 뒤의 모든 원소를 물리적으로 한 칸씩 옮겨야 한다 — 원소 수에 비례하는 $O(n)$이다. `deque`는 고정 크기 청크들의 배열이라 앞쪽 청크에 여유가 있으면 그 자리에 채우기만 하면 되고, 여유가 없어도 새 청크를 하나 할당해서 맵 앞에 등록할 뿐 기존 원소는 하나도 옮기지 않는다 — 상각 $O(1)$이다.
2. `list`의 임의 위치 삽입/삭제가 $O(1)$인 건 앞뒤 노드의 포인터 4개만 바꾸면 되기 때문이다. 하지만 순회는 다음 원소의 주소를 알려면 현재 노드의 `next` 포인터를 실제로 읽어야 하고, 그 주소는 힙 어디에 있을지 예측이 안 돼서 CPU의 프리페처가 무력해진다. `vector`는 다음 주소가 항상 "현재 주소 + 원소 크기"라 프리페처가 미리 캐시에 올려 두는데, `list`는 이게 안 돼 매 노드마다 캐시 미스 확률이 높다 — 같은 $O(n)$이라도 상수항(캐시 미스 비용)이 완전히 다르다.
3. `vector`쪽은 $N$이 30000→60000→120000으로 두 배씩 늘 때마다 시간이 대략 4배씩(제곱 관계) 늘어야 한다. `deque`쪽은 세 구간 모두 거의 같은 수준(이 절의 실측 기준 0.1~0.2ms대)에 머물러야 한다.
4. 청크 경계가 몇 개 원소마다 나타나는지, 그 경계에서 나온 "직전 청크 길이"가 일정한 값(이 절에서는 128)으로 반복되는지 확인한다. 이 값은 `sizeof(T)`와 라이브러리 구현에 따라 달라지므로, 다른 컴파일러·표준 라이브러리에서는 다른 숫자가 나올 수 있다 — 그 자체가 "표준이 보장하지 않는 구현 세부사항"이라는 이 절의 경고를 실감하는 지점이다.
5. 중괄호 없는 `std::array`는 값 초기화를 하지 않으므로 스택에 이미 남아 있던 임의의 비트 패턴이 그대로 보인다 — 이 절의 `array_uninit` 실측처럼 매 실행에서 같은 값이 나올 수도 있지만 그건 우연이고 표준이 보장하는 값이 아니다. `a{}`로 바꾸면 값 초기화가 적용돼 항상 0이 나온다.
:::

이 절의 `front_insert.cpp`, `deque_chunk.cpp`, `traverse.cpp`, `array_decay.cpp`, `array_get_oob.cpp`, `array_stack.cpp`를 전부 직접 타이핑해라. 특히 `deque_chunk.cpp`는 네 환경에서 청크 크기가 몇으로 나오는지 직접 확인하고, `traverse.cpp`는 `N`을 낮춰서(예: 100만) 실행 시간이 함께 줄어드는지, 그래도 비율은 비슷하게 유지되는지 재봐라. 기준 명령: `g++ -std=c++20 -Wall -Wextra -O2 파일.cpp -o 이름 && ./이름`.
