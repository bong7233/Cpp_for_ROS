# 0.4 학습 워크플로: 앱과 IDE의 병행

::: lead
[0.1](#/how-to-read)에서 원칙을 세웠다 — 직접 타이핑하고, 결과를 예측한 뒤 실행하고, 위젯으로 확인한다. [0.2](#/toolchain-setup)와 [0.3](#/first-build)에서 도구도 갖췄다. 이 절은 그 원칙을 **매일 반복 가능한 손 순서**로 바꾼다. 화면을 어떻게 배치하고, 절 하나를 어떤 단계로 돌고, 실습 파일을 어디에 어떤 이름으로 쌓고, 언제 무엇으로 복습하는지. 한 번 정해 두면 앞으로 100개가 넘는 절이 전부 같은 방식으로 굴러간다. 워크플로를 정하는 데 드는 시간은 오늘 30분이고, 아끼는 시간은 그 수백 배다.
:::

## 화면 배치: 읽는 창과 만드는 창

이 책의 학습은 두 창 사이의 왕복이다. 읽는 창(이 앱)과 만드는 창(IDE). 왕복이 잦을수록 창 전환 비용이 학습을 갉아먹으므로, **두 창이 동시에 보이는 배치**가 기본이다.

**듀얼 모니터라면** 왼쪽에 이 앱, 오른쪽에 VSCode를 놓는다. 좌우는 취향이지만 한 번 정하면 바꾸지 마라 — "코드는 항상 오른쪽"이 몸에 배면 시선 이동이 무의식이 된다. 터미널은 별도 창을 띄우지 말고 VSCode 통합 터미널(<kbd>Ctrl</kbd>+<kbd>`</kbd>)을 쓴다. 에디터·터미널·컴파일 에러가 한 화면에 모여야 [0.3](#/first-build)에서 훈련한 에러 읽기 루프가 빨라진다.

**모니터가 하나라면** 화면을 절반으로 가른다. 윈도우는 <kbd>Win</kbd>+<kbd>←</kbd>/<kbd>→</kbd>, 우분투는 <kbd>Super</kbd>+<kbd>←</kbd>/<kbd>→</kbd>, macOS는 창 초록 버튼을 길게 눌러 타일링. 왼쪽 절반에 이 앱, 오른쪽 절반에 VSCode다.

이 앱이 절반 폭에서도 깨지지 않는 것은 우연이 아니다. **절반 폭이 이 앱의 기본 사용 폭이라고 보고 설계했다.** 화면이 좁아지면 오른쪽 절내 목차가 자동으로 숨고, 본문 폭이 코드 블록 기준으로 잡혀 있다. 전체 화면으로 넓게 펴 놓고 읽는 것은 오히려 이 책이 의도한 사용법이 아니다 — 그 배치에는 IDE가 없기 때문이다.

::: tip 앱 쪽 글자를 한 단계 줄여라
절반 폭에서는 상단 오른쪽 아이콘으로 글자 크기를 한 단계 줄이는 쪽이 대개 읽기 편하다. 한 화면에 들어오는 문맥이 넓어져서 코드 블록과 그 설명 문단이 같이 보인다. 코드와 설명이 따로 스크롤되기 시작하면 이해가 끊긴다.
:::

## 절 하나를 도는 루틴

이 책의 최소 학습 단위는 절이다. 절 하나를 도는 절차를 고정하라. 여섯 단계다.

**① lead를 읽고 "왜"를 잡는다.** 모든 절은 `lead` 상자로 시작한다. 여기서 얻어야 할 것은 지식이 아니라 질문이다 — *이 절이 없으면 무엇이 안 되는가.* 질문 없이 본문에 들어가면 읽기는 스캔이 되고, 스캔한 내용은 하루를 못 간다.

**② 코드가 나오면, 결과를 먼저 말한 뒤 IDE에 타이핑한다.** 본문의 코드 블록을 만나면 눈으로 넘기지 말고 멈춘다. 실행 결과를 예측해서 소리 내어 말하거나 주석으로 적는다 — "이건 링커 에러", "출력은 3, 힙 할당은 두 번". 그 다음 오른쪽 창에서 **직접 타이핑한다.** 복사 버튼은 없다고 생각하라. 이유는 [0.1](#/how-to-read)에서 말했다 — 타이핑 실수가 곧 에러 메시지 읽기 훈련이다.

**③ 실행해서 예측과 대조한다.** 맞았으면 지식이 확인된 것이고, **틀렸으면 그 지점이 오늘의 수확이다.** 예측과 실제의 어긋남이 당신의 멘탈 모델에서 정확히 어디가 잘못됐는지 알려 준다. 틀린 예측을 파일 상단 주석에 남겨라 — `// 예측: 컴파일 에러 / 실제: 통과, 경고만` 한 줄이면 된다. 나중에 이 주석이 복습 자료가 된다.

**④ 위젯이 있으면 앞뒤로 오가며 자기 말로 설명한다.** 재생 버튼으로 한 번 흘려 본 뒤, 스텝을 수동으로 되감아라. 임의의 스텝에서 멈추고 *"지금 이 상태가 된 이유"* 를 자기 말로 설명할 수 있는지 확인한다. 설명이 막히는 스텝이 바로 이해가 빈 지점이다. 그 스텝 앞뒤를 오가며 무엇이 변했는지 본다.

**⑤ quiz를 푼다.** 개념 문제는 답을 보기 전에 손으로 쓴다. 실습형 문제의 성공 기준은 답안 확인이 아니라 **당신의 컴퓨터에서 컴파일이 통과하고 새니타이저가 조용한 것**이다. Part II부터는 이런 문제가 절마다 섞인다.

**⑥ '완료로 표시'는 다시 설명할 수 있을 때만 누른다.** 절 맨 아래 완료 버튼은 "읽었음"이 아니라 "설명할 수 있음"의 표시다. 기준은 하나 — *이 절의 핵심을 코드 없이 말로 재구성할 수 있는가.* 안 되면 체크하지 말고 넘어가라. [0.1](#/how-to-read)의 원칙 3대로, 비워 둔 체크는 부끄러운 게 아니라 정확한 상태 기록이다.

::: warn "아는 내용"이 가장 위험하다
②를 건너뛰게 만드는 것은 어려운 절이 아니라 쉬워 보이는 절이다. "이 정도는 안다"며 눈으로만 읽은 절에서 예측 실패가 가장 많이 나온다. 특히 다른 언어 경험이 있다면 — `=` 가 복사라는 것, 정수 나눗셈, 암묵 변환 — C++은 당신의 직관이 다른 언어에서 왔다는 사실을 노린다. 쉬워 보일수록 예측을 명시적으로 하라.
:::

::: tip 한 세션에 한 절이면 충분하다
이 책의 절은 15~30분 분량으로 잡혀 있지만, 위 루틴을 다 돌면 실제로는 40~60분이 든다. 그게 정상이다. 하루에 세 절을 눈으로 읽는 것보다 한 절을 여섯 단계로 도는 쪽이 일주일 뒤에 남는 양이 많다. 진도 그래프의 속도가 아니라 ⑥의 기준을 지키는 것이 진짜 진도다.
:::

## 실습 디렉터리 규율: ~/cpp-lab

②와 ③에서 타이핑한 코드를 아무 데나 버리지 마라. **연습장 디렉터리 하나를 파고, 절마다 파일 하나씩 쌓는다.**

```console
$ mkdir ~/cpp-lab && cd ~/cpp-lab
$ git init
```

파일 이름은 `ch<장>-<절>-<키워드>.cpp` 로 통일한다. 몇 주 뒤에는 이런 모양이 된다.

```text nolines
~/cpp-lab/
├── ch1-1-linker-error.cpp      <- 1.1 링커 에러 재현
├── ch1-2-overflow.cpp          <- 1.2 정수 오버플로
├── ch2-1-stack-frame.cpp       <- 2.1 스택 프레임 관찰
├── ch2-7-move.cpp              <- 2.7 이동 시맨틱
├── ch2-7-quiz.cpp              <- 2.7 연습문제 답안
└── notes.md                    <- 절 번호별 한 줄 메모
```

왜 이렇게 하는가. 첫째, **지난 실습이 곧 나만의 예제 사전이 된다.** "이동 시맨틱 예제가 뭐였지" 싶을 때 남의 블로그가 아니라 `ch2-7-move.cpp` 를 연다. 거기엔 당신이 친 코드, 당신이 틀린 예측 주석, 당신이 고친 흔적이 있다 — 검색으로 나오는 어떤 예제보다 회수율이 높다. 둘째, 파일명에 절 번호가 박혀 있으므로 코드에서 책으로, 책에서 코드로 양방향 검색이 된다. `ls ch2-*` 한 줄이 Part II 복습 목록이다.

컴파일 명령은 매번 치지 말고 셸 알리아스로 박아 둔다.

```bash
# ~/.bashrc 에 추가 — 이 책의 표준 컴파일 명령
alias cpx='g++ -std=c++20 -Wall -Wextra -fsanitize=address,undefined'
```

이후 실습은 `cpx ch2-7-move.cpp && ./a.out` 으로 끝난다. 새니타이저를 기본 포함시키는 이유는 [2.11](#/ub-sanitizers)에서 다루지만, 지금은 "켜 두면 미래의 버그가 지금 잡힌다"만 알면 된다.

::: tip 커밋이 복습 기록이 된다
절 하나를 끝낼 때마다 `git add -A && git commit -m "ch2-7 move semantics"` 한 줄. 이러면 `git log --oneline` 이 학습 일지가 되고, 몇 달 뒤 `git diff` 로 초기 코드를 열어 보면 자신의 변화가 보인다. 원격에 올릴 필요도 없다 — 로컬 저장소로 충분하다.
:::

## 복습 루틴: 세 개의 고리

복습을 "시간 나면 하는 것"으로 두면 안 한다. 기존 루틴에 붙어 있는 세 개의 고리로 고정하라.

**고리 1 — 다음 절에 들어가기 전, 직전 절의 quiz를 다시 푼다.** 5분이면 된다. 어제의 답을 기억해서 맞히는 것은 의미가 없고, **풀이 과정을 다시 만들어낼 수 있는지**가 검사 대상이다. 두 번째에도 막히는 문제는 해당 절의 완료 체크를 해제하라 — ⑥의 기준을 소급 적용하는 것이다.

**고리 2 — 주말에 그 주에 읽은 절의 interview 상자만 훑는다.** 왼쪽 목차에서 이번 주에 체크한 절들을 열고 🎤 상자만 골라 읽는다. interview 상자는 그 절의 핵심을 "설명하라"는 형태로 압축해 놓은 것이라, 답변 뼈대를 소리 내어 말해 보는 것만으로 ⑥의 기준 검사가 된다. 이 습관이 쌓이면 [12.4 기술면접 총정리](#/interview-cpp)에 도달했을 때 할 일이 "정리"가 아니라 "확인"이 된다.

**고리 3 — 용어가 가물하면 그 자리에서 용어 사전을 연다.** 왼쪽 아래 용어 사전에 모든 용어의 한 줄 정의와 최초 등장 절이 있다. "lvalue가 뭐였지" 상태로 본문을 계속 읽는 것은 모래 위에 쌓는 것이다. 30초를 써서 바닥을 다지고 돌아와라.

::: note 복습 도구는 이 세 가지가 전부다
별도의 복습 화면이나 자동 알림 같은 것은 이 앱에 없다. quiz 재풀이, interview 상자, 용어 사전 — 전부 이미 본문에 있는 것을 다시 여는 방식이다. 도구가 단순한 대신 루틴을 당신이 지켜야 한다. 고리 1은 "다음 절 열기 전"에, 고리 2는 "주말"에 붙어 있으므로 별도의 의지력이 거의 안 든다.
:::

## 폰에서의 복습: EPUB

이동 중까지 학습을 끌고 가고 싶다면, 이 앱을 폰 브라우저로 여는 대신 **EPUB로 내보내서 폰 기본 e리더로 읽는다.** 이 앱은 PC 전용이다 — 폰에는 IDE가 없으므로 여섯 단계 루틴의 절반이 불가능하고, 그렇다면 폰에서 할 일은 순수 텍스트 복습뿐이기 때문이다.

```console
$ cd Cpp_for_ROS
$ python tools/export_epub.py
dist/ 에 EPUB 파일이 생성된다
```

생성된 파일을 폰으로 옮겨(이메일, 클라우드 드라이브, USB 아무거나) 기본 e리더로 연다 — iOS는 Apple Books, 안드로이드는 Play Books가 EPUB를 바로 읽는다.

EPUB에는 위젯이 없다. 대신 이 책의 본문은 **위젯 없이도 개념이 전달되도록** 쓰여 있다 — 위젯은 이해를 새기는 도구이지 본문의 구멍을 메우는 도구가 아니라는 것이 이 책의 집필 원칙이다. 그래서 EPUB 복습은 "위젯에서 봤던 그 상태 변화를 글만 읽고 머릿속에서 재생할 수 있는가"를 검사하는, 꽤 좋은 시험이기도 하다.

::: warn 진도의 기준점은 PC 앱 하나다
e리더의 읽은 위치, 북마크는 EPUB 파일 안의 참고 기록일 뿐이다. 완료 체크와 진도 그래프는 PC 앱의 브라우저 `localStorage` 에만 있다 — [0.1](#/how-to-read)에서 말한 대로 **이 PC의 이 브라우저가 유일한 기준점**이다. 폰에서 두 절을 읽었어도 완료 판정(⑥)은 PC로 돌아와 IDE 실습까지 마친 뒤에 하라. 본문을 다시 내보냈다면 옛 EPUB는 지우고 새 파일로 교체한다.
:::

## Hexpider 프로젝트와의 병행

이 책의 종착지는 책이 아니라 로봇이다. 실전 프로젝트 — 6족 보행 로봇 Hexpider — 와 이 책을 어떻게 맞물릴 것인가.

**Part I~II를 도는 동안은 책에 집중하라.** 이 시기에 프로젝트 코드를 열면 모든 줄이 물음표라 병행의 효율이 안 나온다. 단, 구경은 다르다 — `ros2_control` 의 `hardware_interface` 소스를 일주일에 한 번 열어서 "여전히 안 읽히는가"를 확인하는 것은 좋은 진도 측정이다.

**Part II와 VI를 마치면 첫 번째 문이 열린다.** `hardware_interface` 코드의 골격 — 왜 `unique_ptr` 로 하드웨어를 소유하는지, 왜 `read()`/`write()` 루프가 별도 스레드에서 도는지, 왜 그 루프 안에서 할당을 피하는지 — 가 읽히기 시작한다. [2.1 메모리 모델](#/memory-model)에서 시작한 소유권 이야기와 Part VI의 스레드 이야기가 전부 그 코드 안에 있다.

**Part IX~X는 병행이 기본이다.** 역기구학 절을 읽은 주에 헥사포드 다리 IK를 실제로 구현하고, executor 절을 읽은 주에 노드의 콜백 구조를 손보는 식으로, **읽는 것과 만드는 것의 간격을 일주일 이내로** 유지하라. 이 구간은 책이 프로젝트의 해설서 역할을 하도록 설계되어 있어서, 순차(책 다 읽고 프로젝트 시작)보다 병행이 압도적으로 효율이 좋다.

어느 파트가 Hexpider의 어느 단계에 대응하는지 전체 지도는 [12.5 Hexpider 로드맵 매핑](#/hexpider-map)에 있다. 미리 한 번 훑어 두면 "지금 배우는 게 어디에 쓰이는가"라는 질문에 항상 답을 갖고 읽게 된다.

## 요약

- 두 창이 동시에 보이는 배치가 기본이다. 듀얼 모니터면 앱/IDE를 좌우로, 하나면 절반 분할 — 이 앱은 절반 폭을 기준으로 설계됐다.
- 절 하나의 루틴: **lead로 "왜" 잡기 → 예측 말하고 타이핑 → 실행해 대조(틀리면 수확) → 위젯을 오가며 자기 말로 설명 → quiz(실습형은 컴파일 통과가 기준) → 다시 설명할 수 있을 때만 완료 체크.**
