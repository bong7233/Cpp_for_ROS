# 7.1 CMake 기초: 타겟과 프로퍼티

::: lead
[0.3](#/first-build)에서 `cmake -B build`와 `cmake --build build` 두 줄을 "지금은 이유를 묻지 말고 외워서 쓰라"고 미뤄 뒀다. [6.9](#/thread-architecture)까지 이 책의 모든 예제는 `g++ -std=c++20 ...` 한 줄로 끝났다 — 파일이 하나거나 많아야 서넛이었으니 가능한 일이었다. 이 절에서 그 한 줄이 왜 더 이상 버티지 못하는지 실제로 겪고, CMake가 그 문제를 "파일"이 아니라 "타겟"이라는 단위로 어떻게 다시 짜는지 손으로 직접 확인한다. `cmake_minimum_required`/`project`/`add_executable` 세 줄부터 시작해서, `target_include_directories`/`target_link_libraries`가 왜 전역 설정보다 나은지를 실제 컴파일 명령을 뽑아 대조하고, `PUBLIC`/`PRIVATE`/`INTERFACE`가 실제로 컴파일 에러를 내고 없애는 걸 눈으로 본다.
:::

## g++ 한 줄이 무너지는 지점

`g++ -std=c++20 -Wall -Wextra hex_thread_arch.cpp -o hex_thread_arch -lpthread` — [6.9](#/thread-architecture)의 이 명령은 소스 파일이 하나였다. 헥사포드 제어 소프트웨어가 실제로 커지면 이 그림이 된다. `ik_solver.cpp`, `sensor_driver.cpp`, `logging.cpp`, `leg_kinematics.cpp`... 파일이 수십 개로 늘고, 그중 몇몇은 Eigen이나 다른 라이브러리에 의존한다. 이걸 여전히 `g++` 한 줄로 다루려면 명령이 이렇게 된다.

```console
$ g++ -std=c++20 -Wall -Wextra -Iinclude -Ithird_party/eigen \
      main.cpp ik_solver.cpp sensor_driver.cpp logging.cpp leg_kinematics.cpp \
      -o hex_control -lpthread -lyaml-cpp
```

파일 하나를 고쳤을 뿐인데 이 명령은 매번 전체를 다시 컴파일한다 — [0.3](#/first-build)이 이미 지적한 "3초 vs 20분"의 그 문제가 그대로 재현된다. 이걸 손으로 해결하려면 파일마다 `.o`를 따로 만들고, 어떤 `.o`가 어떤 소스보다 오래됐는지 타임스탬프를 비교하고, 바뀐 것만 골라 재컴파일한 뒤 링크하는 스크립트를 직접 짜야 한다 — 이게 전통적인 Makefile을 손으로 쓰는 방식이고, 실제로 이렇게 하면 새 소스 파일을 추가할 때마다, 새 라이브러리를 하나 붙일 때마다, 심지어 다른 컴파일러나 다른 운영체제로 옮길 때마다 이 스크립트를 다시 고쳐야 한다. 파일 수가 늘수록 이 손대야 할 지점도 같이 는다 — 사람이 관리하기에는 너무 빨리 무너지는 규모다.

CMake는 이 문제를 "빌드 절차를 손으로 쓰지 말고, 무엇을 만들 것인지만 선언하라"는 방향으로 푼다. 파일 의존성 추적, 컴파일러별 플래그 차이, 병렬 빌드 스케줄링 — 이 전부를 CMake가 대신 계산해서 실제 빌드 스크립트를 만들어 준다. 그 빌드 스크립트를 실행하는 게 다음 절의 주제다.

## CMake는 빌드 시스템이 아니라 빌드 시스템 생성기다

여기서 흔한 오해 하나를 바로잡아야 한다. `cmake --build build`를 치면 CMake가 직접 컴파일러를 불러 오브젝트 파일을 만드는 것처럼 보이지만, 실제로 컴파일을 수행하는 건 CMake가 아니다. CMake는 **빌드 시스템 생성기(generator)** 다 — `CMakeLists.txt`를 읽고 실제 빌드 시스템(전통적으로는 Makefile, 요즘은 Ninja 빌드 파일)의 설정 파일을 만들어 낼 뿐이고, 그 설정 파일을 실행해서 진짜 컴파일을 하는 건 `make`나 `ninja` 같은 별도 프로그램이다. `cmake -B build`가 "구성(configure)" 단계이고, 이 단계가 하는 일은 빌드 파일을 **생성**하는 것뿐이다. 최소 프로젝트로 이 구분을 실제로 확인한다.

```cmake title="CMakeLists.txt"
cmake_minimum_required(VERSION 3.16)
project(HexControlDemo LANGUAGES CXX)

add_executable(hex_demo main.cpp)
```

```cpp title="main.cpp"
#include <iostream>

int main() {
    std::cout << "hex_demo running\n";
    return 0;
}
```

```console
$ cmake -B build
-- The CXX compiler identification is GNU 13.3.0
-- Detecting CXX compiler ABI info
-- Detecting CXX compiler ABI info - done
-- Check for working CXX compiler: /usr/bin/c++ - skipped
-- Detecting CXX compile features
-- Detecting CXX compile features - done
-- Configuring done (0.4s)
-- Generating done (0.0s)
-- Build files have been written to: .../build
```

구성이 끝난 `build/` 디렉터리 안을 직접 열어 본다.

```console
$ ls build
CMakeCache.txt  CMakeFiles  Makefile  cmake_install.cmake
$ head -5 build/Makefile
# CMAKE generated file: DO NOT EDIT!
# Generated by "Unix Makefiles" Generator, CMake Version 3.28

# Default target executed when no arguments are given to make.
default_target: all
```

`build/Makefile`이 실제로 존재하고, 첫 줄부터 "Generated by \"Unix Makefiles\" Generator"라고 스스로 밝힌다 — CMake가 직접 만든 게 아니라 **make가 읽을 설정 파일**이다. 여기서 `cmake --build build`를 치면 그제서야 진짜 컴파일이 시작된다.

```console
$ cmake --build build
[ 50%] Building CXX object CMakeFiles/hex_demo.dir/main.cpp.o
[100%] Linking CXX executable hex_demo
[100%] Built target hex_demo
```

`cmake --build build`는 내부적으로 `build/` 안에서 `make`를 실행하는 것과 같다 — CMake는 어떤 하위 빌드 도구를 쓰든 이 명령 하나로 통일해서 감싸 준다는 게 이 래퍼의 존재 이유다. 생성기가 Unix Makefiles뿐이라고 생각하면 안 된다 — `-G` 옵션으로 다른 생성기를 지정할 수 있고, 요즘 많이 쓰는 Ninja로 바꾸면 완전히 다른 빌드 파일이 나온다.

```console
$ cmake -B build_ninja -G Ninja
-- Configuring done (0.2s)
-- Generating done (0.0s)
-- Build files have been written to: .../build_ninja
$ head -5 build_ninja/build.ninja
# CMAKE generated file: DO NOT EDIT!
# Generated by "Ninja" Generator, CMake Version 3.28

# This file contains all the build statements describing the
# compilation DAG.
$ cmake --build build_ninja
[1/2] Building CXX object CMakeFiles/hex_demo.dir/main.cpp.o
[2/2] Linking CXX executable hex_demo
$ ./build_ninja/hex_demo
hex_demo running
```

(cmake 3.28.3 / g++ 13.3.0 / Ubuntu 24.04 x86-64 실측.) 같은 `CMakeLists.txt`에서 `Makefile`이 나올 수도, `build.ninja`가 나올 수도 있다는 것 — 이게 "CMake가 빌드 시스템 그 자체가 아니라 빌드 시스템을 만드는 도구"라는 말의 실체다. 이 책은 특별한 이유가 없으면 기본 생성기(Unix Makefiles)를 쓰지만, ROS 2 생태계에서는 `colcon build`가 내부적으로 Ninja를 선택하는 경우도 흔하다 — 어느 쪽이든 `CMakeLists.txt`를 고칠 필요는 없다.

::: note project() 뒤의 LANGUAGES CXX
`project(HexControlDemo LANGUAGES CXX)`의 `LANGUAGES CXX`는 이 프로젝트가 C++만 쓴다고 못박는 것이다. 생략해도 CMake가 알아서 감지하지만, 명시하면 구성 단계에서 불필요한 C 컴파일러 탐색을 건너뛰어 조금 더 빠르고, 무엇보다 읽는 사람에게 의도를 분명히 전달한다.
:::

## 타겟 중심 사고 — 빌드의 단위는 파일이 아니다

`add_executable(hex_demo main.cpp)` 한 줄이 이 책 CMake 이야기 전체의 핵심을 이미 담고 있다. 이 줄은 "`main.cpp`를 컴파일해라"가 아니라 **"`hex_demo`라는 이름의 타겟을 만들고, 그 타겟이 `main.cpp`로 만들어진다"** 는 선언이다. 이후 이 절에서 나올 모든 함수 — `target_include_directories`, `target_link_libraries`, `target_compile_options`, `target_compile_features` — 는 전부 파일이 아니라 이 타겟에 속성을 붙이는 함수다. CMake를 배운다는 것은 결국 "이 파일에 어떤 옵션을 줄까"가 아니라 "이 타겟은 무엇을 필요로 하고, 무엇을 다른 타겟에 넘겨줄까"를 사고하는 훈련이다.

이 타겟 중심 사고를 다음 위젯으로 직접 조작해서 확인한다. 정적 라이브러리 타겟(`ik_solver`)과 그걸 링크하는 실행파일 타겟(`hex_control`) 두 개만으로, 소스 파일이 오브젝트 파일이 되고 그 오브젝트 파일들이 타겟 단위로 묶여 최종 실행파일까지 이어지는 전체 그래프를 스텝별로 재생한다.

::: widget cmake-build-graph
{}
:::

위젯의 마지막 스텝이 요점을 그대로 말한다 — 소스 3개, 오브젝트 3개, 아카이브 1개, 링크 1개인데 이 전체 그래프를 만드는 `CMakeLists.txt`는 단 세 줄이다. `add_library`와 `add_executable`이 타겟을 선언하고, `target_link_libraries`가 타겟 사이의 의존 관계를 선언한다 — 컴파일 순서, 병렬화 가능 여부, 실제 `g++ -c`와 `ld` 호출까지는 전부 CMake가 계산해서 대신 실행한다.

## 옛날 방식: 전역 설정이 프로젝트 전체를 오염시킨다

CMake 초기 예제에 흔히 나오는 `include_directories()`와 `CMAKE_CXX_FLAGS`는 **타겟이 아니라 프로젝트 전체**에 적용되는 전역 설정이다. 문제가 실제로 어떻게 나는지 보려고, 타겟이 둘인 프로젝트를 만든다 — 헥사포드 제어 실행파일 `hex_control`(자체 헤더 `legacy_math.hpp`가 필요)과, 그 프로젝트와 아무 관계도 없는 서드파티 벤더 코드 `vendor_stub`(둘 다 필요 없다) 다.

```cmake title="CMakeLists.txt — 전역 방식"
cmake_minimum_required(VERSION 3.16)
project(GlobalStyleDemo LANGUAGES CXX)

# 옛날 방식: 이 두 줄은 "프로젝트 전체"에 적용된다 -- 이 줄 아래에서 선언되는
# 모든 타겟이 강제로 물려받는다. hex_control이 필요해서 넣은 설정인데,
# 나중에 추가된 vendor_stub 타겟도 예외 없이 적용받는다.
include_directories(include)
set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -Wall -Wextra")

add_executable(hex_control main.cpp ik_solver.cpp)
add_executable(vendor_stub vendor_stub.cpp)   # legacy_math.hpp도 -Wall/-Wextra도 필요 없다
```

`vendor_stub.cpp`는 미사용 변수 하나를 일부러 남겨 뒀다 — 프로젝트가 통제할 수 없는 서드파티 코드를 흉내낸 것이다. `--verbose`로 실제 컴파일 명령을 뽑아 본다.

```console
$ cmake --build build --verbose
...
[ 80%] Building CXX object CMakeFiles/vendor_stub.dir/vendor_stub.cpp.o
/usr/bin/c++  -I/.../step2_global/include -Wall -Wextra -MD -MT ... -c vendor_stub.cpp
vendor_stub.cpp: In function 'int vendor_entry()':
vendor_stub.cpp:2:9: warning: unused variable 'unused_by_vendor' [-Wunused-variable]
```

(cmake 3.28.3 / g++ 13.3.0 실측.) `vendor_stub.cpp`를 컴파일하는 명령에 `-I.../include`와 `-Wall -Wextra`가 그대로 들어가 있다 — `include_directories()`와 `CMAKE_CXX_FLAGS`가 `hex_control`을 위해 넣은 설정인데도, 이 파일과 아무 상관 없는 `vendor_stub`까지 강제로 물려받았다는 증거다. 그 결과로 벤더 코드의 미사용 변수 경고가 이 프로젝트의 경고 목록에 섞여 들어온다 — 프로젝트가 원래 통제하려던 것도 아니고 고칠 권한도 없는 코드에 대한 경고다.

## 타겟 방식: target_* 함수로 속성을 해당 타겟에만 붙인다

같은 프로젝트를 `target_include_directories`와 `target_compile_options`로 다시 짠다. 이번에는 속성이 타겟 이름을 인자로 받는다 — "어디에 적용할지"가 함수 이름 자체에 박혀 있다.

```cmake title="CMakeLists.txt — 타겟 방식"
cmake_minimum_required(VERSION 3.16)
project(TargetStyleDemo LANGUAGES CXX)

add_executable(hex_control main.cpp ik_solver.cpp)
target_include_directories(hex_control PRIVATE include)
target_compile_options(hex_control PRIVATE -Wall -Wextra)

add_executable(vendor_stub vendor_stub.cpp)   # include도, 경고 플래그도 물려받지 않는다
```

```console
$ cmake --build build --verbose
...
[ 80%] Building CXX object CMakeFiles/vendor_stub.dir/vendor_stub.cpp.o
/usr/bin/c++    -MD -MT ... -c vendor_stub.cpp
```

(cmake 3.28.3 / g++ 13.3.0 실측.) `-I`도 `-Wall -Wextra`도 이번에는 `vendor_stub.cpp`의 컴파일 명령에 전혀 없다 — 경고도 나지 않는다. 두 프로젝트의 `hex_control` 쪽 컴파일 명령은 사실상 똑같다(둘 다 `-Iinclude -Wall -Wextra`가 붙는다). 달라진 건 딱 하나, **그 설정이 새는가 안 새는가** 다. 타겟이 둘, 셋일 때는 이 차이가 작아 보이지만, 실제 로봇 소프트웨어처럼 자체 코드·서드파티 라이브러리·테스트 실행파일이 뒤섞인 프로젝트에서는 전역 설정 한 줄이 의도치 않게 서드파티 코드의 빌드 방식까지 바꿔 버리는 사고로 이어진다. `target_include_directories`/`target_compile_options`가 나은 이유는 코드가 더 짧아서가 아니라, **의존성이 코드에 명시적으로 드러나기 때문**이다 — `hex_control` 타겟 선언을 읽는 사람은 그 타겟이 `include/`를 쓰고 `-Wall -Wextra`로 빌드된다는 걸 그 자리에서 바로 안다. 전역 설정은 파일 어딘가에 있는 `set()` 한 줄을 찾아야 알 수 있다.

::: perf 알아두면 좋은 부수 효과
전역 플래그는 `project()` 이후 선언되는 모든 타겟에 무조건 적용되므로, 프로젝트가 커질수록 "이 타겟은 왜 이 플래그가 붙었지"를 추적하기 어려워진다. 타겟 방식은 각 `target_*` 호출이 그 자체로 문서다 — 별도 주석 없이도 코드가 의존관계를 말해 준다.
:::

## PUBLIC/PRIVATE/INTERFACE — 전파 규칙을 컴파일 에러로 확인한다

`target_link_libraries`와 `target_include_directories`에는 `PRIVATE`/`PUBLIC`/`INTERFACE` 키워드가 붙는다. 이 셋은 "이 설정이 이 타겟 자신에게 적용되는가"와 "이 타겟을 링크하는 다른 타겟에도 전파되는가"를 각각 따로 결정한다.

| 키워드 | 이 타겟 자신에 적용 | 이 타겟을 링크하는 타겟에 전파 |
|---|---|---|
| `PRIVATE` | O | X |
| `INTERFACE` | X | O |
| `PUBLIC` | O | O |

말로는 추상적이니 타겟 세 개짜리 프로젝트로 직접 확인한다. `mathlib`(벡터 연산 라이브러리) → `physics`(내부적으로 `mathlib`를 쓰는 라이브러리) → `app`(`physics`만 링크하는 실행파일) 순서로 의존한다.

```cmake title="physics/physics.cpp — physics 구현 내부"
#include "physics.hpp"
#include "mathlib/vec3.hpp"   // physics 구현 내부에서만 mathlib를 쓴다
double leg_force_sum(double a, double b, double c) {
    Vec3 v = add(Vec3{a, 0, 0}, Vec3{b, c, 0});
    return v.x + v.y + v.z;
}
```

```cpp title="app/main.cpp"
#include <iostream>
#include "physics.hpp"
#include "mathlib/vec3.hpp"   // app이 mathlib 헤더를 직접 쓰려고 시도한다
int main() {
    Vec3 v{1.0, 2.0, 3.0};
    std::cout << leg_force_sum(v.x, v.y, v.z) << "\n";
}
```

`app/main.cpp`가 `physics.hpp`뿐 아니라 `mathlib/vec3.hpp`도 직접 포함한다는 데 주목하라 — `app`은 `physics`만 링크할 뿐 `mathlib`를 링크한다고 선언한 적이 없다. 이 상태에서 `physics`가 `mathlib`를 `PRIVATE`로 링크하면 어떻게 되는가.

```cmake title="CMakeLists.txt"
add_library(mathlib STATIC mathlib/src/vec3.cpp)
target_include_directories(mathlib PUBLIC mathlib/include)

add_library(physics STATIC physics/physics.cpp)
target_include_directories(physics PUBLIC physics)
target_link_libraries(physics PRIVATE mathlib)   # <- PRIVATE

add_executable(app app/main.cpp)
target_link_libraries(app PRIVATE physics)
```

```console
$ cmake --build build
[ 16%] Building CXX object CMakeFiles/mathlib.dir/mathlib/src/vec3.cpp.o
[ 33%] Linking CXX static library libmathlib.a
[ 50%] Building CXX object CMakeFiles/physics.dir/physics/physics.cpp.o
[ 66%] Linking CXX static library libphysics.a
[ 83%] Building CXX object CMakeFiles/app.dir/app/main.cpp.o
app/main.cpp:3:10: fatal error: mathlib/vec3.hpp: No such file or directory
    3 | #include "mathlib/vec3.hpp"   // app이 mathlib 헤더를 직접 쓰려고 시도한다
      |          ^~~~~~~~~~~~~~~~~~
compilation terminated.
gmake[2]: *** [CMakeFiles/app.dir/build.make:76: CMakeFiles/app.dir/app/main.cpp.o] Error 1
```

(cmake 3.28.3 / g++ 13.3.0 실측.) `mathlib`도 `physics`도 정상적으로 빌드됐지만, `app`이 `mathlib/vec3.hpp`를 찾다가 그 자리에서 죽었다 — `target_include_directories(mathlib PUBLIC mathlib/include)`가 `mathlib`를 직접 링크하는 타겟에는 그 경로를 넘겨주지만, `physics`가 `mathlib`를 `PRIVATE`로 링크했으므로 그 전파가 `physics`에서 끊긴다. `app`은 `physics`만 링크했을 뿐 `mathlib`를 링크한 적이 없으니, `mathlib`의 include 경로도 물려받지 못한다. **`PRIVATE`는 "내가 이 라이브러리를 쓴다는 사실을 나를 쓰는 쪽에는 숨긴다"는 선언이다** — `physics`의 구현이 내부적으로 `mathlib`를 쓰든 말든, `physics`의 사용자 입장에서는 알 필요도 없고 실제로 접근할 수도 없어야 한다는 뜻이다. 이 한 줄만 `PUBLIC`으로 바꾼다.

```console
$ sed -i 's/target_link_libraries(physics PRIVATE mathlib)/target_link_libraries(physics PUBLIC mathlib)/' CMakeLists.txt
$ cmake --build build
[ 16%] Building CXX object CMakeFiles/mathlib.dir/mathlib/src/vec3.cpp.o
[ 33%] Linking CXX static library libmathlib.a
[ 50%] Building CXX object CMakeFiles/physics.dir/physics/physics.cpp.o
[ 66%] Linking CXX static library libphysics.a
[ 83%] Building CXX object CMakeFiles/app.dir/app/main.cpp.o
[100%] Linking CXX executable app
$ ./build/app
6
```

(cmake 3.28.3 / g++ 13.3.0 실측.) 이번에는 `app`이 `mathlib/vec3.hpp`를 문제없이 찾았고, 실행 결과 `6`(1+2+3)도 정확하다 — `PUBLIC`은 `physics`가 이 의존성을 **자신의 인터페이스의 일부로 다시 내보낸다**는 선언이라서, `physics`를 링크하는 `app`도 `mathlib`의 include 경로를 자동으로 물려받는다. 실무에서는 `physics.hpp`가 `Vec3`를 파라미터나 반환 타입으로 쓰는 경우(헤더에 `mathlib/vec3.hpp`가 `#include`돼 있는 경우)에 `PUBLIC`을 쓴다 — `app`이 `physics.hpp`를 포함하는 순간 `mathlib`의 타입도 같이 필요해지기 때문이다. 반대로 이 절의 첫 버전처럼 `mathlib`가 `physics`의 `.cpp` 안에서만 쓰이고 `physics.hpp`에는 전혀 노출되지 않는다면 `PRIVATE`가 맞는 선택이다 — 나중에 `physics`의 내부 구현을 다른 라이브러리로 바꿔도 `app`은 전혀 영향받지 않는다.

`INTERFACE`는 세 번째 경우다 — 이 타겟 자신은 컴파일할 것이 없지만(헤더만 있는 라이브러리), 이걸 링크하는 타겟에는 설정을 넘겨줘야 할 때 쓴다.

```cmake title="CMakeLists.txt — 헤더 전용 라이브러리"
add_library(units INTERFACE)
target_include_directories(units INTERFACE units/include)

add_executable(app app/main.cpp)
target_link_libraries(app PRIVATE units)
```

```console
$ cmake --build build --verbose
/usr/bin/c++  -I/.../units/include  -MD -MT ... -c app/main.cpp
[100%] Linking CXX executable app
$ ./build/app
1.5708
```

(cmake 3.28.3 / g++ 13.3.0 실측.) `units` 타겟 자체는 컴파일 로그에 단 한 줄도 등장하지 않는다 — `.cpp`가 없으니 컴파일할 것도 없다. 그런데도 `app/main.cpp`를 컴파일하는 명령에 `-I.../units/include`가 실제로 붙었다. `INTERFACE`는 "이 설정은 나 자신에게는 적용할 게 없지만, 나를 쓰는 쪽에는 반드시 넘겨준다"는 선언이다 — 헤더 전용(`constexpr` 상수, 템플릿 등) 라이브러리를 표현하는 CMake의 표준적인 방법이다.

::: interview PUBLIC/PRIVATE/INTERFACE를 한 문장으로
면접에서 "CMake의 PRIVATE와 PUBLIC 차이를 설명하라"는 질문에는 표를 그리기보다 이렇게 답하는 게 명확하다 — "이 타겟 자신의 컴파일에 필요한가"와 "이 타겟을 링크하는 다른 타겟에도 필요한가"라는 두 개의 독립된 질문에 각각 예/아니오로 답한 조합이 `PRIVATE`(예/아니오), `INTERFACE`(아니오/예), `PUBLIC`(예/예)이다. 실무에서 가장 흔한 실수는 습관적으로 전부 `PUBLIC`을 쓰는 것 — 이러면 내부 구현 세부사항(예: 특정 수학 라이브러리를 쓴다는 사실)이 그 라이브러리의 모든 사용자에게 새어 나가고, 나중에 그 구현을 바꾸면 사용자 코드까지 재컴파일해야 하는 범위가 쓸데없이 넓어진다.
:::

## target_compile_features — 표준 버전도 타겟 단위로

[0.2](#/toolchain-setup)에서 확인했듯 g++ 13의 기본 표준은 C++20이 아니라 gnu++17이다. [0.2](#/toolchain-setup)와 [0.3](#/first-build)의 예제는 이 문제를 `set(CMAKE_CXX_STANDARD 20)`이라는 전역 변수로 해결했다 — 이것도 앞서 본 전역 설정의 일종이라, 프로젝트에 C++17로 남아 있어야 하는 레거시 타겟이 섞여 있으면 이 전역 변수 하나로는 대응할 수 없다. 타겟 단위로 표준을 지정하는 함수가 `target_compile_features`다.

```cpp title="main.cpp"
#include <compare>
#include <iostream>

struct JointAngle {
    double deg;
    auto operator<=>(const JointAngle&) const = default;   // C++20 삼중 비교
};

int main() {
    JointAngle a{30.0}, b{45.0};
    std::cout << (a < b) << "\n";
}
```

```cmake title="CMakeLists.txt"
add_executable(hex_demo main.cpp)
target_compile_features(hex_demo PRIVATE cxx_std_20)   # 표준 버전을 이 타겟에만 지정
```

```console
$ cmake --build build --verbose
/usr/bin/c++   -std=gnu++20 -MD -MT ... -c main.cpp
$ ./build/hex_demo
1
```

(cmake 3.28.3 / g++ 13.3.0 실측.) `target_compile_features`가 요구한 `cxx_std_20`이 실제 컴파일 명령의 `-std=gnu++20`으로 번역됐고, C++20 전용 문법인 `operator<=>`가 `= default`로 정상 컴파일됐다. `target_compile_features`도 `PUBLIC`으로 지정하면 이 표준 요구사항이 이 타겟을 링크하는 다른 타겟에도 전파된다 — 라이브러리 헤더가 C++20 문법을 쓴다면 그 라이브러리의 표준 요구는 `PUBLIC`으로 선언해야, 그 라이브러리를 링크하는 실행파일도 자동으로 C++20 이상으로 컴파일되게 강제할 수 있다.

## 로보틱스 도메인: ament_target_dependencies도 결국 이 위에 얹힌 것

[10.10](#/ament-colcon)에서 제대로 다룰 내용을 미리 한 줄만 짚는다. ROS 2 패키지의 `CMakeLists.txt`에 흔히 나오는 `ament_target_dependencies(내_노드 rclcpp sensor_msgs)` 같은 호출은 새로운 메커니즘이 아니다 — `find_package`로 찾은 각 ROS 2 패키지의 include 경로와 링크 대상을 모아서 이 절에서 배운 `target_include_directories`/`target_link_libraries`를 대신 불러 주는 헬퍼 함수일 뿐이다. `ros2_control`의 `hardware_interface`([10.9](#/ros2-control))가 실제 헥사포드 컨트롤러 패키지에서 어떤 타겟에 어떤 프로퍼티를 붙이는지는 그 절에서 실제 `CMakeLists.txt`를 열어 확인한다 — 이 절을 직접 짜 본 사람이라면 그 파일이 "왜 이렇게 생겼는가"를 이미 절반은 알고 시작하는 셈이다.

## 요약

- `g++` 한 줄로 파일 수십 개를 관리하는 건 재컴파일 범위도, 플래그 관리도 감당이 안 된다 — CMake는 그 절차를 손으로 짜는 대신 "무엇을 만들지"만 선언하게 해 준다.
- CMake는 빌드 시스템이 아니라 **빌드 시스템 생성기**다. `cmake -B build`가 실제로 Makefile(또는 `-G Ninja`로 지정하면 `build.ninja`)을 만들고, `cmake --build build`가 그 파일을 통해 진짜 컴파일을 수행한다.
- `add_executable`/`add_library`는 파일이 아니라 **타겟**을 선언한다 — 이후 모든 `target_*` 함수는 그 타겟에 속성을 붙인다. 이게 CMake 사고방식의 전부다.
- `include_directories()`/`CMAKE_CXX_FLAGS` 같은 전역 설정은 프로젝트의 모든 타겟에 새어 들어간다(실측: 관계없는 타겟이 불필요한 `-I`와 경고까지 물려받음). `target_include_directories`/`target_compile_options`는 지정한 타겟에만 적용된다.
- `PRIVATE`/`PUBLIC`/`INTERFACE`는 "이 타겟 자신에게 적용되는가"와 "이 타겟을 링크하는 타겟에 전파되는가"를 독립적으로 결정한다(실측: `PRIVATE` 링크는 컴파일 에러를 내고, `PUBLIC`으로 바꾸면 해결됨). `INTERFACE`는 헤더 전용 라이브러리처럼 자신은 컴파일할 게 없지만 설정만 넘겨줘야 할 때 쓴다.
- `target_compile_features(타겟 PRIVATE/PUBLIC cxx_std_20)`으로 C++ 표준 버전도 타겟 단위로 지정한다 — 실제로 `-std=gnu++20`으로 번역되는 것을 확인했다.
- ROS 2의 `ament_target_dependencies`는 새 개념이 아니라 이 절의 `target_link_libraries`/`target_include_directories` 위에 얹힌 헬퍼다.

::: quiz 연습문제
1~2번은 개념 문제, 3번은 예측 문제, 4번은 네 컴퓨터에서 직접 프로젝트를 짜고 컴파일 에러를 실제로 재현하는 실습이다.

1. "CMake는 빌드 시스템이다"라는 문장의 어디가 부정확한지, 이 절에서 실제로 확인한 `build/Makefile` 또는 `build.ninja`를 근거로 고쳐 써라.

2. `target_include_directories(A PRIVATE include)`와 `target_include_directories(A PUBLIC include)`의 차이를, "A 자신의 컴파일"과 "A를 링크하는 다른 타겟"이라는 두 대상으로 나눠 설명하라.

3. (예측) 이 절의 `mathlib`/`physics`/`app` 예제에서, `physics`는 그대로 `mathlib`를 `PRIVATE`로 링크해 두고 `physics.hpp`를 고쳐서 `#include "mathlib/vec3.hpp"`를 이 헤더 안에 넣고 `Vec3`를 반환하는 함수를 선언한다면, `app/main.cpp`가 `physics.hpp`만 포함해도 컴파일이 되는지 예측하고 근거를 써라. 힌트: 컴파일 에러는 "링크 여부"가 아니라 "헤더를 찾을 수 있는가"에서 난다.

4. (실습, 코드 작성형) 정적 라이브러리 타겟 하나(`imu_filter` — 헤더 하나와 `.cpp` 하나)와 그걸 쓰는 실행파일 타겟(`hex_app`)을 직접 만들어라. 처음에는 `target_link_libraries(hex_app PRIVATE imu_filter)`로 링크하되 `imu_filter`의 include 경로를 `PRIVATE`로 선언하고, `hex_app`의 소스에서 `imu_filter`의 헤더를 직접 포함시켜 컴파일이 실패하는 것을 확인하라. 그다음 `imu_filter`의 include 경로 선언을 `PUBLIC`으로 바꿔 같은 코드가 컴파일되는 것까지 확인하라. `cmake -B build && cmake --build build`로 두 상태를 각각 재현한다.
:::

::: answer 해설
1. "CMake는 빌드 시스템이다"는 CMake가 직접 컴파일을 수행한다고 오해하게 만든다. 정확히는 CMake는 **빌드 시스템 생성기**이고, 실제 컴파일은 CMake가 만들어 낸 `build/Makefile`(기본 생성기 기준)이나 `build/build.ninja`(`-G Ninja` 지정 시)를 `make`나 `ninja`가 읽어서 수행한다 — 두 파일 모두 첫 줄에 "Generated by ... Generator"라고 스스로 밝히는 걸 이 절에서 직접 확인했다. `cmake --build build`는 이 하위 도구 호출을 감춰 주는 통일된 인터페이스일 뿐이다.
2. `PRIVATE`는 이 include 경로가 A 자신의 소스 파일을 컴파일할 때만 쓰이고, A를 링크하는 다른 타겟에는 전혀 전달되지 않는다. `PUBLIC`은 A 자신에게도 적용되고, 동시에 A를 링크하는 모든 타겟의 컴파일 명령에도 같은 include 경로가 자동으로 추가된다 — 이 절의 `mathlib`/`physics`/`app` 실험에서 `PRIVATE`일 때는 `app`이 `mathlib`의 헤더를 못 찾아 `fatal error`가 났고, `PUBLIC`으로 바꾸자 그 경로가 `app`까지 전파돼 컴파일이 통과했다.
3. 컴파일된다 — 다만 이유가 미묘하다. `app/main.cpp`가 `physics.hpp`만 포함해도, `physics.hpp` 안에 `#include "mathlib/vec3.hpp"`가 있으므로 전처리기는 그 include 지시문을 그대로 따라가 `mathlib/vec3.hpp`를 찾으려 시도한다. 이때 필요한 건 "`mathlib`를 링크했는가"가 아니라 "그 헤더 파일을 찾을 수 있는 include 경로가 있는가"이다. `physics`가 `mathlib`를 `PRIVATE`로 링크한 상태라면 `mathlib`의 include 경로가 `app`에 전파되지 않으므로, `physics.hpp`가 아무리 `mathlib/vec3.hpp`를 포함해도 `app`의 컴파일 명령에는 그 경로가 없다 — 즉 여전히 실패한다. `physics.hpp`가 다른 타겟에 노출되는 헤더에서 `mathlib`를 쓰는 순간, `mathlib` 링크는 `PUBLIC`으로 바꿔야 한다는 게 이 문제의 핵심이다.
4. 실제로 실행하면, `imu_filter`의 include 경로가 `PRIVATE`인 첫 상태에서는 `hex_app`의 소스 파일을 컴파일하는 단계에서 "`No such file or directory`" 에러가 나야 한다 — 이 절의 `mathlib`/`app` 실험과 정확히 같은 형태다. include 경로 선언을 `PUBLIC`으로 바꾸고 다시 `cmake -B build && cmake --build build`를 실행하면 그 에러가 사라지고 링크까지 끝나야 한다. 두 상태 모두 직접 재현했다면 `PRIVATE`/`PUBLIC`이 만드는 차이를 눈이 아니라 컴파일러 에러 메시지로 확인한 것이다.
:::

이 절의 세 CMake 프로젝트(전역 방식 대 타겟 방식, `PRIVATE`/`PUBLIC` 대조, `INTERFACE` 라이브러리)는 전부 직접 타이핑하고 컴파일해 봐야 남는다 — 특히 `PRIVATE`에서 컴파일 에러가 실제로 나는 순간과, `PUBLIC`으로 바꿔서 그 에러가 사라지는 순간을 두 눈으로 봐야 이 절의 표가 추상적인 규칙이 아니라 실제로 컴파일러가 지키는 규칙이라는 게 몸에 남는다. 기준 명령: `cmake -B build && cmake --build build`.
