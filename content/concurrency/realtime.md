# 6.8 실시간 제약과 제어 루프

::: lead
[6.1](#/threads)에서 스레드 생성 비용이 마이크로초 단위로 흔들린다는 걸 봤고, [6.3](#/mutex)에서 뮤텍스 경합이 futex 대기로 넘어가면 비용이 7배 넘게 뛴다는 걸 실측했고, [6.6](#/lockfree)에서는 그 경합 자체를 피하려고 SPSC 링 버퍼를 락 없이 짰다. 세 절 모두 이 절을 예고하며 끝났다 — 지금까지 본 "느려질 수 있다"는 문제들이 헥사포드의 관절 제어 루프에서는 왜 "느려지면 안 된다"는 문제로 바뀌는지, 이 절이 정면으로 다룬다. 실시간(real-time)이 정확히 무엇을 뜻하는지부터 바로잡고, 1ms 주기 제어 루프를 실제로 돌려 지터를 측정하고, 그 지터를 만드는 원인(힙 할당, 뮤텍스 경합, 페이지 폴트) 하나하나를 실측으로 재현한 다음, 이걸 피하는 실전 기법(사전 할당, lock-free, 실시간 스케줄링 정책)과 우선순위 역전이라는 새로운 함정까지 전부 실측으로 확인한다.
:::

## "실시간"은 빠르다는 뜻이 아니다

헥사포드가 걷는 동안 제어 루프는 이런 일을 반복한다 — IMU와 다리 센서에서 현재 자세를 읽고, 여섯 다리 각각의 목표 관절각을 계산하고([9.5](#/inverse-kinematics)의 역기구학이 여기 쓰인다), 그 값을 액추에이터로 내보낸다. 이 사이클을 1ms나 몇 ms 같은 고정 주기로 계속 돈다. 여기서 "실시간"이 뜻하는 건 "이 계산이 빠르다"가 아니라 **"이 계산이 정해진 마감시간(deadline) 안에 반드시 끝난다"**는 것이다. 이 둘은 완전히 다른 요구다 — 평균 실행시간이 100마이크로초로 매우 빠른 루프라도, 10만 번에 한 번 20밀리초짜리 지연이 섞이면 그 루프는 실시간이 아니다. 반대로 평균이 좀 느려도 매번 정확히 예측 가능한 시간 안에 끝난다면 그건 실시간 시스템이다. **속도의 문제가 아니라 예측 가능성의 문제다.**

이 구분을 몸으로 느끼려면 실제로 마감을 놓치면 무슨 일이 나는지 생각해 봐야 한다. 다리 하나의 목표 관절각 계산이 이번 주기 안에 안 끝나면, 그 다리는 지난 주기의(즉 낡은) 목표를 향해 계속 움직이거나 아예 명령을 못 받은 채로 멈춘다. 나머지 다섯 다리는 이번 주기 명령대로 움직이는데 그 다리만 어긋나면, 로봇 전체의 무게 중심을 지지하던 다리 배치가 한순간 깨진다 — 몸이 기울거나, 심하면 넘어진다. 이게 **hard real-time**이 말하는 것이다: 마감을 한 번이라도 놓치면 그 자체로 시스템이 실패한 것으로 친다. 평균이 아무리 좋아도 예외가 없다. (참고로 "얼마 놓쳐도 성능만 조금 떨어지는" soft real-time도 있지만 — 예를 들어 화면 렌더링 프레임 하나가 늦으면 화면이 좀 끊길 뿐이다 — 로봇 제어 루프는 이 절에서 계속 hard real-time을 기준으로 이야기한다.)

## 1ms 제어 루프의 지터를 실측한다

말로만 하지 말고 직접 재 본다. 1ms 주기로 계속 도는 루프를 만들어서, 매 주기가 **의도한 시각보다 실제로 얼마나 늦게 시작됐는지**를 측정한다. 이 지연을 **지터(jitter)**라고 부른다.

```cpp title="rt1_jitter.cpp — 1ms 주기 루프의 지터 측정"
#include <algorithm>
#include <chrono>
#include <iostream>
#include <thread>
#include <vector>

using clk = std::chrono::steady_clock;

int main() {
    constexpr int N = 5000;
    constexpr auto period = std::chrono::microseconds(1000);   // 1ms 주기
    std::vector<double> jitter_us;
    jitter_us.reserve(N);

    auto t0 = clk::now();
    for (int i = 0; i < N; ++i) {
        auto target = t0 + period * i;          // 이번 주기가 시작됐어야 할 시각
        std::this_thread::sleep_until(target);
        auto actual = clk::now();                // 실제로 깨어난 시각
        jitter_us.push_back(
            std::chrono::duration<double, std::micro>(actual - target).count());

        volatile double x = 0;                   // 제어 루프 본체 흉내 -- 순수 연산만
        for (int k = 0; k < 200; ++k) x += k * 0.0001;
    }
    std::sort(jitter_us.begin(), jitter_us.end());
    std::cout << "p50=" << jitter_us[N / 2] << " p90=" << jitter_us[N * 9 / 10]
              << " p99=" << jitter_us[N * 99 / 100] << " 최악=" << jitter_us.back() << " (us)\n";
}
```

```console
$ g++ -std=c++20 -O2 -Wall -Wextra rt1_jitter.cpp -o rt1_jitter -lpthread
$ ./rt1_jitter
p50=124.782 p90=480.509 p99=9161.67 max=84917.9 (us)
```

(g++ 13.3.0 / Ubuntu 24.04 / x86-64, 가상화된 실습 환경 실측. 여러 번 반복하면 p50은 대략 110~130us로 안정적이지만, p99와 최악값은 반복마다 수백에서 10만 마이크로초까지 크게 흔들렸다.) 이 숫자를 그대로 읽으면 이 절의 요점이 바로 보인다 — **중앙값(p50)은 130us 근처로 멀쩡한데, 최악의 경우는 그보다 600배 넘게 큰 8만 4천 마이크로초(84.9ms)까지 튄다.** 1ms 주기 루프에서 84.9ms짜리 지연이 단 한 번이라도 나온다는 건, 이 루프가 84번의 주기를 통째로 건너뛴 것과 같다 — 앞 절에서 본 "평균이 아니라 최악이 기준"이라는 원칙이 정확히 이 수치로 나타난다.

::: warn 이 환경 자체가 실시간 검증에 부적합하다
이 최악값(84.9ms)의 상당 부분은 이 실습 환경이 여러 사용자가 나눠 쓰는 가상화된 컨테이너라는 사실에서 온다 — 게스트 OS의 스케줄러가 아무리 잘 짜여 있어도, 호스트 하이퍼바이저가 이 가상 CPU에 실제 물리 코어 시간을 배분하지 않는 구간(steal time)은 게스트 커널이 전혀 통제할 수 없다. 실제 로봇에 들어가는 실시간 제어 소프트웨어는 이런 공유 가상화 환경이 아니라 전용 하드웨어(또는 최소한 전용 코어로 격리된 환경)에서, 필요하면 `PREEMPT_RT` 패치가 적용된 커널 위에서 검증한다. 이 절의 수치는 원인별 상대적 차이(할당 있음/없음, 락 있음/없음)를 보여주는 데는 유효하지만, 절대적인 "이 정도면 실시간으로 충분하다"는 기준으로 읽으면 안 된다.
:::

## 무엇이 최악의 경우를 만드는가

p50과 최악값 사이의 이 간극을 만드는 원인은 여러 개가 겹친다. 이 절은 그중 코드로 직접 통제할 수 있는 세 가지 — 힙 할당, 뮤텍스 경합, 페이지 폴트 — 를 하나씩 따로 떼어 실측한다.

### 힙 할당: 경합이 붙으면 최악값이 100배 뛴다

[5.1 vector](#/vector)에서 재할당 없는 `push_back` 한 번은 25ns인데 재할당 한 번(원소 524,288개를 옮기는 경우)은 1.33ms — 5만 배 넘게 느리다는 걸 실측했다. 그 비대칭이 단일 스레드에서도 이 정도인데, 제어 루프가 도는 동안 **다른 스레드들도 같은 힙 할당자를 동시에 두드리면** 어떻게 되는지 잰다.

```cpp title="rt2_heap_tail.cpp — 힙 할당 1회의 지연 분포, 경합 없음 vs 있음(핵심부)"
std::atomic<bool> stop_bg{false};
void background_allocator() {                 // 다른 스레드(로거, 진단 스레드 등)가 계속 malloc/free
    while (!stop_bg.load(std::memory_order_relaxed)) {
        auto* p = new char[256]; p[0] = 1; delete[] p;
    }
}

int main(int argc, char** argv) {
    bool contended = argc > 1;
    std::vector<std::thread> bg;
    if (contended) for (int i = 0; i < 8; ++i) bg.emplace_back(background_allocator);

    std::vector<double> lat_us;
    for (int i = 0; i < 200000; ++i) {
        auto s = clk::now();
        auto* buf = new double[64];             // 제어 루프 본체가 직접 할당하는 임시 버퍼
        for (int k = 0; k < 64; ++k) buf[k] = k * 0.001;
        delete[] buf;
        lat_us.push_back(std::chrono::duration<double, std::micro>(clk::now() - s).count());
    }
    // ... 정렬 후 p50/p99/최악 출력
}
```

```console
$ g++ -std=c++20 -O2 -Wall -Wextra rt2_heap_tail.cpp -o rt2_heap_tail -lpthread
$ ./rt2_heap_tail
[경합 없음(단독)] 평균=0.0605us p50=0.054us p99=0.158us p99.9=0.291us 최악=46.3us
$ ./rt2_heap_tail x
[경합 있음(백그라운드 할당 스레드 8개)] 평균=0.102us p50=0.054us p99=0.170us p99.9=0.399us 최악=4041.9us
```

(g++ 13.3.0 실측, 여러 번 반복.) p50(0.054us)은 경합 유무와 무관하게 거의 그대로다 — 즉 **"보통의 경우"만 보면 힙 할당이 위험해 보이지 않는다.** 최악값을 보면 얘기가 달라진다 — 경합이 없을 때 46~71us 사이였던 최악값이, 백그라운드 스레드 8개가 같은 할당자를 두드리는 순간 4~12밀리초까지 치솟는다(반복 실행에서 4041us, 8083us, 12031us 등으로 계속 확인됨). **약 100배 차이다.** glibc의 `malloc`은 여러 스레드가 동시에 힙을 건드리면 내부 잠금(아레나 락)으로 순서를 맞추는데, 이 잠금에서 경합이 생기면 [6.3](#/mutex)에서 본 futex 대기 경로를 그대로 타기 때문이다 — 제어 루프가 필요로 한 건 64개짜리 `double` 배열 하나뿐인데, 그 배열을 받는 순간이 하필 다른 스레드의 할당과 겹치면 대기열에 줄을 서야 한다.

### 뮤텍스 경합이 지터로 바로 번진다

같은 실험을 뮤텍스로 반복한다. 제어 루프가 매 주기 공유 자세 추정치를 읽어야 한다고 하고([6.3](#/mutex)의 `mut8_cost.cpp`와 같은 구조), 다른 스레드들이 그 값을 계속 갱신한다.

```console
$ g++ -std=c++20 -O2 -Wall -Wextra rt3_mutex_tail.cpp -o rt3_mutex_tail -lpthread
$ ./rt3_mutex_tail
[경합 없음(단독)] 평균=32.3ns p50=27ns p99=101ns p99.9=241ns 최악=63.3us
$ ./rt3_mutex_tail x
[경합 있음(백그라운드 갱신 스레드 8개)] 평균=2029.1ns p50=64ns p99=44692ns p99.9=212194ns 최악=32622.7us
```

(g++ 13.3.0 실측, 백그라운드 갱신 스레드 8개.) [6.3](#/mutex)에서 본 "경합이 붙으면 futex가 커널 대기 큐를 탄다"는 것이 여기서 지터로 직접 나타난다 — 경합 없는 최악값이 63.3us였는데, 경합이 붙자 최악값이 **32.6밀리초**까지 뛰었다(약 500배). p99도 101ns에서 44,692ns로 400배 넘게 벌어졌다. 제어 루프가 뮤텍스 한 번 잠그는 데 32ms가 걸린다는 건, 1ms 주기 루프 입장에서는 32번의 주기를 놓쳤다는 뜻이다 — [6.6](#/lockfree)이 "실시간 제약이 있으면 락을 없앨 강력한 이유가 된다"고 말한 게 바로 이 숫자다.

### 페이지 폴트: 처음 만지는 메모리는 비싸다

`new`/`malloc`이 돌려준 주소는 가상 메모리일 뿐, 그 주소를 실제로 처음 **쓰기**로 건드리는 순간에야 커널이 물리 페이지를 배정한다(페이지 폴트). 이미 한 번 만져 둔 메모리를 재사용하는 것과, 매번 새로 매핑받은 메모리를 처음 만지는 것을 비교한다.

```console
$ g++ -std=c++20 -O2 -Wall -Wextra rt4_pagefault.cpp -o rt4_pagefault
$ ./rt4_pagefault              # 이미 터치된 매핑을 재사용
[이미 터치된 매핑 재사용] p50=0.241us p99=0.442us p99.9=3.363us 최악=5.013us
$ ./rt4_pagefault fresh        # 매번 새 mmap을 받아 처음 만짐
[매번 새 mmap, 최초 터치] p50=75.956us p99=148.306us p99.9=227.237us 최악=290.0us
```

(g++ 13.3.0 실측, 256KB 버퍼 64페이지 기준.) 이미 터치된 메모리를 다시 쓰는 건 p50이 0.24us지만, 처음 만지는 메모리는 p50부터 76us — **약 315배** 느리다. 페이지 하나당 커널이 물리 프레임을 찾아 매핑하고 0으로 채우는 작업이 필요하기 때문이다. 힙 할당·뮤텍스 경합과 달리 이건 "가끔 튀는" 게 아니라 **첫 접근마다 항상** 발생한다는 점이 다르다 — 그래서 회피법도 다르다: 아예 미리 다 만져 두면 된다.

### 예외 던지기와 시스템 콜도 마찬가지다

힙·뮤텍스·페이지 폴트만큼 자주 걸리지는 않지만, 제어 루프 안에서 똑같이 피해야 할 것이 둘 더 있다 — 예외를 던지는 것과 시스템 콜(파일 I/O, 로깅)을 부르는 것이다. 둘 다 직접 재 본다.

```console
$ g++ -std=c++20 -O2 -Wall -Wextra rt7_exception_cost.cpp -o rt7_exception_cost
$ ./rt7_exception_cost
정상 리턴 (3단계 호출): 2.40 ns/회
예외 throw+catch (3단계 되감기): 2228.4 ns/회
배율: 927.6x
```

(g++ 13.3.0 실측, 3단계 깊이의 호출 스택에서 예외를 던지고 잡는 경우.) C++ 예외는 "안 던지는 경로는 공짜"라는 설계(zero-cost exception)로 유명하지만, 이건 **안 던졌을 때** 얘기다. 실제로 던지는 순간에는 스택을 되감으며 각 프레임의 소멸자를 호출하고, 어떤 `catch`가 이 타입을 받을 수 있는지 RTTI로 찾는 과정이 필요하다 — 이 절 실측에서 정상 리턴보다 **900배 넘게** 느렸다. 제어 루프 안에서 "예외적인 상황"(센서 타임아웃, 범위 밖 값)을 예외로 처리하면, 그 상황이 실제로 발생하는 주기만 마감을 놓친다 — 대신 [5.9 optional/variant/expected](#/vocabulary-types)의 값 기반 오류 표현을 쓰면 이 되감기 비용 자체가 없다.

```console
$ g++ -std=c++20 -O2 -Wall -Wextra rt8_syscall_cost.cpp -o rt8_syscall_cost
$ ./rt8_syscall_cost
순수 연산만: 57.8 ns/회
write() 시스템 콜 1회(파일 로깅): 605.1 ns/회
배율: 10.5x
```

(g++ 13.3.0 실측, 파일에 24바이트 한 줄을 `write()`하는 경우 — 터미널 출력의 추가 지연은 포함하지 않은, 순수 시스템 콜 비용만이다.) 로그 한 줄을 남기는 것뿐인 `write()` 호출도 순수 연산보다 10배 이상 느리다 — 커널로 넘어갔다 돌아오는 문맥 전환 자체가 비용이고, 로그 대상이 디스크나 느린 콘솔이면 이보다 훨씬 나빠질 수 있다(이 절의 나머지 실측처럼 최악값은 특히 예측하기 어렵다). 제어 루프 안에서 매 주기 로그를 남기고 싶다면, 그 자리에서 직접 `write`/`printf`를 부르지 말고 [6.6](#/lockfree)의 SPSC 큐로 로그 메시지만 별도의 로깅 스레드에 넘기고, 실제 시스템 콜은 그 로깅 스레드가 실시간 제약 밖에서 처리하게 한다.

::: danger 제어 루프 본체 체크리스트 — 이 다섯을 부르고 있다면 멈춰라
`new`/`malloc`(직접 호출은 물론, `std::vector::push_back`처럼 내부에서 재할당을 부를 수 있는 모든 STL 연산 포함 — [5.1](#/vector) 참고), 경합 가능성이 있는 뮤텍스 `lock()`([6.3](#/mutex)), `throw`(그리고 `throw`를 부를 수 있는 모든 하위 호출), 파일 I/O·로깅·네트워크 같은 시스템 콜, 그리고 처음 만지는 메모리(사전 터치 안 된 버퍼). 이 다섯 중 하나라도 제어 루프 본체 안에 있다면, 이 절에서 실측한 대로 그 주기의 최악값이 수백~수만 배 튈 수 있다.
:::

## 회피 기법 1: 사전 할당 — 제어 루프에 들어가기 전에 전부 끝낸다

앞의 세 실측이 가리키는 결론은 하나다 — **제어 루프 본체 안에서 `new`/`malloc`을 부르지 않는다.** 필요한 버퍼는 프로그램 시작 시점, 즉 제어 루프에 진입하기 전에 전부 할당하고 한 번 이상 실제로 써서 페이지 폴트까지 미리 끝내 둔다. 이게 **사전 할당(pre-allocation)**이다. [5.1 vector](#/vector)의 `reserve()`가 정확히 이 원칙의 한 사례였다 — 최종 원소 개수를 미리 알면 재할당이 일어날 여지 자체를 없애서, 그 절 실측에서 3배 넘는 속도 차이를 만들었다. 제어 루프에서는 이걸 한 단계 더 철저하게 적용한다: 목표 관절각 배열, IK 계산용 임시 행렬, 로깅 버퍼 — 전부 루프 진입 전에 `reserve`하거나 고정 크기 `std::array`로 잡고, 그 이후로는 루프 안에서 크기가 바뀌는 컨테이너를 아예 쓰지 않는다.

::: perf 사전 할당은 새로 배우는 기법이 아니다
[5.1 vector](#/vector)의 `reserve()` 실측(재할당 있음 41.7~47.8ms 대 없음 13.1~13.5ms, 3배 이상 차이)이 이미 사전 할당의 한 사례였다는 걸 이 절에서 다시 확인한 셈이다. 제어 루프에서 새로 요구되는 건 원칙이 아니라 **철저함**이다 — 컨테이너 하나 정도가 아니라 루프 안에서 쓰는 모든 동적 자료구조에 예외 없이 적용해야 한다는 것.
:::

## 회피 기법 2: lock-free 자료구조로 락 자체를 없앤다

뮤텍스 경합의 근본 해법은 "잠글 필요가 없게 만드는" 것이다. [6.6](#/lockfree)에서 만든 SPSC 링 버퍼가 정확히 이 자리를 위한 도구다 — 센서 드라이버 스레드가 데이터를 채우고 제어 루프 스레드가 소비하는 구조는 생산자·소비자가 각각 하나로 고정되므로, `try_pop`이 데이터가 없으면 즉시 `false`를 돌려줄 뿐 절대 블록되지 않는다. 이 절에서 실측한 32.6ms짜리 뮤텍스 최악값이 제어 루프에서는 원천적으로 나올 수 없게 된다 — 커널 대기 큐에 들어갈 일 자체가 없기 때문이다.

## 회피 기법 3: 실시간 스케줄링 정책 — SCHED_FIFO

지금까지는 애플리케이션 코드가 만드는 지연이었다. 하지만 아무리 코드가 깨끗해도, 리눅스의 기본 스케줄링 정책인 `SCHED_OTHER`는 "공평하게 나눠 쓰기"가 목표이지 "이 스레드는 반드시 먼저 실행돼야 한다"를 보장하지 않는다. `SCHED_FIFO`(그리고 `SCHED_RR`)는 실시간 정책으로, 더 높은 우선순위의 `SCHED_FIFO` 스레드가 실행 가능해지면 낮은 우선순위 스레드를(그리고 모든 `SCHED_OTHER` 스레드를) 즉시 선점한다. 셸에서는 `chrt`, 코드에서는 `pthread_setschedparam`으로 적용한다.

```cpp title="이 스레드를 SCHED_FIFO 우선순위 90으로 올린다"
sched_param sp{};
sp.sched_priority = 90;                                    // SCHED_FIFO 범위: 1~99
int rc = pthread_setschedparam(pthread_self(), SCHED_FIFO, &sp);
```

```console
$ ulimit -r
0
$ ./rt_sched_check
pthread_setschedparam 성공, SCHED_FIFO priority 80 적용됨
```

(g++ 13.3.0 / 이 실습 환경 실측.) `ulimit -r`(실시간 우선순위 상한)이 0인데도 성공했다 — 이 실습 환경이 root 권한으로 실행되고 있어서 `CAP_SYS_NICE`가 `RLIMIT_RTPRIO` 제한을 우회하기 때문이다. **일반 사용자 계정에서는 이 호출이 `EPERM`으로 실패할 수 있다** — 그럴 때는 `/etc/security/limits.conf`에 `<사용자> - rtprio 99` 같은 줄을 추가하거나, `CAP_SYS_NICE` capability를 부여받아야 한다. 추측하지 않고 실제로 시도해서 나온 결과를 그대로 적는다 — 이 환경에서는 됐고, 일반 계정에서는 안 될 수 있다는 것이 정직한 결론이다.

이 정책이 실제로 최악값을 줄이는지, 모든 코어에 CPU 부하를 걸어 스케줄러가 실제로 선점 결정을 내리게 만든 뒤 잰다.

```console
$ ./rt5_sched                 # SCHED_OTHER(기본), 전 코어 부하 상태
p50=82.294 p90=95.503 p99=122.233 max=7466.29 (us)
$ ./rt5_sched fifo             # SCHED_FIFO(90), 전 코어 부하 상태
p50=32.040 p90=44.698 p99=56.979 max=1598.09 (us)
```

(g++ 13.3.0 실측, 4코어 전부에 CPU 바운드 스레드를 띄운 상태.) p50이 82us에서 32us로, p90이 95.5us에서 44.7us로 줄었다 — `SCHED_FIFO`가 실제로 다른 부하 스레드들에 밀리지 않고 제때 실행됐다는 뜻이다. 최악값(7466us → 1598us)도 개선됐지만 이 환경의 하이퍼바이저 노이즈가 여전히 섞여 있어 앞의 p50/p90만큼 깨끗하지는 않다 — 그래도 방향은 반복 실행마다 일관되게 `SCHED_FIFO` 쪽이 나았다.

::: tip 지금 이 프로세스의 스케줄링 정책을 바로 확인한다
`chrt -p <PID>`로 실행 중인 프로세스(또는 스레드)의 현재 스케줄링 정책과 우선순위를 즉시 확인할 수 있다 — 코드를 다시 컴파일하지 않고도 "이 스레드가 정말 SCHED_FIFO로 떴는가"를 검증하는 가장 빠른 방법이다. 배포 스크립트나 launch 파일에 `chrt -f 80 ./control_loop_node`처럼 실행 명령 자체를 감싸는 방법도 있다 — 코드 안에서 `pthread_setschedparam`을 호출하는 대신, 프로세스를 실행하는 시점에 정책을 정하고 싶을 때 쓴다.
:::

## 우선순위 역전: 낮은 우선순위가 높은 우선순위를 막는다

`SCHED_FIFO`로 우선순위를 매겼다고 다 끝난 게 아니다. 우선순위가 다른 스레드들이 뮤텍스를 공유하면 **우선순위 역전(priority inversion)**이라는 역설적인 상황이 생긴다 — 가장 높은 우선순위 스레드가, 자신보다 우선순위가 훨씬 낮은 스레드 때문에 막힌다. 시나리오는 이렇다. 낮은 우선순위 스레드(low)가 뮤텍스를 잠그고 짧은 임계 구역을 실행한다. 높은 우선순위 스레드(high)가 같은 뮤텍스를 기다리며 블록된다. 그런데 그 뮤텍스와는 전혀 무관한, 우선순위가 low보다는 높고 high보다는 낮은 스레드(medium)가 CPU 바운드 작업으로 끼어든다 — SCHED_FIFO는 엄격한 우선순위 선점이므로 medium이 low를 그냥 밀어낸다. low는 CPU를 못 받으니 임계 구역을 끝낼 수도, 뮤텍스를 풀 수도 없다. high는 뮤텍스를 쥔 low가 아니라, 뮤텍스와 아무 상관도 없는 medium 때문에 대기가 무한정 길어진다.

실제로 재현한다 — low(우선순위 10)는 뮤텍스를 잠그고 방해 없이 돌면 약 56ms에 끝나는 고정된 작업을 한다. high(우선순위 90)는 그 뮤텍스를 잠그려 한다. medium(우선순위 50)은 뮤텍스와 무관한 약 300ms짜리 CPU 작업을 한다. 세 스레드를 코어 하나에 묶어야 우선순위가 실제로 경쟁에 영향을 준다.

```cpp title="rt6_priority_inversion.cpp — 우선순위 역전 재현(핵심부)"
void* low_thread(void*) {
    pthread_mutex_lock(&g_mutex);
    for (long i = 0; i < kLowWorkIters; ++i) x += 1.0000001;   // 방해 없으면 약 56ms
    pthread_mutex_unlock(&g_mutex);
}
void* high_thread(void*) {
    auto t_try = clk::now();
    pthread_mutex_lock(&g_mutex);                               // low가 쥐고 있으면 블록
    auto wait_ms = /* t_try부터 여기까지 */;
    pthread_mutex_unlock(&g_mutex);
}
void* medium_thread(void*) {
    for (long i = 0; i < kMediumHogIters; ++i) x += 1.0000001;  // 뮤텍스는 전혀 안 건드림, 약 300ms
}
```

```console
$ ./rt6_priority_inversion
-- 실험 1: 우선순위 상속 없는 일반 뮤텍스 --
[상속 없음] high가 락을 기다린 시간: 347.323 ms   (low의 실제 작업 시간: 347.703 ms)
```

(g++ 13.3.0 실측, 세 스레드 모두 CPU 0에 고정, `SCHED_FIFO` 적용.) low가 진짜로 필요로 한 시간은 56ms인데, high는 **347ms**를 기다렸다 — low 자신의 작업 시간도 347ms로 늘어났다. medium이 뮤텍스와 전혀 무관한데도 low를 계속 밀어낸 탓에, low는 자기 임계 구역을 끝내지 못했고, high는 그 사이 아무것도 못 하고 대기했다. 반복 실행에서 340~365ms 사이로 일관되게 나타났다(다만 medium의 개입 타이밍이 어긋나면 어쩌다 한 번은 지연이 거의 없기도 했다 — 이 재현 여부 자체가 스케줄링 타이밍에 좌우된다는 게 우선순위 역전이 테스트로 잡기 어려운 이유다).

**우선순위 상속(priority inheritance)** 뮤텍스는 이 문제를 이렇게 고친다 — 높은 우선순위 스레드가 뮤텍스를 기다리며 블록되는 순간, 그 뮤텍스를 쥔 스레드의 우선순위를 일시적으로 대기자 수준까지 끌어올린다. 그러면 low는 잠깐 동안 medium보다 우선순위가 높아지므로 medium에게 밀리지 않고 자기 임계 구역을 끝낼 수 있다. POSIX는 `pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT)`로 이 프로토콜을 켠다.

```console
$ ./rt6_priority_inversion
-- 실험 2: PTHREAD_PRIO_INHERIT 뮤텍스 --
[상속 있음] high가 락을 기다린 시간:  56.384 ms   (low의 실제 작업 시간: 57.329 ms)
```

(g++ 13.3.0 실측, 뮤텍스 생성 시 `PTHREAD_PRIO_INHERIT`만 추가하고 나머지는 동일한 코드.) high의 대기시간이 56.4ms로 줄었다 — low가 "진짜로" 필요로 했던 56ms와 거의 정확히 일치한다. 반복 실행에서도 55.8~56.5ms로 항상 이 범위에 머물렀다. medium은 이제 high가 뮤텍스를 받은 뒤에야 실행되기 시작한다 — 뮤텍스를 쥔 동안만 우선순위를 잠깐 빌려주는 것으로, 원래 있으면 안 될 지연을 없앤 것이다.

::: interview 우선순위 역전이 뭐고 어떻게 막는가
답변 뼈대: ① 낮은 우선순위 스레드가 뮤텍스를 쥔 채로, 그 뮤텍스와 무관한 중간 우선순위 스레드에 CPU를 뺏겨 임계 구역을 못 끝내면, 그 뮤텍스를 기다리는 더 높은 우선순위 스레드까지 덩달아 무한정 대기하게 된다 — 이게 우선순위 역전이다. ② 이 절의 실측처럼, 일반 뮤텍스에서는 high의 대기시간이 (low의 진짜 필요 시간이 아니라) medium의 작업 시간에 좌우돼 300ms대까지 늘어났다. ③ 해법은 우선순위 상속 프로토콜(`PTHREAD_PRIO_INHERIT`)이다 — 뮤텍스를 쥔 스레드의 우선순위를 대기자 수준으로 일시적으로 끌어올려, 그 스레드가 무관한 중간 우선순위 스레드에 밀리지 않게 만든다. 이 절의 실측에서 대기시간이 347ms에서 56ms로, low의 실제 필요 시간과 거의 같아진 게 그 증거다.
:::

## 로보틱스 연결: ros2_control이 이 모든 걸 한곳에 모은다

이 절에서 실측한 원인(힙 할당, 뮤텍스 경합, 페이지 폴트)과 회피법(사전 할당, lock-free, `SCHED_FIFO`, 우선순위 상속)은 낱개로 보면 각자의 최적화 기법처럼 보이지만, `ros2_control`의 `hardware_interface`는 이 전부를 한 아키텍처 안에 강제한다 — 하드웨어 I/O를 처리하는 스레드와 제어 계산을 하는 실시간 스레드를 분리하고, 그 실시간 스레드에는 `SCHED_FIFO`를 걸고, 스레드 사이의 데이터 교환은 [6.6](#/lockfree)의 lock-free 자료구조로 하고, 필요한 버퍼는 초기화 단계에서 전부 사전 할당한다. [6.9 스레드 아키텍처 설계](#/thread-architecture)에서 이 구조를 처음부터 직접 설계해 보고, [10.9 ros2_control과 hardware_interface](#/ros2-control)에서 실제 프레임워크가 이걸 어떻게 구현했는지 코드로 확인한다.

## 요약

- 실시간은 "빠르다"가 아니라 "마감시간을 반드시 지킨다"는 뜻이다 — 평균이 아무리 좋아도 어쩌다 한 번 마감을 놓치면 hard real-time 기준으로는 실패다.
- 1ms 주기 제어 루프의 지터를 실측하면 p50은 100us대로 멀쩡해도 최악값은 그보다 수백~수천 배 큰 밀리초 단위까지 튄다(실측: `rt1_jitter`).
- 힙 할당은 경합이 없으면 위험해 보이지 않지만(p50 변화 없음), 다른 스레드가 같은 할당자를 두드리면 최악값이 약 100배 뛴다(실측: 46us → 4~12ms).
- 뮤텍스 경합은 [6.3](#/mutex)의 futex 대기 경로를 그대로 타 최악값이 500배까지 뛴다(실측: 63.3us → 32.6ms).
- 처음 만지는 메모리는 페이지 폴트 때문에 p50부터 약 300배 느리다(실측: 0.24us → 76us) — 힙/뮤텍스와 달리 이건 매번 확실히 발생하므로 사전 할당으로 미리 없앤다.
- 회피 기법 셋: 제어 루프 진입 전에 전부 할당·터치해 두는 사전 할당, 락 자체를 없애는 lock-free 자료구조([6.6](#/lockfree)), `SCHED_FIFO`/`SCHED_RR` 실시간 스케줄링 정책(이 환경에서는 root 권한으로 성공, 일반 계정은 권한 설정이 별도로 필요할 수 있음).
- 우선순위 역전은 낮은 우선순위 스레드가 뮤텍스를 쥔 채 무관한 중간 우선순위 스레드에 밀려 임계 구역을 못 끝내는 상황이다(실측: high 대기시간 347ms). 우선순위 상속 뮤텍스(`PTHREAD_PRIO_INHERIT`)는 뮤텍스를 쥔 스레드의 우선순위를 일시적으로 끌어올려 이걸 막는다(실측: 347ms → 56ms, low의 실제 필요 시간과 거의 일치).

::: quiz 연습문제
1~2번은 개념 문제, 3번은 예측 문제, 4번은 네 컴퓨터에서 직접 코드를 짜고 컴파일·실행으로 확인하는 실습이다.

1. "이 함수는 평균 200us에 끝나니 실시간 요구사항을 만족한다"는 주장이 왜 틀렸는지, hard real-time의 정의를 근거로 설명하라.

2. 이 절의 힙 할당·뮤텍스·페이지 폴트 실측에서, 페이지 폴트만 다른 두 원인과 성격이 다르다고 말한 이유를 p50 값의 변화 양상을 근거로 설명하라.

3. (예측) 어떤 팀이 제어 루프 스레드에는 `SCHED_FIFO`를 걸었지만, 그 스레드가 공유 상태를 읽을 때 쓰는 뮤텍스는 일반 뮤텍스(우선순위 상속 없음) 그대로 뒀다. 다른 낮은 우선순위 스레드가 그 뮤텍스를 가끔 잠근다면 무슨 문제가 생길지 예측하고 근거를 써라. 힌트: 이 절의 `rt6_priority_inversion.cpp` 실험 1을 떠올려라.

4. (실습, 코드 작성형) 이 절의 `rt2_heap_tail.cpp` 구조(단독 실행 vs 백그라운드 할당 스레드 여러 개)를 직접 타이핑해서, `new`/`delete` 대신 `std::mutex` 하나를 여러 스레드가 두드리는 버전으로 바꿔 짜라. 경합 없음/있음 각각의 p50과 최악값을 직접 측정하고, 최악값이 몇 배 차이 나는지 계산해서 이 절의 배율(약 500배)과 비교하라. `g++ -std=c++20 -O2 -Wall -Wextra main.cpp -o main -lpthread`로 컴파일하고, 백그라운드 스레드 개수를 4개, 8개로 바꿔가며 경합 강도에 따라 최악값이 어떻게 변하는지까지 확인하면 완전한 실습이다.
:::

::: answer 해설
1. hard real-time은 "평균 실행시간"이 아니라 "매 실행이 마감시간을 지키는가"로 판단한다. 평균이 200us라도 그 분포의 꼬리(최악의 경우)가 마감시간을 단 한 번이라도 넘으면 그 시스템은 실패한 것으로 친다 — 이 절의 `rt1_jitter` 실측에서 p50은 100us대였지만 최악값은 그 수백 배인 걸 봤다. 평균은 이 판단과 아예 무관한 지표다.
2. 힙 할당과 뮤텍스 경합은 p50(전형적인 경우)은 경합 유무와 거의 무관하게 그대로였고, 최악값만 경합이 있을 때 크게 튀었다 — 즉 "가끔, 운이 나쁠 때만" 발생하는 문제다. 반면 페이지 폴트는 처음 만지는 메모리라면 p50부터 이미 약 300배 느렸다 — 조건(처음 접근인가)만 맞으면 매번 확실하게 발생한다는 점에서 나머지 둘과 다르다. 그래서 회피법도 다르다 — 경합성 문제는 애초에 경합을 없애야 하고(lock-free, 사전 할당으로 할당 자체를 없앰), 페이지 폴트는 그 조건(첫 접근)을 미리 충족시켜 버리면(사전 터치) 끝난다.
3. 이 설정은 `rt6_priority_inversion.cpp`의 실험 1과 정확히 같은 함정이다 — 제어 루프(높은 우선순위)가 일반 뮤텍스를 통해 낮은 우선순위 스레드와 상태를 공유하면, 그 사이에 중간 우선순위의 무관한 스레드가 끼어드는 순간 우선순위 역전이 발생할 수 있다. `SCHED_FIFO`를 걸었다는 사실 자체는 이 문제를 전혀 막아주지 않는다 — 오히려 엄격한 우선순위 선점이 medium을 low보다 무조건 우선시키기 때문에 역전이 더 확실하게 일어난다. 제어 루프가 쓰는 모든 뮤텍스는 우선순위 상속을 켜거나(`PTHREAD_PRIO_INHERIT`), 애초에 [6.6](#/lockfree)의 lock-free 자료구조로 바꿔야 한다.
4. 뮤텍스 버전은 이 절의 `rt3_mutex_tail.cpp`와 사실상 같은 구조다 — 경합 없음의 최악값은 수십 마이크로초, 경합 있음(백그라운드 스레드 8개)의 최악값은 수 밀리초에서 수십 밀리초로 나와야 하고, 배율은 이 절에서 실측한 약 500배와 자릿수가 비슷하게 나오면(정확히 같은 숫자가 아니어도 된다 — 기기·부하마다 다르다) 정답이다. 백그라운드 스레드를 4개로 줄이면 8개일 때보다 경합이 약해져 최악값이 작아지는 경향을 직접 확인했다면 완전한 실습이다.
:::

이 절의 코드는 전부 직접 쳐라. 특히 `rt6_priority_inversion.cpp`는 스레드 세 개를 코어 하나에 고정하고 `SCHED_FIFO` 우선순위를 정확히 설정해야 재현된다 — `pthread_attr_setinheritsched`로 스레드 생성 시점부터 우선순위를 못박지 않으면(스레드가 시작된 뒤 스스로 우선순위를 올리게 하면) 그 사이의 경쟁 때문에 재현이 안 될 수 있다. 기준 명령: `g++ -std=c++20 -O2 -Wall -Wextra main.cpp -o main -lpthread && ./main`. `pthread_setschedparam`이 `EPERM`으로 실패한다면 일반 사용자 계정이기 때문이다 — `sudo chrt -f 80 ./main`으로 실행하거나 `/etc/security/limits.conf`에 rtprio 권한을 추가해라.
