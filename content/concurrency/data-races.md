# 6.2 데이터 레이스의 해부

::: lead
[6.1](#/threads)에서 스레드를 만들고 join하는 법을 봤다. 그런데 스레드 두 개가 정말로 "동시에" 뭔가를 건드리기 시작하면 그때부터 코드가 이상해진다 — 정확히는, 코드가 대부분의 실행에서는 멀쩡해 보이다가 가끔, 실행할 때마다 다른 값을 내놓는다. 이 절은 세상에서 가장 단순해 보이는 코드 한 줄 `counter++`가 두 스레드에서 왜 깨지는지를 실측으로 끝까지 파헤친다 — 어셈블리 명령 단위로 쪼개서 보고, 위젯으로 그 인터리빙을 손으로 재생하고, ThreadSanitizer로 실제 리포트를 받아 본다. 마지막에는 이게 왜 "버그"가 아니라 "미정의 동작"인지, 그 차이가 왜 중요한지까지 간다.
:::

## counter++ 를 두 스레드가 백만 번씩 돌리면

전역 변수 하나를 두 스레드가 각자 500만 번씩 증가시키는 코드다. 동기화는 전혀 없다.

```cpp title="race1_counter.cpp — 동기화 없는 공유 카운터"
#include <iostream>
#include <thread>

constexpr long N = 5'000'000;
long counter = 0;   // 동기화 없이 두 스레드가 공유한다

void increment_loop() {
    for (long i = 0; i < N; ++i) {
        counter++;   // 이 한 줄이 이 절의 전부다
    }
}

int main() {
    std::thread t1(increment_loop);
    std::thread t2(increment_loop);
    t1.join();
    t2.join();

    std::cout << "기대값: " << (2 * N) << ", 실제값: " << counter << "\n";
    return 0;
}
```

기대값은 뻔하다 — 스레드 둘이 500만 번씩 더했으니 1,000만이어야 한다. 같은 바이너리를 다섯 번 그대로 돌린 결과다.

```console
$ g++ -std=c++20 -O0 -Wall -Wextra race1_counter.cpp -o race1_counter -lpthread
$ ./race1_counter
기대값: 10000000, 실제값: 5995839
$ ./race1_counter
기대값: 10000000, 실제값: 6982480
$ ./race1_counter
기대값: 10000000, 실제값: 5892284
$ ./race1_counter
기대값: 10000000, 실제값: 5721345
$ ./race1_counter
기대값: 10000000, 실제값: 7004446
```

(g++ 13.3.0 / Ubuntu 24.04 / x86-64 / `-O0` 실측.) **소스 코드도 바이너리도 한 글자 안 바꿨다.** 그런데 다섯 번 실행해서 다섯 번 다른 숫자가 나왔다 — 600만대, 698만, 589만, 572만, 700만. 대략 40~50%의 증가가 통째로 증발했다. 어느 실행이 "맞고" 어느 실행이 "틀렸는지"를 가릴 수 없다는 게 핵심이다. **전부 같은 프로그램의 정당한 실행 결과다.** [1.3 변수, 초기화, 스코프](#/variables)에서 초기화 안 된 변수를 읽었을 때도 실행마다 다른 값이 나왔던 것과 같은 부류의 비결정성이지만, 그때는 메모리 하나를 잘못 읽은 것이었고 이번엔 **두 스레드가 같은 메모리를 서로 밟고 지나가면서** 생긴 손실이라는 점이 다르다.

같은 코드를 `-O2`로 다시 컴파일하면 어떻게 되는지 봐 둔다 — 나중에 이 차이가 UB 이해에 결정적인 단서가 된다.

```console
$ g++ -std=c++20 -O2 -Wall -Wextra race1_counter.cpp -o race1_counter_opt -lpthread
$ ./race1_counter_opt
기대값: 10000000, 실제값: 10000000
$ ./race1_counter_opt
기대값: 10000000, 실제값: 10000000
```

(g++ 13.3.0 실측, 5회 반복 모두 동일.) `-O2`에서는 다섯 번, 열 번을 돌려도 항상 정답이 나온다. **그렇다고 이 코드에 레이스가 없는 게 아니다** — 왜 그런지는 몇 절 아래에서 어셈블리로 확인한다. 지금 붙잡아야 할 건 하나다. 같은 소스 코드가 최적화 레벨 하나만 바꿔도 "항상 틀림"에서 "항상 맞음"으로 겉모습이 바뀐다면, **눈으로 본 실행 결과만으로는 이 코드에 레이스가 있는지 없는지 절대 판단할 수 없다.**

## 데이터 레이스의 정확한 정의

지금까지 본 현상에 이름을 붙인다. **데이터 레이스(data race)**는 다음 세 조건이 동시에 성립할 때 생긴다.

1. 두 개 이상의 스레드가 **같은 메모리 위치**에 접근한다.
2. 그 중 **최소 하나가 쓰기**다.
3. 두 접근 사이에 **동기화에 의한 순서 관계가 없다**(뮤텍스, atomic, `join` 등 어느 것으로도 "이 접근이 저 접근보다 먼저다"를 못박지 못한 상태).

`race1_counter.cpp`는 세 조건을 정확히 만족한다 — T1과 T2가 같은 `counter`에 접근하고(1), 매 반복이 읽기+쓰기고(2), 두 스레드 사이엔 아무 동기화 장치도 없다(3).

::: note 읽기만 여러 스레드가 하면 레이스가 아니다
두 번째 조건이 핵심이다. 스레드 열 개가 같은 `const int& config`를 동시에 읽기만 한다면 데이터 레이스가 아니다 — 아무도 값을 바꾸지 않으니 "누가 먼저 읽었나"가 결과에 영향을 안 준다. 문제는 늘 **적어도 하나의 쓰기가 동기화 없이 끼어들 때** 시작된다. 읽기 전용 공유 상태는 이 절 내내 걱정할 필요가 없다.
:::

이 정의에서 "인터리빙(interleaving)"이라는 말이 나온다 — 두 스레드의 명령들이 실제 실행 시점에 어떤 순서로 섞이는지를 뜻한다. `counter++` 같은 한 줄짜리 코드도 CPU 입장에서는 여러 개의 개별 명령이고, 그 명령들이 스레드 사이에서 어떤 순서로 인터리빙되느냐에 따라 최종 결과가 달라진다. 다음 절에서 그 명령들이 정확히 무엇인지 눈으로 확인한다.

## counter++ 는 왜 원자적이지 않은가 — 어셈블리로 확인

`counter++`가 "하나의 연산"처럼 보이는 건 C++ 소스 코드 층위에서만 그렇다. CPU가 실행하는 기계어 층위로 내려가면 이야기가 다르다. 최적화를 끄고(`-O0`) 컴파일한 오브젝트 파일을 역어셈블해서 확인한다 — 최적화가 없어야 컴파일러가 명령을 합치지 않고 소스 그대로 옮긴다.

```console
$ g++ -std=c++20 -O0 -c race1_counter.cpp -o race1_counter_O0.o
$ objdump -d --no-show-raw-insn -M intel race1_counter_O0.o
```

```text nolines
0000000000000000 <_Z14increment_loopv>:
   0:  endbr64
   4:  push   rbp
   5:  mov    rbp,rsp
   8:  mov    QWORD PTR [rbp-0x8],0x0
  10:  jmp    29 <_Z14increment_loopv+0x29>
  12:  mov    rax,QWORD PTR [rip+0x0]        # counter 를 레지스터로 load
  19:  add    rax,0x1                        # 레지스터 안에서 add
  1d:  mov    QWORD PTR [rip+0x0],rax        # 레지스터를 counter 로 store
  24:  add    QWORD PTR [rbp-0x8],0x1
  29:  cmp    QWORD PTR [rbp-0x8],0x4c4b3f
  31:  jle    12 <_Z14increment_loopv+0x12>
  33:  nop
  34:  nop
  35:  pop    rbp
  36:  ret
```

(g++ 13.3.0 / x86-64 / `-O0` 실측.) 12~1d번 주소 세 줄이 `counter++;` 정확히 그 자체다 — **load, add, store** 세 개의 별개 명령이다. `0x4c4b3f`는 십진수로 4,999,999(`N - 1`)로, 반복 횟수가 소스와 정확히 일치하는 것도 확인된다. 두 스레드가 각자 이 세 줄짜리 코드를 500만 번 반복하는 동안, CPU 스케줄러가 두 스레드를 실제로 어떻게 번갈아 실행시키는지는 프로그램이 통제할 수 없다. T1이 load(0을 읽음) → add(1을 만듦) → store(1을 씀)를 마치기 전에 T2가 자기 load를 끼워 넣으면, T2도 똑같이 0을 읽어서 결국 둘 다 "1"을 쓰게 된다 — 두 번 증가시켰는데 결과는 1이다. 이 절 맨 앞의 실측에서 봤던 손실이 바로 이 패턴이 500만 번 중 수백만 번 누적된 결과다.

::: warn -O2 는 이 레이스를 없앤 게 아니라 확률을 극단적으로 낮췄을 뿐이다
같은 함수를 `-O2`로 역어셈블하면 완전히 다른 그림이 나온다.

```text nolines
0000000000000000 <_Z14increment_loopv>:
   0:  endbr64
   4:  add    QWORD PTR [rip+0x0],0x4c4b40   # counter += 5000000, 통째로 한 명령
   f:  ret
```

`0x4c4b40`은 정확히 5,000,000(`N`)이다 — 컴파일러가 "이 루프는 결국 `counter`에 5,000,000을 한 번에 더하는 것과 같다"고 판단해서 500만 번의 load-add-store 루프 전체를 **단 하나의 메모리 덧셈 명령**으로 접어 버렸다. 왜 이래도 되는지는 바로 다음 절에서 설명한다 — 지금 확인할 건, T1과 T2가 각자 "명령 하나"만 실행하게 되면서 두 스레드가 그 찰나의 순간에 정확히 겹칠 확률이 극도로 낮아졌을 뿐이라는 사실이다. **레이스 자체는 여전히 있다.** 스레드 스케줄링, 코어 개수, 시스템 부하가 바뀌면 이 "단일 명령 add"조차 겹칠 수 있고, 애초에 x86의 메모리 피연산자 `add` 명령은 `lock` 접두어 없이는 원자적이지 않다. "최적화 빌드에서는 안 터진다"는 절대 안전 증명이 아니다.
:::

## 위젯: 같은 코드가 스케줄링에 따라 다르게 흐른다

방금 어셈블리로 본 load-add-store 세 단계가 두 스레드 사이에서 실제로 어떻게 겹치는지, 스텝 단위로 재생해서 본다. 아래 위젯은 정확히 같은 코드(load→add→store)를 두 가지 다른 인터리빙으로 재생한다.

::: widget thread-timeline
{ "scenario": "race" }
:::

여기서는 T2의 load가 T1의 store보다 먼저 일어나 T1의 +1이 통째로 사라진다 — 두 번 증가시켰는데 최종값은 1이다. 그런데 **정확히 같은 두 함수, 같은 코드**를 스케줄러가 조금 다르게 배치하면 결과가 달라진다.

::: widget thread-timeline
{ "scenario": "race-lucky" }
:::

이번엔 T1의 load→add→store 세 단계가 T2가 끼어들기 전에 전부 끝났다 — 순전히 스케줄러가 그렇게 배치했기 때문이고, 소스 코드에는 그걸 보장하는 어떤 장치도 없다. 두 시나리오는 코드가 완전히 동일하고, 차이는 오직 "어느 스레드가 몇 번째 명령에서 끼어드는가"뿐이다.

::: danger 레이스는 재현이 안 될 수도 있다 — 그래서 테스트로는 못 잡는다
`race-lucky` 시나리오가 이 절에서 가장 중요한 그림이다. **레이스가 있는 코드를 백 번 실행해서 백 번 다 정답이 나왔다고 해도, 그 코드에 레이스가 없다는 증거가 되지 않는다.** 스케줄링은 CPU 부하, 다른 프로세스, 심지어 그날의 시스템 상태에 따라 달라진다 — 개발 머신에서 아무 문제 없이 통과한 코드가 로봇에 배포된 뒤, 다른 프로세스가 CPU를 더 많이 쓰는 상황에서 처음으로 이 인터리빙을 만나 오동작할 수 있다. "테스트를 통과했다"가 "레이스가 없다"를 보장하지 않는 이유가 바로 이 비결정성이다. 이 문제를 구조적으로 해결하는 게 [6.3 mutex와 락 가드](#/mutex)의 몫이다 — 위 위젯의 세 번째 시나리오("뮤텍스")가 그 해결책을 미리 보여준다.
:::

## UB로서의 데이터 레이스 — 왜 컴파일러가 저렇게 접었나

[2.11 댕글링, UB, 새니타이저](#/ub-sanitizers)에서 C++ 표준이 프로그램의 동작을 네 등급(정의됨/미지정/구현 정의/미정의)으로 나눈다는 걸 봤다. **데이터 레이스가 있는 프로그램은 표준상 미정의 동작(UB)이다.** 표준은 이 상황에서 프로그램이 어떻게 동작해야 하는지 아무 요구도 하지 않는다 — "레이스가 나면 값을 잃는다"조차 표준이 보장하는 게 아니다. 값이 이상하게 나오는 것도, 프로그램이 크래시하는 것도, 겉보기엔 완벽하게 정상 동작하는 것도 전부 "표준이 허용하는 결과"에 포함된다.

이게 왜 중요한지는 방금 본 `-O2` 결과가 정확히 보여준다. [2.11](#/ub-sanitizers)에서 확인한 원칙 그대로다 — **컴파일러는 UB가 없다고 가정하고 최적화한다.** `counter`에 데이터 레이스가 없다고(즉 이 함수가 실행되는 동안 다른 스레드는 절대 `counter`를 건드리지 않는다고) 가정하면, "루프 안에서 매번 메모리를 다시 읽고 다시 쓸 필요 없이 레지스터에 누적하다가 마지막에 한 번만 메모리에 반영해도 결과가 같다"는 최적화가 완전히 합법적이다. 이 코드에 실제로 데이터 레이스가 있다는 사실은 컴파일러의 관심사가 아니다 — **표준이 "레이스가 있으면 동작은 정의되지 않는다"고 정해 놓았으므로, 컴파일러는 레이스가 없다는 전제 위에서 마음껏 코드를 바꿀 권리를 갖는다.** `-O2`가 "레이스를 감지해서 고쳐준 것"이 절대 아니라, "레이스가 없다고 믿고 최적화했더니 우연히 레이스가 드러날 확률이 낮아진 것"이라는 게 정확한 설명이다.

::: interview "레이스 컨디션"과 "데이터 레이스"는 같은 말인가
면접에서 자주 뒤섞이는 두 용어다. **레이스 컨디션(race condition)**은 더 넓은 개념이다 — 여러 스레드/프로세스의 실행 순서에 따라 프로그램의 결과가 달라지는 상황 전체를 가리킨다. 파일 두 개를 서로 다른 프로세스가 쓰는 순서에 따라 최종 내용이 달라지는 것도 레이스 컨디션이다. **데이터 레이스(data race)**는 이 절에서 정의한 세 조건(같은 메모리, 동기화 없는 접근, 최소 하나의 쓰기)을 만족하는 특정한 경우이고, C++에서는 이게 곧 UB라는 표준상의 지위를 갖는다. 즉 모든 데이터 레이스는 레이스 컨디션이지만, 모든 레이스 컨디션이 데이터 레이스는 아니다 — 예를 들어 두 스레드가 각자 락을 정상적으로 잡고 큐에 넣고 빼는데 그 처리 "순서"에 따라 로직 결과가 달라지는 것은 레이스 컨디션이지만, 메모리 접근 자체는 뮤텍스로 동기화돼 있으니 데이터 레이스는 아니다. 이 구분을 명확히 답하면 "동시성 버그를 언어 표준 층위에서 이해하고 있다"는 인상을 준다.
:::

## ThreadSanitizer로 레이스를 실제로 잡는다

눈으로 값을 비교하는 방법은 이 절 맨 앞에서 봤듯 신뢰할 수 없다 — `-O2`에서는 다섯 번 다 정답이 나왔다. 레이스를 확정적으로 잡으려면 도구가 필요하다. [7.5](#/sanitizers)에서 정식으로 다룰 ASan/UBSan과 같은 계열의 **ThreadSanitizer(TSan)**를 `-fsanitize=thread`로 지금 붙여 본다.

```console
$ g++ -std=c++20 -g -O0 -fsanitize=thread -Wall -Wextra race1_counter.cpp -o race1_counter_tsan -lpthread
$ ./race1_counter_tsan
```

```text nolines
==================

    #0 increment_loop() race1_counter.cpp:9 (race1_counter_tsan+0x135a)
    #1 ... std::thread 내부 호출 스택 (invoke/Invoker/_State_impl) ...

  Previous write of size 8 at 0x5570cac5e158 by thread T1:
    #0 increment_loop() race1_counter.cpp:9 (race1_counter_tsan+0x1374)
    #1 ... std::thread 내부 호출 스택 ...

  Location is global 'counter' of size 8 at 0x5570cac5e158 (race1_counter_tsan+0x5158)
```

(g++ 13.3.0 / `-fsanitize=thread` / `-g -O0` 실측. 스레드 ID·주소·표준 라이브러리 내부 호출 스택 줄은 지면상 줄였다 — `race1_counter.cpp:9`가 정확히 `counter++;` 줄이라는 것과 종료 코드가 실측의 핵심이다.) `$?`로 확인한 종료 코드는 `66`이다 — TSan은 레이스를 발견하면 정상 종료(0)가 아니라 넌제로 코드로 끝난다. **CI 파이프라인([7.8](#/ci))에서 빌드를 실패시키는 신호로 그대로 쓸 수 있다는 뜻이다.** 리포트는 "`Read`"와 "`Previous write`"가 같은 메모리 위치(`Location is global 'counter'`)에서 서로 다른 스레드(T1, T2)에 의해, 동기화 없이 일어났다는 것을 정확히 지목한다 — 이 절 도입부에서 값으로만 추측했던 것을 TSan은 **소스 줄 번호(`race1_counter.cpp:9`)까지 찍어서 확정**해 준다.

::: perf ThreadSanitizer는 공짜가 아니다
같은 프로그램, 같은 `-O0`을 TSan 없이 돌리면 0.023초, TSan을 붙이면 0.201초 걸렸다(각 3회 반복 중 대표값, `time` 실측) — 이 실행에서는 약 **9배** 느려졌다. TSan은 메모리 접근마다 셰도우 메모리에 접근 기록을 남기고 비교하기 때문에 실행 시간과 메모리 사용량이 둘 다 크게 늘어난다. 그래서 [6.8 실시간 제약](#/realtime)에서 다룰 실제 제어 루프에는 TSan을 절대 그대로 배포하지 않는다 — 개발 중 테스트 빌드에서만, 특히 CI에서 상시 돌리는 용도로 쓴다.
:::

## 로보틱스 도메인: 센서 콜백과 공유 자세 추정치

핵사포드 같은 다리 로봇에서는 IMU 콜백 스레드, 각 다리의 인코더 콜백 스레드, 카메라 처리 스레드가 각자 독립적으로 실행되면서 로봇의 자세 추정치(위치·자세·속도를 담은 공유 상태) 하나를 갱신하는 구조가 흔하다. 이 절의 `counter`를 "추정된 헤딩 각도"로, `counter++`를 "IMU 콜백이 새 자이로 적분값을 더하는 연산"으로 바꿔서 생각해 보면 위험이 바로 보인다 — IMU 콜백 스레드와 인코더 콜백 스레드가 동기화 없이 같은 자세 추정치 구조체를 동시에 갱신하면, 이 절에서 본 것과 똑같은 이유로 갱신 하나가 조용히 증발하거나, 더 나쁘게는 `struct`의 절반은 IMU 콜백이 쓴 값, 나머지 절반은 인코더 콜백이 쓴 값이 섞인 **일관성 없는 자세**가 만들어질 수 있다. 문제는 이게 이 절 맨 앞의 실측처럼 매번 재현되지 않고, 실험실 벤치 테스트에서는 멀쩡하다가 로봇이 실제로 걸으면서 콜백 타이밍이 미묘하게 달라지는 순간에만 튀어나올 수 있다는 것이다 — 걷는 도중 헤딩이 갑자기 튀는 버그를 재현 스크립트로 못 잡는 이유가 바로 이거다. [6.3](#/mutex)부터 이 문제를 실제로 해결하는 도구(뮤텍스, atomic)를 다룬다.

## 요약

- 데이터 레이스는 (1) 두 스레드 이상이 같은 메모리 위치에 접근하고, (2) 최소 하나가 쓰기이고, (3) 동기화로 순서가 정해지지 않았을 때 생긴다. 읽기만 여러 스레드가 공유하는 건 레이스가 아니다.
- `counter++`는 소스 코드로는 한 줄이지만 어셈블리로는 load, add, store 세 개의 별개 명령이다(`-O0` 실측: `race1_counter_O0.o`). 두 스레드의 이 세 단계가 인터리빙되면 갱신 하나가 통째로 사라질 수 있다.
- 같은 소스 코드를 500만×2회 반복 실행한 실측에서 `-O0`은 매번 다른(그리고 항상 기대값보다 작은) 결과를, `-O2`는 매번 정확한 결과를 냈다 — `-O2`가 레이스를 없앤 게 아니라, 루프 전체를 단일 `add` 명령으로 접어서 겹칠 확률을 극단적으로 낮췄을 뿐이다(실측: `0x4c4b40` = 5,000,000).
- 데이터 레이스는 C++ 표준상 미정의 동작(UB)이다 — [2.11](#/ub-sanitizers)에서 본 "컴파일러는 UB가 없다고 가정하고 최적화한다"는 원칙이 여기서도 그대로 적용된다. "최적화 빌드에서 안 터진다"는 안전의 증거가 아니다.
- 레이스 컨디션은 실행 순서에 따라 결과가 달라지는 상황 전체를 가리키는 넓은 개념이고, 데이터 레이스는 그중 세 조건을 만족하는(그리고 C++에서 UB인) 좁은 개념이다.
- ThreadSanitizer(`-fsanitize=thread`)는 레이스를 실행 한 번만으로도 소스 줄 번호와 함께 리포트하고 넌제로 종료 코드(실측: 66)를 낸다 — 실행 결과 비교로는 절대 확정할 수 없는 레이스를 CI에서 기계적으로 잡는 방법이다. 대가로 실행 속도가 크게(실측: 약 9배) 느려진다.

::: quiz 연습문제
1~2번은 개념 문제, 3번은 예측 문제, 4번은 네 컴퓨터에서 TSan을 직접 돌려 확인하는 실습이다.

1. 이 절의 세 조건(같은 메모리, 최소 하나의 쓰기, 동기화 없음)을 근거로, 스레드 다섯 개가 같은 `const std::vector<double>& params`를 동시에 읽기만 하는 코드에 데이터 레이스가 있는지 판단하고 이유를 한 문단으로 써라.

2. `race1_counter.cpp`를 `-O2`로 컴파일하면 매번 정확한 결과가 나왔다. 이 사실이 "이 코드에는 데이터 레이스가 없다"는 뜻인지 판단하고, 이 절에서 실측한 어셈블리 결과를 근거로 답하라.

3. (예측) `race1_counter.cpp`의 `increment_loop` 함수 안에서 `counter`를 지역 변수 `long local = 0;`으로 바꾸고 루프가 끝난 뒤 `counter += local;` 한 줄만 남긴다면(각 스레드가 자기 몫을 다 더한 뒤 마지막에 한 번만 전역 변수에 반영), 이 코드에도 여전히 데이터 레이스가 있는지 예측하고 근거를 써라. 힌트: `counter += local;` 자체가 읽기+쓰기다.

4. (실습, 코드 작성형) 이 절의 `race1_counter.cpp`를 그대로 타이핑하고, `g++ -std=c++20 -g -O0 -fsanitize=thread -Wall -Wextra race1_counter.cpp -o race1_counter_tsan -lpthread`로 컴파일해서 직접 실행하라. `WARNING: ThreadSanitizer: data race`가 뜨는지, `echo $?`로 확인한 종료 코드가 0이 아닌지 두 가지를 모두 확인하는 것이 성공 기준이다. 그 다음 3번에서 예측한 "지역 변수로 누적 후 마지막에 한 번만 반영" 버전도 같은 방식으로 TSan을 붙여 실제로 리포트가 뜨는지 확인하라.
:::

::: answer 해설
1. 데이터 레이스가 아니다. 세 조건 중 두 번째("최소 하나가 쓰기")가 성립하지 않는다 — 다섯 스레드 모두 읽기만 하고 아무도 `params`를 바꾸지 않으므로, 어떤 순서로 읽어도 결과가 같다. 이 절의 note 박스에서 짚었듯 읽기 전용 공유 상태는 동기화가 필요 없다.
2. 아니다. `-O2`에서 매번 정답이 나온 건 이 절에서 실측한 대로 컴파일러가 500만 번의 load-add-store 루프 전체를 단일 `add [counter], 5000000` 명령으로 접었기 때문이다(오브젝트 코드에서 `0x4c4b40` 확인). 두 스레드가 각자 "명령 하나"만 실행하게 되면서 그 찰나에 겹칠 확률이 낮아졌을 뿐, 소스 코드 자체는 여전히 세 조건(같은 메모리 `counter`, 읽기+쓰기, 동기화 없음)을 그대로 만족하는 데이터 레이스다. 표준 관점에서 이 프로그램의 동작은 여전히 정의되지 않았다.
3. 여전히 데이터 레이스가 있다. `counter`에 대한 접근이 500만 번의 개별 `counter++`에서 스레드당 딱 한 번의 `counter += local;`로 줄었을 뿐, `counter += local;` 자체가 여전히 "같은 전역 메모리를 읽고 쓰는" 연산이고 두 스레드 사이에 동기화가 없다는 사실은 그대로다. 다만 레이스가 발생하는 "창(window)"이 500만 번에서 스레드당 1번으로 극도로 좁아져서, 실제로 값 손실이 관측될 확률은 이 절 맨 앞의 버전보다 훨씬 낮아진다 — 이게 정확히 `-O2`가 한 일과 같은 종류의 위험이다. 확률이 낮아진 것과 레이스가 없어진 것은 다르다.
4. TSan은 `race1_counter.cpp:9`(`counter++;`)를 가리키는 `WARNING: ThreadSanitizer: data race` 리포트를 내고, 종료 코드는 `66`(또는 실행 환경에 따라 0이 아닌 값)이 된다. "지역 변수 누적 후 한 번만 반영" 버전도 `counter += local;` 줄에서 똑같이 리포트가 뜬다 — 레이스 발생 확률이 낮아졌을 뿐 TSan은 실행 중 단 한 번이라도 동기화 없는 동시 접근이 실제로 일어나면 그 자리에서 잡아낸다(단, 그 낮아진 확률 때문에 이 버전은 몇 번 반복 실행해야 리포트가 뜰 수도 있다 — 그 자체가 3번 문제의 답을 실측으로 재확인해 주는 셈이다).
:::

이 절의 `race1_counter.cpp`는 반드시 직접 타이핑하고, TSan 플래그까지 그대로 붙여서 돌려 봐라 — 리포트가 소스 줄 번호를 정확히 찍어 주는 걸 눈으로 봐야 다음 절부터 나올 "이렇게 하면 레이스가 없어진다"는 해결책들이 왜 필요한지 실감이 난다. 기준 명령: `g++ -std=c++20 -g -O0 -fsanitize=thread -Wall -Wextra race1_counter.cpp -o race1_counter_tsan -lpthread && ./race1_counter_tsan; echo "exit=$?"`.
