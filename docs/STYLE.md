## 1. 독자와 목표

- 독자는 **일반 프로그래밍 경험이 있고 C++을 실무 수준으로 익히려는 사람**이다.
  변수, 함수, 조건문, 반복문이 무엇인지는 안다. C++ 특유의 것(포인터, 템플릿, RAII)은 모른다고 가정한다.
- 목표는 *"ROS 2를 C++만으로 자유롭게 구현하는 로봇 애플리케이션 개발자"* 다.
- 그러므로 **"~하는 법"이 아니라 "왜 그런가"** 를 쓴다. 사용법은 검색하면 나온다.
  검색으로 안 나오는 것을 써라: 내부 동작, 설계 이유, 함정, 실제 비용.

### 밀도의 기준

한 절은 **읽는 데 15~30분**, 본문 6,000~12,000자(코드 제외) 정도다.
짧으면 얕고, 길면 안 읽는다. 다만 **분량을 채우려고 늘리지 마라.** 할 말이 없으면 짧게 끝낸다.

## 2. 말투

- **평서형 반말체.** "~한다", "~이다". `~습니다`, `~해요` 금지.
- 독자를 부를 때는 "당신". 명령형은 "~하라", "~해라" 를 쓴다.
- **단정적으로 쓴다.** "~인 것 같다", "~일 수도 있다" 같은 헤지를 피한다.
  단, 모르면 모른다고 쓴다. 아는 척이 최악이다.
- 문장은 짧게. 한 문장에 한 가지만.
- **비유는 정확할 때만.** 틀린 비유는 잘못된 모델을 심는다.

### 좋은 예

> C++에서 `=` 는 기본적으로 값을 복사한다. 파이썬과 정반대다.

> `shared_ptr` 를 습관적으로 쓰지 마라. 대부분의 경우 `unique_ptr` 로 충분하다.

### 나쁜 예

> C++에서는 대입 시 복사가 일어날 수도 있으니 주의가 필요합니다.

> shared_ptr는 유용하지만 상황에 따라 성능 이슈가 발생할 수 있습니다.

## 4. C++ 특화 절대 규칙

### 4-1. 위젯 우선 원칙 (가장 중요)

**공간적으로 표현 가능한 개념은 아스키 다이어그램이 아니라 반드시 인터랙티브 위젯으로
만든다.** 스택/힙, 포인터, 소유권 이동, 상속/vtable, 템플릿 인스턴스화, 스레드 타임라인,
좌표 변환 — 전부 여기 해당한다. 아스키 다이어그램은 위젯을 만들기엔 과한 단순 구조에만
쓴다. **"글로 설명 가능하니 됐다"는 이 책에서는 근거가 안 된다.**

단, EPUB(폰 복습용)에서는 위젯이 제거되므로 **위젯을 감싼 본문 산문이 그 자체로 개념을
전달해야 한다.** 위젯은 이해를 새기는 도구이지, 본문의 구멍을 메우는 도구가 아니다.

위젯 문법:

````markdown
::: widget stack-heap
{ "scenario": "basic-call" }
:::
````

### 4-2. 코드 블록과 실행 예시

코드 블록은 **표시·복사 전용**이다. 이 앱에는 실행 기능이 없고, 그게 설계다 — 실제
컴파일·실행은 독자의 IDE에서 한다(§4-5).

실행 결과·컴파일러 출력·ASan/UBSan 리포트를 보여줄 때는 반드시 **실제로 그 환경에서
실행해서 나온 진짜 출력**을 싣는다. 추정 금지. 수치가 기기마다 다르면 그렇다고 쓴다.

```markdown
(g++ 13 / -O2 / Linux x86-64 기준 실측. 절대값은 기기마다 다르지만 자릿수 차이는 어디서나 같다.)
```

특히 조심할 것:

- 주소값, 스택/힙 배치는 실행마다 다르다(ASLR) — 예시임을 알 수 있게 쓴다.
- UB는 "이렇게 나온다"라고 단정하지 마라. **"이 환경에서는 이렇게 나왔지만 보장이 없다"**
  가 정확한 서술이고, 그 차이가 UB 개념의 핵심이다.
- 최적화 레벨(-O0/-O2)에 따라 달라지는 것은 레벨을 명시한다.
- 컴파일러 버전에 따라 달라지는 에러 메시지는 버전을 명시한다.

### 4-3. 외부 링크 검증

영상/자료 링크는 **넣는 시점에 실제로 접속해서 확인한 것만** 넣는다. 개별 영상보다
채널/재생목록 단위를 우선한다. 링크를 추가한 커밋에 `docs/LINK_LOG.md` 한 줄을 같이
남긴다. 확인 못 하면 링크 없이 비워 둔다 — **없는 것이 틀린 것보다 낫다.**

### 4-4. 로봇 도메인 연결은 필수, 남발은 금지

**Part I~VIII(언어 자체)의 각 절은 최소 한 번, "이게 Part IX~XII(로보틱스 도메인)의
어디에 실제로 쓰이는가"를 짚는다.** 억지 연결은 금지 — 관련 없으면 안 쓰고, 그 사실을
마일스톤 보고에 한 줄 남긴다.

### 4-5. 타이핑 원칙

"복사 버튼을 누르지 말고 직접 쳐라"를 이 책은 한 단계 더 강하게 적용한다 — **타이핑
실수(세미콜론 빠뜨림, 헤더 안 붙임)가 곧 컴파일 에러 메시지를 읽는 연습이기 때문이다.**

- 모든 실습형 코드 예제는 절 끝에 "네 IDE에서 해보라"는 **구체적 지시**로 마무리한다.
  명령줄을 그대로 준다: `g++ -std=c++20 -Wall -fsanitize=address main.cpp && ./a.out`
- Part II 이후 실습 비중이 높은 절은 `::: quiz` 안에 최소 1개 **코드 작성형** 문제를
  넣는다 — 성공 기준이 "답을 눈으로 확인"이 아니라 **"네 컴퓨터에서 컴파일이 통과하고
  새니타이저가 조용한가"** 인 문제.

### 4-6. 공개 레포 신원 보호 (예외 없음)

이 레포는 public이다. 개인 이력·경력·"몰랐다"류 고백·과거 기술 스택 언급·구직 활동
서술을 **본문·주석·커밋 메시지 어디에도 넣지 않는다.** 상세는 CLAUDE.md §4-6.
"일반 프로그래밍 경험이 있는 독자" 같은 **타깃 독자층** 서술은 괜찮다 — 콘텐츠 설계
기준이지 저자 상태 고백이 아니다.

## 5. 마크다운 확장 문법

### 코드 블록

````markdown
```cpp title="파일명.cpp"
컴파일 가능한 완결된 코드
```

```cpp title="설명" {3,5-7}
3번, 5~7번 줄이 강조된다
```

```console
$ g++ -std=c++20 main.cpp     <- $ 로 시작하는 줄은 명령
main.cpp: error: ...          <- 나머지는 출력으로 칠해진다
```

```cmake
CMakeLists.txt 내용
```

```text nolines
아스키 다이어그램 (줄 번호 없음)
```
````

지원 언어: `cpp` `c` `cmake` `console` `bash` `json` `yaml` `xml` `python` `js` `sql` `text`

### 표시 상자

| 문법 | 쓰는 때 |
| --- | --- |
| `::: note` | 알아두면 좋은 보충. 몰라도 진도는 나간다 |
| `::: tip` | 실무 요령 |
| `::: warn` | 틀리기 쉬운 지점 |
| `::: danger` | **버그를 만드는** 지점. 실제 장애로 이어지는 것만 |
| `::: deep` | 컴파일러 내부, 표준 문서, ABI — 한 단계 아래 이야기 |
| `::: perf` | 실행 속도·메모리. **반드시 측정치와 함께** |
| `::: interview` | 기술면접에서 이 개념이 실제로 어떻게 질문되는가 + 모범 답변 뼈대 |
| `::: hist` | 왜 이렇게 설계됐는가 |
| `::: quiz` | 연습문제 |
| `::: answer` | 해설 (접혀서 나온다) |
| `::: lead` | 절 머리말 |
| `::: widget <type>` | 인터랙티브 위젯 (본문 JSON 파라미터, §4-1) |

**한 절에 상자가 8~16개** 정도가 적당하다. 너무 많으면 본문이 사라진다.

`::: interview` 는 **실제 면접에서 나올 법한 지점에만** 넣는다. 억지로 만들지 마라.

### 기타

- 수식: `$O(n \log n)$`, `$$블록 수식$$`
- 형광펜: `==강조==`
- 인라인 태그: `<kbd>Ctrl</kbd>` 만 허용
- 링크: 같은 책 안은 `[2.5 RAII](#/raii)` — **toc.json 의 id를 쓴다**

### 아스키 다이어그램 안에는 ASCII만

한글은 고정폭 글꼴에서 **두 칸**을 차지한다. 선(`│ ─ ┌ └`)보다 **왼쪽에** 한글이 있으면
그림이 깨진다. 선보다 오른쪽의 꼬리 주석은 괜찮다. `python tools/check_diagrams.py` 가 검사한다.

### 코드 규칙

- **모든 코드는 컴파일 가능해야 한다.** 조각이면 조각임을 밝힌다.
- 주석은 **왜**를 쓴다. 무엇을 하는지는 코드가 이미 말한다.
- 나쁜 예/좋은 예는 `// ❌` / `// ✅` 로 표시한다.
- 변수명·주석은 한국어로 써도 된다. 단 식별자는 영어.
- 기본 스타일: C++20, `-Wall -Wextra` 에서 경고 없는 코드.

## 6. 이 책의 성격을 만드는 것

다른 책과 이 책을 가르는 지점이다. 매 절에서 최소 하나는 해라.

1. **한 층 아래로 내려간다.** "vector는 동적 배열이다"에서 멈추지 말고,
   capacity 성장 패턴을 실측해서 보여준다.
2. **실제 비용을 말한다.** $O(1)$ 이라고만 하지 말고 상수가 얼마나 큰지,
   캐시 미스 한 번이 몇 사이클인지 쓴다.
3. **함정을 먼저 보여준다.** 독자가 실제로 당할 것을 미리 당하게 한다.
   C++은 함정의 언어다 — 이 항목이 파이썬판보다 훨씬 무겁다.
4. **설계 이유를 말한다.** "왜 C++은 이렇게 했나"에 답한다. 제로 오버헤드 원칙,
   C 호환성, 위원회의 결정 — 맥락을 알면 잘 안 잊힌다.
5. **로보틱스와 잇는다.** 이 지식이 Part IX~XII의 어디에 쓰이는지. (§4-4)
