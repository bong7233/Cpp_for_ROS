## 0. 이 프로젝트가 존재하는 이유

C++은 실시간 로봇 제어·ROS2 코어 스택(rclcpp, Nav2, `ros2_control`, MoveIt2)의 표준 언어다. 이 프로젝트는 C++을
언어 기초부터 로보틱스 실전 적용까지 전 범위로 다루는 학습·레퍼런스 시스템을 구축하며, Hexpider(헥사포드 로봇
플랫폼) 프로젝트에서의 실전 적용으로 검증한다.

이 앱의 존재 이유는 하나다. **시중의 어떤 C++ 책·강의보다 나은 학습 자료를, 팔아도 될 수준의 완성도로 만든다.** 텍스트만
늘어놓는 자료는 이미 세상에 넘친다. 시각·공간적 학습이 효과적이라는 전제 위에, 이 앱은 **조작 가능한 시각 위젯 + 엄선된
영상 링크**로 개념을 몸에 새긴다.

**실제 코딩은 이 앱 안에서 하지 않는다.** PC에서는 듀얼 모니터 한쪽에 이 앱, 다른 한쪽에 실제 IDE(VSCode + Hexpider의
ROS2 Humble Docker 환경)를 띄워 놓고 병행한다. **이 앱 자체는 PC 전용이다.** 모바일 복습(이동 중 자투리 시간)은
이 앱과 완전히 분리된 EPUB 파일로 처리한다 — 같은 `content/` 소스에서 자동 생성되는 파생물이다. §1·§2-3에서 이 결정의
배경을 다룬다.

### 결정 변경 ①: 코드 실행 샌드박스 — 도입했다가 폐기

직전 라운드에서 로컬 `/run` 엔드포인트(g++ 서브프로세스 컴파일·실행)와 CodeMirror 에디터를 앱에 넣기로 했었다.
**이번 라운드에서 폐기했다.** 이유를 숫자와 함께 남긴다 — 사용자가 다시 이 논의를 반복할 필요가 없도록.

- **사용자의 실제 작업 방식이 바뀌었다.** PC에서는 듀얼 모니터로 이 앱과 실제 IDE를 동시에 띄워 놓고 쓴다. 즉 "이
  앱 안에서 실행해야 하는 이유"였던 "다른 데를 켤 필요 없이 바로 시도해본다"는 전제가 애초에 성립하지 않는다.
- **위젯의 효과는 라이브 실행에 의존하지 않는다.** §5의 위젯(스택/힙, 포인터, 소유권 이동, 스레드 타임라인 등)은
  임의의 사용자 코드를 실시간 컴파일해서 그 실행 상태를 뽑아내는 구조가 아니라, **정해진 예시 코드에 맞춰 손으로
  설계한 애니메이션**이다. 라이브 컴파일러가 있으나 없으나 위젯의 학습 효과는 동일하다 — 오히려 없는 쪽이 구현
  리스크가 낮다(실행 중인 바이너리의 메모리 상태를 실시간으로 계측해 위젯에 반영하는 건 별개의, 훨씬 어려운 작업이고
  이번 스코프에 없다).
- **실제 개발 도구(gdb, ASan, CMake, colcon)를 쓰는 것 자체가 훈련이다.** Hexpider 프로젝트에서 최종적으로 써야 할
  도구는 브라우저 안의 장난감 컴파일러가 아니라 진짜 IDE·진짜 빌드시스템이다. 처음부터 그 도구로 연습하는 게 전이
  효과가 크다.
- **부수 효과: 서버 의존성이 사라졌다.** `/run` 엔드포인트가 없으면 앱은 순수 정적 파일(Pythonic과 동일 구조)이
  된다.

**단, ASan/UBSan·GoogleTest 같은 도구 자체는 Part VII에서 계속 가르친다.** 다만 "이 책 안에서 눌러서 실행"이 아니라
"이 명령을 당신의 IDE 터미널에서 쳐 보라"는 형태로, 실제 출력 예시를 (Pythonic STYLE.md §5.1처럼) **실측해서** 텍스트로
보여준다.

### 결정 변경 ②: 폰용 오프라인 캐싱(PWA/Service Worker) — 도입했다가 폐기, EPUB로 대체

직전 라운드에서 Service Worker 기반 오프라인 캐싱(폰이 한 번 LAN 접속하면 이후 PC 없이도 열리는 방식)을 설계했다.
**이번 라운드에서 폐기했다.** 사실관계: 모바일에서의 실제 학습 시간은 이동 중 짧은 자투리 시간 위주이고, 순수
이론 복습 용도다. 이 정도 사용량을 위해 다음 복잡도를 감수하는 건 배보다 배꼽이 크다 — Service Worker 구현, 보안 컨텍스트
(HTTPS/localhost) 우회를 위한 안드로이드 Chrome 플래그 설정 또는 iOS 자체서명 인증서, 모바일 반응형 위젯 축소
UI, 플랫폼(안드로이드/iOS)별 분기 유지보수.

**대신 EPUB 내보내기로 대체한다.** 상세는 §2-3. 요지: 같은 `content/*.md` 소스에서 위젯·인터랙션을 제거한 순수
텍스트만 뽑아 EPUB 파일 하나로 만든다. 폰 기본 앱(iOS 내장 Apple Books, 안드로이드 Play Books 등)으로 열리고,
서버·와이파이·특정 브라우저 설정이 전혀 필요 없다. 파일 하나를 클라우드 드라이브나 메신저 "나에게 보내기"로
옮기면 끝이다. **잃는 것:** 위젯 애니메이션(본문 설명 텍스트가 이미 그 자리를 채우고 있어야 한다 — §4-1 참고),
PC 앱과의 진도/SRS 동기화(폰에서 읽은 건 PC 쪽 학습 기록에 반영되지 않는다 — PC 앱이 진도의 유일한 기준점).
이 정도는 자투리 시간의 수동적 복습 용도에 문제가 되지 않는다.

**★ 링크 검증 절대 규칙 (사용자가 과거에 겪은 문제 — 반드시 지킬 것):**
콘텐츠에 들어가는 모든 외부 영상/자료 링크는 **작성 시점에 실제로 접속해서 (1) 살아있는지 (2) 제목이 아니라 실제 내용이
주제와 일치하는지**를 확인한 뒤에만 넣는다. 확인 못 한 링크, 추측으로 만든 URL은 **넣지 않는다.** 개별 영상보다
**채널/재생목록 단위 링크**(예: The Cherno 채널, CppCon "Back to Basics" 재생목록)를 우선한다 — 개별 영상보다 수명이
길다. 확인이 안 되면 링크 없이 비워 둔다. **없는 것이 틀린 것보다 낫다.** 상세 규칙은 §4-3.

## 2. 아키텍처 — Pythonic 패턴 계승 (순수 정적 구조, PC 전용)

### 2-1. 유지하는 것 (Pythonic에서 그대로 가져옴 — 검증된 패턴, 재발명 금지)

- **의존성 제로 원칙.** `index.html` 더블클릭으로 PC에서 바로 열린다. 빌드 도구·npm·프레임워크·백엔드 서버 불필요.
- **`content/` 마크다운 → `build.py` → `assets/bundle.js`** 번들링 구조. `file://`의 CORS 제약을 우회하기 위해
  마크다운을 JS 파일 안에 JSON으로 박아 넣는 것도 동일.
- **`toc.json`이 책의 뼈대.** Part → Chapter 계층, id 기반 링크.
- **`python build.py --serve`.** Pythonic에서는 폰 접속용이었지만, 이 프로젝트는 폰 대응을 안 하므로 **개발 중
  미리보기·디버깅 용도로만** 남긴다. 진도는 `localStorage`.
- **마크다운 확장 문법**(코드 블록 title/강조줄, `pyrepl`류 REPL 세션, `::: note/tip/warn/danger/deep/perf/hist/quiz/answer/lead`
  표시 상자, 수식, 형광펜, 링크 규칙)을 그대로 계승하되 아래 `::: widget` 문법과 새 상자 타입 `::: interview`를
  추가한다. `::: interview`는 Pythonic의 `::: cote`(코딩테스트 포인트)와 대칭인 상자 — "기술면접에서 이 개념이
  실제로 어떻게 질문되는가 + 모범 답변 뼈대"를 담는다. Part I~XII 어디든, 실제 면접에서 나올 법한 지점에만 붙인다
  (남발 금지, §4-4와 동일한 원칙).
- **`docs/STYLE.md` 집필 규범**을 이 프로젝트용으로 그대로 이식하고 §4에서 C++ 특화 규칙을 더한다.

### 2-2. 새로 추가하는 것

**① 인터랙티브 시각 위젯**
```text
::: widget <type> {json 파라미터}
:::
```
`assets/widgets/<type>.js`가 해당 블록을 커스텀 엘리먼트로 마운트한다. 의존성 없는 순수 Canvas/SVG + vanilla JS
(React 등 프레임워크 금지 — §2-1의 의존성 제로 원칙은 위젯에도 동일). 상세 스펙은 §5.

**② 폰용 EPUB 내보내기**
Pythonic에는 없던 기능. §2-3에서 상세 설명. 웹앱과 무관한 별도 산출물이다.

### 2-3. 폰용 자료 — EPUB 내보내기 (웹앱과 완전히 분리)

**웹앱은 폰을 신경 쓰지 않는다.** 반응형 모바일 UI도, 오프라인 캐싱도, LAN 서버 의존성도 없다 — PC에서
`index.html`을 열어 쓰는 것 하나만 최적화한다(§6). 폰은 대신 **같은 `content/*.md` 소스에서 자동 생성한 EPUB
파일 하나**로 대응한다.

**생성 파이프라인:**
```
tools/export_epub.py
  1. content/*.md를 toc.json 순서대로 이어붙인다
  2. 폰에서 의미 없는 것을 제거/치환한다:
     - `::: widget ...` 블록 → 제거 (위젯을 설명하던 본문 산문이 이미 개념을 전달하고 있어야 한다. §4-1)
     - `cpp runnable` 등 실행 관련 메타 → 무시(코드는 어차피 표시 전용이었으므로 내용은 그대로 유지)
     - `::: quiz` / `::: answer` → 굵게 표시된 "문제" / "해설" 절로 평문 변환 (접기 기능 없이 그대로 순서대로,
       EPUB 리더에 접기 UI가 없으므로 순서로 스포일러를 최소화 — quiz 다음 answer가 오되 사이에 페이지 구분)
     - 나머지 `::: note/tip/warn/danger/deep/perf/hist/lead` → 라벨 붙은 인용구(`> **주의:** ...`)로 변환
  3. `pandoc`(무료 오픈소스 CLI, 빌드 타임 전용 — §2-1 의존성 제로 원칙은 배포되는 앱 자체에 적용되는 것이지
     저자의 빌드 도구체인에는 적용되지 않는다. build.py가 Python을 요구하는 것과 동일한 성격)으로 마크다운 →
     EPUB 변환. 챕터 구분·목차(TOC)·기본 타이포그래피 CSS 포함.
  4. 출력: dist/cpp-mastery.epub (버전 파일명에 콘텐츠 해시 포함 — 갱신 여부를 파일명으로 바로 확인)
```

**전달 방법:** 사용자가 알아서 한다(클라우드 드라이브 동기화 폴더에 넣기, 메신저 "나에게 보내기", 케이블 복사 등) —
이 부분은 앱이 관여할 필요 없는 사용자 워크플로다.

**전략:** `python tools/export_epub.py`를 콘텐츠를 크게 갱신했을 때(마일스톤 완료 시점 등) 수동으로 돌리는 것으로
충분하다. 매 커밋마다 자동 생성할 필요 없음 — Pythonic의 `bundle.js`처럼 "빌드 산출물이지만 커밋할지"는 사용자
선택(용량이 크지 않으면 커밋해도 무방, `dist/`를 `.gitignore`해도 무방).

**기기 무관 확인:** EPUB은 Service Worker와 달리 브라우저·OS 보안 정책에 전혀 얽히지 않는다. 사용자의 폰(갤럭시,
안드로이드)에서도 아이폰에서도 동일하게 열린다 — 안드로이드는 Google Play Books(무료, 표준 EPUB 리더, 갤럭시
포함 모든 안드로이드에서 설치 가능)나 Moon+ Reader 등으로 열면 된다. 이게 §1 결정 변경 ②의 핵심 이득이다:
플랫폼 분기 코드가 원천적으로 필요 없다.

### 2-4. 용어 사전 (크로스커팅 기능)

Hexpider 프로젝트의 `GLOSSARY.md` 관행(프로젝트_지침_v5 §13)과 같은 이유 — 이 책은 120개 이상의 챕터가 될
것이고, "vtable이 뭐였지"를 다시 찾을 때 챕터를 뒤지게 하면 안 된다. `content/glossary.json`에 `{term, 정의
한 줄, 최초 등장 챕터 id}`를 챕터 집필과 동시에 누적한다(집필 후 별도 정리 작업이 아니라, STYLE.md 체크리스트에
"이 절에서 처음 정의한 용어를 glossary.json에 추가했는가" 항목을 넣어 그때그때 채운다). 앱에는 검색과 별개로
전용 "용어 사전" 페이지를 하나 둔다 — 알파벳/가나다 순 목록, 클릭하면 최초 정의 챕터로 이동. 별도 위젯·프레임워크
불필요, 기존 `toc.json`/검색 인프라를 재사용하는 정적 페이지 하나로 충분하다.

## 4. 집필 규범 — Pythonic `docs/STYLE.md` 계승 + C++ 특화 추가 규칙

기본 규범(반말 평서형 단정문, 문제 먼저 정의는 나중, 절당 6,000~12,000자 밀도, 실측 수치만, 아스키 다이어그램 규칙,
`::: quiz`/`## 요약`/"다음 절" 링크 구조, 체크리스트)은 Pythonic `docs/STYLE.md`를 **그대로** 이 프로젝트의
`docs/STYLE.md`로 이식한다. 아래는 C++ 전용으로 추가되는 규칙이다.

### 4-1. 위젯 우선 원칙 (가장 중요)

**공간적으로 표현 가능한 개념은 아스키 다이어그램이 아니라 반드시 §5의 인터랙티브 위젯으로 만든다.** 스택/힙, 포인터,
소유권 이동, 상속/vtable, 템플릿 인스턴스화, 스레드 타임라인, 좌표 변환 — 전부 여기 해당한다. 아스키 다이어그램은
위젯을 만들기엔 과한 단순 구조(예: 함수 호출 스택 한 프레임 정도)에만 쓴다. **"글로 설명 가능하니 됐다"는 이 책에서는
근거가 안 된다** — 사용자의 학습 방식 자체가 시각·공간 우선이라는 게 이 프로젝트의 전제다.

### 4-2. 코드 블록과 실행 예시

코드 블록은 표시·복사 전용이다(§1 결정 변경 ① 참고). 실행 결과·컴파일러 출력·ASan/UBSan 리포트를 보여줄 때는 반드시
**실제로 그 환경에서 실행해서 나온 진짜 출력**을 싣는다(Pythonic STYLE.md §5.1과 동일 규칙 — 추정치 금지). 필요하면
"이 코드를 당신의 터미널에 그대로 붙여넣고 `g++ -std=c++20 -fsanitize=address ...`로 실행해 보라"는 지시문을 덧붙인다.

### 4-3. 링크 검증 로그

콘텐츠 파일에 영상 링크를 추가할 때는 같은 커밋에 `docs/LINK_LOG.md`에 한 줄을 남긴다: 확인 날짜, URL, 확인 방법(직접
접속/검색 결과 교차확인), 확인자(Claude Code 세션). 이후 다른 세션이 콘텐츠를 갱신할 때 오래된 링크를 재검증할
근거가 된다.

### 4-4. 로봇 도메인 연결은 필수, 남발은 금지

Pythonic STYLE.md의 "뒤의 파트와 잇는다" 원칙을 계승하되 더 구체적으로 강제한다. **Part I~VIII(언어 자체)의 각 절은
최소 한 번, "이게 Part IX~XII(로보틱스 도메인)의 어디에 실제로 쓰이는가"를 짚는다.** 억지 연결은 금지 — 관련 없으면
안 쓴다.

### 4-5. 타이핑 원칙 — 앱에 실행 기능이 없는 이유의 긍정적 재정의

§1 결정 변경 ①은 "코드 실행을 뺐다"가 아니라 "코드를 손으로 타이핑하는 게 원래 더 효과적인 학습법이다"로 다시
읽는다. Pythonic `how-to-read.md`의 원칙 1("복사 버튼을 누르지 말고 직접 쳐라")을 이 책에서도 그대로 채택하고,
C++은 한 단계 더 강하게 적용한다 — **타이핑 실수(세미콜론 빠뜨림, 헤더 안 붙임)가 곧 컴파일 에러 메시지를 읽는
연습이기 때문이다.** Part 0에서 이 원칙을 명시한다.

**모든 실습형 코드 예제는 절 끝에 "네 IDE에서 해보라"는 구체적 지시로 마무리한다.** `::: quiz`(개념 이해 확인)와는
별개로, Part II 이후 실습 비중이 높은 절은 `::: quiz` 안에 최소 1개는 **코드 작성형** 문제를 포함한다 — "이 함수를
직접 구현하고 `g++ -fsanitize=address`로 돌려서 에러가 안 뜨는지 확인하라" 같은 형태(Pythonic도 이미 이 스타일을
쓴다 — decorators.md 연습문제 참고). 답을 눈으로 확인하는 게 아니라 **네 컴퓨터에서 실제로 컴파일이 통과하는가**가
성공 기준이 되는 문제를 섞는다.

### 4-6. 공개 레포 신원 보호 원칙 (예외 없는 강제 규칙)

이 레포는 public이다(Hexpider와 동일). **기술 면접관을 포함한 누구나 이 레포의 모든 문서·챕터·주석·커밋 메시지를
볼 수 있다는 전제로 쓴다.** 헤더의 경고를 여기서 구체적인 작성 규칙으로 못박는다.

**절대 넣지 않는 것:**
- 개인 이력(학력, 경력, 이전 직무, 나이, 커리어 전환 서사).
- "나는/저자는 이걸 몰랐다", "비전공자라서", "완전 초보였다" 류의 자기 능력에 대한 방어적·고백적 서술. Pythonic
  STYLE.md의 GLOSSARY 원칙("독자를 위한 문서이지 저자의 학습 노트가 아니다. '나는 몰랐다' 류 문구 금지")을 이
  책 전체, 특히 §0/Part 0/커밋 메시지에 예외 없이 확장 적용한다.
- 특정 과거 기술 스택(이 프로젝트 이전에 쓰던 언어·툴)에 대한 언급 — 지금 이 레포가 증명하는 역량과 무관하다.
- 채용 시장 조사, 특정 회사명, "이직 준비 중이다" 류의 구직 활동 서술.

**대신 쓰는 프레이밍:** 이 프로젝트는 "로보틱스 C++ 엔지니어링 역량을 체계적으로 구축·검증하기 위해 설계한
학습·레퍼런스 플랫폼"이다 — 완성된 콘텐츠(Part 0~XII 전체, 정확한 설명, 잘 만든 위젯)가 그 자체로 역량의
증거다. **역량을 얻어가는 과정을 고백하지 말고, 역량이 담긴 결과물을 보여준다.** Part 0의 대상 독자 서술처럼
"일반 프로그래밍 경험이 있고 C++을 실무 수준으로 익히려는 독자"라는 **책의 타깃 독자층**을 서술하는 것은
괜찮다 — 이건 저자의 상태 고백이 아니라 콘텐츠 설계 기준이다.

**적용 범위:** 이 CLAUDE.md 자신을 포함해 레포에 들어가는 모든 파일. README, 챕터 본문, 코드 주석, 커밋 메시지,
`docs/LINK_LOG.md` 같은 부속 문서까지 전부.

## 5. 인터랙티브 위젯 라이브러리 — 구현 목록

각 위젯은 `assets/widgets/<name>.js`, 의존성 없는 Canvas 또는 SVG + vanilla JS. **정해진 예시 코드에 맞춰 손으로
설계한 애니메이션**이며 라이브 컴파일러에 연결되지 않는다(§1 결정 변경 ①). PC 전용 앱이므로 풀 인터랙션(드래그,
스텝 실행, 값 편집)만 신경 쓰면 된다 — 모바일 축소 모드는 불필요하다(§1 결정 변경 ②, 폰은 EPUB).

| 위젯 | 무엇을 보여주는가 | 우선순위 | 쓰이는 곳 |
|---|---|---|---|
| `stack-heap` | 함수 호출에 따라 스택 프레임이 쌓이고 사라지는 것 + `new`/`malloc`으로 힙에 블록이 할당되는 것을 애니메이션으로. 스텝 단위 재생/되감기 | ★★★ 최우선 | Part II 메모리 모델 |
| `pointer-diagram` | 변수(박스) → 주소값 → 가리키는 대상(화살표). `*`, `&`, 포인터 산술을 조작하면 화살표가 실시간으로 갱신 | ★★★ 최우선 | Part I~II 포인터/레퍼런스 |
| `ownership-move` | `std::move` 전후로 객체가 한 변수에서 다른 변수로 "이동"하고 원본이 빈 상태(moved-from)가 되는 것을 시각화 | ★★★ 최우선 | Part II 소유권/이동 시맨틱 |
| `smart-ptr-refcount` | `shared_ptr` 참조 카운트가 복사/소멸에 따라 오르내리는 것, `weak_ptr`가 카운트에 영향 안 주는 것 | ★★ | Part II 스마트 포인터 |
| `vtable-diagram` | 클래스 상속 구조와 vtable, 가상함수 호출이 실제로 어떤 함수 포인터를 따라가는지 | ★★ | Part III 다형성 |
| `template-instantiation` | 제네릭 코드 한 벌 → 호출되는 타입마다 컴파일러가 별도 코드를 "찍어내는" 과정 | ★★ | Part IV 템플릿 |
| `thread-timeline` | 여러 스레드의 실행 타임라인 + 락 획득/대기 구간 + 레이스 컨디션이 실제로 발생하는 인터리빙을 재생 | ★★★ 최우선 | Part VI 동시성 (D-019 실시간 스레드 분리 이해에 직결) |
| `cmake-build-graph` | 소스 파일 → 오브젝트 파일 → 링크 → 실행파일/라이브러리의 의존 그래프 | ★ | Part VII 빌드 시스템 |
| `frame-transform-3d` | 좌표 프레임(TF) 트리와 회전/이동 변환을 3D로 조작. Three.js 없이 순수 WebGL 최소 구현 또는 CSS 3D transform으로 단순화 가능 — 구현 난이도 보고 판단 | ★★ | Part IX 로보틱스 수학, Part X tf2 |
| `kinematics-chain` | 링크-조인트 체인이 각도 변화에 따라 움직이는 것 (Hexpider 다리 IK와 직결) | ★★ | Part IX 운동학 |

## 6. UI/UX 요구사항 (PC 전용)

Pythonic의 좌측 목차 + 우측 인북 목차 + 검색(`/`, `Ctrl+K`) + 테마 토글 + 단축키를 그대로 유지. 여기에 추가:
위젯이 포함된 절은 위젯이 본문 폭을 넘어서도(화면이 넓으면) 최대한 크게 렌더링 — 텍스트보다 위젯이 주인공인
섹션에서는 2단 레이아웃(왼쪽 설명, 오른쪽 위젯 고정)도 허용. **듀얼 모니터 사용을 전제로, 창을 세로로 좁게 줄여도
레이아웃이 안 깨지도록** 반응형을 신경 쓴다 — 한쪽 모니터 전체가 아니라 절반 폭에서 쓰는 게 기본값이다(모바일
브레이크포인트는 불필요 — §1 결정 변경 ②).

**SRS(간격 반복 학습) 복습 큐**는 이 앱 안에 유지한다 — PC에서 공부를 시작할 때 "오늘의 복습" 카드를 먼저
훑고 진도로 들어가는 흐름. 절 안의 `::: quiz` 항목 + 핵심 정의를 카드로 쪼개 SM-2 알고리즘으로 스케줄링,
데이터는 `localStorage`.

## 7. 커리큘럼 구조 (Part 0~XII) — Pythonic `toc.json`과 대칭·확장

아래는 Part 단위 골격이다. **각 Part의 챕터 목록은 Claude Code가 이 골격과 Pythonic의 밀도(Part당 5~26개 챕터,
챕터당 6,000~12,000자)를 기준으로 확장 설계한다.** 대표 챕터 예시만 표기했다 — 전체 챕터 리스트를 이 문서에 미리

**빌드 타임 도구 의존성:** `pandoc`(EPUB 변환). 앱 자체(브라우저에서 실행되는 것)의 의존성이 아니라 저자의
로컬 환경에 한 번 설치하는 CLI 도구다 — `apt install pandoc` 또는 동급. `build.py`가 `--epub` 관련 동작 시
`pandoc`이 없으면 크래시 대신 "pandoc이 설치돼 있지 않다, `apt install pandoc`로 설치하라"는 친절한 에러로
안내한다(Pythonic이 없는 챕터를 `[경고]`로 부드럽게 알리는 것과 같은 태도).

### 8-1. `tools/check_links.py` — 신규 자동 검사

Pythonic의 `check_diagrams.py`(아스키 다이어그램 정렬)와 같은 위치의 도구를 하나 더 둔다. `build.py`가 빌드
끝에 호출하되 실패해도 빌드는 막지 않는다(Pythonic의 `lint_content()`와 동일 패턴). 검사 항목:

1. **내부 링크 무결성**: 본문의 `[텍스트](#/id)` 패턴을 전부 긁어 `toc.json`에 실재하는 id인지 확인. STYLE.md
   §7 체크리스트("링크한 id가 전부 toc.json에 있는가")를 사람이 놓쳐도 기계가 잡는다.
2. **위젯 참조 무결성**: `::: widget <type>` 블록의 `type`이 `assets/widgets/<type>.js`로 실재하는지 확인.
   콘텐츠가 위젯보다 먼저 쓰였을 때(§9 마일스톤상 흔히 발생) 이 검사가 "아직 구현 안 된 위젯"을 목록으로
   보여줘 §9 마일스톤 우선순위를 다시 확인하는 계기로 쓴다.
3. **용어 사전 갱신 여부(느슨한 검사)**: 새로 추가된 챕터 대비 `glossary.json`에 추가된 항목이 있는지 개수만
   비교 — 0개면 경고만(강제 실패 아님). §2-4 체크리스트를 사람이 잊었을 때의 안전망.

### 8-2. 코딩·커밋 컨벤션

Conventional Commits(`feat:`, `docs:`, `fix:` 등)를 쓴다. **커밋 메시지·코드 주석에 AI 어트리뷰션을 넣지 않는다**
— Hexpider 프로젝트_지침_v5 §13과 동일 규칙. 이 레포도 같은 GitHub 계정에 남고, 포트폴리오 성격도 있으므로
일관되게 적용한다. 위젯 JS는 함수/클래스 단위로 "무엇"이 아니라 "왜 이렇게 구현했는가"를 주석으로 남긴다
(예: "왜 requestAnimationFrame이 아니라 스텝 기반인가" 같은 설계 판단).

**레포 가시성:** public(Hexpider와 동일 정책). **그래서 §4-6 신원 보호 원칙이 예외 없는 강제 규칙이다** — 공개
저장소이므로 커밋 이력에 한 번 들어간 문장은 사실상 되돌리기 어렵다고 가정하고 쓴다.

## 11. Claude Code 자율 실행 지침 — 언제 질문하고 언제 그냥 진행하는가

사용자의 명시적 요구: **"의사결정은 필요할 때만 진행하고, 이 CLAUDE.md로 프로그래밍에 최대 효율을 낼 수 있어야
한다."** 즉 이 문서에 이미 답이 있는 것은 절대 다시 묻지 않는다. 아래는 이미 이 문서 안에서 결정이 끝난 것과,
정말로 남은 것을 명시적으로 가른다.

### 11-1. 이미 결정 끝남 — 그냥 실행할 것

- 아키텍처(§2), 위젯 목록과 우선순위(§5), 커리큘럼 Part 골격(§7), 리포 구조(§8), 마일스톤 순서(§9),
  집필 규범 전체(§4), 링크 검증 절차(§1·§4-3), EPUB 파이프라인(§2-3), 용어 사전 방식(§2-4), 커밋 컨벤션(§8-2).
- **각 Part의 챕터 세부 목록·순서**: §7은 골격만 제시했다. 세부 챕터 제목·개수·순서는 Pythonic의 밀도 기준
  안에서 Claude Code가 직접 설계한다 — 이건 "물어볼 결정"이 아니라 "저술 판단"이다. Pythonic의 대응 Part를
  참고해 유사한 세분화 수준으로 맞춘다.
- **`frame-transform-3d` 위젯의 구현 기법**(순수 WebGL vs CSS 3D transform): §5에 "구현 난이도 보고 판단"이라고
  이미 위임했다. 착수 시점에 더 간단히 되는 쪽으로 정하고 진행한다. 나중에 부족하면 그때 고도화한다.
- **레포 이름**: `cpp-mastery`가 기본값(§8). 그대로 쓴다.

### 11-2. 진행하되 사용자에게 통보만 하면 되는 것 (질문 아님, 완료 후 한 줄 보고)

- 특정 챕터를 쓰다가 §4-4(로봇 도메인 연결) 지점을 못 찾은 경우 — 억지로 만들지 말고 "이 챕터는 로보틱스
  연결이 약하다"고 마일스톤 보고에 한 줄 남기고 진행.
- 영상 링크를 찾다가 검증 가능한 게 없는 경우 — §1 규칙대로 링크 없이 진행하고 "이 개념엔 검증된 영상이
  없었다"고 보고.

### 11-3. 진짜로 물어야 하는 것 (드물어야 정상)

- **하드웨어·계정·금전이 걸린 결정** (예: 유료 도구·서비스 도입 제안이 실제로 생기는 경우 — 현재 스코프엔
  없음, pandoc도 무료다).
- **이 문서의 명시적 결정과 정면으로 충돌하는 상황을 실제로 발견한 경우** (예: Pythonic 구조를 그대로 못 쓸
  기술적 이유를 발견했다면, 조용히 다르게 만들지 말고 왜 안 되는지 설명하고 대안을 제시한 뒤 확인받는다 —
  이건 §1의 "결정 변경" 사례들과 같은 급의 사안이라 기록·확인이 필요하다).
- **Hexpider 본 프로젝트의 진행 상황과 이 학습앱의 우선순위가 실제로 충돌할 때** (예: Phase 1이 이미 시작됐는데
  이 앱의 Part II도 안 끝났다면, 순서를 어떻게 조정할지는 사용자의 시간 배분 판단이 필요하다).
- **§4-6(신원 보호) 위반 여부가 애매한 문장을 발견했을 때** — 예를 들어 어떤 서술이 "책의 타깃 독자층 설명"인지
  "저자의 개인 고백"인지 스스로 판단하기 애매하면, 조용히 아무 쪽으로나 정하지 말고 물어본다. 이 원칙만큼은
  틀렸을 때 되돌리기 어렵다(공개된 커밋 이력에 남는다) — 애매하면 보수적으로 쓰고, 그래도 애매하면 확인받는다.

**이 넷에 해당하지 않으면 질문하지 말고 §9 마일스톤을 그대로 진행한다.**
